## <a name="introduction"></a>簡介

IUI 是一種使用者介面模型，它會建議如何將功能分解成容易解釋和瞭解的螢幕或頁面，以簡化軟體應用程式。 Microsoft 已在2000（大型商務軟體應用程式）中實施此模型。 非正式的測試建議使用者可以像傳統介面一樣，在此模型中快速執行工作，而且可以更輕鬆地找到專案。

許多商務軟體應用程式都包含使用者介面，其中的螢幕會顯示一組控制項，但會將其保留給使用者，以推算頁面的用途，以及如何使用控制項來完成該目的。

本檔中所述的原則不需要或暗示任何特定的固定設計、控制項或視覺元素集合。 如同一般的圖形化使用者介面，本檔中的原則會在設計中留下許多彈性和創意的空間。

我們會示範採用的一般使用者介面原則，並提供從2000貨幣中繪製的範例。

> [!IMPORTANT]
> IUI 的整體概念是在其成形中。 採用這些技術的設計人員會學習，並在其軟體使用時探索更多相關資訊。 這份檔中的資訊會隨著時間而進化，因為這一段時間的研究和知識也會增加。 本主題提供 IUI 的簡介，而不是一組完整的指導方針。

 

## <a name="iui-in-action-solving-a-common-design-problem"></a>IUI 實務：解決常見的設計問題

本節討論今日軟體產品的常見設計問題，並引進 IUI 作為解決問題的技術。

### <a name="the-problem-software-is-hard-to-use"></a>問題：軟體很難使用

大部分的軟體太難使用。 這項結論是從可用性測試、有趣辨識項，以及軟體設計人員的個人經驗所打造。 IUI 的概念是藉由執行研究來建立的，並對導致目前軟體難以使用的情況進行猜測，然後提出解決方案的建議。 使用 IUI 的設計人員應該依賴客戶滿意度，以判斷設計的最終成功與否。

基於下列一般原因，最新的軟體產品很難使用：

-   使用者似乎不會建造出適合產品的精神模型。

    最新軟體產品的介面設計會假設使用者將瞭解設計工具精心設計的概念模型。 可惜的是，大部分的使用者似乎都不會取得完整且精確的精神模型，以引導其導覽。 這些使用者可能非常忙碌，並有資訊超載。 他們沒有時間、能源或想要研究其軟體的概念模型。

-   即使有許多長時間的使用者永遠都不會主控一般程式。

    設計師知道新的使用者一開始可能會有問題，但預期這些問題會在使用者學習一般工作時消失。 可用性資料表示通常不會發生這種情況。 在一份研究中，研究人員在家裡將自動設備設定為磁帶使用者。 磁帶顯示著重于手邊工作的使用者不一定會注意到他們所關注的程式，也不會從經驗中學習。 當使用者下一次執行相同的作業時，他們可能會以完全相同的方式遇到。

-   使用者必須努力找出每個功能或畫面。

    大部分的軟體產品都是針對 (瞭解其概念模型並具有一般程式的幾個) 使用者所設計。 對於大部分的客戶而言，每項功能或程式都是令人沮喪的不必要拼圖。 使用者可能會假設這些謎題是使用電腦的無可避免成本，但在沒有此負擔的情況下，一定會感到滿意。

這些問題的最佳解決方法是找出一般策略，讓軟體產品的功能更容易理解且一目了然。 使用者必須在每次需要時都能找到功能，而且每次需要使用該功能時，都必須能夠使用該功能。

### <a name="deductive-user-interface"></a>Deductive 使用者介面

現今軟體中的大部分元素都會要求使用者研究它們並推算其行為，如下列螢幕擷取畫面所示。

![範例 [屬性] 對話方塊的螢幕擷取畫面。](images/iuiguidelines01.png)

有經驗的電腦使用者（包括軟體設計人員）可快速辨識此對話方塊，讓他們能夠管理專案清單。 他們瞭解清單下方的按鈕可以新增、移除和提供清單專案的相關資訊。 不過，請注意，此行為在對話方塊本身中未明確陳述。

現在從使用者的觀點來看看對話方塊的另一種方式。 許多使用者在遇到此對話方塊時，會問：「我應該怎麼做？」 當對話方塊出現時，使用者必須停止並找出接下來該怎麼做。 首先，使用者必須推算大型白色矩形是要填入專案的空白清單方塊。 方塊的小型文字標籤「事物」提供了不清楚的提示。 有些使用者會嘗試在清單方塊中輸入，因為它看起來像是 [編輯] 文字方塊。

接下來，使用者必須推算清單下方的按鈕會影響其內容。 某些按鈕一開始是停用的，這是另一個可能的使用者混淆來源。 使用者必須試驗控制項，才能瞭解對話的運作方式。

使用者也可能提出其他問題：「我應該在清單中放入多少專案？ 我是否應該以特定順序輸入專案？ 為什麼我會在一開始就取得這個對話方塊呢？ 這是什麼？」

當使用者必須找出畫面的用途及其使用方式時，就會從其目標中進行工作。 這最終會以時間和使用者滿意度表示成本。 更糟的是，使用者每次使用某項功能時，都會在介面上重複使用這項成本。

畫面應該會有清楚說明其用途的標題。 當設計工具建立螢幕時，很少會要求它有明顯的可表達用途。 相反地，它可能只是使用者必須推算較大的概念模型的一部分。

研究顯示，即使是軟體的基本作業，也會混淆許多使用者。 他們難以瞭解產品的用途、執行作業的位置，以及如何在找到該作業之後，執行該操作。 藉由進行基本變更來簡化軟體，是更能完全滿足現有客戶並吸引新使用者的強大方式。

### <a name="a-solution-inductive-user-interface"></a>解決方案：感應消費者介面

作為設計軟體的新方式，IUI 的目標是要減少使用者在產品的各部分之間成功移動並使用其功能所需執行的無關思維。 感應的這個字來自動詞引發，這表示會影響或 persuasion。

IUI 是通用 Web 樣式介面的延伸。 在 web 環境中，頁面必須是簡單且以工作為基礎，因為每一項資訊都必須透過相對較慢的連接傳送至伺服器。 然後伺服器會以下一個步驟回應，依此類推。 良好的 web 設計表示焦點在每頁一項工作，並提供流覽頁面。 同樣地，「感應」導覽一開始會將每個頁面上的活動焦點放在單一的主要工作。

設計完善的「感應」介面可協助使用者在查看畫面時，回答兩個他們所面臨的根本問題：

-   我現在應該怎麼做？
-   哪裡可以完成下一項工作？

根據這些原則所設計的軟體，可透過基本的部署來回答這些問題：具有單一、明確陳述之明確用途的畫面，在沒有這類用途的情況下，會比網頁更容易瞭解。 如果螢幕更容易瞭解，使用者就可以更輕鬆地知道要做什麼，以及接下來該怎麼做。

這個基本的部署可以擴充成一連串的四個步驟，以設計使用 IUI 的軟體：

-   將每個畫面焦點放在單一工作上。
-   陳述工作。
-   使畫面的內容符合工作。
-   提供次要工作的連結。

雖然本檔說明 IUI 的一般原則，但它也會顯示 Money 2000 和其他軟體的範例，以示範這些原則的作用。 您應該將這些範例視為 IUI 的特定運算式，而不是執行的嚴格模型。

除了上面所列的四個步驟之外，您還可以遵循下列五個指導方針來強化介面：

-   使用一致的螢幕範本。
-   提供啟動工作的畫面。
-   讓它清楚說明如何使用畫面上的控制項來執行工作。
-   提供簡單的方法來完成工作並開始新的工作。
-   讓下一個導覽步驟顯而易見。

### <a name="processes"></a>處理序

許多工需要使用者流覽一連串的畫面。 執行工作的使用者可能會按一下移至次要工作的連結，該工作會從組成主要工作的螢幕順序中移開。 當使用者完成次要工作時，應該會有很簡單的方法，將使用者直接傳回原始工作中的分支點。 使用者可能無法使用傳統導覽控制項（例如 [ **上一頁** ] 和 [ **下一頁** ] 按鈕）回到其開始位置。

為了提供這項功能，IUI 會定義稱為進程的導覽概念、執行工作的畫面或畫面系列。 處理常式會作為一種導覽副程式。 使用者可以開始進行程式、處理其一系列的畫面，然後在最後一頁按一下 [完成] 按鈕，以快速回到開始處理的頁面。

## <a name="steps-for-creating-an-inductive-user-interface"></a>建立感應消費者介面的步驟

本節將深入說明您可用來建立 IUI 的四個步驟。

### <a name="step-one-focus-each-page-on-a-single-task"></a>步驟1：將每個頁面焦點放在單一工作上

設計 IUI 的第一個步驟是取得功能或一組功能，並將其分成不同的畫面。 每個畫面都應著重于單一工作，稱為畫面的主要工作。

這個概念聽起來很簡單，但應用程式會跟著關注。 大部分的應用程式都會顯示所有相關功能都已完成的畫面。 這種設計需要使用者找出 (推算) 可以完成的工作，以及如何進行。

主要工作可以是「特定」或「開啟」結束。 例如，在個人財務程式中，特定工作可能是「選取您想要支付的帳單」，而開放式工作可能是「複習您投資的效能」。

主要工作必須是對使用者有意義的事物，而不是反映執行的詳細資料或其他抽象概念。 這項工作應該是使用者可能想要做的事，最好是以自己的單字描述。

### <a name="example"></a>範例

本節將比較兩個不同版本的貨幣。 這些範例會顯示非常類似的功能，可讓使用者查看和管理財務帳戶。

IUI 模型是在建立 Money 2000 （用來管理個人財務的應用程式）期間開發的。 Money 2000 是產品的第八個主要版本。 Money 2000 是大型的 Windows 程式，並具有超過1000000行程式碼。

Money 2000 是 Web 樣式的應用程式。 它不是網站，而是與網站共用許多屬性。 其使用者介面是由共用框架中顯示的全螢幕頁面所組成，內含可在導覽堆疊中向前和向後轉的工具。 在此基礎上，Money 2000 新增了一組新的使用者介面慣例，以建立更具結構化的使用者體驗。

雖然 IUI 是首次用於 Money 2000 的 Web 樣式設計，但也可以搭配傳統介面專案使用，例如視窗和對話方塊。

在99年的中，使用者通常會在單一畫面上執行許多工。 例如，下列螢幕擷取畫面顯示的是在單一畫面上顯示所有帳戶相關功能的 **帳戶管理員** ，在99年的費用。

![費用99的帳戶管理員螢幕擷取畫面。](images/iuiguidelines02.png)

此畫面會將一般工作、流覽至帳戶，以及建立和刪除帳戶等不頻繁的工作分組。 在 [ **帳戶管理員**] 中，這些特定工作都不會直接以畫面的標題表示。 許多使用者可能會發現這個畫面與 [圖 1] 中的範例對話很有挑戰性。 在這兩種情況下，使用者都必須推算畫面的用途及其使用方式。

IUI 之後的 Money 2000 會提供與帳戶相關的一組幾乎相同的功能，但會在兩個不同的畫面上提供它們。 下列螢幕擷取畫面會顯示這些畫面的第一個畫面，這些畫面是完全專注于讓使用者挑選帳戶。

![2000貨幣中帳戶選取畫面的螢幕擷取畫面。](images/iuiguidelines03.png)

[Money 2000] 畫面包含的視覺元素數目大約與先前的 [Money 99] 螢幕相同，但頁面現在完全專注于讓使用者挑選帳戶。 例如，在99貨幣的版本中，使用者必須按兩下滑鼠來開啟帳戶：一個用來選取帳戶，另一個則用來選取開啟作業。 在2000年的版中，使用者按一下帳戶的唯一理由是開啟帳戶，只要按一下就可以足夠。 如此一來，即使畫面數目可能會增加，執行一般工作所需的點擊次數通常也會降低。

有時候，使用者會想要新增或移除帳戶。 若要在2000年的成本中執行這項工作，使用者可以流覽至第二個畫面， (將著重于設定帳戶的下列螢幕擷取畫面) 所示。

![[帳戶設定] 畫面的螢幕擷取畫面（money 2000）。](images/iuiguidelines04.png)

在 IUI 版本中，每個畫面的用途都比較清楚2000。 此外，每個畫面都有更多空間來滿足其用途。 例如，「金錢99」 **帳戶管理員** 可能會對 [ **刪除帳戶** ] 按鈕提供很少的空間，因為與螢幕上的其他命令相比，它很少使用。 相反地，Money 2000 的帳戶設定畫面可以更明確地設定此命令，使其更容易探索且一目了然。

### <a name="what-is-a-single-task"></a>什麼是單一工作？

您如何知道螢幕是否真的專注于一項工作？ 支援許多工作的畫面可能會被解釋為只有一個用途（如果該目的已足夠抽象化）。 以下是經驗法則：當設計工具可以使用簡潔、有意義且自然發音的畫面標題來表達該目的時，畫面會著重于一個用途。

Money 2000 的設計師將這些畫面視為中斷 (**挑選要使用的帳戶** ，並 **將您的帳戶以金錢**) 到更多畫面。 不過，由於每個畫面都已經有簡潔、有意義且自然發音的標題，因此設計人員認為畫面的焦點已足夠。 當您設計畫面時，如果您無法考慮清楚且簡單的標題，您可能會嘗試在畫面上完成太多工作。

### <a name="step-two-state-the-task"></a>步驟二：陳述工作

每個畫面都應以簡單且明確的主要工作說明作為標題。 這可以是直接指示 ( 「請選取您想要平衡的帳戶」 ) 或您想要讓使用者回答的問題 ( 「您要平衡哪個帳戶？」) 。

這是另一個通常不會練習的簡單發音原則。 例如，較舊版本的 Money 具有具有標題的畫面，例如 **線上金融服務管理員** 和 **餘額帳戶**。 使用者必須從控制項的排列和標籤，推算這些畫面的用途和行為。

畫面或頁面的標題非常重要。 無論產品使用的是 windows、web 樣式的頁面、對話或其他設計，都不應該允許標題離開。

### <a name="usable-screens-have-clear-titles"></a>可用的畫面有清楚的標題

執行許多工的畫面需要抽象或複雜的標題。 例如，[圖 2] 所示的 [Money 99] 畫面可讓使用者流覽至帳戶並設定帳戶。 為此頁面提供抽象標題「帳戶管理員」，以嘗試同時捕捉這兩個用途。 雖然使用者可能會對「帳戶管理員」頁面的功能有一些構想，但可能不知道此畫面最常見的工作是選擇帳戶。

某些螢幕或命令具有抽象用途，因此不會輕易地建議清楚的標題。 在這些畫面中，設計人員可以選擇刻意模糊的名稱，例如「設定」、「創造聽這些和」（例如 "QuickStep;"），或顯示執行詳細資料 ( 「資料庫壓縮」 ) 的術語。 這些類型的名稱通常會令人困惑或誤導使用者。 此外，這類名稱通常不表示使用者想要完成的動作，這會增加混淆。

在接近設計程式結尾之前，通常不會決定畫面標題和其他名稱。 設計人員通常會要求作者在設計及撰寫程式碼之後，為螢幕提供適當的名稱。 屆時，如果找不到正確的名稱，就沒有要是，因此小組可能必須針對不清楚的名稱進行結算。 這項瑕疵的解決方法是讓設計人員在設計過程中及早思考螢幕功能和標題的清楚清楚。

畫面功能和標題應該專注于客戶所執行的最常見工作。 設計人員通常會想要提供大量的功能，以嘗試滿足最多的客戶數目，以及設計小組本身的期望。 不過，額外的功能一律會增加複雜度和其他成本。

### <a name="screen-title-indicates-design-clarity"></a>螢幕標題表示設計清楚明瞭

在 IUI 模型中，設計工具會選擇設計流程最早階段中的畫面標題。 標題用來判斷畫面是否有意義，而不是挑選標題來證明畫面的運作方式。 如果找不到適合的標題，則會重新設計該功能。 如果沒有任何設計允許清楚且簡潔的標題 (也就是，如果沒有辦法解釋) 的功能，設計人員可能會放棄該功能。 在以下螢幕擷取畫面中，比較左邊的 [99 帳單付款] 畫面，其提供頁面的靜態標籤 ( [即將推出的帳單 & 存款] ) 以及右邊的對應 [Money 2000] 畫面，其中有明確的標題 ( [按一下您想要支付的帳單] ) ：

![顯示貨幣99中靜態標題的螢幕擷取畫面，以及 money 2000 中的有效標題。](images/iuiguidelines05.png)

畫面標題（當然只是一個片語或句子）比設計或程式碼更容易變更。 儘管這種情況，IUI 的體驗也會讓工作在清晰的螢幕標題早期產生更好的設計。 您應該選擇使用者教育和可用性小組成員以及產品設計人員的輸入。

小組成員有時可能會嘗試延後這項決策，假設客戶會分享他們對畫面用途的瞭解。 不過，當強制提供此用途的清楚且簡潔的陳述時，通常會顯示意見的差異。 藉由解決這些差異並及早挑選標題，設計討論可以更順暢地進行。

一旦選擇標題之後，您就不應該將它視為不可變更。 設計人員可能會隨著時間而修改畫面標題，就像任何設計一樣。 但是，第一個選擇的標題應該盡可能強，因為這是在開發階段。

### <a name="guidelines-for-choosing-screen-titles"></a>選擇畫面標題的指導方針

本節說明選擇良好畫面標題的簡單技巧。 若要使用這項技術，設計人員可以想像一下「此畫面的功能為何？」的朋友 然後提出清楚、實用的回應，就是「這是您的畫面？」這個句子。 完成句子的單字會變成畫面標題。

在開發金錢2000期間，小組的檔作者建立了螢幕標題指導方針，以確保品質和一致性。 例如，這些指導方針建議使用動詞的標題，並以問題或直接指示的形式片語。 設計工具會避免允許更抽象的靜態名稱，而且可能會是不清楚的名稱。

為了簡化標題，設計人員可以避免使用複合句子，並嘗試使用交談語言，避免使用不容易的詞彙和術語。 如果設計工具無法在不需要結合 (」、「」或「) 的情況下描述工作，則畫面可能會嘗試進行一項以上的工作，而且較不可能使用者可以立即瞭解該怎麼做。

即使是謹慎選擇標題，標題區域可能太小，無法適當地說明複雜的工作。 若要減輕這個問題，您可以在介紹于工作上的畫面內容區域頂端，包含簡短的描述性段落。

### <a name="making-the-screen-title-prominent"></a>使畫面標題醒目顯示

一旦您針對實用的螢幕標題進行了確認，請務必確定使用者已對其顯示使用者的注意力。 有些研究顯示，使用者很少閱讀解說文字。 為了解決這個問題，應該將畫面標題設計成明顯且吸引人，以吸引使用者的注意力。 畫面的視覺效果設計應該會通知使用者，標題是最重要的一件事。

### <a name="step-three-make-the-pages-contents-suit-the-task"></a>步驟三：使頁面的內容符合工作

當您建立遵循 IUI 指導方針的軟體時，最困難的設計工作通常會涉及將功能分解成螢幕或頁面。 下一步是要判斷每個畫面上將使用哪些控制項來完成其主要工作。 這些控制項會組成頁面的內容，使用者可在其中完成工作。 畫面標題和內容是程式與使用者之間的兩個對話的一半。 標題會顯示計畫的問題或提供指示，而使用者會透過畫面的介面回應。

如果畫面標題很清楚且簡單，則設計畫面通常很簡單。 例如，稍早所示的其中一個 Money 2000 畫面標題為「挑選要使用的帳戶」。 在此標題中，畫面顯然應該包含使用者可從中選擇的簡單帳戶清單。 另一個 Money 2000 畫面的標題為「檢查要包含在稅金中的專案」。 當然，此畫面包含專案的檢查清單。

使用者應該能夠輕鬆地找出如何使用控制項來完成畫面的主要工作。 當使用者被告知選取帳戶時，他們可以查看畫面來尋找帳戶清單，他們會確認他們是否瞭解該工作。 這樣可以提高使用者成功的機率，這也會增加他們對執行其他工作的信心。

### <a name="screen-content-areas"></a>螢幕內容區域

螢幕內容區域的確切位置和圖形取決於軟體的設計。 在2000貨幣中，螢幕內容區域是畫面標題和工作清單右邊的所有專案。 此區域可能會在長螢幕上滾動。 某些非必要的內容可能也會出現在 [工作清單] 底下的 [狀態] 區域中。

設計人員可能會選擇在內容區域頂端的段落中詳細說明畫面的主要工作。 使用者永遠不需要閱讀這段文字，但可能會覺得很有用。 許多使用者都可以略過它，但仍能順利使用畫面。 與標題不同的是，如果畫面是可滾動的，則此描述可以退出。 如需詳細資料，請參閱 [選擇畫面標題的指導方針](#guidelines-for-choosing-screen-titles)。

如果設計工具想要讓頁面顯示非必要的提醒、警示或其他狀態資訊，可以在畫面左側的 [工作清單] 底下，將它顯示在 [主要內容] 區域的左邊。 在功能上，此狀態區域是螢幕內容的其他區域。 此區域的功能不夠顯著，無法包含基本控制項。

### <a name="provide-a-clear-exit-from-the-page"></a>從頁面提供清楚的離開

順利完成工作之後，使用者會面臨另一個問題：何時及如何離開螢幕。 若為主要工作為流覽的畫面，執行工作本身會將使用者移至下一個畫面。 在其他畫面上，使用者可能更難知道如何進行。 例如，在要求使用者將資訊輸入欄位的畫面上，使用者可能需要協助來找出何時及如何繼續進行。 在這類頁面上，在一致的位置提供明確的 **下一個或 [** **完成** ] 按鈕，通常會很有用。

可用性研究顯示，即使有全域導覽按鈕（例如工具列上的 [ **上一頁** ] 或 [ **首頁** ] 按鈕），使用者也偏好使用此類按鈕。 使用者通常不會在沒有清除結束的畫面上感到不安，即使是只是要提供讀取資訊的畫面。

如需此主題的詳細資訊，請參閱 [提供簡單的方法來完成工作，並](#provide-screens-for-starting-tasks) 在其他指導方針區段中開始新的工作。

### <a name="step-four-offer-links-to-secondary-tasks"></a>步驟四：提供次要工作的連結

設計畫面的最後一步是提供次要工作的連結，這些是不會直接完成主要工作但與螢幕相關的功能。 比方說，如果螢幕上的主要工作是要撰寫信件，該畫面上的次要工作可能是查詢郵寄地址或列印信封。

次要工作可能會顯示對話方塊、變更畫面內容的視覺呈現，或將使用者流覽至不同的畫面。 次要工作可能會間接完成主要工作，或可能會將遺失的使用者重新導向至他們搜尋的位置。

如果頁面是電腦與使用者之間的交談，次要工作可讓使用者略過電腦的目前問題，並要求電腦改為進行其他作業。 例如，假設此對話方塊：電腦：「您要支付哪個帳單？」 使用者：「實際上，我真正想要做的就是找出我一回的帳單。」

您產品中的某些畫面將沒有次要工作，有些則會有數個。 您應該避免建立很長的工作清單，讓使用者很難進行掃描。 如果螢幕的次要工作有相當長的清單，則應該先放置最常見的工作，將其分組在不同的區段中，或是以視覺化方式強調。

此清單不應包含每個可想像的次要工作，只要它讓下一個導覽步驟變得很明顯。 畫面可提供次要工作，以流覽至列出更多工具的子頁面，而不是提供許多次要工作。

### <a name="visual-design-of-secondary-tasks"></a>次要工作的視覺化設計

次要工作應該列在畫面上的從屬位置中，視需要可存取，但不會讓使用者與主要工作進行干擾。 將此清單放在每個畫面上一致的位置，可協助使用者在需要時快速找到清單。

如果您在螢幕左側顯示次要工作的清單，清單本身不應該是可滾動的，也不應該在頁面上滾動，如下列從 Money 2000 的 **帳單付款** 畫面螢幕擷取畫面所示。

## <a name="additional-guidelines"></a>其他指導方針

本節說明根據上一節所述的四個步驟來建立 IUI 的五個實用指導方針。

### <a name="use-consistent-screen-templates"></a>使用一致的螢幕範本

設計遵循 IUI 模型的軟體時，您應該建立範本，以作為每個畫面的指南。 「感應」模型不會指示您使用任何特定的範本。 有許多可能的變化可以滿足一種形式的設計。 您的產品可能只需要一個適用于其所有畫面的範本，或者您可能會基於各種用途建立數個不同的範本。

好的範本可讓新使用者快速瞭解產品畫面的運作方式。 在產品的螢幕中一致使用範本，可讓您從螢幕到螢幕提供良好的使用者介面流程。 當使用者想要讓相同的專案出現在同一個位置時，他們可以掃描並開始更快速地使用每個新畫面。

### <a name="provide-screens-for-starting-tasks"></a>提供啟動工作的畫面

使用 IUI 設計的產品通常會使用專門用來對工作集啟動使用者的特殊畫面。 這些畫面稱為活動頁面，因為它們會組織相關的一般工作群組。 活動頁面提供使用者的起點。 活動頁面通常會顯示使用者實際完成工作的其他頁面連結。 活動頁面會詢問使用者「您現在要怎麼做？」 並提出可能的答案清單。 活動頁面可能會遵循特殊範本以協助使用者辨識。

活動頁面會為產品提供良好的預設起始頁。 當使用者啟動應用程式時，他們通常會想到他們想要完成的工作。 通常，啟動產品的原因只是少數很常見的工作之一。 產品的預設起始畫面可讓您清楚瞭解如何開始一般工作，以辨識這項作業。

[ **Money 2000] 首頁是** 活動頁面的範例。 依預設，使用者會看到此畫面，在此畫面中，您會在啟動應用程式時顯示一般財務工作（例如支付帳單和平衡帳戶）的存取權。

### <a name="make-it-obvious-how-to-carry-out-the-task-with-the-controls-on-the-screen"></a>讓它清楚說明如何使用畫面上的控制項來執行工作

遵循此指導方針的最佳方式是選擇適當的畫面標題，並將主要工作的範圍限制為最常見的範圍。 一旦您到達頁面的清楚標題和用途之後，選擇正確的控制項集將會很簡單。

### <a name="provide-an-easy-way-to-complete-a-task-and-start-a-new-one"></a>提供簡單的方法來完成工作並開始新的工作

使用者在螢幕上所面對的最後一個障礙，就是要找出何時及如何離開。 使用者通常會離開螢幕，方法是按一下連結或執行流覽至另一個畫面的命令。 這些連結可以出現在 [螢幕內容] 區域、[工作清單] 或導覽工具列上。 使用者也可以藉由關閉目前的檔案或應用程式本身來離開螢幕。

在某些畫面上的工作是準備使用者必須確認或取消的作業。 這類畫面通常會提供一個執行和認可作業的連結，還有另一個取消的連結。 如果使用者忽略這些選項並按一下另一個連結，程式應該會執行較少的破壞性選項。 畫面應該指出當使用者採用此路徑時，會發生什麼事。 您可以將連結組成更清楚的連結。 例如，標示為「儲存變更」的 [認可] 按鈕表示在按下該按鈕之前，在畫面上所做的變更將不會生效。

即使使用者隨時都可以離開，您仍然可能會提供一個連結，以建議從頁面中有明顯的離開。 這同樣適用于只顯示靜態資訊的頁面。 如需此主題的詳細資訊，請參閱一節 [從頁面提供清楚](#provide-a-clear-exit-from-the-page)的結束。

### <a name="starting-and-completing-processes"></a>開始和完成進程

基於本文的目的，處理常式是處理將使用者帶到多個畫面之工作的技術。

假設使用者按一下螢幕內容或工作清單中的連結，並移至另一個畫面。 接著，該頁面可能是完成一些整體結果的一系列畫面的第一頁。 在這一系列的畫面結束時，使用者想要回到程式之前的畫面。 使用者至少有兩種方法可以取回？ 按一下 [上一頁] 按鈕重複或返回首頁，然後從該處流覽？ 但是這兩種方法都很明顯或自然。 大部分的使用者預期會在最後一個畫面上找到一個按鈕，將它們直接傳回原始畫面。

IUI 模型透過進程的概念來支援此案例，將螢幕或畫面系列視為導覽單位。 使用者可以輸入處理常式、流覽其畫面，然後在最後一個畫面上找到將其傳回至開始位置的按鈕。 重要的是，使用者可以從產品中的多個位置啟動流程。 無論使用者開始處理的位置為何，都一律會傳回使用者的位置。

### <a name="process-name"></a>程序名稱

每個進程都應該有名稱，而且名稱應該會出現在程式中每個畫面的某個位置。 Money 2000 使用這種方法。 整個進程中的每個畫面都包含該進程在頂端的名稱。 此進程名稱的顯示方式比螢幕的唯一標題更不重要，因為它比較不重要。 進程名稱會提醒使用者他們正在進行的處理常式，並強調該程式中的所有畫面都是單一功能的一部分。 例如， **Money** 區域包含橫跨數個畫面的 **稅務估算器** 流程。 此程式中的每個畫面都會顯示集體進程名稱及其唯一的畫面標題。

### <a name="implementation-of-processes"></a>進程的執行

您可以從不同畫面上的各種連結啟動相同的程式，使用者一律會返回正確的起始頁。 此行為無法透過程式最後畫面上的硬式編碼連結來達成，因為連結的目的地會有所不同。 相反地，應用程式可以藉由維持作用中進程的堆疊，而不受 **Back** 和 **Forward** 命令所使用的一般導覽堆疊，來執行此行為。 當使用者啟動進程時，啟動畫面會推送至進程堆疊。 當使用者在處理常式的最後一個畫面上按一下 [ **完成** ] 按鈕時，應用程式會將最新的啟動畫面從堆疊中取出，並將使用者返回該畫面。

當使用者離開進程中的螢幕時，進程會在進程堆疊上保持作用中狀態。 使用者可以完成備份到畫面中的程式，然後再繼續進行。 這可讓使用者進行繞道、備份，然後繼續進行程式。 若要瞭解此行為的運作方式，請在 World Wide Web 上開始任何線上購物程式、離開網站，然後按 [ **上一步** ] 按鈕。 您通常可以從離開的地方繼續進行。

### <a name="done-button"></a>[完成] 按鈕

若要完成畫面並移至程式中的下一個畫面，螢幕可以在頁面底部附近顯示按鈕。 此按鈕的標籤為「下一步」、「完成」或類似的標籤。 如果按鈕結束進程，而且可以從多個位置呼叫進程，則 [ **完成** ] 按鈕的標題可以包含呼叫位置的名稱。

### <a name="navigation-bar"></a>導覽列

在任何畫面上，使用者都可以決定是否已完成產品的目前區域，並且想要啟動其他內容。 它們可能不會想要在移至產品的另一個部分之前明確完成目前的畫面。 導覽工具列可提供使用者一組啟動新工作的連結。 與其他工作連結清單一樣，這些都應該遵循下一步導覽步驟的原則，如下一節中將詳細討論。

### <a name="make-the-next-navigational-step-obvious"></a>讓下一個導覽步驟顯而易見

少數程式可同時讓所有功能都可供使用。 使用者通常必須流覽程式以尋找特定功能。 如果使用者可以輕鬆地查看如何取得至少一個步驟，更接近其所需的結果，則會更成功地進行流覽。 使用 IUI 的畫面是以這個原則為考慮而設計的。

例如，活動頁面不一定會顯示使用者可能想要從該時間點達成的每個可想像的工作或目的地。 相反地，活動頁面會提供已足夠完成的工作清單，讓使用者可以輕鬆地判斷要按一下的適當連結，即使它只會將他們帶到另一個連結頁面。 最常見的工作應該最重要，而且需要最少的流覽量。 較不頻繁的工作可能需要更多步驟。

以下是 Money 2000 的範例。 假設使用者想要執行他們只會偶爾執行的作業，例如檢查下一年的收入款項預估量。 使用者第一次在2000年的 [] 首頁上開始搜尋這項功能。 因為功能不會出現在常見的工作清單中，所以他們必須掃描財務區的清單。 **稅金** 區域聽起來很有可能，因此按一下它。 [ **含稅中心** ] 頁面包含其所尋找之稅金估計功能的連結，因此按一下該頁面即可完成其工作。 藉由套用 IUI 原則，Money 2000 可讓使用者以直覺的方式找出他們要尋找的內容。

## <a name="user-assistance"></a>使用者協助

本節說明一組建議的指導方針，以將使用者協助文字整合至使用 IUI 的產品。

主要協助是指畫面上可見的所有文字 (如下列螢幕擷取畫面所示) 。 主要協助提供工作導向的文字線索，讓使用者可以輕鬆地瞭解畫面上顯示的所有資訊。 他們瞭解頁面的目的，以及物件彼此之間的關聯性，以協助完成工作。 由於文字是直接顯示在畫面上，因此回答「我該怎麼辦？」這類問題的資訊 很容易存取且高度可見，無需使用者採取任何動作。

![money 2000 的帳戶設定畫面螢幕擷取畫面。](images/iuiguidelines11.png)

次要協助是由畫面上看不到的所有文字所組成，而且需要一些使用者互動才能存取，例如按一下或暫留在使用者介面元素上。 這項內容並不重要，無法完成手邊的工作，但仍會直接相關。

### <a name="primary-assistance"></a>主要協助

主要協助可以包含下列部分或所有元件：

-   螢幕標題

    範例：變更您的圖片

    畫面標題是顯示在畫面上的第一個和最重要的專案。 其目的是要以使用者自己的語言描述可在此頁面上完成的工作。 畫面標題應避免說明如何完成工作的詳細資料。 螢幕標題中的文字應該只參考畫面的核心工作。 根據經驗法則，工作的描述越簡單且更短，工作的定義越好。 如需有關本主題的詳細資訊，請參閱步驟二： [陳述工作](#step-two-state-the-task)。

-   螢幕子標題

    範例：您也可以從網際網路下載新的圖片。

    即使有了審慎的工作，螢幕標題可能還不足以充分說明複雜的工作。 子標題可讓您根據畫面的用途進行詳細說明。 您可以使用子標題來協助闡明頁面的用途、提供其他工作描述，或協助設定預期。 未讀取副標題的使用者應該能夠順利使用頁面。 就像標題一樣，子標題應該避免描述完成工作的詳細資料。

-   工作

    範例：變更您的螢幕保護裝置

    工作可以顯示為需要使用者互動的文字連結或圖形影像。 以文字連結呈現的命令應以動詞為基礎，並以明確且簡潔的工作撰寫。

-   命令按鈕的標籤

    範例：建立密碼

    命令按鈕有三種類型：

    -   取消
    -   完成
    -   Commit

    [取消] 和 [完成] 按鈕只會使用 [取消] 和 [完成] 作為其標籤。 認可按鈕應使用活動文字標籤，而不是「確定」。 例如，使用「建立密碼」而不是「確定」。

-   其他控制項的標籤

    範例：輸入您的密碼

    選項按鈕、核取方塊和文字方塊等控制項的標籤應該清楚且簡潔地撰寫，讓使用者確切知道控制項的用途、要使用的控制項，以及需要提供哪些資訊才能完成工作。

-   「相關工作」連結

    範例：相關工作-變更其他帳戶

    「相關工作」連結是與目前功能相關之其他工作的明確進入點。 它們應該撰寫為以工作為基礎的連結。

-   「另請參閱」連結

    範例：另請參閱-變更您的主題

    「另請參閱」連結是次要工作。 這些與主要工作相關，但會將使用者從目前的內容中取出。 這些應該會顯示為一般的工作連結。 如需次要工作的詳細資訊，請參閱 [次要工作的視覺化設計](#visual-design-of-secondary-tasks)。

### <a name="secondary-assistance"></a>次要協助

次要協助可以包含下列部分或所有元件：

-   提示

    您可以使用資訊提示，為使用者提供工作連結或命令按鈕的其他相關資訊。 例如，工作連結的資訊提示可能會讀取：「顯示頁面，您可以在其中選擇要與您的帳戶搭配使用的圖片」。 當使用者將滑鼠停留在相關聯的物件上方時，就會出現提示。 您應該為使用者可以按一下的所有使用者介面元素建立提示。

-   「瞭解」說明主題

    範例：瞭解-下載檔案

    「深入瞭解」連結可開啟說明主題，例如功能總覽、概念資訊、支援資訊，以及程式資訊。 若要減少雜亂的情況，您應該將畫面上的「瞭解」說明主題的數目降至最低。

### <a name="designing-and-testing-money-2000"></a>設計和測試金錢2000

### <a name="clearer-titles-and-designs"></a>更清楚的標題與設計

2000年的設計師發現，他們通常會使用未實際顯示在畫面上的文字來描述功能。 在 IUI 模型中，畫面應該會說明本身。 例如，小組說明標示為 **付款行事曆** 的畫面是用於支付帳單。 在2000貨幣中，該畫面稱為「 **付款帳單**」。 所有與該目的無關的元素都已移至附屬畫面，以產生更清楚的設計。

另一個範例牽涉到一個稱為「 **線上金融服務管理員**」的畫面。 小組努力提出此螢幕用途的簡單說明。 當他們無法抵達時，會移除此畫面，並在以邏輯方式定義的頁面之間散發其功能。

### <a name="helping-new-designers"></a>協助新的設計工具

小組發現 IUI 的設計技術，可讓您輕鬆地對新的、缺乏經驗的軟體設計人員教授。 這項技術可讓設計人員在所有經驗層級上，使用螢幕標題作為更清楚的測試來評估其設計。 當強制將清楚且簡潔的標題放在設計不良的畫面時，設計人員很快就能辨識出沒有足夠的標題可供頁面使用。 他們發現問題騙你不是選擇標題的單字，而是在有瑕疵的螢幕設計中。 有了這種瞭解之後，他們就可以重新設計畫面來支援更清楚的使用者互動，因此更清楚的標題。

### <a name="including-writers"></a>包含寫入器

在設計的過程中，小組發現檔作者和編輯器應該牽涉到建立畫面標題。 寫入器在舊版中無法挑選良好的標題，因為它們只牽涉到最晚的階段。 螢幕通常是由設計人員或程式設計師提供暫時的工作標題。 這些標題是在要求作者提出最終畫面標題的時候，一直使用到產品週期的遲到。 屆時，修改設計不良的畫面太晚。

相較之下，2000團隊的團隊在設計程式的最早階段中牽涉到寫入器。 當它仍有助於設計時，這會在螢幕標題上帶來寶貴的輸入。 如果螢幕太複雜而無法允許清楚的標題，寫入器可能會建議您重新設計頁面。

在專案結束時，作者和設計人員認為螢幕標題比舊版更清楚且更強。 作者也發現更容易解釋新的頁面，讓記錄產品的工作更簡單。 所有小組成員都認為牽涉到設計階段中的所有專業領域，讓產品更好且更容易使用。

### <a name="comparison-with-web-sites"></a>與網站比較

許多設計完善的網站會使用與本檔中所述的 IUI 模型類似的原則。 這可能是 Web 運作方式的副作用。 因為很難在單一網頁上的控制項之間執行複雜的互動，所以設計人員通常會將工作分成牽涉到伺服器的多個旅程來取得新頁面的片段。 某些網站甚至包含可明確陳述頁面用途的頁面標題。

傳統應用程式的設計工具提供了更豐富的工具組。 這可提供更大的彈性，但也提供更多機會來建立複雜且令人困惑的頁面。 建立可感應的使用者介面時，設計人員應該謹慎使用這種功能，並記得將價值明確和簡化。
