## <a name="introduction"></a>簡介

開發人員通常無法考慮使用者的觀點。 開發人員會努力讓應用程式順利運作，客戶只期望它能正常運作，而且其對軟體中心的認知也會圍繞這項需求。 如果您開發的是零售軟體，或非技術人員將使用的產品，則更是如此。 開發人員必須留意整個軟體設計流程中的使用者需求。

軟體應用程式必須盡可能容易地流覽和使用。 在建立的軟體數量中，預估4個軟體應用程式中有一個很棒的 UI，讓終端使用者真正喜歡並立即熟悉。

針對公司建立大量的內部使用軟體。 無論是在內部開發，或在顧問的護理下，通常只需投入最少的時間、工作或金錢，就能建立更好的 UI。 「設計工具」角色在開發週期中很罕見，尤其是在 Windows 應用程式的世界中。

有一些基本規則可以遵循，讓您的應用程式有更更好的功能和更佳的 UI。 在您的過程中，不需要太多時間或金錢的投資，並增加投資報酬率。

在我們進一步討論之前，讓我們先區分出使用者介面和使用者經驗，至少在本文的討論範圍內。 使用者介面（或 UI）是指應用程式的視覺效果和控制項，而使用者經驗或 UX 則包含 ui 以及與 UI 相關的應用程式行為，以及使用者從您的應用程式取得的「感覺」。 這並不只是設計美觀的 UI，還能確保它也能正常運作。

我們將在這裡討論20個 UX 設計，您可以輕鬆地將其整合到您的應用程式設計階段。 結果會是更豐富的應用程式，具有更直覺的功能，也就是「人類 UX」。 因為我們全都知道，Windows Vista 產生的應用程式必須以不同的方式來尋找和運作。 本主題將協助您為未來的應用程式做好準備，同時為您目前的使用者提供更好的未來體驗。

下列各節將討論適當 UI 設計的基本概念。

## <a name="the-basic-principles-of-proper-ui"></a>適當 UI 的基本原則

專業型式的 UX 取決於下列四項因素：

-   間距和定位
-   大小
-   群組
-   Intuitiveness

在8.0 之前的 Microsoft Visual Studio 版本中，間距和調整大小是次佳。 4x4 或8x8 方格不一定都能運作。 在包含對齊線的情況下，已大幅簡化處理常式。 將標籤與文字方塊對齊或更糟的是，將多個標籤與其對應的文字方塊對齊，在舊版的 Visual Studio 中相當困難。 對齊線大幅簡化了這個程式。

下列各節說明專業 UX 設計更重要的四方面。

### <a name="spacing-and-positioning"></a>間距和定位

兩個控制項之間的間距相當重要。 下列螢幕擷取畫面示範了設計不佳的 **使用者資訊** 輸入表單—前兩個文字方塊太接近，其下的清單太遠，而且表單上有許多未使用的空間。

![更妥善設計的表單螢幕擷取畫面。](images/humanux-02.png)

雖然沒有確切的間距規則，對齊線也提供非常實用的指導方針來提供適當的間距。 其他可協助您保留適當間距的工具，都是 [容器工具箱] 群組底下的版面配置控制項。 TableLayoutPanel 在建立專案表單樣式對話方塊時也非常有用。

### <a name="size"></a>大小

相同的考慮也適用于大小。 當您將按鈕從 [工具箱] 拖曳到表單上時，它的高度和寬度都是完美的。 建議的最大寬度 (會限制任何嚴重的重要原因) 是要將原始寬度加倍。

如果您查看 [**開始**] 功能表上的 [**執行**] 視窗，或任何 Windows 檔案總管物件的 [**屬性**] 對話方塊，則按鈕大小為「恰好是右」。 如果您有一個非常重要的功能，您需要終端使用者注意而不會失敗，則會有其他方法，而不是使用大型按鈕或花俏非標準色彩 (稍後) 。

在下圖中，您可以看到三個按鈕。 第一個按鈕是最建議的大小，而且是從 [工具箱] 拖曳 (或按兩下) 時預設建立的大小。 有時候額外的文字會要求您將按鈕放大。 第二個按鈕會顯示夠大但可接受的大小。 它不會建立配置其他控制項的混亂。 不過，第三個按鈕是完全無法接受的大小。 您可以看到，它甚至稍微扭曲了主題點陣圖 Windows 用來繪製主題控制項。 它也可以讓其他控制項以直覺方式在其周圍進行調整。

![三個按鈕的影像，從左至右增加大小。](images/humanux-03.png)

### <a name="grouping"></a>群組

一般而言，應用程式包含許多控制項。 只有透過適當的直覺分組，您才能讓這些控制項更容易使用。 以函式或分類的群組是由索引標籤控制項完成的最佳方式。 例如，「帳戶」、「報表」、「員工」和「專案」是一般商務應用程式中的索引標籤的絕佳候選項目。 同級群組（構成相同最終結果的控制項）最好是由群組控制項來完成。 不建議針對這類群組使用具有框線的面板。 群組控制項可為您節省標籤控制項的額外權數，特別是如果您的子控制項一目了然。

除非一個大型群組控制項內沒有超過2或3個群組控制項，否則不建議群組控制項內的群組控制項。

### <a name="intuitiveness"></a>Intuitiveness

這是絕佳使用者體驗的最重要層面。 直覺化 UX 可減少說明的需求。 使用者只知道控制項的功能。

直覺化設計中的重要主題是以色彩撰寫程式碼。 Windows XP 中提供一個很好的範例，其中顯示了函式的新虛方塊按鈕，例如主題應用程式 **中的導覽、登出和關閉****電腦** 對話方塊等等。

這些控制項的色彩是根據該按鈕所推送的結果嚴重性來決定。 流覽為綠色，相當於「Go」交通燈。 關機（這會導致可能的工作遺失）會以紅色標示，如同警告符號。 半關鍵性按鈕（例如登出或休眠）為黃色。 未對使用者的工作進程產生任何重大影響的中性按鈕（例如 [說明]）是一種軟藍色。 建立 skinned UI 時，請記住這些色彩層面。

以色彩辨識內容的絕佳範例是 Microsoft Office OneNote。 應用程式的索引標籤可設定為不同的色彩，但仍看起來像是整體 Windows XP 樣式設計的適當部分。

另一個重要的層面是您應用程式中的文字。 最近，為了簡化 Windows 軟體中書面指示所使用的語言，已有許多努力。 稍後會討論軟體內的文字用法，但請注意下列小型但重要的詳細資料。

MSN Messenger 在 [ **選項** ] 對話方塊中有一個標示為「共用網路攝影機功能」的核取方塊。 開發人員和技術方便的人員都知道這是什麼意思，但是新手的使用者可能會認為您也可以讓另一位使用者在聊天的另一端使用您的 Web 凸輪。 在最新版本中，他們會將它變更為「我的網路攝影機：讓其他人看到我有網路攝影機」。 這非常適合可能沒有技術知識且習慣于簡單語言的目標物件。

雖然更簡單的語言可讓您更容易瞭解，但也有額外的好處。 科學研究顯示當您嘗試瞭解新的語言時，我們的想法可以更輕鬆地使用較簡單的語言。 通常，我們的大腦會處理像是「it」、「」、「」、「speedily」和其他常見字組等單字，如此就能更輕鬆地配置「頻寬」來瞭解彼此的單字，例如「網路攝影機」或「其他」。

訊息方塊標題、分組標題和其他這類文字區塊，可讓您輕鬆地將控制項群組的函式傳達給終端使用者，只需幾個字。

Intuitiveness 也從熟悉度來誕生。 例如，[ **確定] 和 [** **取消** ] 按鈕的位置是一致的，而且在我們的思維中，如果對話方塊以反向順序保存這些按鈕 (**取消**]，然後是 **[確定]**;而不是 **[確定]**，然後 **取消**) —您可能會改為按下 [ **取消** ]。 使用特定的標準來執行工作（例如 Windows 型應用程式），習慣開發一年以上的習慣。 遵循業界標準 (不過，unstated 可能會) 讓您的軟體更容易使用。

在先前的 Windows Vista preview 組建中，任何視窗的 **最小化**、**最大化** 和 **關閉** 按鈕變得不同。 在舊版 Windows 中 (尤其是在使用單一監視器) 時，您會習慣將游標移至畫面右上角，然後按一下。 這一律會導致視窗關閉。 現在在 Windows Vista 的這個特定組建中，視窗的 [關閉] 按鈕與最右邊的框線之間大約有8個圖元的空間。 額外的空間會讓它看起來很酷， (但按鈕能運用同樣) 的酷炫動畫可能是必要的，但因為它不允許使用者輕易地關閉開啟的視窗，所以煩人。 Reconditioning 您的想法可能很困難。 幸好在下列組建中，已解決此問題。 現在視窗的框線和 [關閉] 按鈕之間還是有空格，但是按一下該空間也會導致視窗關閉。

直覺設計的一個非常重要的因素是，「精神 bandwidth'」的數量，也就是我們想到理解某個事物的時間。 「頻寬」使用量越低，使用者體驗就越好。

這些是對使用軟體應用程式的「經驗」有所貢獻的小部分。 下列範例提供使用真實的秘訣和訣竅來改善應用程式的秘訣。

## <a name="20-tips-for-a-better-functional-user-experience"></a>20提示更好的功能使用者體驗

更好的使用者體驗的目標是讓更簡單、更簡單的功能 UI 看起來很不錯。 這些秘訣可協助您塑造 UI，使其更有效率。

### <a name="use-standards"></a>使用標準

無論是在作業系統層級、品牌層級或應用層級上，所有軟體環境所建立的標準都非常重要。 除了商標之外，標準在使用者的想法中也扮演有名的架構。 當使用者花很長一段時間來使用軟體應用程式時，會因為不斷成長而自動提高生產力。

在討論標準之前，讓我們先討論這些標準到底有什麼。 標準包括以特定方式在對話方塊中配置控制項的所有內容，例如 [確定] 和 **[** **取消**] 按鈕，以及使用者介面的圖形—視窗頂端的圓角，如 Windows XP 對話、圖示樣式、任何其他圖形的樣式、應用程式的互動式行為等等。

如果您的應用程式落在特定的範圍內，則它可能會費用得更好，以遵循一組不同的標準。 例如，如果您的應用程式支援的應用程式或附加元件 Office OneNote 2003，則最好遵循 Office 的 UI 和互動標準的樣式），特別是 OneNote 本身。 這包括使用 Office 樣式的命令列，而不是標準的工具列，以及其他這類專案，也就是視覺效果和行為。 如果您的應用程式是 Microsoft Visual Studio .net 分類的一部分，則您會有一組不同的標準。 事實上，對於這類附加元件或支援的應用程式，Microsoft 之類的公司都有發行書面指導方針。 另請注意，圖形和設計概念有時是受保護的智慧屬性。 請務必檢查適當的檔，以確定您有建立這類設計的授權。

標準的第三個範例是 Tablet PC 環境。 這些標準跨越作業系統方針和應用程式指導方針之間的界限。 [TABLET PC SDK 檔](/previous-versions/ms840465(v=msdn.10))包含「規劃您的應用程式」主題中一些非常有用的資訊。 相較于 theOffice 2003 或 Visual Studio 的指導方針，這些設計建議會直接影響使用者與您的應用程式互動的方式，以及它的行為。 比方說，如果您在應用程式中有停駐視窗，檔建議您確定它可以偵測螢幕方向何時變更，而且停駐視窗會視需要適當地在直向或橫向重新組織。 即使您未將應用程式設計為平板電腦專用，您還是應該移至這些指導方針。

隨著智慧型用戶端的增加，應用程式現在會跨越不同硬體（一般電腦、平板電腦、行動裝置或 Ultra 行動裝置、Media Center 電腦等）之間的界限。 每種情況下，都會呼叫不同 (或其他) 組標準。

當應用程式共用作業系統層級或應用層級標準時，使用者會在家裡使用軟體，讓您更容易學習和使用。 這項結果可直接提升生產力。 使用者希望能夠儘快獲得新軟體的生產力。

### <a name="draw-attention-to-important-buttons"></a>強調重要按鈕

有時候您需要將使用者導向最重要的按鈕，即使您有四個或五個其他按鈕。 如果您使用大小、色彩或字型進行實驗，就會中斷標準（不建議這麼做）。 相反地，您可以使用幾個簡單的訣竅來完成這項工作。

第一個是將其他非關鍵性按鈕轉換成 LinkLabels，如下圖所示。 如此一來，使用者就知道這些連結會執行工作，但他們的注意會先進入指出的按鈕，而不會破壞標準的設計方針。

![轉換成 linklabels 的三個按鈕的影像。](images/humanux-04.png)

第二個是將 [重大] 按鈕排在第一行，也就是水準配置的左邊（如下圖所示），或在垂直配置的上方。 請注意，這可能會隨著目標使用者的文化特性而變更，如果您的應用程式是由右至左書寫的任何語言，則您可能會在最右邊的按鈕中，費用更好的效果。

![三個按鈕的影像，其中關鍵按鈕是水準配置的最左邊。](images/humanux-05.png)

建議的選項是設定為預設接收焦點。 例如，在 [刪除確認] 對話方塊中，[ **否** ] 選項應該會反白顯示，因為這樣可避免使用者不小心刪除某個內容。

### <a name="simplify-recognition-with-icons"></a>使用圖示簡化辨識

圖示（尤其是 Windows XP 和 Office 2003 圖示和工具列點陣圖）有助於加速 UI 的認知以及使用者必須執行的工作。

例如，當您在訊息方塊上看到最常看到的驚嘆號圖示時，您會立即注意到該圖示旁邊與控制項相關聯的風險層級。 同樣地，當您的應用程式組成大量控制項時，無論是如何適當排列，都可能會很難找到您要尋找的控制項集合。

在 Windows XP Service Pack 2 中，已更新的索引標籤會新增至 [**系統屬性**] 控制台小程式，稱為 "自動更新"。 有四個選項可供選擇：自動下載更新、下載更新，但讓使用者決定何時要安裝、通知使用者是否有可用的更新，但不會開始下載，以及完全停用自動更新。

新的電腦使用者可能不知道這些更新是什麼，而且可能不知道最適合選擇的選項。 因此，Microsoft 會在最建議的方塊旁邊放置一個具有大核取記號的綠色盾牌圖示，表示「安全」的選項，並在其旁邊有一個大的「x」紅色盾牌圖示，它會對使用者造成危害。 這在重大情況下很有説明，特別是當使用者沒有時間讀取太多文字時。

在相同的 [ **系統屬性** ] 小程式中，每個索引標籤都有多個 GroupBoxes 具有不同的控制項，不同的 相對圖形放置在每個群組旁邊，方便表示控制項群組的工作。 這類型的圖形化程式碼類似于實體檔案或停車場中的色彩編碼。 這也適用于在雜誌文章中至少具有部分視覺效果的相同原則，可讓讀者的興趣。

選擇正確的圖示也很重要。 Microsoft 在 Visual Studio 2005 中提供許多標準圖形。 這些都是最佳選擇。 如果您建立自己的圖示，強烈建議您遵循上述「 [使用標準](#use-standards) 」一節中所述的作業系統層級或應用層級標準來進行這些圖形。

[Windows 的使用者經驗互動指導方針](/windows/apps/desktop/)包含了非常實用的指南，可協助您建立 Windows 樣式的[圖示](https://msdn.microsoft.com/library/aa511280.aspx)。

### <a name="simplify-recognition-with-headers"></a>使用標頭簡化辨識

Windows Installer 的嚮導可能是最受歡迎的標頭：最右邊的簡單圖示;描述對話方塊的標題標籤 (例如，選取 [安裝資料夾]) ;以及描述對話方塊用途的子標題 (例如，選取將安裝軟體檔案的資料夾) 。

假設我們有一個具有帳戶區段的一般商務應用程式。 依照 Windows Vista 所探討的設計典範，我們可以在標頭 (或頁尾中提供任務關鍵性資訊和相關命令（如果案例) 本身的情況下）。 我們的使用者已經開啟「大型公司」的帳戶檔案，而且標頭看起來會像下列螢幕擷取畫面所示。

![包含詳細頁尾之對話方塊的螢幕擷取畫面。](images/humanux-06.png)

![對話方塊的螢幕擷取畫面，其中包含詳細的標頭。](images/humanux-07.png)

同樣地，您也可以避免必須加入 Windows XP 樣式的工作窗格，特別是當您只有幾個命令，這會將這些命令移至標頭，這會浪費許多垂直空間。

在設計標頭時，您應該牢記幾件事：

-   請確定背景色彩與對話方塊背景色彩不同。 比起，標準 Windows 內建控制項臉部色彩的白色標頭則較不常。 但是，如果您真的想要確定沒有任何特殊的主題或自訂色彩會讓您的標題混亂，請使用 **FromKnownColor** 搭配 **ControlLight** 和 **ControlDark** 的色彩來繪製 **LinearGradient** 。

### <a name="use-custom-message-boxes"></a>使用自訂訊息方塊

標準 Windows 訊息方塊中提供的選項非常有限。 當您必須詢問使用者無法使用簡單的 [是]/[否] 或 [取消] 來回答的問題時，就會變得複雜。

Windows 的應用程式現在會變得更容易使用，因為這是大量非技術性的使用者。 有時候，提供更容易使用的文字，甚至還有一些其他控制項（例如 LinkLabels），讓您更輕鬆地完成工作。

Microsoft .NET Framework 可讓您輕鬆地執行自訂對話方塊。 只要在自訂對話方塊表單上指派幾個屬性，或使用一行程式碼，您的表單就可以像標準的訊息方塊一樣運作。 在按鈕按下事件中，將對話方塊的 [ **DialogResult** ] 屬性設定為 **[DialogResult] 或 [** **DialogResult]。** 使用來自父表單的 **ShowDialog (\[ OwnerForm \])** 方法。 這個方法方法會傳回 **DialogResult** 值。

您可以使用所有的 **DialogResult** 成員。 標準 **MessageBox. Show** 方法會使用這些相同的選項。

或者，您可以只將對話的 **>acceptbutton** 屬性設定為 **btnOK**，並將 **CancelButton** 屬性設定為 **btnCancel**。 這會自動將 **Enter** 和 **Esc** 鍵對應至 **btnOK** 和 **btnCancel** 按鈕的個別 Click 事件。

-   針對複雜的主題，請提供本機或線上說明的連結，其中 LinkLabel 說明適當的文字標籤下的「深入瞭解」。
-   您可以 /  / 使用文字來明確陳述按一下按鈕的結果，例如「儲存檔案並結束」、「離開但不儲存 **」和「** 不要結束」，而不是 [是] 取消按鈕。 不過，請盡可能保持標準 [**是** / ****]、 **[確定]、[** / **取消**] 和 [標準] 按鈕。 熟悉是否可提供絕佳的生產力。
-   根據目標文化特性設定) ，讓左邊界空間的 (或右邊保留50圖元，並新增代表對話方塊案例的圖示。 如果是 [資訊] 對話方塊，您可以使用標準訊息方塊所使用的 "i" 圖示;如果是 [安全性] 對話方塊，您可以使用鎖定圖示或按鍵圖示。 Visual Studio 2005 隨附一些絕佳的高品質圖形。

### <a name="include-alternate-commands"></a>包含替代命令

有兩個重要因素需要替代的輸入方法，也就是挫折和酷。 挫折是電腦使用者經常發生的事情。 當您感到挫折時，您想要快速完成工作。 幾秒鐘的額外點擊或額外等候真的 infuriates 一位人的壓力—您知道它是怎麼的，我們一直都在那裡。 酷通常會主張切勿您使用鍵盤或滑鼠來完成工作—無論您目前使用的是哪一種。 除了這兩個因素之外，具有替代輸入方法可讓使用者輕鬆地完成工作。

例如，如果您有一個具有兩個按鈕（[新增] 和 [移除]）的清單方塊，則應該在任一側新增該清單方塊的內容功能表，其中包含類似這些按鈕的功能表命令。 這讓使用者有機會選擇最適合的方法。 身為 Windows Vista 使用者體驗指導方針狀態的新手使用者，請使用內容功能表很多，而且希望他們在其上按一下滑鼠右鍵。

同樣地，我們會使用視覺控制項進行文字或數位輸入。 例如，滑杆可用來指定整數，而行事曆控制項則用於日期輸入。 有時候只要輸入，就可以更輕鬆地輸入。 如果您加入連結至滑杆的數值 Up-Down 控制項，或使用 DateTimePicker 而不是行事曆控制項，則通常會讓使用者有所差異。

### <a name="how-to-handle-critical-actions"></a>如何處理重要的動作

執行重要、無法復原的函式時，通常最好是讓訊息方塊快顯視窗確認動作。 讓我們來看一個凹槽。 在下列螢幕擷取畫面中，您可以看到類似的自訂訊息方塊，但有一個額外的優點，也就是以視覺化方式顯示的計時器，並顯示進度列的協助。

![[重要動作] 對話方塊的螢幕擷取畫面。](images/humanux-08.png)

您可以使用一些案例特定的變化。 如果要採取的動作非常重要 (範圍從多載的，到永久刪除檔案) ，則計時器執行之後的預設動作應該是 [取消]。 對話方塊不應消失，但是文字標籤會變更，以顯示已取消動作。 使用者可以選擇確認命令或取消。

務必清楚地標示執行重要作業的按鈕，一律使用可精確描述動作的純文字。 如果動作是刪除檔案，請勿寫入「從存放庫移除檔案」;撰寫「刪除存放庫中的檔案」。 使用檔案清單時，如果 [刪除] 功能表命令從硬碟本身刪除選取的檔案 (而不只是從 [檔案清單] 中移除) ，您應該適當地強調這項操作的關鍵本質，並明確地強調該動作將會永久刪除檔案。

有人說，「您的工作越糟越好。」 軟體應用程式也同樣適用。 使用您的應用程式時，單一的不良體驗可能對使用者造成嚴重的負面印象。 為了確保不會發生這種情況，您可以做的一件事就是確定您的應用程式損毀時，它會正常運作。 如果您可以新增資料復原，或允許使用者嘗試儲存該資料的複本，它可能是很大的加法。 如果應用程式當機，則應適當通知使用者。 JIT-Debugger 或嚴重錯誤對話方塊不是好的事。 雖然討論如何處理當機的範圍超出本文的討論範圍，但我建議您由衷抱歉到使用者的簡單對話，並通知他或她 (的情況，並可能提供有關如何從此損毀中復原的詳細資訊連結) 對使用者而言很有説明。

如果您想要進一步瞭解，您可以執行我最愛的圖形設計應用程式。 如果發生損毀，則會顯示 [復原] 對話方塊，讓您儲存正在處理的個別檔案複本，然後提供您意見反應對話方塊，您可以在其中輸入損毀的相關資訊 (個人資訊，當然) 並傳送給建立者。

### <a name="radiobuttons-or-comboboxes"></a>選項按鈕或下拉式方塊？

乍看之下，進行一對多選取的方法似乎有點困難或不重要。 有時可能是—特別是當案例是用於時間緊迫工作的應用程式時。

讓我們看看一個實際的範例。 Microsoft 最近發行了預覽版本的圖形應用程式、Expression 圖形設計工具 (先前代號 "Acrylic" ) 。 我有大約20個繪圖物件，我必須在此應用程式中分開指派特定屬性。 它是一種單調的流程。 若要這麼做，我必須選取物件，按一下按鈕以開啟 [設定] 視窗，並設定選項。 在其中一個選項中，您必須從下拉式方塊挑選兩個選項，如下列螢幕擷取畫面所示。

![[運算式圖形設計工具] 的 [附加材質] 對話方塊的螢幕擷取畫面。](images/humanux-09.png)

當您必須下拉下拉式方塊清單，然後選取第二個專案 (僅有2個專案時) 可以真的煩人。 我們通常不知道的是下拉式清單顯示所需的時間。 這可能會浪費許多時間，而且可能很令人沮喪。 您可以使用兩個選項按鈕放在群組方塊中，輕鬆地解決這種情況，特別是當有這麼多空間可用時。 我在 CorelDRAW、Microsoft Access 和其他應用程式中遇到類似的問題。

由於下拉式動畫的浪費時間，它也浪費了「精神頻寬」。 在兩個「永遠可見」選項按鈕的情況下，我們的想法會 subliminally 知道游標要按一下的位置。 使用 ComboBox，它只會在繪製清單之後處理。 雖然可能有點不重要，但其實非常重要。

有時候最好使用選項按鈕，特別是當您有4個或更少的選項時。

### <a name="never-disrupt-the-user"></a>永遠不會干擾使用者！

很簡單地說，這是開發人員可以對使用者做的最具破壞性的事情。 如果您的應用程式（非必要或其他）在使用訊息方塊或工作列閃爍的另一個應用程式上工作時中斷使用者，您就會從使用者獲得負面的點。

當然，工作列閃爍可能很有用，但應該只在應用程式的程式需要使用者輸入才能繼續時呼叫，或者，如果您有重要的東西要傳達給使用者。 如果使用者已將工作列保持在自動隱藏，則閃爍的工作列按鈕可以對使用者存取狀態列或其他低錨定的控制項（因為工作列會在其頂端），直到使用者按下閃爍按鈕為止，都不會再次隱藏。

![快顯通知視窗的螢幕擷取畫面。](images/humanux-10.png)

「快顯」 windows (請參閱 [圖) 10]，這是由立即訊息用戶端（例如 MSN Messenger）所構成的絕佳解決方案，可通知使用者不會討厭或破壞其工作流程。 在建立快顯通知視窗時，帳單 Wagner (有很棒的文章 https://docs.microsoft.com/archive/msdn-magazine/2005/september/sprinkle-some-pizzazz-on-your-plain-vanilla-windows-forms-apps) 。 這是很好的原則 (和形式) 不會打擾任何其他應用程式的快顯通知。 這類 windows 的障礙物可能很麻煩且免除沒有效率。 其中一個解決方法是使用作業系統所提供的 ToastSemaphore Mutex (/library/WinMessenger/winmessenger/overview/toast.asp) ，以避免快顯衝突。

有時您可能需要在快顯通知中顯示多個專案。 彈出3個或更多的快顯通知並不是建議的方法。 相反地，藉由在另一個快顯中快顯/淡化快顯通知來逐一迴圈。 當通知使用者內送電子郵件時，Microsoft Outlook 會實行類似的解決方案。

### <a name="provide-progress-status"></a>提供進度狀態

通常會有需要使用者等待的工作。 當然，這是使用者單純討厭要做的其中一件事。 但是更糟的是，在不知道發生什麼狀況的情況下等待。 有時候您的應用程式可能需要連線至 Web 服務或遠端電腦，或可能會處理大量的資料，無論原因為何，使用者都應該知道，或至少模糊知道，幕後發生什麼事。 根據情況，有不同的方法可以執行這項操作。

如果您要連接到一些物件（例如 Web 服務），或是放置在網路或網際網路伺服器上的某個位置，建議您顯示簡單的進度對話方塊， (查看下圖) ，或在狀態列中裝載的進度列。 伴隨的標籤應該描述進程的目前狀態。 例如，如果您要連接到 Web 服務來處理某些資料，只要說「正在連接到 Web 服務 .。。「或」請稍候，正在處理 .。。「如果這個程式是同步的，建議您停用使用者可以存取的所有控制項，直到程式完成為止，或只將進度顯示為強制回應對話方塊。

![[進度列] 對話方塊的螢幕擷取畫面。](images/humanux-11.png)

如果您使用進度列且處理時間不明，或如果您沒有最大值，強烈建議您將進度列樣式設定為 [天棚] 模式。

另一個變得很熱門的方法是顯示進度的固定「快顯」視窗。 Microsoft 反間諜軟體下載程式/更新程式或諾頓防毒軟體掃描快顯通知是很好的範例。 當然，這應該僅用於非同步處理常式。 否則使用者可能會覺得 disconcerted。 這類視窗最適合用來進行背景處理，例如下載更新或執行排定的工作，而且永遠不會設定為「永遠開啟最上層」。

### <a name="simplify-complex-steps-with-wizards"></a>使用嚮導簡化複雜的步驟

您可以放心地假設如果在單一表單上有控制項的眾多，一般的使用者就不會混淆。 有時候，當您有許多重要的控制項時，不會有任何數量的群組、大小調整或間距可提供協助。

Wizard 是這類案例的最佳做法。 您可以依適當情況將控制項除以工作或類別，並將它們放在個別的步驟中。 這可協助使用者保持焦點，而不是由工作 daunted。 您可以使用 [說明] 按鈕來提供步驟或工作特定的說明。 您可以在 MSDN Library 中找到 Wizard 建立指導方針。

您也可以使用嚮導來協助設定應用程式的初始設定。 許多應用程式會在安裝程式完成或第一次使用時，使用這類嚮導來設定個人化設定。 這種初始的 wizard 也應設為選擇性（如果可能的話）。如果使用者在任何時間點取消，未指定的設定會移至預設值。 如果您可以讓 wizard 成為一個圖形 (請參閱 [ [使用美觀的圖形](#use-pretty-graphics) ] 區段) ，使設定工作更容易。

### <a name="get-the-tone-of-your-text-right"></a>取得文字右邊的語氣

在[Windows 使用者經驗互動指導方針](/windows/apps/desktop/)中，已對「文字語氣」有很重要的一點。 這是您應用程式中的文字所提供的印象和感覺。 這可以是從簡單工具提示到指令標籤控制項的任何作業。

稍早，我們已討論過 MSN Messenger 中網路攝影機選項的文字變更。 這稱為適當的文字語氣。 處理非技術性或新手的使用者時，跨不同的層面取得訊息。

如果您在自我解壓縮應用程式中的文字方塊上方寫入「目的地路徑」，則技術使用者很容易就能知道您輸入的內容像是「C： \\ Temp \\ MyPath」。 新手的使用者 (認為「Mom」 ) 可以輕易地 baffled，而且必須參考手動、致電技術支援或更糟的—放棄。 最好的方法是指定您要讓使用者執行的動作：「選取您要放置這些檔案的資料夾」。 您甚至可以重新命名 [流覽 ...]"的按鈕位於該文字方塊旁的 [選取資料夾 ...]"

清楚描述您想要讓使用者進行的工作，也能減少說明檔的需求，或至少減少您需要包含在說明檔中的詳細資料。

[Windows 的使用者經驗互動指導方針](/windows/apps/desktop/)中的一項非常好的建議適用于任何軟體。 它指出寫入器應該保持文字交談。 這些指導方針將此項定義為：「避免他人說不到別人的單字」。

撰寫文字的一些秘訣：

-   避免在第三個人中談論使用者。 說「您」，而不是「使用者」。
-   請盡可能謹慎地使用 "My name：" 或 "My E-Mail address："，而不是 "Name：" 或 "E-Mail："
-   提供多個選項時，請從使用者的觀點來撰寫文字。 例如，如果您的標籤底下有兩個選項按鈕（例如「這個網路上的使用者名稱的選取許可權」） \[ \] 上方的兩個選項按鈕（例如「允許」和「拒絕」），請將選項按鈕文字取代為「我想要允許使用者 \[ 名稱」和「 \] 我想不 \[ 允許使用者名稱」 \] 。
-   只有在用於連結時才會將文字加上底線。 如果加底線的文字不是連結，則會會混淆使用者。
-   使用粗體標籤吸引重要的資訊，但請小心使用。 太多粗體文字會造成混淆，並減少表單的整體影響。
-   當您撰寫核取方塊的文字時，請確定它很容易知道當選取此選項時，以及未選取或清除時所發生的情況。 建議的選項是直接將文字寫入作為選取之核取方塊的結果。 例如，撰寫「從您的夥伴傳送有用的資訊」，而不是「不要向我的夥伴傳送有用的資訊」。 雖然我可以想像許多行銷人員爭論這個特定的範例，但我相信您知道我的意思。
-   如果您有類似按鈕的控制項 (通常是具有命令按鈕外觀) 控制項已啟用/停用的選項按鈕，請務必將它標示為正確。 如果程式已啟用，請寫入 "Enabled" 而非 "Enable" 或 "Disable"。 如果您已啟用寫入，則會顯示目前的狀態。 如果按一下按鈕 ([啟用]) ，而按鈕顯示 [啟用]，則可能會造成混淆且有問題。 [啟用] 可能會提示使用者按一下，以為進程不在使用中狀態。

### <a name="sometimes-a-listview-is-better"></a>有時 ListView 更好

針對選取工作，我們通常會使用 DataGrid、ListBox 或甚至 ComboBox，但是 Windows XP 和更新版本的 Windows，則使用 ListView 可以提供更大的選項。

ListView 控制項的細微點：

-   使用圖示和點陣圖加速專案辨識。
-   顯示具有詳細資料或磚視圖的其他資訊。
-   使用 Visual Studio 2005 時，您甚至可以有群組以進行其他分類。 群組跨所有視圖，而且有彈性。 群組也可用來壓平合併階層視圖 (例如 TreeView) ，其中的子節點數目超過父節點。 其中一個很好的例子，就是 Windows XP 中的 [網路連線] 對話方塊（當您使用 [在群組中顯示]，並將視圖設定為 [詳細資料] 時）。
-   若要自訂 ListView 控制項，請設定 **OwnerDraw** 屬性，並使用 **DrawItem** 和 **DrawSubItem** 事件，以手動方式進行繪製。
-   支援 ListView 專案的快速就地編輯。
-   輕鬆支援手動重新排列。
-   讓使用者選取 (大型圖示、小圖示、清單等) 最適合使用的視圖。

### <a name="simplify-navigation-with-breadcrumb-controls-and-sidebars"></a>使用階層連結控制項和提要欄簡化導覽

「子導覽」是複雜 UI 的關鍵。 有時候您不能以複雜的 UI 來進行 escape。 在這種情況下，最佳的作法是讓使用者的體驗盡可能簡單。 包含連結標籤的提要欄位，或以階層為基礎之導覽的樹狀檢視，建議目前對話工作的同級層級導覽。 這讓使用者很容易就能在程式的步驟之間跳躍，同時知道他或她的所在位置。

如果您進入以階層為基礎的 Treeview 或其他類似的複雜導覽，則使用者的良好公用程式會是階層式控制項。 雖然 Visual Studio 並未隨附內建的控制項，請參閱建立階層連結[控制項](/archive/msdn-magazine/2005/july/advanced-basics-creating-a-breadcrumb-control)，以取得您自己建立的連結。 階層連結控制項可讓您輕鬆地找到相對於階層的目前位置。

如果表單有一個，則可以輕鬆地將階層連結導覽合併到標頭。 請參閱上一節中的 [標頭](#simplify-recognition-with-headers)。

### <a name="use-pretty-graphics"></a>使用美觀的圖形

每個人都熱愛具有非經常性圖形的應用程式，最重要的就是大部分。 雖然具有美觀圖形的 UI 並不是所有應用程式的邏輯選擇，但它確實有助於您的印象，並可讓您輕鬆地使用。 當然，圖形應該不會妨礙生產力，但如果正確使用，則可以增加！

不需要有許多圖形，也不一定需要太多工作。 經過專業設計的啟動顯示畫面或標頭 (，就像我們在前面提到的) 會執行這項技巧。 如果您的預算允許，您可以針對工具列、嚮導等使用設計良好的圖形。 它們讓您的應用程式看起來很美觀，也更專業。 這是一項很微妙的效果，但是專業的外觀會傳達信心和穩定性。 如果您是建立零售應用程式的相對小型公司，這是要考慮的重要層面。

一律讓重點成為使用專業設計的圖形。 免支付權利的圖形很容易使用且價格實惠。 您也可以雇用設計工具。 但是，如果圖形不是您的 forte，請勿親自試試看。 如果您無法取得或使用專業設計圖形，則最好不要使用它們。

針對小型圖形，您可以一律移至 Visual Studio 2005 隨附的圖示和點陣圖。 不建議 (隨附于舊版的圖形！ ) 

### <a name="provide-resizable-forms-when-possible"></a>盡可能提供可調整大小的表單

可調整大小的視窗與解析度無關的視窗有點類似。 無論您是使用96DPI 或300DPI 畫面，解決獨立的視窗看起來都一樣。 無論應用程式的 UI 是否與解析度無關，如果可調整大小，則會費用得更好。 當然，這並不適用于許多案例，而是良好的一般用途規則。

如果您的視窗處理任何排序（尤其是 Listview）的清單，這會變得更重要。 調整大小可讓使用者同時查看更多資料。

例如，我們有一個應用程式，使用者必須從大型集合中選取一個影像。 [開啟] 對話方塊可讓您選取縮圖視圖，但對話是固定大小，而縮圖清單一次只顯示4個縮圖。 如果集合具有一百個影像、滾動和查看（重複的工作），可能會相當一定乏味且效率降低。 如果對話方塊可以調整大小，則使用者可以將它設得很大，就像螢幕允許的一樣，也可以快速完成工作。 如果您的清單有水準滾動，例如詳細的 ListView 或 DataGrid，則更一定乏味！ 可調整大小的視窗在這種情況下非常有用。

### <a name="provide-more-functionality-with-sidebarstask-panes"></a>使用側邊欄/工作窗格提供更多功能

如同我們先前提到的標頭，提要視窗和工作窗格是提供額外功能和公用程式命令的絕佳方式。 例如，Office Word 2003 中的工作窗格很方便、可供存取，且不會干擾。 它們也會在連接到線上資源時以非同步方式運作，並提供使用者多工的選項。

建立工作窗格或提要欄位就像建立停駐面板一樣簡單，而且可以選擇將巧妙的圖形放在頂端，以作為標題列。 您甚至可以使用彩色的標籤控制項。 工作窗格的商機很多！

如果您有其他功能，而且想要將它提供給您的使用者，則不會有任何地方，像是工作窗格。 您也可以讓工作窗格「自動隱藏」或像 Visual Studio 工具視窗一樣折迭。

### <a name="give-a-notification-choice"></a>提供通知選擇

先前我們已看到如何建立自訂訊息方塊。 如果您的應用程式中的訊息方塊會經常顯示給使用者，則可以謹慎地新增核取方塊，讓使用者可以選擇停用該對話方塊，使其無法在未來顯示。 這種選項特別適用于更明顯的訊息。

其中一個常見的範例是 Visual Studio 尋找] 對話方塊。 當您搜尋或取代文字時，Visual Studio 會顯示一個訊息方塊來指出結果。 但是您也可以選擇停用該訊息方塊。 如果您必須按 Enter 鍵，或每次搜尋時都按一下 [確定]，可能會很麻煩。

Visual Studio 的另一個很酷的事，就是即使對話方塊已停用，仍然會在狀態列中顯示該作業的結果。

### <a name="provide-tooltips"></a>提供工具提示！

有時候工具提示可以節省很多時間。 按鈕、核取方塊和其他控制項可能不明確，而且使用者可能不確定該怎麼做。 工具提示只會在單一行中提供內容相關説明的最佳形式。 使用者可以快速決定要怎麼做，而不需要搜尋說明檔中的任何內容，或開啟另一個視窗。

人們通常會在其應用程式中略過此程式。 如果可能的話，請將工具提示新增至所有不明確的控制項（或所有控制項）。 請勿重複伴隨標籤的文字或該控制項本身的文字，但是提供該控制項的其他資訊。 文字應該只以幾個字解釋控制項的功能。

### <a name="do-not-forget-the-little-things"></a>別忘了一點

很多人都可能會對您造成問題，但是忽略它們可能會影響您所做的印象。 我曾經使用的應用程式是由軟體產業中重要的人員所組成，其表單的 BorderStyle 設為相當大，但表單右側的控制項並未錨定。 因此，產業所建立的應用程式具有非專業的風格。

這類「小事物」是整體印象的核心。 您應用程式的 UI 和 UX 是使用者將在一開始就判斷您應用程式的目標。 如果他們在您的 UI 中看到明顯的錯誤，可能會認為您的應用程式較不強大且更有效。

## <a name="conclusion"></a>結論

我們只觸及一小部分的人為使用者體驗。 由於使用者體驗變得更簡單、有效、有趣，而且更容易使用，因此建立該使用者體驗的工作會變得更複雜。 但有一些先見之明和良好的規劃，您就可以建立絕佳的使用者體驗。

建立最佳使用者體驗的最佳方式，是進行以特別是在 UI 上為目標的可用性測試，無論是使用特殊測試群組或您自己。 在發行應用程式之前，您花在測試使用者體驗的時間愈多，就愈好。 它會在稍後為您節省很多問題。
