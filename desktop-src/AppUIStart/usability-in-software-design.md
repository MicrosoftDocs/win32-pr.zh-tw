## <a name="introduction"></a>簡介

建立軟體之內容中的「可用性」一詞，是指將使用者（而不是系統）放在程式中心的方法。 這項原理（稱為「以使用者為主的設計」）結合了設計程式開頭的使用者關注和推動，並規定使用者的需求應該是任何設計決策最重要的一項。

這種方法最明顯的層面是可用性測試，在此測試中，使用者可以工作並與產品介面互動，並與設計人員和開發人員分享其觀點和疑慮。

## <a name="defining-usability"></a>定義可用性

本節定義軟體發展環境中的可用性是什麼，以及它與開發流程的其他層面有何關聯。

### <a name="ease-of-use"></a>便於使用

可用性是指使用產品來執行指定工作有多麼容易的方法。 這與公用程式和 likeability 的相關概念不同。

### <a name="usability-vs-utility"></a>可用性與公用程式

決定產品質量的主要屬性是實用性。 這會測量產品的實際使用是否可達成設計人員想要達成的目標。 實用性的概念會進一步進入公用程式和可用性。 雖然這些詞彙是相關的，但它們並不是可互換的。

公用程式指的是產品執行工作的能力。 產品設計的工作越多，它所擁有的公用程式就越多。

請考慮從1980年代晚開始的一般 Microsoft MS-DOS 字處理器。 這類程式提供許多強大的文字編輯和操作功能，但需要使用者學習並記住數十個難懂按鍵來執行這些功能。 這些應用程式可說是具有高效用的 (它們可為使用者提供必要功能) 但低可用性 (使用者必須花大量時間和精力來學習和使用這些應用程式) 。 相較之下，設計完善的簡單應用程式（例如計算機）可能很容易使用，但不提供許多公用程式。

這兩種品質都是市場接受的必要條件，而兩者都是整體實用性概念的一部分。 很明顯地，如果程式具有高可用性，但沒有任何值，則沒有人會使用它。 而且，如果使用者看到的強大程式很難使用，則可能會使其無法使用或找出替代方案。

可用性測試可協助您決定讓使用者執行特定工作有多麼容易。 但是，它不會直接協助您判斷產品本身是否有價值或公用程式。  (使用者可能會在可用性測試期間自願使用與公用程式相關的批註，但任何批註都應以其他更穩固的研究方法進行驗證。 ) 

### <a name="liking-it-vs-using-it"></a>喜好與使用它

Likeability 一律是產品中的理想特徵。 如果是產品，則比較有可能使用它，並建議其他人使用。 但就像使用公用程式一樣，您應該小心不要混淆 likeability 與可用性。

人們通常會像是與公用程式和可用性無關的原因。 它們可能會被吸引其樣式，或是他們認為產品會在其上授的狀態。 人們通常會喜歡高度可用的產品，但您不應該假設這表示有很好用的產品可供使用。

可用性是指某個人是否可以使用該產品來執行所需執行的工作。 可用性測試主要是測量效能，而非喜好設定。 不過，標準化的問卷可以用來測量各產品的喜好設定。

### <a name="discovery-vs-learning-vs-efficiency"></a>探索與學習與效率的比較

可用性有許多方面，但傳統上的一詞是指探索、學習和效率等特性。

-   探索包含尋找及尋找產品的功能，以回應特定的需要。 可用性測試可以決定使用者尋找功能的時間長度，以及使用者在該過程中)  (錯誤選項的錯誤數目。
-   學習指的是使用者瞭解如何使用探索到的功能來完成工作的進程。 可用性測試可判斷此程式所需的時間，以及使用者在學習此功能時所進行的錯誤數目。
-   效率是指使用者具有「主要」功能並加以使用的時間點，而不需要進一步的學習。 可用性測試可以決定有經驗的使用者執行使用此功能所需的步驟所需的時間。

這三個基本層面的可用性會受到手邊工作的本質以及使用者執行的頻率所影響。 有些功能很少使用或太複雜，因此使用者基本上每次都會 relearns 它們;針對這些功能，Microsoft 通常會開發嚮導來引導使用者完成程式。

### <a name="slogans-do-not-work"></a>Slogans 無法運作

軟體發展人員有時認為簡單的 slogans （例如「讓產品更容易使用」）將有助於解決可用性問題。 雖然對可用性有正面的態度很重要，但在建立的特定產品內容中，只有一般使用者的適當可用性測試才可提供開發人員所需的資訊，以建立可滿足使用者需求的產品。 「讓產品更容易使用」應該是每個軟體發展人員的標語，但只有在開發人員知道可用性的意思時，才有意義。 使用一般使用者進行測試是最可靠的方法來找出。

## <a name="why-is-usability-important"></a>可用性為何重要？

本節將回答一些有關可用性為何重要的常見問題，以及如何將以使用者為主的設計原則納入開發程式中。

### <a name="why-should-you-care"></a>您為什麼要在意？

如果未在產品設計程式中併入可用性考慮，您可能會想知道為何需要或更有需要。 畢竟，您當然可以釋出可運作、無 bug 的產品，而不需要執行任何可用性工作。 但是，合併以使用者為主的設計原則，可能會導致許多方面的產品獲得改良。

執行可用性測試的最佳原因是減少使用者的支援呼叫次數。 使用不良的原因是使用者呼叫軟體技術支援人員的主要原因，而且每個軟體公司的主管和資訊服務管理員都知道產品支援的昂貴程度。 此外，向使用者收取支援會增加產品的潛在不滿。 如果使用者發現您可以輕鬆地使用您的產品，則不需要經常呼叫技術支援。

針對內部使用所產生的軟體，讓可用性成為開發程式的重要部分的下一個最佳原因是降低定型成本。 使用高度可用的產品，可讓使用者更容易瞭解可用性不是高優先順序的產品。 使用者可以更快速地學習功能，並讓其知識更久，這會直接與降低的定型成本和時間相互關聯。

可用性測試有助於改善使用者的接受度。 從數個因素（包括可用性、公用程式和 likeability）的接受結果。 對於零售產品而言，使用者接受通常會直接相互關聯以重複購買或忠誠度，這表示使用者可能會向其他人推薦產品。 對內部應用程式而言，使用者接受與使用軟體來執行其設計工作的意願相互關聯，有助於提高生產力。 提高可用性是可提升使用者驗收的因素之一。

可用性有助於將您的產品與競爭對手的產品區別開來。 如果有兩項產品在公用程式中大致相同，則具有較高可用性的產品可能會被視為高階產品。 此外，Windows 的外觀和操作和隨附的程式設計指導方針都已為基本使用者介面提供了「播放」欄位，讓許多服務相似函式的程式看起來像是有點相似。 這些相似性表示可用性的差異很小，對使用者喜好設定可能會有重大影響。

最後，每個產品最終都會測試可用性。 使用者會在每次使用產品時，對其執行可用性測試，並透過持續使用或缺少這些產品來呈現其結論。 在將產品釋出到市場之前測試產品，可協助確保使用者的產品體驗是正面的。

### <a name="what-does-it-cost"></a>它的費用為何？

軟體發展人員和專案管理人員通常會擔心啟始以使用者為主的設計程式，以及執行適當的可用性測試將需要無法接受的時間和金錢。 實際的情況是，將焦點放在使用者上的時間和金錢成本通常相對較小，而且與未執行它的成本相比，也是如此。

例如，請考慮在開發週期中，在開發週期延遲設計修訂的時間和成本，而不是稍早的時候，產品仍在繪圖面板上。 等到測試期間向產品公開使用者，以供可用性測試使用，可能會導致程式的部分 dismantling 花很多時間進行開發。 並等到產品實際發行，然後根據負面意見反應或支援的不良設計進行變更，才能讓成本 immeasurably 變得更高，因為產品支援的成本較高，或使用者的接收不佳。

合理的可用性研究通常可以在大約兩周或更短的時間內執行，而且可以大幅減少在開發週期中進行變更的時間和成本。 執行測試的成本會根據產品的本質和測試的介面元件而有所不同。

將可用性測試視為類似程式碼測試。 成功的專案管理器帳戶，可在規劃開發專案時進行程式碼測試。 他們不會看到它是必須附加至專案排程和預算的額外部分。 相反地，專案經理接受程式碼測試做為業務執行的成本，因為替代方案的成本較高。 這同樣適用于可用性測試。

### <a name="how-do-i-increase-usability"></a>如何提高可用性？

在閱讀及瞭解可用性的重要性時，軟體發展人員有時會想要增加可用性，就像是可以直接新增至產品以使其更容易使用的成分一樣。 相反地，可用性應該是設計流程本身的一部分，而不是在這裡或該處新增至程式的「事物」。 可用性專家指的是「使用者焦點」和「以使用者為主的設計」的原因，在於可用性取決於將使用者的需求集中到設計流程。 以使用者為中心的設計只需要遵循一組規則來管理介面中的按鈕和功能表位置。 可用性測試是檢查設計工作的機會。 這不是「將」可用性新增至產品的方法。

Gould、Boies 和 Lewis (1991) 找出四個以使用者為中心之設計的重要原則：

-   搶先專注于使用者。

    開發人員應該專注于瞭解使用者在設計過程中的初期需求。

-   整合式設計。

    設計的所有層面都應該以平行方式進行，而不是依序進行。 將產品的內部設計與使用者介面的需求保持一致。

-   早期且持續的測試。

    目前唯一可行的軟體設計方法是經驗一：如果真實的使用者決定運作，設計就能運作。 在整個開發過程中併入可用性測試，讓使用者有機會在產品發行之前，在設計上提供意見反應。

-   反復的設計。

    大問題通常會遮罩小問題。 設計人員和開發人員應該透過四捨五入測試來反復修訂設計。

### <a name="why-should-i-involve-users"></a>為什麼我應該牽涉到使用者？

開發人員應該要知道它們並非一般使用者。 他們對開發的系統比一般使用者所開發的系統更有深入的知識與瞭解。 對大部分的使用者而言，介面可能不清楚或令人困惑的層面，因此對於已處理專案的人員而言可能完全清楚。 有些軟體發展人員能夠以平均使用者的角度體諒前，但不會取代實際使用者與產品的實際互動。

因此，藉由經常專注于一般使用者的需求，並根據使用者測試來修訂設計，以使用者為主的軟體發展人員會產生更佳的設計，因此更適合產品。

使用更好的設計，更能接受使用者的接受。 零售軟體的好處很明顯：增加了銷售額。 接受對於針對內部使用所開發的軟體也很重要：提高使用者中心設計的焦點會導致提高生產力，並降低支援需求。 從開發開始，與使用者相關的說明，也會示範其關注和需求，進而提高其在開發工作方面的意願。

### <a name="cant-i-just-follow-guidelines"></a>我不是遵循指導方針嗎？

Microsoft 開發了一組適用于 Windows 運算平臺的介面指導方針，以確保 Windows 程式具有一致的外觀與風格。 其他公司也開發了類似于其他計算平臺的指導方針，而 Jakob Nielsen 等可用性專家已廣泛撰寫，以設計可用的網頁。 有了這些主題所提供的豐富資訊，設計人員有時會認為需要嚴格遵循指導方針和標準，以產生可用的產品。

這種方法的問題是，指導方針本質上是一般的。 指導方針必須適用于各種不同的情況，因此不一定會針對開發中的特定應用程式規定最佳的動作。 遵循一組妥善撰寫的指導方針，可能有助於設計一致的介面，但無法保證其 usablility，除非它已與實際使用者進行測試。 使用指導方針時，請不要使用這些指導方針，例如指導方針指向所有結果的最佳方式。 有兩個開發人員可以用兩種不同的方式來執行相同的指導方針，而且這兩個執行可能不會同樣適用于這種情況。 有時候，嚴格遵循指導方針可能會導致結果不佳，或指導方針之間的衝突。 只有以使用者為中心的設計可以在這些問題變成問題之前，協助將這些問題排清。

考慮這一點的另一種方式是：讓以使用者為中心的設計是設計決策的仲裁端，而不是使用者介面指導方針。

### <a name="do-i-need-to-build-a-usability-lab"></a>我是否需要建立可用性實驗室？

請勿假設可用性測試表示認可至昂貴的實驗室，並使用天花板掛接的攝影機、單向鏡像和其他焦點群組 trappings。 要確定，進行大量測試的公司通常會覺得建立專用實驗室很方便，而可用性顧問通常會有各式各樣的設施和設備來提供用戶端。 但是，有效的可用性測試可在各種不同的設定和情況下執行。

其中一種方法是直接擁有測試人員嗎？有人精通執行人為參與者研究和收集資料？坐在使用者的工作中，並觀察使用者執行工作。 您可以輕鬆地在會議室或辦公室內執行這項作業。 如需透過觀察測試的詳細資訊，請參閱 [其他資源](other-resources.md)中的 Dumas 和 Redish 專案。

隨著可用性測試的開發和發展更多，您也可以加入攝影機，例如攝影機、單向鏡像，或可讓您即時觀看並記錄使用者監視器的工具。

或者，您也可以將測試外包給可用性顧問。 下一節包含尋找正確顧問的秘訣。

### <a name="how-do-i-get-started"></a>我該如何開始？

一旦決定將使用者置中的設計原則納入開發程式之後，您必須決定是要雇用可用性專業人員，還是將可用性測試外包給廠商。

 (UPA) 的「可用性專業人員」關聯有廠商指南，可協助您尋找可用性顧問。

某些諮詢小組也可以協助設定可用性實驗室，或開發內部可用性程式，將可用性準則納入設計程式中。

如果雇用可用性專業人員，人類因素和體認協會都有放置服務，可協助尋找潛在的員工。 許多可用性專家也都屬於 Computer-Human 互動 (SIGCHI) 和 UPA 的限時特殊興趣群組。 將雇用廣告放在其出版作品或會議中。

無論採用哪一種路線，請記住這些都是測試服務。 設計人員並非一般使用者的原則也適用于可用性專業人員。

如需有關這些公司和組織的詳細資訊，並深入瞭解可用性測試和以使用者為主的設計，請參閱 [其他資源](other-resources.md)。
