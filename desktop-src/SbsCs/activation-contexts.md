# <a name="activation-contexts"></a><span data-ttu-id="946b0-103">啟用內容</span><span class="sxs-lookup"><span data-stu-id="946b0-103">Activation Contexts</span></span>

<span data-ttu-id="946b0-104">[*啟用*](a-sbscs-gly.md) 內容是記憶體中的資料結構，其中包含系統可用來將應用程式重新導向以載入特定 DLL 版本、COM 物件實例或自訂視窗版本的資訊。</span><span class="sxs-lookup"><span data-stu-id="946b0-104">[*Activation contexts*](a-sbscs-gly.md) are data structures in memory containing information that the system can use to redirect an application to load a particular DLL version, COM object instance, or custom window version.</span></span> <span data-ttu-id="946b0-105">啟用內容的其中一個區段可能包含 dll 載入器所使用的 DLL 重新導向資訊;另一個區段可能包含 COM 伺服器資訊。</span><span class="sxs-lookup"><span data-stu-id="946b0-105">One section of the activation context may contain DLL redirection information which is used by the DLL loader; another section may contain COM server information.</span></span> <span data-ttu-id="946b0-106">啟用內容函式會使用、建立、啟動和停用啟用內容。</span><span class="sxs-lookup"><span data-stu-id="946b0-106">The activation context functions use, create, activate, and deactivate activation contexts.</span></span> <span data-ttu-id="946b0-107">啟用函式可以將應用程式的系結重新導向至指定特定 DLL 版本、視窗類別、COM 伺服器、類型程式庫和介面的版本命名物件。</span><span class="sxs-lookup"><span data-stu-id="946b0-107">The activation functions can redirect the binding of an application to version-named objects that specify particular DLL versions, window classes, COM servers, type libraries, and interfaces.</span></span> <span data-ttu-id="946b0-108">如需啟用內容函式和結構的詳細資訊，請參閱 [啟用內容參考](activation-context-reference.md)。</span><span class="sxs-lookup"><span data-stu-id="946b0-108">For more information about the activation context functions and structures, see the [Activation Context Reference](activation-context-reference.md).</span></span>

<span data-ttu-id="946b0-109">從 Windows XP 開始，啟用內容函數可讓 Windows 使用 [資訊清單](manifests.md) 中的資訊來建立版本命名的物件。</span><span class="sxs-lookup"><span data-stu-id="946b0-109">Beginning with Windows XP, activation context functions enable Windows to use information in [manifests](manifests.md) to create version-named objects.</span></span> <span data-ttu-id="946b0-110">如果應用程式藉由呼叫 [**CreateProcess**](/windows/desktop/api/processthreadsapi/nf-processthreadsapi-createprocessa)來建立進程，Windows 會檢查 [應用程式資訊清單](application-manifests.md)是否存在。</span><span class="sxs-lookup"><span data-stu-id="946b0-110">If an application creates a process by calling [**CreateProcess**](/windows/desktop/api/processthreadsapi/nf-processthreadsapi-createprocessa), Windows checks for the existence of an [application manifest](application-manifests.md).</span></span> <span data-ttu-id="946b0-111">如果資訊清單存在，Windows 會使用資訊清單中的資訊來填入啟用內容。</span><span class="sxs-lookup"><span data-stu-id="946b0-111">If a manifest exists, Windows uses the information in the manifest to populate the activation context.</span></span> <span data-ttu-id="946b0-112">因為資訊清單會描述應用程式對 [並存元件](about-side-by-side-assemblies-.md) 版本的相依性，所以在資訊清單中指定沒有版本的物件會對應至版本命名的物件。</span><span class="sxs-lookup"><span data-stu-id="946b0-112">Because manifests describe an application's dependence on [side-by-side assembly](about-side-by-side-assemblies-.md) versions, objects specified without versions in the manifest are mapped to version-named objects.</span></span> <span data-ttu-id="946b0-113">例如，資訊清單可能會描述 Dll、檔案、視窗類別、COM 伺服器、類型程式庫和介面。</span><span class="sxs-lookup"><span data-stu-id="946b0-113">For example, the manifest may describe DLLs, files, window classes, COM servers, type libraries, and interfaces.</span></span>

<span data-ttu-id="946b0-114">當全域物件是在啟用內容中建立時，系統會透過查閱資訊清單，自動為物件提供特定版本的名稱。</span><span class="sxs-lookup"><span data-stu-id="946b0-114">When a global object is created within the activation context, the system automatically gives the object a version-specific name by consulting the manifest.</span></span> <span data-ttu-id="946b0-115">當應用程式執行並要求命名物件時，它會取得版本命名的物件。</span><span class="sxs-lookup"><span data-stu-id="946b0-115">When the application executes and requests a named object, it gets the version-named object.</span></span> <span data-ttu-id="946b0-116">這可讓程式碼模組的多個版本在系統上同時執行，而不會干擾彼此。</span><span class="sxs-lookup"><span data-stu-id="946b0-116">This enables multiple versions of a code module to run on the system at the same time without interfering with each other.</span></span> <span data-ttu-id="946b0-117">例如， [Windows Shell](/previous-versions/windows/desktop/legacy/bb776778(v=vs.85)) 會使用資訊清單來描述6.0 版 comctl32.dll 的相依性，以及建立視窗類別的版本。</span><span class="sxs-lookup"><span data-stu-id="946b0-117">For example, [Windows Shell](/previous-versions/windows/desktop/legacy/bb776778(v=vs.85)) uses a manifest to describe a dependence on version 6.0 of COMCTL32 and to create versions of window classes.</span></span>

<span data-ttu-id="946b0-118">如果應用程式藉由呼叫 [**CreateWindow**](/windows/win32/api/winuser/nf-winuser-createwindowa)來建立資源，進程會指定該函式的類別名稱。</span><span class="sxs-lookup"><span data-stu-id="946b0-118">If an application creates a resource by calling [**CreateWindow**](/windows/win32/api/winuser/nf-winuser-createwindowa), the process specifies a class name to that function.</span></span> <span data-ttu-id="946b0-119">對 [**GetCurrentActCtx**](/windows/desktop/api/Winbase/nf-winbase-getcurrentactctx) 的呼叫會取得目前的啟用內容，並檢查指定類別名稱是否存在對應。</span><span class="sxs-lookup"><span data-stu-id="946b0-119">The call to [**GetCurrentActCtx**](/windows/desktop/api/Winbase/nf-winbase-getcurrentactctx) gets the current activation context and checks to see if a mapping exists for the given class name.</span></span> <span data-ttu-id="946b0-120">如果有對應存在，則會使用該版本的呼叫進程來解析對應，並提供版本特定的類別名稱。</span><span class="sxs-lookup"><span data-stu-id="946b0-120">If a mapping exists, it will use that version of the calling process to resolve the mapping and provide the version-specific class name.</span></span> <span data-ttu-id="946b0-121">Windows 會建立一個視窗，其中含有與該類別名稱和版本相關聯的視窗程式、樣式和其他屬性。</span><span class="sxs-lookup"><span data-stu-id="946b0-121">Windows creates a window with the window procedure, styles, and other attributes associated with that class name and version.</span></span>

<span data-ttu-id="946b0-122">在大部分情況下，啟用內容是由系統所管理。</span><span class="sxs-lookup"><span data-stu-id="946b0-122">The activation context is managed by the system in most cases.</span></span> <span data-ttu-id="946b0-123">應用程式開發人員和元件提供者通常不需要對堆疊進行呼叫。</span><span class="sxs-lookup"><span data-stu-id="946b0-123">Application developers and assembly providers do not commonly need to make calls to the stack.</span></span> <span data-ttu-id="946b0-124">應用程式可以直接呼叫啟用內容來管理啟用內容。</span><span class="sxs-lookup"><span data-stu-id="946b0-124">Applications can manage an activation context by directly calling the activation context.</span></span> <span data-ttu-id="946b0-125">如需詳細資訊，請參閱 [使用啟用內容 API](using-the-activation-context-api.md)。</span><span class="sxs-lookup"><span data-stu-id="946b0-125">For more information, see [Using the Activation Context API](using-the-activation-context-api.md).</span></span>
