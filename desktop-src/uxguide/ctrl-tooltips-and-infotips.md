# <a name="tooltips-and-infotips"></a>工具提示與提示

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

工具提示是一個小型的快顯視窗，它會標記所指向的未標記控制項，例如未標記的工具列控制項或命令按鈕。

![顯示 [列印] (Ctrl + P) ' 工具提示的 [列印] 按鈕的螢幕擷取畫面。](images/ctrl-tooltips-and-infotips-image1.png)

工具列按鈕的一般工具提示。

因為工具提示已證明如此實用，所以有一個稱為提示的相關控制項存在，它比工具提示更能提供更多的描述性文字。

提示是一個小型的快顯視窗，可簡潔地描述所指向的物件，例如工具列控制項、圖示、圖形、連結、Windows 檔案總管物件、[開始] 功能表專案和工作列按鈕的描述。 提示是一種 [漸進式洩漏控制項](ctrl-progressive-disclosure-controls.md)，不需要在螢幕上一律具有描述性文字。

![具有提示的 [共用] 按鈕的螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image2.png)

一般的提示。

基於本文的目的，工具提示和提示統稱為提示。

秘訣可協助使用者瞭解在使用者介面中未直接描述的未知或不熟悉物件 (UI) 。 當使用者將指標停留在物件上方時，就會自動顯示這些物件，並在使用者按一下控制項或移動滑鼠時，或是當提示時間結束時移除。

**開發人員：** 沒有任何提示控制項;提示會使用工具提示控制項來執行。 不同之處在于使用方式，而不是執行。

> [!Note]  
> 與 [氣球](ctrl-balloons.md)、 [工具列](cmd-toolbars.md)和說明相關的指導方針會顯示在 [不同的文章](winenv-help.md) 中。

 

## <a name="is-this-the-right-control"></a>這是正確的控制項嗎？

若要決定使用時機，請考量下列問題：

-   **是否會根據指標停留顯示資訊？** 如果不是，請使用其他控制項。 只有當使用者互動的結果不會自行顯示提示時，才會顯示提示。 相反地， [氣球](ctrl-balloons.md) 可以在自己的 (上顯示) 的通知，因此它們具有可識別其來源的結尾。
-   **控制項是否有文字標籤？** 如果沒有，請使用工具提示提供標籤。 請注意，大部分的控制項都應該加上標籤，因此沒有工具提示。 具有圖形標籤的工具列控制項和命令按鈕應該有工具提示。
-   **物件是否受益于補充描述或進一步的資訊？** 若是如此，請使用提示。 不過，文字必須是補充的，而不是主要工作的必要項。 如果是必要的文字，請將它直接放入 UI 中，使用者就不必到處尋找這些文字。
-   **補充資訊是「錯誤」、「警告」或「狀態」？** 如果是的話，請使用另一個 UI 專案，例如球形、 [錯誤訊息](mess-error.md)或 [狀態列](ctrl-status-bars.md)。 通知區域圖示提示是一種例外狀況，因為它們可以用來顯示狀態資訊。
-   **使用者是否需要與提示互動？** 如果是的話，請使用另一個控制項，例如「球」。 使用者無法與提示互動，因為移動滑鼠會讓提示消失。
-   **使用者是否需要列印補充資訊？** 如果是的話，請使用另一個控制項，例如靜態批註欄位。 不過，您也可以使用提示來提供此資訊的更直接存取權。

    ![批註批註提示的螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image3.png)

    在此範例中，Microsoft Word 中的靜態批註欄位可讓使用者列印批註。

-   **這是因為使用者可能會發現很討厭或混亂的秘訣？** 如果是，請考慮使用其他解決方案，包括完全不執行任何動作。 如果您在這類內容中使用提示，可讓使用者將其關閉。

適當地使用時，秘訣會改善與使用者的通訊。 **絕對不要使用秘訣來取代良好的設計。** 如果圖形、按鈕或其他物件需要使用者繼續檢查提示以瞭解它，則設計就不正確。 請改為修正設計。

## <a name="design-concepts"></a>設計概念

秘訣是簡化使用者介面的強大方式。 他們會在需要時提供使用者所需的資訊，並對其部分進行極低的工作。 秘訣可協助您更有效率地使用螢幕空間，並減少螢幕雜亂。 不過，設計不良的秘訣可能會很討厭、令人困擾、無用、困擾或的方式。 下列設計概念旨在顯示差異。

### <a name="discoverability"></a>可搜尋性

當使用者將指標停留在某個時間點的物件上時，就會自動顯示提示。 這段時間延遲機制會讓秘訣變得非常方便，但也可減少其發現性。

經過一段時間之後，使用者會瞭解某些標準物件（例如工具列按鈕、圖形按鈕、[開始] 功能表專案和通知區域圖示）有提示，因此您可以取得其可搜尋性。

使用者會花更長的時間來探索非標準位置的秘訣。 沒有視覺提示，例如作用點或指標變更，表示物件具有秘訣。 更糟的是，有些使用者會四處移動滑鼠，尤其是在學習流覽 UI 時。 使用者必須知道某物件的秘訣，是透過過去的經驗或試驗。

您可以使用一致的秘訣來改善可搜尋性，進而促進可預測性。 如果您為某些物件提供提示，您應該針對使用者可能想要補充資訊的所有類似物件提供提示。 有時候這樣做可能是一項挑戰，因為您也必須確定秘訣有説明且不明顯。

如果提供可探索且一致的實用秘訣證明是問題，請考慮使用替代的設計，例如自我理解的控制項標籤或就地補充文字。

### <a name="appropriate-information"></a>適當資訊

適用于提示的資訊具有下列特性：

-   **簡潔。** 提示使用的快顯視窗最適合簡短的句子和句子片段，以及格式化的文字。 大型、未格式化的文字區塊很難閱讀。
-   **有用。** 提示文字必須有資訊。 它應該很明顯，或只是重複畫面上的內容。
-   **補充。** 因為不一定會顯示提示文字，所以它應該是使用者不必閱讀的補充資訊。 重要資訊應使用簡單易懂的控制項標籤或就地補充文字來傳達。
-   **靜態。** 使用者不會預期從某個實例變更為下一個實例的秘訣，因此不太可能注意到動態內容中的變更，例如狀態資訊。 通知區域圖示提示是值得注意的例外狀況：使用者更有可能發現提示資訊中的變更，因為這些圖示主要會傳達狀態。

### <a name="appropriate-timeouts"></a>適當的超時

適當的提示自動顯示和移除，對於維護其 UI 環境控制的使用者而言非常重要。 秘訣有三個 timeout 值：

-   **初始。** 指標必須保持靜止以顯示提示的時間。 預設時間為0.5 秒。
-   **Reshow.** 指標在指標移至另一個目標時必須保持靜止的時間。 預設時間為0.1 秒。
-   **去除。** 提示自動移除的時間。 預設時間為5秒。

使用太短的初始和 reshow 值會導致令人討厭的干擾式體驗，因為它們通常會不慎顯示出來，但很長的結果會導致提示感覺沒有回應或無法探索。 預設的移除時間適用于簡短提示文字，如工具提示中所使用。 提示具有較長的文字，因此需要較長的顯示時間。

### <a name="appropriate-placement"></a>適當放置

秘訣應放置在要停留的物件附近，通常是在指標的 tail 或 head （如果可能的話）。 不過，它們絕對不應該透過遮蔽感興趣的物件，來干擾使用者執行的工作。 若要避免這個問題，您可能需要將秘訣移離指標，但與物件相鄰。 只要物件與其秘訣之間的關聯性很清楚，這就不成問題。 請確定使用者不會移動指標，只是要讓您的程式提示消失。

### <a name="accessibility"></a>協助工具選項

秘訣對存取範圍有不尋常的影響。 雖然通常會藉由將指標暫留在物件上來觸發，但是 [螢幕讀取器](inter-accessibility.md) 會針對具有鍵盤存取權的控制項來處理提示。 當您針對簡潔、有用、靜態、補充資訊適當地使用時，秘訣可以改善整體存取範圍。 事實上，替代文字提示模式是讓圖形變成可供存取的慣用方式。 不過，當您不當使用時，會造成重要或動態資訊難以取得，進而損害存取範圍。

如果控制項需要無法存取鍵盤的提示，請提供一個以上的方法來存取控制項。

![具有工具提示的 [列印] 按鈕的螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image4.png)

在此範例中，使用者可以使用工具列按鈕來列印 (這無法透過鍵盤存取) 或 [列印命令] 鍵盤快速鍵。

**如果您只做一件事 .。。**

設計可搜尋的秘訣，在適當的時間于適當的位置，顯示簡潔、實用、靜態的補充資訊。

## <a name="guidelines"></a>指導方針

### <a name="timeouts"></a>逾時

-   **使用預設的初始和 reshow 超時。例外：**
    -   不是多餘且顯示在其相關物件側邊的縮圖，可以立即顯示 (不會有任何延遲) 。 不過，使用重複縮圖的預設初始超時 (例如小型繪圖物件的大型縮圖提示) 或涵蓋其相關物件的縮圖。
-   **針對工具提示，請使用預設的五秒提示移除超時。**
-   若 **為提示，請關閉 tip 移除超時。開發人員：** 因為您無法在技術上關閉移除超時，請將它設定為最大的值。
-   針對協助工具，如果您需要將超時值設定為最大值以外的值，請讓它們成為 SPI \_ GETMOUSEHOVERTIME 和 spi \_ GETMESSAGEDURATION 系統參數的倍數，而不是使用固定的時間。 這麼做會將超時調整為使用者的速度。

### <a name="placement"></a>放置

-   **避免涵蓋使用者即將查看或互動的物件。** 一律將提示放置在物件的側邊，即使這需要在指標與提示之間區隔。 只要物件與其秘訣之間的關聯性很清楚，某些分隔就不會有問題。

    -   **例外狀況：** 清單和樹狀目錄中使用的完整名稱工具提示。

    **不正確：**

    ![[提示] 遮蔽搜尋方塊的螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image21.png)

    **正確：**

    ![放在搜尋方塊下方的提示螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image22.png)

    在正確的範例中，會將提示字元放在搜尋方塊以外的位置，即使這需要它和插入號之間的空格也一樣。

    **不正確：**

    ![顯示修訂文字的提示螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image23.png)

    **正確：**

    ![放在修訂文字上方的內容提示螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image24.png)

    在正確的範例中，基礎文字遠比提示更實用，因此資訊提示的運作方式會很好。

    **不正確：**

### <a name="infotips"></a>提示

-   **針對非標準位置中的提示，優先于效益以改善可搜尋性。** 提供提示給使用者可能需要補充資訊的所有物件，即使有一些提示可能很明顯。 這麼做可避免讓使用者等待永遠沒有的提示。
    -   **例外狀況：** 如果只有少數物件有實用的提示，請不要使用提示。 相反地，請改用一目了然的控制項標籤或就地補充文字。
-   使用具有結束標點符號的完整句子。
    -   **例外狀況：** 通知區域 [圖示提示](winenv-notification.md) 請勿使用結束標點符號。
-   使用句型大寫。
-   使用現在時，而不是未來。
-   使用平行語法結構。 平行處理原則需要具有相同函式的單字和片語具有相同的格式。
    -   **例外狀況：** 針對完整名稱提示模式，提示文字完全符合基礎控制項的片語、大小寫和標點符號。
-   **避免大型提示。** 大型提示很難讀取，而且很難在不幹擾基礎物件的情況下定位。
-   **將提示格式化，讓內容更容易閱讀和掃描。** 大型的未格式化文字區塊可能難以閱讀。

    **不正確：**

    ![長、非結構化、執行工具提示的螢幕擷取畫面 ](images/ctrl-tooltips-and-infotips-image34.png)

    **正確：**

    ![相同工具提示的螢幕擷取畫面，其中每個專案都有一行 ](images/ctrl-tooltips-and-infotips-image35.png)

    在正確的範例中，已格式化的文字更容易讀取和掃描。

-   第一次在資訊提示中使用時，請將縮寫名稱加上括弧中的縮寫。 範例：「動態主機設定通訊協定 (DHCP) 」。
