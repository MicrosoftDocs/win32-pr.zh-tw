# <a name="confirmations"></a>確認

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

確認是一個強制回應對話方塊，會詢問使用者是否要繼續執行動作。

![顯示 [您要儲存變更嗎？] 的 [記事本] 的螢幕擷取畫面 對話方塊中指定選項。](images/mess-confirm-image1.png)

一般確認。

確認具有這些基本特性：

-   它們會顯示為使用者所起始之動作的直接結果。
-   他們會確認使用者是否想要繼續執行動作。
-   它們包含一個簡單的問題，以及兩個或多個回應。

當動作需要使用者進行相關且不同的選擇，但稍後無法進行時，確認最有用。 這項選擇通常牽涉到使用者不明顯的風險元素，但不一定要確認風險。 需要這些元素，以證明回應強制回應對話方塊的中斷。

相反地， [警告訊息](mess-warn.md) 會出現可能會在未來發生問題的狀況。 其基本特性是它們牽涉到風險：

-   它們牽涉到可能遺失下列一或多項：
    -   有價值的資產，例如資料遺失或財務損失。
    -   系統存取或完整性。
    -   隱私權或對機密資訊的控制。
    -   使用者的時間 (大量的時間，例如30秒或更多的) 。
-   它們有非預期或非預期的結果。
-   因為錯誤無法輕易地修正，甚至可能無法復原，因此現在它們需要正確的處理。

如果確認牽涉到風險，也可以將其視為警告。 因此，也適用警告訊息指導方針。

**注意：** 與 [對話方塊](win-dialog-box.md)、[錯誤訊息](mess-error.md) [、配置和](vis-layout.md)[警告訊息](mess-warn.md)相關的指導方針會顯示在不同的文章中。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用時機，請考量下列問題：

-   **使用者是否有問題繼續進行有兩個或多個回應的動作？** 如果沒有，則不會確認訊息。
-   **UI 是否呈現發生的錯誤或問題？** 若是如此，請改用 [錯誤訊息](mess-error.md) 。
-   **繼續進行動作需要使用者選擇沒有適當的預設值？** 如果是的話，則可能會有適當的確認。
-   **是否有替代的設計可避免確認的需求？** 確認的需要有時表示設計瑕疵。 通常有更好的設計替代方案不需要確認。
-   **使用者是否要執行具風險的動作？** 如果是這樣，如果動作有重大的後果或無法輕易復原，則會有適當的確認。
-   **使用者是否要放棄工作？** 若是如此，請不要確認。 假設使用者瞭解未完成工作的後果。
-   **動作是否會造成使用者可能不知道的後果？** 如果是的話，則可能會有適當的確認。
-   **由於目前的內容，使用者是否有可能執行錯誤的動作？** 如果是的話，則可能會有適當的確認。
-   **使用者是否經常執行動作？** 如果是，請考慮使用替代的設計。 頻繁的確認是討厭的，且有很少的價值，因為使用者會學習回應而不需要思考。
-   **動作是否有安全性含意？** 若是如此，即使先前的測試另有指示，也可能需要確認。

## <a name="design-concepts"></a>設計概念

### <a name="unnecessary-confirmations-are-annoying"></a>不必要的確認很討厭

第一次建立的 Windows 確認看起來像這樣：

![[確定嗎？] 的螢幕擷取畫面 確認 ](images/mess-confirm-image2.png)

原始討厭的確認。

這是非常糟糕的起點。 如果您想要讓使用者討厭您的程式，只要在整個小量確認。 若要瞭解原因，請考慮使用者的觀點。 使用者只是要求由確認定義來執行動作，如此一來，使用者就會想要繼續進行。

不只是不必要的確認，也不是保護使用者免于犯錯的效用。 當程式有不必要的確認時，使用者很快就會發現，而其自然回應是盡可能快速地關閉（通常不會讀取）。 因此，這類確認只需要為這些工作新增額外的步驟。

請不要使用確認，因為使用者可能會犯錯誤。 **相反地，確認是在用來確認具有顯著或非預期結果的動作時最有效。** 良好確認永遠不會陳述明顯的;他們應該傳達使用者需要注意的事項，以找出不能繼續的好理由。 而且只有在動作真正需要時，才會使用它們，例如只有在有變更值得儲存的時候，才要求使用者儲存變更。 這麼做只會要求使用者在真正的保證時，才需要注意。

針對其他類型的確認，通常會有比強制使用者回答問題更好的設計替代方案。

### <a name="consider-the-design-alternatives"></a>考慮設計替代方案

以下是一些不再需要進行例行確認的設計替代方案：

-   **防止錯誤。** 設計工作，讓嚴重錯誤難以意外地發生。 例如，實際將破壞性命令與其他命令分開，並且需要多個動作才能完成。
-   **提供復原。** 提供還原動作的能力。 例如，在 Microsoft Windows 中刪除檔案時，通常不需要確認，因為已刪除的檔案可以從資源回收筒復原。 請注意，如果動作很容易執行，則讓使用者重做動作可能就已足夠。
-   **提供意見反應。** 明顯地產生不想要的結果。 如果使用者在發生錯誤時不知道，單獨提供復原並不足夠。 例如，直接操作 (的效果（例如拖放作業) ）應該一律很明顯。
-   **假設可能的結果，但可讓您輕鬆地進行變更。** 如果您不確定使用者需要什麼，但是有很可能、安全且安全的選擇，請採用該選擇，讓它清楚瞭解發生什麼事，並使用操作功能表輕鬆地進行變更。 例如，Microsoft Word 假設使用者想要正確拼寫文字。 如果它辨識拼錯的字，而且它知道可能正確的拼寫，Word 會自動進行更正，但允許使用者還原。
-   **完全消除選擇。** 如果選擇不重要，使用者就不會在意。 更妥善地 [簡化](/previous-versions//dn742474(v=vs.85)) 您的程式並排除選擇。

### <a name="make-confirmations-require-thought"></a>確認需要考慮

為了確認有價值，使用者必須瞭解不會繼續的原因。 有時候原因很明顯，因為當使用者以尚未儲存的變更來關閉檔時：

![顯示「您要儲存變更嗎？」繪圖的螢幕擷取畫面 回應。](images/mess-confirm-image3.png)

在此範例中，確認的原因很明顯。

在其他情況下，原因可能不明顯。

選擇對話方塊的 [認可按鈕標籤] 時， [一般的指導方針](win-dialog-box.md) 是選擇對主要指令有特定回應的標籤。 這會導致有效的決策，因為使用者必須讀取最少量的文字才能繼續進行。 不過，這種效率目標可這些確認。 請考慮此範例：

**不正確：**

![使用 [卸載] 按鈕確認的螢幕擷取畫面 ](images/mess-confirm-image4.png)

在此範例中，需要考慮正確的回應。

如果您在使用者提供卸載命令之後立即提出這項確認，使用者的回應可能會是「當然我要卸載！」 使用者會按一下 [卸載]，而不需要再做一次考慮。

針對確認，我們不希望使用者做出惟恐、情緒決策。 為了鼓勵使用者思考其回應，我們需要提供小規模的決策加速。 在可行的情況下，最好是藉由仔細地措辭認可按鈕來完成這項作業。 例如，我們可以使用其他語言來指出不會繼續的原因。

**較佳：**

![[仍要卸載] 按鈕的螢幕擷取畫面 ](images/mess-confirm-image5.png)

在此範例中，「仍」會新增至認可按鈕標籤，以指出確認提供的原因不會繼續。

如果該方法不實用，我們可以使用 [是]/[否] 認可按鈕。

**還有更好的：**

![確認具有 [是]/[否] 按鈕的螢幕擷取畫面 ](images/mess-confirm-image6.png)

在此範例中，使用 [是]/[否] 認可按鈕會強制使用者至少讀取主要指令。

### <a name="provide-all-the-information"></a>提供所有資訊

如果您要詢問問題，您必須為使用者提供足夠的資訊，讓使用者以智慧方式回答該問題。 請考慮 Windows XP 中的 [確認檔案取代] 對話方塊：

![[確認檔案取代] 對話方塊的螢幕擷取畫面 ](images/mess-confirm-image7.png)

[Windows XP 確認檔案取代] 對話方塊。

這種確認是否提供使用者可能需要的所有資訊來回答問題？ 在您回答之前，請考慮最常見的使用者案例：

-   複製 (或移動) 另一個檔案，取代現有的檔案。
-   保留現有的檔案，而不復制或移動其他檔案。
-   保留或複製較新的檔案 (主要案例) 。
-   您可以保留現有的檔案，或複製另一個檔案，視像是檔案內容和大小的準則而定。
-   保留現有的檔案，並使用不同的名稱複製其他檔案。
-   如果發生錯誤或非預期的情況，請取消作業。

使用者只要按一下 [是] 和 [案例 2]，就可以按一下 [否] 來達成案例1。 他們可以藉由比較檔案日期和按一下適當的按鈕來達成案例3，但請注意，判斷較新的檔案需要多少想法，然後判斷出適合最常見案例的適當按鈕。

案例4、5和6也很難。 檔案大小會舍入，因此，舉例來說，您無法判斷這些檔案的大小是否相同，或即使它們是相同的檔案也是一樣。 圖示適用于用來開啟檔案的應用程式，因此使用者必須開啟檔案，才能檢查並比較其內容。 有了檔案內容的縮圖，對回答問題而言會有更大的説明。

從 Windows Vista 的複製檔案確認可提供詳細資訊，並新增保留這兩個檔案的選項，以更好的方式處理這些案例：

![[複製檔案] 對話方塊的螢幕擷取畫面 ](images/mess-confirm-image8.png)

Windows Vista 複製檔案確認。

### <a name="provide-specific-useful-information"></a>提供特定的實用資訊

如果您要詢問問題，請確定使用者瞭解問題以及替代回應的含意。 請考慮此 Windows Internet Explorer 安全性確認：

![確認有模糊問題的螢幕擷取畫面 ](images/mess-confirm-image9.png)

模糊的安全性確認。

此確認會詢問使用者無法智慧地回答的問題。 使用者已要求 Windows Internet Explorer 顯示頁面，而這則訊息會透過文字的用語以隱含的方式提供建議，並藉由反白顯示 [否] 作為預設選項。

頁面所帶來的特定安全性問題並沒有充分的說明，所以繼續進行的風險並不明顯。 確認中的哪些資訊會導致使用者按 [否]？ 由於訊息的 vagueness，確認不太可能會讓使用者無法繼續，但會讓他們感覺不到。

若要確認這是很有用的功能，必須提供更詳細的資訊，這些資訊可能會導致使用者決定不繼續。 一般來說，對於確認中的每個回應，請考慮需要它的案例，並確定有足夠的資訊可供使用者選擇。 提供選項，而非困境。

### <a name="how-to-determine-if-a-confirmation-is-necessary"></a>如何判斷是否需要確認

思考案例，以及選擇每個回應的可能性，都建議以系統化的方式判斷是否需要確認。 如果使用者可能會選取所有回應，則必須確認，而且很有用。 但是，如果只有一個回應可能 (表示) 的時間是98%，則確實不需要確認，而且應該移除。 請注意，與安全性、法律和安全性問題相關的確認是可能發生的例外狀況。

![[您要變更設定嗎？] 的螢幕擷取畫面 ](images/mess-confirm-image10.png)

這是必要確認嗎？ 使用者是否會選取 [否]？ 有可能是很未必的。 應移除此確認。

**如果您只進行三件事 .。。**

1. 確定確實需要您的確認。 應該有合理且明確的原因不會繼續進行，有時使用者可能不會這麼做。

2. 如果確認的原因無法立即察覺，請選擇鼓勵使用者思考其回應的 [認可] 按鈕。 一般來說，這是藉由將確認視為「是」或「否」的問題，並提供完全易懂或「是/否」答案來完成。

3. 請考慮所有案例，並提供以智慧方式回答問題所需的資訊。

## <a name="guidelines"></a>指導方針

### <a name="general"></a>一般

-   **只有在有重大變更時，才使用 [儲存變更] 確認。** 請勿確認使用者未直接進行的變更，例如自動格式化檔。

**不正確：**

![顯示 Microsoft Office Outlook [您要儲存變更嗎？] 的螢幕擷取畫面 對話方塊中指定選項。](images/mess-confirm-image23.png)

當使用者未變更空白電子郵件或檔時，此範例會不正確。

### <a name="icons"></a>圖示

-   確認不使用標題列圖示。
-   **確認的內容區域圖示是根據其設計模式：**



    | 模式                                                | 圖示                                                                                                                                             |
    |--------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|
    | 例行確認 <br/>                | 無圖示。 <br/>                                                                                                                         |
    | 具風險的動作確認 <br/>           | 警告圖示。 <br/>                                                                                                                    |

## <a name="text"></a>Text

### <a name="general"></a>一般

-   **移除多餘的文字。** 在標題、主要指示、補充指示、內容區域、命令連結和認可按鈕中尋找重複的文字。 一般而言，請在指示和互動式控制項中保留完整文字，然後從其他位置移除任何多餘的。
-   **請勿在文字中使用「警告」或「警告」。** 如果使用者需要特別小心，請改為使用警告圖示來表示。

**不正確：**

![磁片區格式化確認的螢幕擷取畫面 ](images/mess-confirm-image36.png)

在此範例中，「警告」一詞是不必要的。

### <a name="titles"></a>標題

-   **使用標題來識別確認來自的命令或功能。異常：**
    -   如果有許多不同的命令顯示確認，請考慮改為使用程式名稱。
    -   如果該標題是多餘的，或與主要指示混淆，請改用程式名稱。

但是，如果確認來自長時間執行的工作，且在工作開始之後可能會顯示正常，請一律使用命令或功能清楚地識別內容。

-   **請勿使用標題來說明在主要指令用途的對話方塊中要執行** 的動作。
-   如果新增了明確的文字，請使用 [確認] 來啟動標題。
-   **針對具風險的動作確認，您可以新增涉及其他強調的物件名稱。**

![[格式化 f 磁片磁碟機] 對話方塊標題的螢幕擷取畫面 ](images/mess-confirm-image13.png)

在此範例中，標題中包含要格式化的磁片磁碟機。

-   使用 [標題樣式的大小寫](glossary.md)，但不含結束標點符號。

-   **請務必只使用單一的完整句子。** 將主要指示帶到基本資訊。 如果您必須進一步說明，請使用補充指示。
-   **如果有相關的物件，請特別提供其完整名稱。**
-   **使用正片語。** 對使用者來說，可以更輕鬆地理解正面片語。

**不正確：**

您要停用檔案和印表機共用嗎？

不過，片語必須符合相關聯的命令，即使命令有負面片語;例如，使用 disable 來確認 Disable 命令。

    | 片語                                                            | 內涵                                                            |

    | 您要 \[ 選取結果 \] 嗎？ <br/>          | 需要澄清。 <br/>                           |
    | \[要執行動作 \] 嗎？ <br/>                          | 無含意。 <br/>                                 |
