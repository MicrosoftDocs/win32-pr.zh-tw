# <a name="control-panels"></a>控制台

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

使用控制台專案來協助使用者設定系統層級的功能，並執行相關工作。 具有使用者介面的程式應該改為從其 UI 直接設定。

有了 Microsoft Windows 的主控台，使用者就可以設定系統層級的功能並執行相關工作。 系統層級功能設定的範例包括硬體和軟體設定、安全性、系統維護以及使用者帳戶管理。

主控台一詞指的是整個 Windows 主控台功能。 個別的控制台稱為控制台專案。 當您直接從 [控制台] 首頁或類別頁面存取 [控制台] 專案時，會將其視為最上層。

![控制台語音分類的螢幕擷取畫面 ](images/winenv-ctrl-panels-image1.png)

一般的 [控制台] 專案。

[控制台] 首頁是所有控制台專案的主要進入點。 它會依類別目錄列出專案，以及最常見的工作。 當使用者按一下 [開始] 功能表中的主控台時，就會顯示此畫面。

[控制台類別] 頁面會列出單一類別內的專案，以及最常見的工作。 當使用者按一下首頁上的類別名稱時，就會顯示此頁面。

控制台專案是使用工作 [流程](glossary.md) 或屬性工作表來執行。 針對 Windows Vista 和更新版本，工作流程是 (UI) 慣用的使用者介面。

**開發人員：** 若要瞭解如何建立 [控制台] 專案，請參閱 [主控台專案](/previous-versions//bb776838(v=vs.85))。

**注意：** 與 [屬性工作表](win-property-win.md) 相關的指導方針會在個別的文章中顯示。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用時機，請考量下列問題：

-   **設定系統層級功能的目的是什麼？** 如果沒有，請使用另一個整合點。 您可以使用 [選項] 對話方塊，直接從 UI 設定您的應用程式功能，而不是使用主控台。 針對未用於安裝、設定或相關工作的公用程式 (例如疑難排解) ，請使用 [開始] 功能表作為整合點。
-   **系統層級功能是否有它自己的 UI？** 如果是這樣，則使用者應該前往該 UI 進行變更。 例如，系統備份公用程式應從其程式選項中設定，而不是從主控台。
-   **使用者是否需要經常變更設定？** 如果有 (說，每週) 幾次，請考慮替代的解決方案，甚至是使用主控台。 例如，您可以直接從通知區域中的圖示來設定 [Windows 主要磁碟區] 設定。 某些設定可以自動設定。 例如，在 Windows 檔案總管中，應用程式屬性的 [相容性] 索引標籤可讓應用程式在256色彩模式中執行，而不需要使用者手動變更影片模式。
-   **目標使用者是 IT 專業人員嗎？** 若是如此，請改用 [Microsoft Management Console 的 (mmc) ](/previous-versions/windows/desktop/mmc/microsoft-management-console-start-page) 嵌入式管理單元，這是專為系統管理工作而設計的。 在某些情況下，最好的解決方法是讓一般使用者都能使用 [控制台] 專案，並為 IT 專業人員提供 MMC 嵌入式管理單元。

    ![[電腦管理] 視窗的螢幕擷取畫面 ](images/winenv-ctrl-panels-image2.png)

    在此範例中，[本機使用者和群組] MMC 嵌入式管理單元提供以 IT 專業人員為目標的使用者管理。 其他使用者更有可能使用主控台中的 [使用者帳戶] 專案。

-   **這項功能是否只有在初始系統組態期間才會使用 OEM 功能？** 若是如此，請使用 Windows 歡迎中心作為整合點。

需要控制台專案，因為許多系統層級功能都沒有更明顯或直接的整合點。 但主控台不應視為所有設定的「單一位置」。 **具有使用者介面的程式應該直接從其 UI 進行設定，而不是使用控制台專案。**

**不正確：**

![[控制台網際網路選項] 專案的螢幕擷取畫面 ](images/winenv-ctrl-panels-image3.png)

在此範例中，Windows Internet Explorer 不應以主控台表示，因為它自己的 UI 是更好的整合點。

### <a name="create-a-new-control-panel-item-or-extend-an-existing-one"></a>建立新的控制台專案或擴充現有的控制台專案？

若要決定使用時機，請考量下列問題：

-   **這項功能可以用來表示可以插入現有、可擴充的控制台專案的工作嗎？** 下列控制台專案可延伸：藍牙裝置、顯示器、網際網路、鍵盤、滑鼠、網路、電源、系統、無線 (紅外線) 。
-   **[屬性] 和 [工作] 取代了現有可擴充控制台專案的功能嗎？** 若是如此，您應該擴充現有的 [控制台] 專案，因為這樣會產生更簡單的使用者體驗。 如果沒有，請建立新的控制台專案。

## <a name="design-concepts"></a>設計概念

**主控台的概念是以真實世界比喻為基礎。** 真實世界的控制台是控制項的集合， (旋鈕、切換、量測計，以及用來監視和控制裝置) 的顯示器。 這類控制台的使用者通常需要訓練才能瞭解其使用方式。

不同于其真實世界的對應專案， **Windows 的控制台設計已針對第一次使用者進行優化。** 使用者不會經常執行大部分的控制台工作，因此通常不會記得如何執行這些工作，而且每次都必須破解它們。

設計實用且容易使用的控制台專案：

-   請確定屬性是必要的。
-   以使用者目標為依據，而不是以技術呈現屬性。
-   將屬性顯示在正確層級。
-   特定工作的設計頁面。
-   標準使用者和受保護系統管理員的設計頁面。

當您設計和評估要包含在主控台中的專案時，請判斷使用者執行的一般工作，並確定有明確的路徑可以執行這些工作。 使用者通常會使用 [控制台] 專案來執行下列類型的工作：

-   初始設定
-   大部分設定 (不頻繁的變更) 
-   頻繁的變更 (有幾個重要的設定) 
-   將設定回復為初始或先前狀態
-   疑難排解

**如果您只做一件事 .。。**

設計特定工作的控制台頁面，並以使用者目標而非技術呈現這些頁面。

## <a name="usage-patterns"></a>使用模式

您可以使用工作流程或屬性工作表，針對 [控制台] 專案。 以下是其使用模式：

### <a name="task-flow-patterns"></a>工作流程模式

工作流程專案使用中樞頁面來呈現高階選項，以及用來執行實際設定的輪輻頁面。

**中樞頁面**

-   以工作為基礎的中樞頁面。 這些中樞頁面會顯示最常使用的工作。 它們最適合用於一些常用或重要的工作，使用者需要更多指引和說明。 中樞頁面沒有認可按鈕。 混合式以工作為基礎的中樞頁面也有直接的一些屬性或命令。 當使用者最有可能使用主控台來存取這些屬性和命令時，強烈建議使用混合式中樞頁面。
-   以物件為基礎的中樞頁面。 這些中樞頁面會使用清單視圖控制項來顯示可用的物件。 當可能有數個物件時，最好使用它們。 中樞頁面沒有認可按鈕。

**輪輻頁面**

-   工作頁面。 這些輪輻頁面會在工作中顯示工作或步驟，其中包含以工作為基礎的特定主要指令。 它們最適合用於受益于其他指引和說明的工作。
-   表單頁面。 這些輪輻頁面會根據一般主要指示來呈現相關屬性和工作的集合。 它們最適合用於具有許多屬性的功能，並受益于直接的單一頁面呈現，例如 advanced properties。

### <a name="property-sheet-patterns"></a>屬性工作表模式

-   屬性工作表最適合用於具有許多設定（以 advanced users 為目標）的舊版專案。 使用表單頁面模式的工作流程，新的專案可以達到相同的效果。

## <a name="guidelines"></a>指導方針

### <a name="property-sheet-control-panel-items"></a>屬性工作表的控制台專案

-   **請勿將屬性工作表用於新的控制台專案。** 相反地，請使用工作流程來建立順暢的體驗，並充分利用 [控制台] 首頁的分類和搜尋功能。

### <a name="task-flow-control-panel-items"></a>[工作流程] 控制台專案

**一般**

-   **將最重要的內容和控制項保持可見，而不需要滾動。** 除非使用者有原因，否則使用者不會看到頁面內容。 您可以將認可按鈕放在 [命令區域](glossary.md) 中，而不是在內容區域中，讓它們保持可見。 請勿只分割頁面以避免滾動。
    -   **您可以垂直捲動長頁，只要可以** 看到最重要的控制項而不需要滾動。
    -   **請勿使用水準滾動。** 請改為重新設計頁面內容，並使用垂直捲動。 頁面只有在變得很窄時，才會有水準捲軸。
-   **在頁面之間流覽：**
    -   使用工作 [連結](glossary.md) 來啟動工作。
    -   使用 [工作連結] 或 [下一步] 按鈕，流覽至多步驟工作中的下一個頁面。
    -   使用 [認可] 按鈕來完成工作。
    -   使用功能表列中的 [上一頁] 按鈕，以返回先前看到的頁面。 請勿將 [上一頁] 按鈕新增至命令區域。
    -   使用網址列直接返回控制台的首頁。
    -   使用工作窗格中的 [另請參閱連結]，流覽至其他控制台專案中的頁面。 否則，流覽應該會保留在單一控制台專案內。
-   **只將 [控制台] 首頁放在網址列中。** 按一下該連結會返回 [控制台] 首頁，放棄任何進行中的工作而不進行 [確認](https://msdn.microsoft.com/library/windows/desktop/aa511273.aspx)。
-   **不要在 [控制台] 頁面上放置 [關閉命令] 按鈕。** 使用者可以使用標題列上的 [關閉] 按鈕，關閉 [控制台] 視窗。

**工作連結和按鈕**

-   **當頁面有一組較小的固定選項時，請使用工作連結，而不是選項按鈕和 [下一步] 按鈕的組合。** 如此一來，使用者只要按一下就可以選取回應。
-   您可以在下列位置中放入工作連結和按鈕， (可探索) 的順序：
    -   只有) 的輪輻頁面上的命令按鈕 ([命令區](glossary.md) 。
    -   [內容區域](glossary.md)：
        -   命令按鈕
        -   工作連結
        -   其他連結
    -   工作 [窗格](glossary.md) 中的連結只 (中樞頁面) 。
-   **將工作連結和按鈕的位置作為重要性，以及可搜尋性的需求。**
    -   **只將認可按鈕放在命令區域中。**
    -   **將重要的工作放在內容區域中。** 命令按鈕通常會繪製最多注意，因此請為使用者必須看到的命令保留它們。 工作連結也會強調，但小於命令按鈕。
    -   **保留工作窗格，以及次要 (不重要) 工作的純連結。** 工作窗格是工作頁面中最不容易探索的區域，而純連結不像命令按鈕和工作連結一樣可見。
-   針對內容區域中顯示的工作連結：
    -   **如果有7個以上的連結，請將連結分組為類別。** 為每個群組提供標題。
    -   **針對少於7個的連結，請在沒有標題的單一群組中顯示連結。**
-   **以邏輯順序呈現工作連結和按鈕。** 垂直列出工作連結，以水準方式列出命令按鈕。
-   在類別中， **將命令分成相關的群組。** 先放置最常用的工作群組，然後在每個群組內，先放置最常使用的工作。 **產生的順序大致上應該遵循使用的可能性，但也有邏輯流程。**
    -   **例外狀況：** 導致執行所有作業的工作連結應先放置。
-   如果有許多工具的連結，請使用24x24 圖元圖示和兩行文字 **，讓最重要的工作成為更顯著的外觀**。 針對較不重要的工作，請使用16x16 圖元圖示或無圖示，以及單行連結文字。

    ![具有大型和小圖示的專案螢幕擷取畫面 ](images/winenv-ctrl-panels-image4.png)

    在此範例中，重要的命令會獲得更顯著的外觀。

-   **在經常使用的命令和破壞性命令之間有清楚的實體分隔。** 否則，使用者可能會不小心按一下破壞性命令。 您可能需要重新排列命令的順序，以將破壞性命令放在一起。
-   **提供在頁面上直接復原命令的機制。** 使用者不需要在其他地方流覽以復原錯誤。
-   **針對工作連結，請使用所有預設工作連結圖示或所有自訂圖示。** 請勿混用。 請考慮只在下列情況使用自訂圖示：
    -   它們可協助使用者熟練工作。
    -   它們符合 [Aero 圖示標準](vis-icons.md)。
    -   它們具有不顯眼的外觀。

**對話方塊**

使用工作流程時，您通常會想要讓工作在單一視窗內從頁面流向頁面，但下列情況除外。 在下列情況下，請使用對話方塊：

-   以提示使用者輸入系統管理員的使用者名稱和密碼。 基於此目的，請一律使用 [認證管理員] 對話方塊。  ([應為強制](glossary.md)回應。 ) 
-   使用工作對話方塊或訊息方塊確認就地命令。  (應為強制回應。 ) 
-   取得就地命令的輸入，例如新的、新增、另存新檔、重新命名和列印命令。

    ![[刪除網路位置] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image5.png)

    在此範例中，Delete 命令會在對話方塊中執行，而不是在不同的頁面上執行。

-   執行次要的獨立工作。 使用非 [模式](glossary.md)的次要視窗可讓您獨立執行這類工作，並在主要工作流程之外執行。

### <a name="hub-pages"></a>中樞頁面

**一般**

-   在下列情況中使用以工作為基礎的中樞頁面：
    -   **有少量的常用或重要工作。**
    -   設定 **包含一或兩個物件** (範例：監視器、鍵盤、滑鼠、遊戲控制器) 。
    -   設定會 **套用整個系統的** (範例：日期和時間、安全性、電源選項) 。
-   使用以物件為基礎的中樞頁面的時機：
    -   設定 **可能涉及數個物件** (範例：使用者帳戶、網路連線、印表機) 。
    -   設定 **只會套用至選取的物件**。
-   如果 [控制台] 專案具有包含所有相關工作和屬性的 **單一頁面，請勿使用 [中樞] 頁面**。

**物件清單**

-   **依邏輯順序列出專案。** 依字母順序排序命名物件、依數位順序排序的數位和日期（以時間順序排列）。
-   如果是以物件為基礎的中樞，請 **在工作窗格中提供 [物件檢視] 命令（如果變更視圖的能力對工作很重要**）。 如果有許多物件，且預設的簡報在所有案例中都無法正常運作，則變更視圖的能力很重要。 即使沒有明確的命令透過清單視圖操作功能表，使用者也可以變更清單視圖，但較不容易找到。

如需有關呈現物件清單的詳細指導方針，請參閱 [清單視圖](ctrl-list-views.md)。

**互動**

-   **不要在中樞頁面上放置認可按鈕。** 中樞頁面基本上是啟動點。 使用者永遠不會「認可」中樞頁面，而不會使用它們來進行。 中樞頁面上的 [認可] 按鈕讓任何從中樞啟動的工作變得令人困惑 (使用者會想要知道這些工作是否需要) 認可。
    -   **例外狀況：** 如果變更設定需要提高 [許可權](glossary.md)，請提供具有 [安全性保護盾圖示](winenv-uac.md)的 [套用] 按鈕。 一旦套用變更之後，請停用 [認可] 按鈕。
-   **請考慮直接在中樞頁面上放置最有用的屬性。** 當使用者最有可能使用主控台來存取這些屬性時，強烈建議使用這類混合式中樞頁面。

    ![電源選項中樞頁面的螢幕擷取畫面 ](images/winenv-ctrl-panels-image6.png)

    在此範例中，電源選項控制台專案在中樞頁面上直接具有最有用的設定。

-   **針對混合式中樞頁面上的任何設定使用立即認可模型，以便立即進行變更。** 同樣地，使用者永遠不會認可中樞頁面。 如果設定需要 [認可] 按鈕，請不要將它放在中樞頁面上。
-   **請考慮直接在中樞頁面上放置簡單的「單一步驟」命令，而不是使用導覽連結。**
-   **確認無法輕鬆復原其效果的就地命令。** 使用工作 [對話方塊](win-dialog-box.md) 或 [訊息方塊](glossary.md)。

    ![[確認刪除] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image7.png)

    在此範例中，會使用對話方塊確認刪除命令。

-   **如果是以工作為基礎的中樞頁面，請使用工作連結和圖示來識別每項工作。** 您也可以為每個連結提供選擇性的描述。 不過，請試著讓工作連結一目了然，並只針對真正需要的連結提供選擇性描述。

    ![[電腦效能中樞] 頁面的螢幕擷取畫面 ](images/winenv-ctrl-panels-image8.png)

    在此範例中，每個工作都有一個工作連結和一個圖示。

-   **如果是以物件為基礎的中樞頁面，請按一下 [選取物件]，然後按兩下 [選取物件] 並導覽至其預設頁面。** 預設頁面通常是屬性頁或以工作為基礎的中樞頁面。
-   **以物件為基礎的中樞頁面可能會針對選取的物件流覽至以工作為基礎的中樞。** 不過，您應該避免這類次要中樞，因為它們會讓 [控制台] 專案感覺太間接。

**工作窗格**

使用工作窗格來呈現命令、流覽和相關控制台專案的連結。

-   針對以工作為基礎的中樞中的工作窗格，以下列順序呈現連結：
    -   **次要命令**。 只在內容區域中顯示主要工作。 使用工作窗格進行次要、選擇性的工作。 如果使用者必須在重要案例中探索，請考慮使用主要工作;次要（如果使用者可以接受的話）。
    -   另 **請參閱**。 流覽至相關控制台專案的選擇性連結。
-   針對以物件為基礎的中樞中的工作窗格，以下列順序呈現連結：
    -   **物件檢視**。 選用的連結，可用來控制物件的呈現方式。
    -   已 **修正的命令**。 與目前選取的物件無關的命令。
    -   內容 **相關命令**。 這些命令相依于目前選取的物件，因此不一定會顯示。
    -   另 **請參閱**。 流覽至相關控制台專案的選擇性連結。
-   **請勿在輪輻頁面中使用工作窗格。** 不同于中樞頁面，輪輻頁面應著重于完成工作。 您不想要鼓勵使用者在完成前離開。

**另請參閱連結**

-   **提供 [工作] 窗格中的 [另請參閱] 連結，以協助使用者尋找相關的控制台專案，如果有錯誤，則為右邊的 [控制台] 專案。** 連結至使用者可能與您的控制台專案相關聯的專案。

    ![控制中心 [另請參閱] 連結的螢幕擷取畫面 ](images/winenv-ctrl-panels-image9.png)

    在此範例中，「動作中心控制台」專案會連結到相關的控制台專案。

-   **如果是使用者更可能辨識的，則連結至特定的工作頁面。** 否則，連結至整個 [控制台] 專案。 使用 [控制台] 名稱，而不新增 [片語]、[控制台]。

### <a name="spoke-pages"></a>輪輻頁面

**一般**

-   **在使用者需要更多指引和說明的情況下，使用工作頁面進行常用或重要的工作。**
-   **將表單頁面用於具有許多設定的功能，並受益于直接的單一頁面呈現。** 這類頁面的理想工作通常涉及一些簡單屬性的明顯變更。
-   **請勿在輪輻頁面中使用工作窗格。**

**互動**

-   **嘗試將主要工作限制在單一頁面上。** 如果需要一個以上的頁面，您可以：
    -   **使用中繼輪輻頁面進行其他或選擇性的步驟。** 中繼輪輻頁面是由最終輪輻頁面所認可。
    -   **針對獨立的輔助工作使用獨立的視窗。** 獨立的視窗會自行認可，而獨立于主要工作。

這樣做會將主要工作的認可按鈕意義保持明確且明確。 使用者應該一律安心瞭解他們正在認可哪些內容。

-   **請勿使用工作流程中的 [另請參閱] 連結。** 這些連結會連結到相關但不同的控制台專案。 雖然中樞頁面中可接受流覽至不同的專案，但它不在輪輻頁面中，因為這樣做會中斷工作。
-   **請勿使用輪輻頁面來進行簡單的輸入或確認。** 請改用強制回應對話方塊。

**(中繼輪輻頁面) 的互動**

-   **使用 [工作連結] 或 [下一步] 按鈕流覽至下一個頁面。** 繼續進行下一個步驟的方法應該一律很明顯。
-   **您可以有選擇性工作步驟的導覽連結。** 為了避免使用者認可工作時的混淆，這些額外的頁面應該是相同 [控制台] 專案內的中繼頁面。 他們不應該有認可按鈕，但應在認可主要工作時認可。

**(最終輪輻頁面) 的互動**

-   **使用 [認可] 按鈕來完成工作。** 針對輪輻頁面使用 [延遲的認可模型](glossary.md) ，如此一來，在明確認可的情況下才會進行變更 (如果使用者離開使用上一頁、Close 或 Address bar，變更就會) 放棄。 認可按鈕是使用者即將完成工作的視覺線索。 請勿使用連結來進行此用途。
-   **請勿確認認可按鈕 (包括取消) 。** 這樣做可能會很麻煩。 例外狀況：
    -   此動作會產生顯著的後果，如果不正確，則無法立即修復。
    -   此動作可能會導致使用者的時間或工作大幅遺失。
    -   動作與其他動作明顯不一致。
-   請勿使用 [上一步]、[關閉] 或 [網址] 列來 **確認使用者是否放棄變更**。 不過，您可以確認可能的非預期流覽可能會導致使用者的時間或工作大幅遺失。
-   **請勿使用命令或導覽連結** (包括 [另請參閱連結]) 。 在最終輪輻頁面上，使用者應該明確地完成或取消工作。 不建議使用者在其他地方流覽，因為這樣做可能會隱含地取消工作。
-   **當使用者完成或取消工作時，應該會將其傳送回已啟動工作的中樞頁面。** 如果沒有這類頁面，請改為關閉 [控制台] 視窗。 請勿假設輪輻頁面一律會從另一個頁面啟動。
-   當您將使用者回到從中啟動工作的頁面時，請 **從 Windows 檔案總管回堆疊中移除過時的**「已認可」頁面。 當使用者按一下 [上一頁] 按鈕時，應該永遠不會看到他們已經認可的頁面。 使用者應該一律透過完全重做工作來進行其他變更，而不是按一下 [上一步] 來修改過時的頁面。
    -   **開發人員：** 您可以使用 ITravelLog：： FindTravelEntry () 和 ITravelLogEx：:D eleteEntry () Api 來移除這些過時的頁面。

**認可按鈕**

**注意：** 取消按鈕會被視為認可按鈕。

-   **使用對主要指示的特定回應來確認工作，而不是一般標籤（例如 [確定]）。** 認可按鈕上的標籤應該有意義。 請避免使用 [確定]，因為它不是主要指示的特定回應，因此更容易誤解。 此外，[確定] 通常會搭配強制回應對話方塊使用，並不正確地表示關閉 [控制台] 專案視窗。
    -   **異常：**
        -   針對沒有設定的頁面，請使用 [確定]。
        -   當特定回應仍為一般（例如 [儲存]、[選取] 或 [選擇]）時，請使用 [確定]，就像變更特定設定或設定集合一樣。
        -   如果頁面有回應主要指示的選項按鈕，請使用 [確定]。 若要維護延遲的認可模型，您無法使用最終輪輻頁面上的工作連結。

            ![具有 [確定] 按鈕之 web 限制的螢幕擷取畫面 ](images/winenv-ctrl-panels-image10.png)

            在此範例中，選項按鈕（而非認可按鈕）會回應主要指令。
-   **提供 [取消] 按鈕，讓使用者明確放棄變更。** 雖然使用者可以藉由不確認其變更而隱含地放棄工作，但是提供 [取消] 按鈕可讓他們更有信心地進行工作。
    -   **例外狀況：** 請勿為使用者無法進行變更的工作提供 [取消] 按鈕。 在此情況下，[確定] 按鈕的效果與 [取消] 相同。
-   **請勿使用 [關閉]、[完成] 或 [完成] 認可按鈕。** 這些按鈕通常會搭配強制回應對話方塊使用，並不正確地表示關閉 [控制台專案] 視窗。 使用者可以使用標題列上的 [關閉] 按鈕來關閉視窗。 此外，[完成] 和 [完成] 會誤導，因為使用者會回到從中啟動工作的頁面，因此它們並不是真的完成。
-   **不要停用認可按鈕。** 否則使用者必須推算認可按鈕停用的原因。 最好讓認可按鈕保持啟用狀態，並在發生問題時提供實用的錯誤訊息。
-   **請確定頁面上顯示的認可按鈕沒有滾動。** 如果頁面很長，您可以將 [認可] 按鈕放在 [命令區域](glossary.md)中，而不是在內容區域中，藉此顯示這些按鈕。

    ![[自動播放] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image11.png)

    在此範例中，內容區域大小未系結，因此認可按鈕會置於命令區域中。

-   將認可按鈕靠右對齊，並使用此順序 (從左至右) ：正面的認可按鈕、取消及套用。

**預覽按鈕**

-   請確定 **[預覽] 按鈕表示立即套用暫止的變更，但如果使用者離開頁面而未認可變更，則還原目前的設定。**
-   **您可以使用任何輪輻頁面上的 [預覽] 按鈕。** 因為中樞頁面使用 [立即認可模型](glossary.md)，所以不需要預覽按鈕。
-   **請考慮使用 [控制台] 頁面的 [預覽] 按鈕，而不是 [套用] 按鈕。** 預覽按鈕可讓使用者更容易瞭解，並且可用於任何輪輻頁面上。
-   **只有當頁面的設定 (至少一個具有使用者可以看到之效果的) 時，才提供 [預覽] 按鈕。** 使用者應該能夠預覽變更、評估變更，然後根據該評估進行進一步的變更。
-   **一律啟用 [預覽] 按鈕。**

**即時預覽**

當您可以立即看到輪輻頁面上變更的影響時，[控制台] 專案會有即時預覽。

-   **當下列情況時，請考慮使用顯示設定的即時預覽：**
    -   效果很明顯，通常是因為它會套用到整個監視器。
    -   可以套用效果，而不會有明顯的延遲。
    -   效果很安全，而且可以輕鬆地復原。

        ![[變更色彩設定] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image12.png)

        在此範例中，會立即看到 Windows 色彩和外觀設定的效果。 這麼做可讓使用者以極短的時間進行變更。

-   **使用 [儲存變更] 和 [取消] 作為 [認可] 按鈕。** [儲存變更] 會保留目前的設定，而 [取消] 則會還原成原始設定。 使用 [儲存變更]，而不是 [確定]，讓您清楚知道任何預覽的變更尚未套用。
-   **不要提供 [套用] 按鈕。** Live preview 會套用不必要的應用。
-   如果使用者使用 [上一頁]、[關閉] 或 [網址] 列離開，請 **還原任何變更**。 若要保留變更，使用者必須明確地認可這些變更。

**套用按鈕**

-   請確定 **[套用] 按鈕表示套用工作啟動後 (所做的暫止變更，或上次套用) ，但仍留在目前的頁面上。** 這麼做可讓使用者在繼續進行其他工作之前，先評估變更。
-   **只在最終輪輻頁面上使用 [套用] 按鈕。** [套用] 按鈕不應該用於中繼輪輻頁面，以維護立即認可模型。
    -   **例外狀況：** 如果變更設定需要提高 [許可權](glossary.md)，您可以使用混合式中樞頁面上的 [套用] 按鈕。 如需詳細資訊，請參閱 [中樞頁面互動](#hub-pages)。
-   **只有當頁面的設定 (至少一個) 具有使用者可透過有意義的方式評估的效果時，才提供 [套用] 按鈕。** 一般情況下，[套用] 按鈕會在設定進行可見變更時使用。 使用者應該能夠套用變更、評估變更，然後根據該評估進行進一步的變更。
-   **只有在有暫止的變更時，才啟用 [套用] 按鈕;** 否則，請將它停用。
-   **指派 "A" 作為存取金鑰。**

### <a name="control-panel-integration"></a>控制台整合

若要整合您的 [控制台] 專案與 Windows，您可以：

-   **註冊您的控制台專案 (包括其名稱、描述和圖示)**，讓 Windows 知道它。
-   如果您的控制台專案是最上層 (請參閱以下) ：
    -   將它與適當的 **類別頁面** 產生關聯。
    -   **提供工作連結 (包括其名稱、描述、關鍵字及命令列)** 來指出主要工作，並允許使用者直接流覽至工作。
-   **提供搜尋字詞** ，以協助使用者使用主控台搜尋功能來尋找您的工作連結。

    請注意，您只能為個別的控制台專案提供此資訊，您無法針對您延伸的現有控制台專案新增或變更此資訊。

**類別頁面**

-   **只有在下列情況時，才將 [控制台] 專案新增至 [類別] 頁面：**

    -   大部分的使用者都需要它。 範例：網路和共用中心
    -   使用的次數很多。 範例：系統
    -   它提供的重要功能不會在其他地方公開。 範例：印表機

    符合這些準則的控制台專案稱為最上層。

-   **如果有下列情況，請不要將您的控制台專案加入至類別頁面：**

    -   它很少使用或用於一次性設定。 範例：歡迎中心
    -   其目標物件為 advanced users 或 IT 專業人員。 範例：色彩管理
    -   不適用於目前的硬體或軟體設定。 範例： Windows SideShow (目前的硬體) 不支援的情況。

    從類別頁面移除這類的 [控制台] 專案，可讓您更輕鬆地尋找最上層專案。 由於這些控制台專案的使用方式，可以透過搜尋或內容進入點充分探索。

-   **將最上層的 [控制台] 專案與使用者最有可能尋找的類別建立關聯。** 此決定應以使用者測試為基礎。
-   **請考慮將最上層的控制台專案與第二個最可能的類別建立關聯。** 如果使用者可能會在多個位置中尋找其主要工作，您應該將 [控制台] 專案與兩個類別建立關聯。
-   **請勿將您的 [控制台] 專案與兩個以上的類別建立關聯。** 如果 [控制台] 專案出現在數個類別中，則分類的值為破壞。

**工作連結**

-   **將您的 [控制台] 專案與其主要工作產生關聯。** 您最多可以在 [類別目錄] 頁面上顯示五個工作，但會使用其他工作來進行控制台搜尋作業。 使用與工作連結相同的片語，可能會移除一些文字，讓工作連結更簡潔。
-   **偏好讓工作連結出現在 [控制台] 專案中的不同位置。** 有多個相同位置的連結可能會造成混淆。

**搜尋詞彙**

-   **註冊您的控制台專案的搜尋詞彙，使用者最有可能用來描述它。** 這些搜尋詞彙應包括：

    -   已設定的功能或物件。
    -   主要工作。

    這些搜尋詞彙應以使用者測試為基礎。

-   **也包含這些搜尋詞彙的一般同義字。** 例如，「監視」和「顯示」是同義字，因此應該包含這兩個字。
-   **包含替代的拼寫或斷詞。** 例如，使用者可能會搜尋網站和網站。 也請考慮提供常見的拼寫錯誤。
-   **請考慮單數與複數名詞形式。** [控制台] 搜尋功能不會自動搜尋這兩種表單;提供使用者可能搜尋的表單。
-   **使用簡單的現在時動詞動詞。** 如果您將 [連線] 註冊為搜尋字詞，搜尋功能將不會自動尋找連接、連接和連線。
-   **不要擔心大小寫。** 搜尋功能不區分大小寫。

### <a name="standard-users-and-protected-administrators"></a>標準使用者和受保護的系統管理員

**許多設定都需要系統管理員許可權才能變更。** 如果進程需要系統管理員許可權，Windows Vista 和更新版本需要[標準使用者](glossary.md)和[受保護](glossary.md)的系統管理員明確地提升其許可權。 這樣做有助於防止惡意程式碼以系統管理員許可權執行。

如需詳細資訊和範例，請參閱 [使用者帳戶控制](winenv-uac.md)。

### <a name="schemes-and-themes"></a>配置和主題

配置是視覺設定的命名集合。 主題是跨系統設定的命名集合。 配置和主題的範例包括顯示、滑鼠、電話和數據機、電源選項和音效和音訊選項。

-   **允許使用者在下列情況建立配置：**

    -   **使用者很可能會變更設定。**
    -   **使用者最有可能將設定變更為集合。**

    當使用者位於不同的環境（例如 (國家/地區、時區) 的不同實體位置）時，配置會很有用;在不同的情況下使用其電腦 (電池、停駐/取消插接) ;或者，將電腦用於不同的功能 (簡報、影片播放) 。

-   **至少提供一個預設配置。** 在大多數情況下，預設配置應妥善命名並套用至大部分的使用者。 使用者不應該建立自己的配置。
-   **提供預覽** 或其他機制，讓使用者可以查看配置內的設定。

    ![[個人化] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image13.png)

    在此範例中，[個人化] 控制台專案會顯示桌面和外觀設定的預覽。

-   **提供 [另存新檔] 和 [刪除] 命令。** 重新命名命令不是必要的，使用者可以藉由儲存所需的名稱並刪除原始配置來重新命名配置。
-   如果無法在沒有配置的情況下套用設定，則 **不允許使用者刪除所有配置。** 使用者不應該建立自己的配置。
-   如果配置不是完全獨立的 (例如，電源配置相依于目前的膝上型電腦模式) ，請 **確定有一個簡單的方法可變更套用到所有配置的設定。** 例如，使用電源配置時，請確定使用者可以設定在單一位置中關閉可攜式電腦的蓋時所發生的情況。

### <a name="miscellaneous"></a>其他

-   **使用主控台擴充功能來取代或擴充現有的 Windows 功能。** 下列控制台專案可延伸：藍牙裝置、顯示器、網際網路、鍵盤、滑鼠、網路、電源、系統、無線 (紅外線) 。

### <a name="default-values"></a>預設值

-   **[控制台] 專案內的設定必須反映功能的目前狀態。** 否則會產生誤導，可能會導致不想要的結果。 例如，如果設定反映出建議，而不是目前的狀態，則使用者可能會按一下 [取消]，而不是進行變更，以為不需要任何變更。
-   **選擇最安全的 (，以防止資料遺失或系統存取) 和最安全的初始狀態。** 假設大部分的使用者不會變更設定。
-   **如果安全性和安全性不是因素，請選擇最有可能或方便的初始狀態。**

## <a name="text"></a>Text

### <a name="item-names"></a>專案名稱

-   **選擇明確通訊的描述性名稱，並區分 [控制台] 專案的用途。** 大部分的名稱會描述要設定的 Windows 功能或物件，並顯示在 [控制台] 首頁的 [傳統] 視圖中。
-   **請勿在名稱中包含 "設定"、"Options"、"Properties" 或 "Configuration" 等字。** 這是隱含的，而不是讓使用者能夠更輕鬆地進行掃描。

    **不正確：**

    協助工具選項

    數據機設定

    電源選項

    地區及語言選項

    **正確：**

    協助工具選項

    數據機

    電源

    地區格式和語言

    在正確的範例中，會移除不必要的字組。

-   **如果 [控制台] 專案設定了相關的功能，只會列出識別該專案所需的功能，並列出最可能先辨識或使用的功能。**

    **不正確：**

    資料夾選項

    電話和數據機選項

    **正確：**

    檔案和資料夾

    數據機

    在正確的範例中，主要的 [控制台] 專案功能會獲得強調。

-   使用 [標題樣式的大小寫](glossary.md)。

### <a name="page-titles"></a>頁面標題

**注意：** 如同所有 Explorer 視窗，[控制台] 頁面標題會顯示在 [網址](glossary.md)列上，但不會顯示在標題列上。

-   **若為中樞頁面，請使用 [控制台] 專案名稱。**
-   **針對輪輻頁面，請使用頁面的簡短摘要。** 如果它很簡潔，請使用頁面的主要指示;否則，請使用主要指令的精確重述。

    ![[電源選項] 對話方塊的螢幕擷取畫面 ](images/winenv-ctrl-panels-image6.png)

    在此範例中，電源選項用於頁面標題，而不是主要指令。

-   使用標題樣式的大小寫。

### <a name="task-link-text"></a>工作連結文字

下列指導方針適用于工作頁面的連結，例如類別頁面工作連結及另請參閱連結。

-   **選擇清楚傳達和區分工作的精確工作名稱。** 使用者不應該需要找出工作真正的意義，或與其他工作有何不同。

    **不正確：**

    調整顯示設定

    **正確：**

    螢幕解析度

    在正確的範例中，文字表達的精確度更高。

-   **在工作連結與它們所指向的頁面之間保留類似的語言。** 連結所顯示的頁面應該不會讓使用者感到驚訝。
-   **針對工作頁面，請將主要指示、認可按鈕和工作連結設計為一組相關的文字。**
    

    | 範例                             |    指令                                                   |
    |------------------------------|-------------------------------------------------------|
    | 工作連結：<br/>        | 連接到無線網路<br/>              |
    | 主要指示：<br/> | 選擇要連接的網路<br/>             |
    | 認可按鈕：<br/>    | 連線<br/>                                    |
    | 工作連結：<br/>        | 設定家長監護控制項<br/>                   |
    | 主要指示：<br/> | 選擇使用者並設定家長監護控制項<br/> |
    | 認可按鈕：<br/>    | 套用家長監護<br/>                     |
    | 工作連結：<br/>        | 解決同步衝突<br/>                |
    | 主要指示：<br/> | 您要如何解決這種衝突？<br/>  |
    | 認可按鈕：<br/>    | 解決<br/>                                    |

    

     

    這些範例顯示工作連結文字、主要指令和認可按鈕文字的關聯性。

-   雖然工作通常是以動詞開頭，但 **如果它是泛型、設定相關的動詞，而不能協助傳達工作，請考慮省略類別頁面上的動詞命令。**

    **特定且有用的動詞：**

    加

    勾選

    連線

    複製

    建立

    刪除

    中斷連線

    安裝

    移除

    設定

    開始

    Stop

    疑難排解

    **泛型、無用動詞：**

    Adjust

    變更

    Choose

    設定

    編輯

    管理

    開啟

    挑選

    集合

    選取

    顯示

    檢視

-   **如果工作設定數個相關的功能，只會列出代表該集合的功能。** 省略可以立即推斷的詳細資料。

    **不正確：**

    喇叭音量、靜音、音量圖示

    喇叭、麥克風、MIDI 和音效方案

    **正確：**

    喇叭音量

    喇叭和麥克風

    在正確的範例中，工作連結中只會提供代表性的功能。

-   **只有當目標使用者這麼做時，才應該將工作以技術為依據。**

    **不正確：**

    Connectoids

    圖元深度

    DPI

    **正確：**

    印表機

    掃描器

    滑鼠

    正確的範例是以技術為基礎的目標使用者可能會使用的詞彙，而不正確的範例則不正確。

-   只有在系統可以支援一個以上時，才使用複數名詞。
-   使用 [句子樣式的大小寫](glossary.md)。
-   請勿使用結束標點符號。

### <a name="main-instructions"></a>主要指示

-   **在 [中樞] 頁面上，使用主要指示，透過 [控制台] 專案來說明使用者的目標。** 主要指示應可協助使用者判斷他們是否已選取正確的控制台專案。 請記住，使用者可能已在錯誤中選取您的 [控制台] 專案，而且實際上正在尋找屬於其他 [控制台] 專案的設定。

    **範例：**

    保持電腦的安全和最新狀態

    優化您的電腦，以便更輕鬆地查看、聆聽及控制

-   **針對輪輻頁面，請使用主要指示來說明頁面上要執行的動作。** 指令應該是特定的語句、命令式方向或問題。 良好的指示會以頁面傳達使用者的目標，而不是單純專注于操作它的機制。

    **不正確：**

    挑選通知工作

    **正確：**

    指出如何處理連入資訊

    正確的版本會更妥善地傳達頁面所達到的目標。

-   **盡可能使用特定動詞。** 特定動詞對使用者來說更有意義，而不是泛型。
-   使用 [句子樣式的大小寫](glossary.md)。
-   **如果指令是語句，則不包含最後句號。** 如果指令是問題，請包含最後一個問號。

### <a name="supplemental-instructions"></a>補充指示

-   **針對 [中樞] 頁面，請使用選擇性的補充指示，進一步說明 [控制台] 專案的用途。**
-   **若為輪輻頁面，請使用選擇性補充指示來呈現有助於瞭解或使用頁面的其他資訊。** 您可以提供更詳細的資訊並定義術語。
-   使用完整句子和 [句子樣式的大小寫](glossary.md)。

### <a name="page-text"></a>頁面文字

-   **請勿重新聲明內容區域中的主要指示。**
-   **使用 "page" 一詞來參考頁面本身。**
-   **使用第二個人 (您的) ，告訴使用者要** 在主要指示和內容區域中執行的動作。 通常會隱含第二個人。

    **範例︰**

    選擇您要列印的圖片。

-   **使用第一個人 (我，我的) 讓使用者知道要** 在內容區域中執行的動作，以回應主要指示。

    **範例︰**

    列印相機上的相片。

### <a name="task-links"></a>工作連結

-   **選擇清楚的連結文字，以明確地傳達和區分工作連結的用途。** 它應該是一目了然的，且對應至主要指令。 使用者不應該需要找出連結真正的意義，或與其他連結有何不同。
-   **一律使用動詞啟動工作連結。**
-   使用 [句子樣式的大小寫](glossary.md)。
-   請勿使用結束標點符號。
-   **如果工作連結需要進一步說明，請使用完整的句子和結束標點符號，在個別的文字控制項中提供說明** 。 不過，只有在需要時才新增這類說明，因為另一個工作連結需要一個。

如需詳細資訊和範例，請參閱 [連結](ctrl-command-links.md)。

### <a name="commit-buttons"></a>認可按鈕

-   **使用對自身有意義的特定認可按鈕標籤，並符合主要指令。** 在理想的情況下，使用者不需要讀取任何其他人來瞭解標籤。 使用者比靜態文字更可能讀取命令按鈕標籤。
-   **一律使用動詞來啟動認可按鈕標籤。**
-   **請勿使用 [關閉]、[完成] 或 [完成]。** 這些按鈕標籤更適合其他類型的視窗。
-   使用 [句子樣式的大小寫](glossary.md)。
-   請勿使用結束標點符號。
-   指派唯一的 [存取金鑰](glossary.md)。
    -   **例外狀況：** 請勿將存取金鑰指派給 [取消] 按鈕，因為 Esc 是其存取金鑰。 這麼做可讓您更輕鬆地指派其他存取金鑰。

-   在使用者檔中，請參閱主控台，使用標題樣式的大小寫，並省略前面的明確文章。

    **範例︰**

    在主控台中，開啟 [ **安全性中心**]。

針對 [控制台] 專案：

範例：

-   在主控台中，開啟 [ **家長** 監護]。
-   回到主要的 [ **家長** 監護] 頁面。
