# <a name="ribbons"></a>緞帶

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

功能區是一種現代化的方法，可協助使用者以最少點擊次數的方式，有效率地尋找、瞭解及使用命令，而不需要設法解決試用和錯誤，也不需要參考說明。

功能區是一種命令列，會將程式的功能組織成視窗頂端的一系列索引標籤。 使用功能區可提高功能和功能的探索能力，讓整個程式的學習速度更快，並讓使用者感覺更能掌控程式的使用經驗。 功能區可以取代傳統的功能表列和工具列。

![功能區的螢幕擷取畫面 ](images/cmd-ribbons-image1.png)

一般功能區。

功能區索引標籤是由群組所組成，這些群組是一組已加上標籤的緊密相關命令。 除了索引標籤和群組之外，功能區還包含：

- 應用程式按鈕，其中顯示的命令功能表包含對檔或工作區進行某些操作（例如檔案相關命令）。
- 快速存取工具列，這是一個小型、可自訂的工具列，可顯示常用的命令。
- [核心] 索引標籤是一律會顯示的索引標籤。
- 只有在選取特定物件類型時才會顯示的內容索引標籤。 一律顯示的索引標籤稱為核心索引標籤。
- 索引標籤集是單一物件類型的內容索引標籤集合。 因為物件可以有多個類型 (例如，具有圖片的資料表中的標頭是) 的三種類型，所以一次可能會顯示多個內容索引標籤集合。
- 強制回應索引標籤，這是以特定暫時性模式顯示的核心索引標籤，例如 [預覽列印]。
- 資源庫，這是以圖形方式呈現的命令或選項清單。 以結果為基礎的資源庫說明命令或選項（而非命令本身）的作用。 功能區圖庫會顯示在功能區中，而不是快顯視窗。
- 增強的工具提示，可清楚說明其相關聯的命令並提供快速鍵。 它們也可以包含圖形和參考以提供協助。 增強的工具提示可減少與命令相關的協助需求。
- 對話方塊啟動器，也就是某些群組底部的按鈕，這些按鈕會開啟對話方塊，其中包含與群組相關的功能。

功能區最初是 Microsoft Office 2007 引進。 若要瞭解 Office 需要使用功能區的原因，以及使用功能區所解決的許多問題，請參閱[功能區的案例](/archive/blogs/jensenh/the-story-of-the-ribbon)。

> [!Note]  
> 與 [功能表](cmd-menus.md)、 [工具列](cmd-toolbars.md)、 [命令按鈕](ctrl-command-buttons.md)和 [圖示](vis-icons.md) 相關的指導方針會顯示在不同的文章中。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用功能區，請考慮下列問題：

### <a name="program-type"></a>程式類型

- **您所設計的程式類型為何？** 程式類型是功能區」適當性的良好指標。 功能區適用于檔建立和撰寫程式，以及檔檢視器和瀏覽器。 功能區可能適用于其他類型的程式，但其他形式的命令呈現可能更適合。 一般來說，輕量程式應該要有輕量的命令呈現。

### <a name="discoverability-and-learning-issues"></a>探索和學習問題

- **使用者在尋找命令時遇到問題嗎？使用者是否要求已在程式中的功能？** 如果是這樣，使用功能區可讓您輕鬆地找到命令，方法是使用簡單易懂的標籤和相關命令的群組。 使用功能區的調整速度也優於功能表列和工具列，以供未來成長之用。
- **使用者是否難以理解程式的命令？它們經常會被視為「試用與錯誤」，以選取正確的命令或判斷命令的運作方式嗎？** 若是如此，根據資源庫和即時預覽使用具有結果導向命令的功能區，可讓您更容易瞭解命令。

### <a name="command-characteristics"></a>命令特性

- **這些命令會顯示在數個位置嗎？如果您的程式已經存在，則會在功能表列、工具列、工作窗格和工作區域本身內顯示命令嗎？** 如果是這樣，使用功能區會將命令整合到單一位置，使其更容易尋找。
- **命令適用于整個視窗，還是套用至特定的窗格？** 功能區最適用于套用至整個視窗或特定物件的命令。 就地命令適用于個別視窗窗格。
- **可以直接呈現大部分的命令嗎？也就是說，使用者只要按一下，就可以與他們互動嗎？如果從功能表和對話方塊存取常用的命令，是否可以將它們重構為 direct？** 雖然有些命令可以使用功能表和對話方塊來呈現，但以這種方式呈現大部分的命令破壞功能區的效率，可能讓功能表列成為更好的選擇。

### <a name="command-scale"></a>命令縮放比例

- **有少數幾個命令嗎？可以輕鬆地在單一的簡單工具列上顯示最常使用的命令嗎？** 如果新增 [核心] 和 [內容] 索引標籤會產生簡單的 [首頁] 索引標籤，以單獨用來執行最常見的工作，則使用功能區是值得的。 如果不是，使用功能區的優點可能不會為少量的命令調整其額外權數。
- **有很多命令嗎？使用功能區需要7個以上的核心索引標籤？使用者是否經常需要變更索引標籤來執行一般工作？** 如果是，使用工具列 (不需要變更索引標籤) 和選擇區 [windows](cmd-toolbars.md) (可能需要變更索引標籤，但一次可能會有數個開啟) 可能是更有效率的選擇。
- **使用者是否傾向于大部分的時間都使用少量的命令？** 如果是的話，可以將這類命令放在 [首頁] 索引標籤上，以有效率地使用功能區。不斷變更的索引標籤會使功能區太沒有效率。
- **程式是否能讓程式的內容區域變得更大？** 如果是，使用功能表列和單一工具列比功能區更能節省空間。 但是，如果您的程式需要三個或多個資料列的工具列，或使用工作窗格，使用功能區會更節省空間。
- **使用者是否傾向于在程式中的大型視窗內有很長一段時間的特定區域內工作？** 如果是，則會從迷你工具列、選擇區視窗和 direct 命令的相近程度獲益。 從工作區到功能區進行來回行程會太沒有效率。
- **為了提高效率和彈性，使用者是否需要對命令展示內容、位置或大小進行重大變更？** 如果是，可自訂且可擴充的工具列和選擇區視窗是較佳的選擇。 請注意，有些類型的工具列可以取消停駐，使其成為調色盤視窗，而且可以移動、調整大小和自訂您的調色盤視窗。

最後，請考慮這個最後的問題： **是否改善可搜尋性、簡化學習、效率和生產力，以節省額外空間的成本，以及索引標籤組織命令的需求？** 如果是，使用功能區是絕佳的選擇。 如果您不確定，請考慮使用以功能區為基礎的設計測試，並將它與最佳替代方式進行比較。

功能區是一種全新且吸引人的命令簡報形式，也是將程式現代化的絕佳方法。 但如有吸引力，它們不是每個程式的正確選擇。

**不正確：**

## <a name="seven-most-important-things"></a>七項最重要的事項

1. 選擇適合您的程式類型的命令解決方案。 使用功能區應該讓程式覺得更簡單、更有效率，而且更容易使用，前所未有。 如果不適合使用功能區，請考慮改為使用豐富的命令。  
2. 千萬別低估建立有效功能區的挑戰。 請勿預期它是您現有功能表列和工具列的簡單埠。 而且不需要使用功能區自動讓程式更好。 願意認可命令重新設計所需的時間和精力，是決定使用功能區的重要因素。  
3. 讓命令可供探索。 選擇索引標籤設計，此索引標籤設計在命令與其所在的 descriptively 標示索引標籤之間有明確、明確、唯一的對應。 使用者應該能夠快速且安心地判斷哪一個索引標籤具有他們要尋找的命令，而且很少選擇錯誤的索引標籤。  
4. 讓命令一目了然。 使用者應該瞭解命令的效果，其標籤、圖示、工具提示和預覽。 他們不應實驗或閱讀說明主題，以瞭解命令的運作方式。  
5. 讓命令有效率地使用：
    - 使用者在 [常用] 索引標籤上應花費大部分的時間。
    - 在一般工作期間，使用者很少需要變更索引標籤。
    - 當視窗最大化且使用者位於正確的索引標籤時，最常使用的命令會有最大的視覺效果，使用者只要按一下就可以叫用它們。 使用者可以在索引標籤上執行所有其他命令，最多點擊四次。
    - 使用者不應該開啟對話方塊來提供命令，以及變更一般工作中的屬性。
6. 協助使用者安心選擇命令和選項，並將試用和錯誤的需求降到最低。 在適當的情況下使用以結果為導向的命令，通常是以資源庫和即時預覽的形式來進行。  
7. 請確定功能區可從最大的視窗大小調整為最小。  

## <a name="design-concepts"></a>設計概念

### <a name="adapting-a-ribbon-in-an-existing-program"></a>調整現有程式中的功能區

雖然您可以只將現有程式的傳統功能表列和工具列設計重構為功能區格式，但使用功能區的大部分值都不會遺失。 當功能區用來呈現立即的結果導向命令時，功能區具有最大的價值，通常是以資源庫和即時預覽的形式呈現。 結果導向的命令可讓您更容易瞭解命令，並讓使用者更有效率且更具生產力。 與其重構現有的命令，最好是在程式中完全重新設計命令的執行方式。

千萬別低估建立有效功能區的挑戰。 而且不需要使用功能區自動讓程式更好。 建立有效的功能區需要花費很多時間和精力。 願意認可這類命令重新設計所需的時間和精力，是決定使用功能區的重要因素。

### <a name="the-nature-of-ribbons"></a>功能區的本質

相較于傳統的功能表列和工具列，功能區具有下列特性：

- **針對所有命令 (UI) 的單一使用者介面。** 功能表列是完整且容易學習的，而工具列是有效率且直接的，但為什麼不使用更多的螢幕空間來建立可完成這些工作的單一命令 UI？ 只有一個 UI，功能區不會要求使用者找出哪些 UI 具有所要的命令。
- **可見且一目了然。** 功能表列命令可透過其標籤一目了然，但大部分的時間都不會顯示。 為了節省螢幕空間，工具列按鈕主要是以圖示而不是標籤來表示 (不過，某些工具列按鈕會使用這兩個) ，而且當圖示無法自我理解時，會相依于工具提示。 不過，使用者通常只知道最常使用的命令的圖示。
- 藉由以標記的圖示呈現大部分命令，功能區命令都是可見的，而且只會使用工具提示來提供補充資訊。 您幾乎不需要前往其他地方 (例如協助) 來瞭解命令。
- **標示為群組。** 當您標示功能表分類時，下拉式選單內的群組不會出現，而且只會使用未標記的分隔符號來表示。 工具列內的群組也會以未標記的分隔符號表示。
- 藉由在標記的群組中組織命令，功能區可讓您更輕鬆地尋找命令並判斷其用途。
- **強制回應，但不是階層式。** 功能表列會藉由建立命令階層來進行調整。 具有許多專案的功能表可以使用一或多個子功能表層級，以提供更多命令。
- 功能區命令需要比工具列命令更多的空間，因此會使用索引標籤來調整規模。 這種使用索引標籤可讓功能區強制回應，因此需要使用者偶爾變更模式來尋找命令。 不過，在索引標籤中，大部分的命令都可以直接使用，也可以使用單一分割按鈕或功能表按鈕，而不是階層。
- **Direct 和 immediate。** 如果您只需要按一下就能叫用命令，則會直接使用一個命令 (也就是，不需要流覽功能表) 和立即生效 (也就是不需要對話方塊來收集其他輸入) 。 功能表列命令一律是間接且通常不會立即進行。 和工具列一樣，大部分的功能區命令都是設計成直接且立即的，其中包含只要按一下就能叫用的最常用命令，而不需要對話方塊來收集額外的輸入。
- **寬敞。** 功能表列和工具列主要是設計成具有空間效率。 為了提供其優點，功能區可能會耗用更多的垂直空間，大致上相當於功能表列和三列工具列。 因為少數程式有三個以上的工具列，所以功能區通常耗用的空間比傳統的命令 Ui 更多。
- **具有應用程式按鈕和快速存取工具列。** 功能區一律會顯示應用程式按鈕和快速存取工具列。 這麼做可讓使用者存取檔案相關和常用的命令，而不需要變更索引標籤，並提升程式之間的一致性。
- **基本自訂。** 雖然功能表列有固定的簡報，但許多工具列都可自訂，讓使用者可以設定位置、大小和內容。 功能區本身無法自訂，但快速存取工具列提供有限的自訂。
- **改良的鍵盤協助工具。** 功能表列具有絕佳的鍵盤協助工具，因為直接按下 ALT 鍵可提供功能表列輸入焦點。 不過，工具列沒有這類機制，因為它們會與視窗的內容共用鍵盤流覽。 因此，使用者必須使用 Tab 鍵 (流覽至工具列，並在最後一個索引標籤停止) ，然後使用方向鍵流覽至特定的命令。

相反地，功能區透過 [快速鍵](glossary.md)提供增強的鍵盤協助工具，通常有三個步驟的程式：

- 按下 ALT 鍵以輸入 keytip 模式。
- 在快速存取工具列中，按下字元以選擇索引標籤、應用程式按鈕或命令。
- 在索引標籤中，按一或兩個字母來選擇命令。

    這種方法是高度視覺效果。 它也更有彈性，讓它能夠更妥善地調整規模，並具有更多助憶鍵存取金鑰指派。

    請勿將便捷鍵與快速鍵混淆。 雖然這兩個存取金鑰和快速鍵都提供鍵盤存取權，但它們具有不同的用途和指導方針。 如需詳細資訊，請參閱 [鍵盤](inter-keyboard.md)。

### <a name="the-nature-of-rich-commands"></a>Rich 命令的本質

Rich 命令指的是功能區所使用命令的呈現與互動，而不一定要使用功能區容器。 Rich 命令具有下列特性：

- **標籤。** 所有命令都有一目了然的標籤，只有當圖示非常知名且空間是 premium 時，才會發生例外狀況。

    **正確：**

    ![字元格式功能區的螢幕擷取畫面 ](images/cmd-ribbons-image3.png)

    這些命令非常知名，因此它們不需要標籤。

    **不正確：**

    ![具有少用圖示的功能區螢幕擷取畫面 ](images/cmd-ribbons-image4.png)

    這些無法辨認的圖示需要適用于豐富命令的標籤。

- **施 膠。** 命令的大小相對於其使用頻率和重要性，而不是統一調整大小。 除了讓最常使用的命令更容易尋找並按一下，也會讓它們更 [可觸式](https://msdn.microsoft.com/library/windows/desktop/cc872774.aspx)。

    ![一個大型和三個小按鈕的螢幕擷取畫面 ](images/cmd-ribbons-image5.png)

    在此範例中，最常使用的按鈕大於其他按鈕。

- **動態調整大小。** 豐富的命令控制項會自行調整大小，以充分利用可用的空間，而不是使用固定大小，並且在大小太小時截斷或使用溢位。

    ![寬功能區具有大小相同按鈕的螢幕擷取畫面 ](images/cmd-ribbons-image6.png)![具有混合按鈕的小型功能區螢幕擷取畫面 ](images/cmd-ribbons-image7.png)

    在此範例中，命令按鈕會調整大小，以在可用的空間中運作正常。

- **分割按鈕。** 分割按鈕適合用來在需要時合併一組不同的命令，同時保有最常使用之命令的 directness。

    ![[另存新檔] 命令的螢幕擷取畫面及其選項 ](images/cmd-ribbons-image8.png)

    在此範例中，[另存新檔] 命令使用 [分割] 按鈕，其中 main 按鈕會執行最常見的變化，而功能表部分會顯示具有命令變化的功能表。

- **豐富的下拉式功能表和資源庫。** 下拉式功能表、下拉式清單和資源庫會取得通訊所需的空間，並區分選項的效果，通常是使用圖形和文字描述。 類別是用來組織大型的選項組。

    ![具有圖示的下拉式功能表選項螢幕擷取畫面 ](images/cmd-ribbons-image9.png)

    在這些範例中，按一下功能表按鈕會顯示顯示其效果的選項清單。

- **即時預覽。** 只要使用者將滑鼠停留在格式化選項上，程式就會顯示使用實際內容進行該格式化的結果外觀。

    ![格式化選項之結果的螢幕擷取畫面 ](images/cmd-ribbons-image10.png)

    即時預覽會顯示在停留時套用格式化選項的結果。

- **增強的工具提示。** 這些會清楚說明其相關命令，並提供快速鍵。 它們也可以包含圖形和參考，以協助 (但它們大多不需要命令相關的說明) 。

    ![具有文字和圖形的大型工具提示螢幕擷取畫面 ](images/cmd-ribbons-image11.png)

    增強型工具提示會清楚說明其相關聯的命令。

雖然功能區可能不適合所有程式，但是所有程式可能會因為豐富的命令而受益。

### <a name="ribbons-always-have-an-application-button-and-quick-access-toolbar"></a>功能區一律具有應用程式按鈕和快速存取工具列

應用程式按鈕和快速存取工具列提供了在任何內容中很有用的命令，因此可減少變更索引標籤的需求。 雖然這三個元件在邏輯上是獨立的，但功能區一定要有應用程式按鈕和快速存取工具列。 假設命令可進入功能區或應用程式按鈕，您可能會想要放置命令的位置。 選擇不是任意的。

[應用程式] 按鈕是用來顯示命令的功能表，這些命令牽涉到與檔案進行某些工作，例如傳統上會在 [檔案] 功能表中建立、開啟和儲存檔案、列印及傳送和發佈檔的命令。

相反地，功能區本身適用于影響視窗內容的命令。 範例包括用來讀取、修改或使用內容或變更視圖的命令。

如果您使用功能區，即使您的程式不包含檔或檔案，也必須使用應用程式按鈕。 在這種情況下，請使用 [應用程式] 功能表來顯示列印、程式選項和結束程式的命令。 雖然這類程式不需要應用程式按鈕，但使用它可提供整個程式的一致性。 使用者不需要尋找 [儲存] 和 [復原] 命令或 [程式] 選項，因為它們一律位於相同的位置。

即使功能區只使用一個索引標籤，也需要快速存取工具列。同樣地，雖然這類程式不需要快速存取工具列 (因為所有命令都已出現在單一索引標籤) 上，所以具有可自訂的快速存取工具列可提供整個程式的一致性。 例如，如果使用者習慣按一下列印命令，則應該可以在任何使用功能區的程式中進行。

### <a name="organization-and-discoverability"></a>組織與可探索性

藉由提供索引標籤和群組，功能區可讓您組織命令以協助探索。 面臨的挑戰是，如果組織的執行效能不佳，可能會明顯損害探索能力。 您的命令和它們所在的 descriptively 標籤和群組之間應該有明確、明顯且唯一的對應。

使用者在使用時，會形成功能區的精神模型。 如果該精神模型對使用者而言沒有意義、效率不佳或不正確，就會導致混淆和挫折。 **設計功能區最重要的目標是協助您快速且安心地尋找命令。如果您沒有完成此作業，您的功能區設計將會失敗。** 達成此目標需要謹慎的設計、使用者測試和反復專案。 不要假設它很簡單。

以下是一些常見的陷阱，可避免：

- **避免 [一般] 索引標籤和組名。** 良好的索引標籤或組名必須準確地描述其特定內容，最好是使用以工作和目標為基礎的語言。 避免一般索引標籤和組名，以及以技術為基礎的名稱。 例如，建立及撰寫程式的檔中，幾乎所有的命令都可以屬於標記為編輯、格式化、工具、選項、Advanced 和其他的索引標籤。 依賴特定的描述性標籤，而不是記住需求。
- **避免過度特定的索引標籤和組名。** 雖然我們想要特定的索引標籤和組名，但它們不一定是因為使用者的內容感到驚訝。 使用者通常會使用排除程式來尋找專案，因此請防止他們忽略您的索引標籤或群組，因為名稱會誤導。
- **請避免對相同命令的多個路徑，特別是如果路徑是非預期的路徑，或命令需要多次點擊才能叫用。** 透過多個路徑尋找命令可能有點方便。 但是請記住，當使用者找到所需的內容時，他們就會停止查看。 如果使用者認為第一個路徑是沒有效率或非預期的，則使用者可能會認為第一個找到的路徑是嚴重的問題。 此外，有重複的命令可讓使用者更難找到正在掃描的其他命令。

![[間接的框線路徑] 命令的螢幕擷取畫面 ](images/cmd-ribbons-image12.png)

在此範例中，您可以透過 [頁面框線] 命令變更段落框線，即使 [常用] 索引標籤上有更直接的路徑也一樣。如果使用者想要在這個非預期的路徑上遇到段落框線，可能會很容易假設它是唯一的路徑。

- **避免任意的命令放置。** 假設您認為您有良好的索引標籤和群組設計，但發現有幾個命令不適合。 您的索引標籤和群組設計很有可能不像您想像的一樣，您必須繼續進行調整。 請勿將這些命令放在不屬於的地方來解決這個問題。 如果您這麼做，使用者可能必須檢查每個索引標籤，以找出它們，然後立即忘記其所在位置。
- **避免以行銷為基礎的放置。** 假設您有新版本的程式，且您的行銷小組真的想要推廣其新功能。 將它們放在 [常用] 索引標籤上可能很吸引人，但是如果它不會造成整體的探索能力，這會是代價高昂的錯誤。 請考慮未來的產品版本，以及不斷變動的組織會有多少挫折。

### <a name="tabs"></a>定位點

最佳的第一個步驟是檢查 [標準功能區索引卷](#standard-ribbon-tabs)標。 如果您程式的命令自然地對應到標準索引標籤，請將您的索引標籤組織作為這些標準。 另一方面，如果您的程式命令未自然對應，請不要嘗試強制執行。 判斷更自然的結構，並務必執行大量的使用者測試，以確保您已正確完成。

針對非標準索引標籤，請考慮下列問題：

- **每個索引標籤名稱都應描述其內容。** 選擇特定但不太明確的有意義名稱。 使用者的內容應該永遠不會驚訝。
- **每個索引標籤名稱都應該反映其用途。** 請考慮與命令相關聯的目標或工作。
- **每個索引標籤名稱都應與所有其他索引標籤名稱明確不同。**

[首頁] 索引標籤是這些考慮的例外狀況。 雖然您不需要有 [首頁] 索引標籤，但大部分的程式都應該這麼做。 [首頁] 索引標籤是第一個索引標籤，其中包含最常使用的命令。 如果您經常使用的命令無法放入其他索引標籤中，[常用] 索引標籤就是正確的位置。

如果您無法判斷有意義、描述性的索引標籤名稱，可能是因為索引標籤設計不妥善。 如果您的功能區組織無法運作，請重新考慮您的索引標籤設計。

### <a name="scaling-the-ribbon"></a>調整功能區

調整工具列很簡單：如果視窗太窄而無法顯示工具列，工具列會顯示適合的專案，並可透過溢位按鈕來存取其他專案。 Rich 命令的目標是要充分利用可用的空間，因此調整功能區需要更多設計工作。 沒有預設的功能區大小，因此您不應該考慮使用特定的寬度來設計功能區。 您必須使用各式各樣的寬度來設計版面配置，並瞭解其中任何一種都可能是您的使用者最常看到的。 調整是功能區設計的基本部分，不是最後一個步驟。 當您設計索引標籤時，請為每個群組指定不同的版面配置 (最多三個) 以及可一起使用的組合。 功能區會顯示符合目前視窗大小的最大有效組合。

## <a name="guidelines"></a>指導方針

### <a name="general"></a>一般

- **請勿將功能區與視窗中的功能表列和工具列合併。** 您必須使用功能區來取代功能表列和工具列。 不過，功能區可結合調色盤視窗和導覽元素，例如 [上一頁] 和 [下一頁] 按鈕，以及 [網址] 列。
- **一律將功能區與應用程式按鈕和快速存取工具列合併。**
- **當程式啟動時，請選取最左邊的索引標籤 (通常是 [首頁]) 。** 不要讓最後選取的索引標籤在程式實例之間保持不變。
- **當程式初次開機時，以正常狀態顯示功能區 (不會將) 降至最低。** 使用者通常會讓預設設定保持不變，因此將程式啟動時的功能區最小化可能會導致所有命令的效率較低。 此外，也可以 disorienting 顯示功能區一開始的最小化。
- **讓功能區狀態在程式實例之間保持一致。** 例如，如果使用者將功能區最小化，則在下一次執行程式時，應該會顯示最小化。 但同樣地，請不要以這種方式來保存最後選取的索引標籤。

### <a name="using-tabs"></a>使用索引標籤

一般而言，較少的索引標籤愈好，所以請移除不有助於達成這些目標的索引標籤。

- **只要可行，請使用標準索引標籤。** 使用標準索引標籤可大幅改善可搜尋性，尤其是在程式之間。 請參閱本文稍後的 [標準功能區索引卷](#standard-ribbon-tabs) 標。
- **如有需要，請為第一個索引標籤首頁加上標籤。** [首頁] 索引標籤應該包含最常使用的命令。 如果您經常使用的命令無法放入其他索引標籤中，[常用] 索引標籤就是正確的位置。
- 如果有下列情況，請新增索引標籤：
  - **它的命令與特定工作強烈相關，而且可以由索引標籤標籤正確地描述。** 加入索引標籤應該有助於讓命令更容易找到，而不是更難。
  - **它的命令大部分與其他索引標籤上的工作無關。** 加入索引標籤時，不應該在執行一般工作時需要更多索引標籤切換。
  - **此索引標籤有足夠的命令可證明有更多的外觀。** 沒有只有一些命令的索引標籤。 **例外狀況：** 如果索引標籤與特定工作有強烈的關聯，請考慮加入索引標籤，並新增索引標籤可大幅簡化過於複雜的 [首頁] 索引標籤。
- **針對其餘索引標籤，請先放置最常使用的索引標籤，同時維持跨索引標籤的邏輯順序。**
- **優化索引標籤設計，讓使用者快速且安心地找到命令。** 其他所有考慮都是次要。
- **不要提供** [說明] 索引標籤。相反地，請使用整個程式的說明和增強的工具提示來提供協助。
- **使用最多七個核心索引標籤。** 如果有7個以上，就很難判斷哪個索引標籤有命令。 雖然有七個核心索引標籤可供具有許多命令的應用程式使用，但大部分的程式都應該以四個以上的索引標籤

### <a name="contextual-tabs"></a>內容索引標籤

- **使用 [內容] 索引標籤，即可顯示只有當使用者選取特定物件類型時，才會相關的命令集合。** 如果只有幾個常用的命令，則使用一般索引標籤可能更方便且更穩定，並只在未套用命令時停用它們。
- ![剪下和複製命令的螢幕擷取畫面暗灰色 ](images/cmd-ribbons-image29.png)<br>最好停用剪下和複製以外的一般命令，而不是使用內容相關索引標籤。
- **只包含特定物件類型的特定命令。** 如果使用者不需要先選取物件，請不要將命令放在內容索引標籤上。
- **包含使用特定物件類型時經常使用的命令。** 將常用的一般內容命令放在操作功能表和迷你工具列上，以避免在經常執行的工作期間切換索引標籤。 或者，如果這樣做可避免經常切換索引標籤，請考慮將一般命令放在內容索引標籤上。 但不要濫用這個方法這一點-請勿嘗試包含使用者在處理物件時可能需要的每個命令。
- ![[設計] 索引標籤上 [框線] 命令的螢幕擷取畫面 ](images/cmd-ribbons-image30.png)<br/>在此範例中，[框線] 命令會包含在 [設計] 索引標籤上，以避免在經常執行的工作期間經常切換索引標籤。
- **選擇不同于目前顯示之內容索引標籤的內容索引標籤色彩。** 您稍後可以使用不同的色彩來顯示相同的索引標籤集合，以達成此目的，但請盡可能嘗試在調用之間使用一致的色彩指派。
- 選取內容索引標籤 **會自動** 協助探索、改善穩定性的認知，並且減少切換索引標籤的需求。 當下列情況時，自動選取內容索引標籤：
  - **使用者插入物件。** 在此情況下，請選取集合中的第一個內容索引標籤。
  - **使用者按兩下物件。** 在此情況下，請選取集合中的第一個內容索引標籤。
  - **使用者選取了 [內容] 索引標籤，並按一下物件，然後立即按一下相同類型的物件。** 在此情況下，請返回先前選取的內容索引標籤。
- 移除作用中索引標籤的內容索引標籤 **時，請將 [首頁] 索引標籤或第一個** 索引標籤設為使用中的這麼做會顯示最穩定的。

- **使用 [強制回應] 索引標籤，即可顯示套用至特定暫時模式的命令集合，而且沒有任何 [核心] 索引標籤適用。** 如果某些核心索引標籤適用，請改用內容索引標籤，並停用不適用的命令。 由於強制回應索引標籤非常有限，因此只有在沒有更好的替代方案時才應該使用。
- ![[預覽列印] 索引標籤的螢幕擷取畫面 ](images/cmd-ribbons-image31.png)<br/>[預覽列印] 是常用的模式索引標籤。

- ![[關閉預覽列印] 按鈕的螢幕擷取畫面 ](images/cmd-ribbons-image32.png)<br/>在此範例中，以模式明確標示 Close 命令會移除關閉內容的任何疑慮。
- **若要關閉強制回應索引標籤，也請重新定義視窗標題列上的 [關閉] 按鈕，以關閉模式而不是程式。** 使用者測試顯示有許多使用者預期此行為。

#### <a name="standard-contextual-tabs"></a>標準內容索引標籤

- **格式。** 包含與變更所選物件類型格式相關的命令。 通常會套用至物件的一部分。
- **設計。** 包含通常在資源庫中的命令，以將樣式套用至選取的物件類型。 通常會套用至整個物件。
- **佈局。** 包含用來變更複雜物件（例如資料表或圖表）結構的命令。

如果您有與格式、設計和配置相關的內容相關命令，但無法使用多個索引標籤，則只需提供 [格式] 索引標籤。

- **只要可行，請使用標準群組。** 具有相同名稱和類似位置的通用命令會大幅改善探索能力。 請參閱本文稍後的 [標準功能區群組](#standard-ribbon-groups) 。
- **新增群組，** 如果：
  - **其命令具有強烈的關聯，且可由群組標籤正確描述。** 新增群組應該有助於讓命令更容易找到，而不是更困難。

  - **群組有足夠的命令，可證明有更多的外觀。** 適用于大部分群組的3-5 命令。 請避免使用只有1-2 命令的群組，不過，您可以接受群組中沒有任何其他命令的功能區資源庫。 擁有多個具有單一命令的群組，會建議太多的結構或缺少命令一致性。
- 不要藉由新增不需要的群組來進行 **過度整理**。
- 如果有下列情況 **，請考慮分割群組**：
  - ![雜亂無章命令群組的螢幕擷取畫面 ](images/cmd-ribbons-image35.png)<br/>群組有許多不同大小的命令，而且需要組織。
  - ![兩個長段落組名的螢幕擷取畫面 ](images/cmd-ribbons-image33.png)<br>此群組具有可大幅受益于擁有額外標籤的命令。
- **將最常使用的群組放在最顯著的位置，並確定在索引標籤上的群組有邏輯順序。**
- **將群組設計優化，讓使用者能夠快速且安心地找到命令。** 其他所有考慮都是次要。
- **請勿將包含單一按鈕的群組調整為彈出群組圖示。** 相應減少時，請將其保留為單一按鈕。
- **使用最多七個群組。** 如果有7個以上的群組，則會變得更難以判斷哪個群組具有命令。

### <a name="commands"></a>命令

- ![功能區上 [行號] 命令的螢幕擷取畫面 ](images/cmd-ribbons-image36.png)<br/>**藉由公開所有常用的命令，來利用功能區的探索能力和擴充性。** 適當時，請將經常使用的命令從對話方塊移至功能區，特別是已知難以尋找的命令。 在理想情況下，使用者應該能夠在不使用任何對話方塊的情況下執行一般工作。

- **請勿使用功能區的擴充性來證明增加不必要的複雜性。** 由於您可以的話，請繼續進行操作，不要將命令新增至功能區。 讓整體的命令體驗保持簡單。 以下是簡化簡報的方法：
  - ![迷你工具列和內容功能表的螢幕擷取畫面 ](images/cmd-ribbons-image37.png)</br>**使用內容功能表和迷你工具列進行就地、內容相關的命令。**
  - **移動 (或在對話方塊中保留) 很少使用的命令。** 使用對話方塊啟動器來存取這些命令。 您仍然可以使用對話方塊搭配功能區！ 只要嘗試減少在一般工作中使用它們的需求。
  - **消除多餘且不常使用的功能。**

#### <a name="presentation"></a>簡報

- **只有一個索引標籤會顯示每個命令。請避免相同命令的多個路徑，特別是當命令需要多次點擊才能叫用時。** 透過多個路徑尋找命令可能有點方便。 但是請記住，當使用者找到所需的內容時，他們就會停止查看。 如果使用者認為第一個路徑是沒有效率的，則使用者可能會認為第一個找到的路徑是嚴重的問題。 **例外狀況：** 內容索引標籤可能會從 [主資料夾] 和 [插入] 索引標籤複製一些命令，如果這樣做，可避免變更一般內容工作的索引卷
- **在群組中，將命令放在其邏輯順序中，同時提供最常使用命令的喜好設定。** 整體來說，這些命令應該要有邏輯流程，以方便尋找，同時仍會先顯示最常使用的命令。 一般而言，具有32x32 圖元圖示的命令會出現在具有16x16 圖元圖示的命令之前，以協助跨群組進行掃描。
- **避免在常用的命令旁放置破壞性命令。** 如果命令的效果很大，而且無法輕易復原或效果無法立即察覺，則會將其視為具破壞性的命令。
- **使用分隔符號來指出強相關的命令，例如一組互斥選項。**
- ![字型和段落群組的螢幕擷取畫面 ](images/cmd-ribbons-image3.png)<br/> **針對不需要標籤的強相關、已知命令集，請考慮使用工具列樣式群組。** 這樣做可讓您在簡潔的空間中呈現許多命令，而不會影響探索和簡化學習。 這種情況很著名，這類命令經常被使用、立即辨識，因此通常會在 [首頁] 索引標籤上。

- **針對最常使用和重要的標記命令使用32x32 圖元圖示。** 當調整群組時，請讓這些命令成為最後轉換成16x16 圖元圖示的命令。
- **避免任意的命令放置。** 請仔細考慮您的索引標籤和群組設計，以確保使用者不會浪費時間檢查每個索引標籤，以尋找所需的命令。
- **避免以行銷為基礎的放置。** 新功能促銷的行銷目標通常會隨著時間而改變。 請考慮未來的產品版本，以及不斷變動的組織會有多少挫折。

#### <a name="thumbnails-in-galleries"></a>圖庫中的縮圖

選擇可執行此作業的 **最小標準資源庫縮圖大小**。

- 針對功能區元件庫，請使用16x16、48x48 或64x48 圖元的縮圖。
- 針對下拉式清單庫，請使用16x16、32x32、48x48、64x48、72x96、96x72、96x96 或128x128 圖元的縮圖。

### <a name="command-labels"></a>命令標籤

- **標示所有命令。** 擁有明確的文字標籤可協助使用者尋找並瞭解命令。 **例外狀況：** 如果命令的圖示非常知名且空間是 premium，則命令可以是未標記的。 最有可能未標記的命令會出現在 [首頁] 索引標籤上。在此情況下，請將其 Name 屬性指派給適當的文字標籤。 這可讓輔助技術產品（例如螢幕讀者）為使用者提供圖形的其他相關資訊。
- **針對命令按鈕，請使用簡潔易懂的標籤。** 如果可能的話，請使用單一文字;最多四個字。
- **針對下拉式清單，如果清單一律具有值，請使用目前的值做為標籤。**
- ![搜尋通訊錄提示的螢幕擷取畫面 ](images/cmd-ribbons-image67.png)<br/>如果 [可編輯的下拉式清單](/windows/desktop/uxguide/ctrl-drop) 沒有值，請使用 [提示](glossary.md)。
- **不容易理解或不常使用的下拉式清單需要明確的標籤。** 在標籤的結尾加上冒號。
- ![[自動之後：秒] \[ \] ](images/cmd-ribbons-image69.png)<br >的螢幕擷取畫面 **，請使用明確的標籤。** 在標籤的結尾加上冒號。
- **使用句型大寫。** 這麼做會更適合 Windows 的[色調](text-style-tone.md)。
- **使用命令式動詞來開機磁碟區標。** 除非它與索引標籤或組名相同，或者像是顯示、建立、插入或格式化的一般動詞。
- **請勿使用結束標點符號。**
- **為了節省空間，請勿在功能區命令標籤上放入省略號。** 但是，應用程式按鈕和下拉式功能表中的命令會使用省略號。

### <a name="enhanced-tooltip-labels"></a>增強的工具提示標籤

- **使用標題提供命令名稱及其快速鍵（如果適用）。**
- **針對標題，請勿使用結束標點符號。**
- **使用動詞來啟動描述。** 使用描述可協助使用者判斷特定功能是否為他們要尋找的功能。 您應該片語描述，以完成「這是您想要使用的正確功能」這個句子。
- **請將描述保持簡短。** 從點開始。 長文字不鼓勵閱讀。
-   ![[貼上分割] 按鈕與兩個工具提示的螢幕擷取畫面 ](images/cmd-ribbons-image70.png)<br/>**針對分割按鈕，請使用不同的工具提示來說明 [分割] 按鈕功能表。**
- **使用選擇性的補充描述來說明如何使用控制項。** 這項文字可以包含控制項狀態的相關資訊 (包括停用該控制項的原因) 如果控制項本身並未指出狀態。 將此文字保持簡短，並使用說明主題以取得更詳細的說明。
- ![工具提示的螢幕擷取畫面，其中包含 ](images/cmd-ribbons-image71.png)**描述和補充描述的圖形和文字，請使用完整的句子搭配結束標點符號。**
- 使用句型大寫。

## <a name="documentation"></a>文件

參考功能區時：

- 請以功能區、索引標籤、群組和控制項的形式來參考功能區及其元件。 這些字詞不是大寫的。
- 請參閱 [圓形] 按鈕做為應用程式按鈕，並將其包含的功能表做為應用程式功能表。
- 請參閱工具列作為快速存取工具列。
- 依標籤和 [文字] 索引標籤來參考索引標籤。使用確切的標籤文字，包括其大小寫。
- 依標籤來參考命令。 請依工具提示名稱參考未標記的命令。 使用確切的標籤文字，包括其大小寫，但不包含省略號。 請勿包含單字按鈕或命令。
- 若要描述使用者互動，請使用 tab 鍵和控制項的按一下。 使用 [輸入] 可供編輯的下拉式清單。 請勿使用 [選擇]、[選取] 或 [挑選]。
- 請參閱無法使用的專案，而不是暗灰色、已停用或灰色。 在程式設計檔中，請使用 [已停用]。
- 可能的話，請使用粗體文字來格式化標籤。 否則，請只在必要時才將標籤放在引號中，以避免混淆。

範例：

- 在 [ **首頁** ] 索引標籤上，按一下 [ **貼** 上]。
- 在 [ **首頁** ] 索引標籤的 [ **字型** ] 方塊中，輸入 "Segoe UI"。
- 在 [ **審核** ] 索引標籤上，按一下 [ **顯示標記**]，然後按一下 [ **審核者**]。
- 在 [ **格式** ] 索引標籤的 [ **圖片工具**] 中，按一下 [ **壓縮圖片**]。
