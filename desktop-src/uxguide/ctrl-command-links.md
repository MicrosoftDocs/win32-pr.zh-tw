# <a name="command-links"></a>命令連結

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

使用命令連結，使用者可以選取單一回應的單一回應，並藉由執行此動作，移至工作中的下一個步驟。

命令連結具有乾淨、輕量的外觀，可提供描述性標籤，並以標準箭號或自訂圖示顯示，以及選擇性的補充說明。

![一般命令連結對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image1.png)

一組一般的命令連結。

命令連結與 [選項按鈕](ctrl-radio-buttons.md) 類似，因為它們可用來從一組互斥、相關的選項中進行選取。 和選項按鈕一樣，命令連結一律會呈現在集合中，而不會個別顯示。 在外觀中，命令連結具有類似于一般 [連結](ctrl-links.md)的輕量外觀，而不需要框架或其他強大的點擊 [affordance](glossary.md)。 命令連結也類似于 [命令按鈕](ctrl-command-buttons.md)，因為它們可以是預設的「命令按鈕」，而且可以有指派的存取金鑰。 就像 [認可按鈕](glossary.md)一樣，在按一下按鈕時，會關閉對話方塊的視窗 () 或前進到下一個頁面 (讓嚮導和頁面的流程) 。

> [!Note]  
> [連結](ctrl-links.md)和[版面](vis-layout.md)配置的相關指導方針會顯示在不同的文章中。

 

## <a name="is-this-the-right-control"></a>這是正確的控制項嗎？

若要決定使用時機，請考量下列問題：

-   **選項是否會回應主要指示，以及與視窗或頁面的主要用途相關的選項？** 除了流覽至其他頁面之外，使用者是否必須回應以進行其他動作？ 如果沒有，請使用另一個控制項，例如命令按鈕或連結。 命令連結不適用次要或選擇性選項，或單純導覽。

    ![個人化控制台專案的螢幕擷取畫面 ](images/ctrl-command-links-image2.png)

    雖然個人化主控台專案看起來像是使用命令連結，但是這些選項是一般的連結，因為此 [中樞頁面](winenv-ctrl-panels.md) 是用於純導覽。

-   **控制項是否可用來從一組互斥的回應中選擇一個回應？** 如果不是，請使用其他控制項。 若要讓使用者選擇個別的命令，請使用命令按鈕或連結。
-   **在對話方塊中，按一下控制項會關閉視窗嗎？** 如果沒有，請使用不需要關閉視窗的控制項，例如選項按鈕、命令按鈕或連結。

    **不正確：**

    ![[索引標籤式防火牆設定] 對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image3.png)

    無法在屬性視窗或索引標籤式對話方塊中使用命令連結，因為按一下控制項可關閉視窗。

-   **針對 [嚮導] 和 [頁面流程]，在沒有承諾的情況下按一下 [前進到下一頁] 嗎？** 請勿使用命令連結來認可工作;請改用認可按鈕。 因為命令連結看起來像是連結，而使用者將連結與頁面流程中的導覽建立關聯，所以連結不適合用于 [認可頁面](glossary.md) ，因為使用者應該一律能夠返回。
-   **針對嚮導和頁面流程，是否有其他頁面使用命令連結？** 如果是，而且所有其他因素都相等，則偏好使用命令連結以取得頁面之間的一致性。
-   **介於兩到五之間的回應數目？** 永遠不應該有單一的命令連結。 因為命令連結是大型控制項，而且使用的螢幕空間與選項數目成正比，所以請將回應的數目維持在五個以下。 針對六個以上的選項，請使用選項按鈕、一般連結或單一選取 [清單視圖](ctrl-list-views.md)。

    ![對話方塊的螢幕擷取畫面，其中包含命令清單 ](images/ctrl-command-links-image4.png)

    在此範例中，Microsoft Windows 中的 [自動播放] 功能會使用清單視圖。

-   **選項按鈕和認可按鈕的組合是較佳的選擇嗎？** 當下列任何條件成立時，選項按鈕是較佳的選擇：
    -   **有一個強式預設選項，可讓您大部分的使用者選取。** 在使用者習慣按下 [下一步] 接受適當預設值的情況下，使用者比較不可能變更預設選項按鈕，而不是預設的命令連結。 另一方面，如果您想要鼓勵使用者做出明確的選擇，命令連結是比較好的選擇。
    -   **使用者需要與選擇互動 (可能會在進行決策之前查看其他資訊) 。** 例如，選取選項按鈕可能會動態顯示選項的描述。

        ![具有選項按鈕之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image5.png)

        在此範例中，選取選項按鈕會顯示選項的描述。

    -   **頁面上有次要或相關的選項。** 命令連結傾向于將頁面設為市場，讓您輕鬆地忽略其他所有專案。 此外，按一下命令連結之後，就無法選取次要選項。

        **不正確：**

        ![具有混合控制項之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image6.png)

        在此範例中，有兩種不同的方式可回應主要指令。 命令連結不會用於第一個回應，因為很難選取次要選項。

        **正確：**

        ![具有相同控制項之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image7.png)

        在此範例中，選項按鈕可讓回應變成清楚，並允許使用者選取次要選項。

-   **在對話方塊中，一組認可按鈕是否是較佳的選擇？** 當選項需要較長、更清楚的回應和補充說明，但如果有一些簡單的選項，則 [認可] 按鈕群組是較佳的選擇。

    **不正確：**

    ![對話方塊的螢幕擷取畫面，其中包含儲存並不儲存 ](images/ctrl-command-links-image8.png)

    在此範例中，使用簡單命令的命令連結，會讓對話方塊不必要地複雜。

    **正確：**

    ![顯示具有 [儲存]、[不要儲存] 和 [取消] 認可按鈕之對話方塊的螢幕擷取畫面。](images/ctrl-command-links-image9.png)

    在此範例中，使用簡單的認可按鈕是從點開始。

    不過，當使用文字來說明認可按鈕時，簡單易懂的命令連結一律是較佳的選擇。

    **不正確：**

    ![具有不必要文字之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image10.png)

    在此範例中，會使用文字來說明認可按鈕。

    **正確：**

    ![不需要更多文字之標籤的螢幕擷取畫面 ](images/ctrl-command-links-image11.png)

    在此範例中，命令連結很容易理解。

> [!Note]  
> 命令連結需要 Windows Vista 或更新版本，因此不適用於舊版的 Windows。 您可以使用一般連結來取代。

 

![具有圖示和文字的一般連結螢幕擷取畫面 ](images/ctrl-command-links-image12.png)

在此範例中，使用圖示和補充說明的一般連結，會用來取代 Windows XP 中的命令連結。

## <a name="design-concepts"></a>設計概念

這是因為命令連結可讓您使用更具描述性的標籤，而選擇性的補充說明並不表示您應該。 請考慮下列範例：

**不正確：**

![具有太多文字之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image13.png)

此對話方塊會嚴重過度通訊。

此對話方塊會採取簡單的問題，並不必要地使其與命令連結文字的複雜性。 使用者不想要讀取這類簡單問題的所有文字。

我們可以套用三個命令連結指導方針，以簡化此對話方塊：

-   **請勿使用屬於命令連結之冗長重述的補充說明。** 只有當您無法讓命令連結一目了然時，才使用補充說明。 提供一個命令連結的補充說明，並不表示您必須為所有命令提供它們。
-   **選取最安全的 (，以防止資料遺失或系統存取) ，而且最安全的回應是預設值。** 如果安全性和安全性不是因素，請選取最有可能或方便的回應。
-   **提供明確的 [取消] 按鈕。** 請勿將命令連結用於此用途。

藉由套用這些指導方針，我們可以消除不必要的補充說明、將最方便的回應設為預設值，並提供明確的 [取消] 按鈕。

**較佳：**

![具有命令和標籤之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image14.png)

更簡單的命令連結的改進版本。

雖然此版本不會明確說明未儲存的部分會被視為遺失，但少數使用者會根據這項資訊來變更其決策，因此這是很好的取捨。

您可以藉由分析命令連結是否甚至是在此案例中使用的正確控制項，來建立更好的對話方塊。 認可按鈕實際上是較佳的選擇，因為不需要更多的清楚解釋回應。

**最好：**

![具有認可按鈕之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image15.png)

正確的版本會使用認可按鈕來向右取得點。

命令連結有許多優點，但使用 unwisely 時，會導致過度通訊。 如果是對話方塊，請考慮使用認可按鈕，只有在認可按鈕未執行作業時，才使用命令連結。

**適當地使用時，命令連結應可簡化並闡明您的 UI。** 如果結果相反，請回頭回頭看看其他替代方案，並專注于您真正需要傳達的內容。

**如果您只做一件事 ...** 請勿使用命令連結來進行通訊。 命令連結應可簡化並闡明通訊，而不會讓它變得更複雜。

## <a name="guidelines"></a>指導方針

### <a name="interaction"></a>互動

-   **如果按一下命令連結的結果不是瞬間，則顯示忙碌指標。** 如果沒有意見反應，使用者可能會認為按一下未發生，然後再按一次。

### <a name="presentation"></a>簡報

-   **一律在兩個或以上的集合中顯示命令連結。** 邏輯上，沒有理由要求只有一個答案的問題。

    **不正確：**

    ![具有一個命令連結之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image19.png)

    在此範例中，對話方塊會顯示為使用者提供選擇，但只有一個指示。 這應該是參考對話方塊。

-   **先顯示最常使用的命令連結。** 產生的順序大致上應該遵循使用的可能性，但也有邏輯流程。
    -   **例外狀況：** 導致執行所有工作的命令連結應先放置。
-   **提供明確的 [取消] 按鈕。** 請勿將命令連結用於此用途。 使用者通常會發現他們不想要執行工作。 使用命令連結來取消會要求使用者謹慎地讀取所有命令連結，以判斷哪一個表示取消。 具有明確的 [取消] 按鈕，可讓使用者有效率地取消工作。

    **不正確：**

    ![具有 [不要結束] 連結之對話方塊的螢幕擷取畫面 ](images/ctrl-command-links-image20.png)

    在此範例中，[不要結束命令] 連結應該是 [取消] 按鈕。

-   **如果提供明確的 [取消] 按鈕，則會保留單一命令連結，同時提供 [取消] 和 [取消] 按鈕的命令連結。** 這麼做可讓使用者清楚知道使用者有選擇。 以與第一個回應不同的方式來說明此命令連結，而不只是「取消」或某些變化。

    ![兩個連結和 [取消] 按鈕的螢幕擷取畫面 ](images/ctrl-command-links-image21.png)

    在此範例中，第二個命令連結表示使用者有一個選擇，但它的所有工作都是 cancel。 不過，它的片語方式與第一個命令連結不同。

-   **如果您無法將環境回復為先前的狀態，請使用 [關閉] 而不是 [取消]，而不會產生任何副作用。**
-   **不要顯示已停用的命令連結。** 如果命令連結未套用至目前的內容，請改為將它移除。 如果移除所有未套用的命令連結，請將其保留為單一命令連結、消除視窗或頁面，或在需要明確使用者同意的情況下顯示 [確認](mess-confirm.md) 。

### <a name="icons"></a>圖示

-   **所有命令連結都需要圖示。** 圖示可協助使用者區分命令連結與一般連結和使用者介面文字。
-   **只針對命令連結使用箭號圖示。** 一般連結不應使用箭號圖示，除非在 Windows XP 中用來替代命令連結。
-   **使用「安全防護」圖示來指出回應需要立即提高許可權。** 如需使用安全性保護盾圖示的其他指導方針，請參閱 [使用者帳戶控制](winenv-uac.md)。
-   **只有在協助使用者以視覺化方式識別並區分選項時，才使用自訂圖示。** 如果無法立即辨識或有意義，請不要使用自訂圖示。

    **不正確：**

    ![使用自訂圖示的兩個命令連結的螢幕擷取畫面 ](images/ctrl-command-links-image22.png)

    在此範例中，自訂圖示無法立即辨識。

-   **若為自訂圖示，請使用16x16 或32x32 圖元圖示。** 如果有足夠的空間，請使用較大的圖示，這些圖示會從較大的大小以視覺化方式受益。 如果您需要安全性防護罩，請使用32x32 或48x48 圖元圖示。

    ![具有圖示的三個命令連結的螢幕擷取畫面 ](images/ctrl-command-links-image23.png)

    這個範例會使用32x32 圖元的自訂圖示。

    ![具有較大圖示的兩個命令連結的螢幕擷取畫面 ](images/ctrl-command-links-image24.png)

    此範例會使用48x48 圖元自訂圖示，並加上安全性防護罩。

-   **避免混合自訂圖示與視窗或頁面上的標準箭號圖示。** 如果您在介面上使用自訂圖示，請嘗試使用所有自訂圖示。 不過，偏好使用標準箭號圖示來進行無意義的自訂圖示。

### <a name="default-values"></a>預設值

-   **選取最安全的 (，以防止資料遺失或系統存取) ，而且最安全的回應是預設值。** 如果安全性和安全性不是因素，請選取最有可能或方便的回應。
-   **在可行的情況下，請將第一個回應設為預設選項** ，因為使用者通常會預期，除非該順序不是邏輯的。
-   在對話方塊中，除非有簡單的方法可復原動作，否則請 **不要將破壞性動作設為預設的命令連結**。

## <a name="labels"></a>標籤

> [!Note]  
> 因為命令連結會回應主要指令，所以您應該先製作 [良好的主要指示](text-ui.md) ，再決定其回應。

 

**命令連結標籤**

-   **選擇明確通訊的精確標籤，並區分命令連結的用途。** 它應該是一目了然的，且對應至主要指令。 將標籤的焦點放在回應之間的差異。 使用者不需要知道命令連結真正的意義，或與其他命令連結的不同之處。

    **不正確：**

    ![重複命令連結的螢幕擷取畫面 ](images/ctrl-command-links-image26.png)

    在此範例中，第二個和第三個回應之間有何差異？ 你很高興沒有 [取消] 按鈕嗎？

-   **專注命令連結標籤，協助使用者做出正確的決策。** 省略不會影響選擇的詳細資料。 標籤不需要是將會發生之情況的完整規格。
-   **使用動詞命令來啟動命令連結。** 不過，請不要使用 click，因為標籤應該傳達命令連結的功能，而不是它的運作方式。
    -   **例外狀況：** 如果所有命令連結的開頭都是相同的動詞或片語，請刪除多餘的動詞或片語。
-   一般情況下，請 **使用正** 片語 (提供可執行某些動作的選項) 。 負措辭 (如果讓標籤變得更容易瞭解，就可以選擇不執行某些動作) 。
-   **使用平行措辭和單行標籤。** 長標籤不建議讀取，也不需要。 此外，您也可以在檔中更輕鬆地參考中等大小的標籤。
-   **使用句型大寫。**
-   **除非標籤是問題，否則請勿使用結尾標點符號。**
-   **指派唯一的存取金鑰。** 如需指導方針，請參閱 [鍵盤](inter-keyboard.md)。
-   **請勿使用省略號。** 省略號表示執行動作可能需要更多的資訊。 正確使用的命令連結不需要省略號，因為它們具有立即的作用。
-   **如果強烈建議回應，請將「 (建議的) 」新增至標籤。** 請務必新增至標籤，而不是補充說明。
-   **如果回應僅供 advanced 使用者使用，請考慮將「 (advanced) 」新增至標籤。** 請務必新增至標籤，而不是補充說明。

**秘訣：** 您可以藉由想像 friend 指出的主要指示，並以命令連結回應，來評估命令連結。 如果回應命令連結會非自然或麻煩，請修改命令連結，也可能是主要指令。

**補充說明**

-   如果命令連結需要進一步的說明，請 **提供補充說明**。 補充說明說明為何使用者可能會想要選擇回應，或在選擇回應時所發生的情況。

    ![描述選項結果之文字的螢幕擷取畫面 ](images/ctrl-command-links-image27.png)

    在此範例中，補充說明會描述選項的含意。

-   **請勿使用冗長重述命令連結的補充說明。** 只有當您無法讓命令連結一目了然時，才使用補充說明。 提供一個命令連結的補充說明，並不表示您必須提供這些連結。
-   **專注補充說明，協助使用者做出正確的決策。** 省略不會影響選擇的詳細資料。 補充說明不一定是將會發生什麼事的完整規格。
-   **使用平行措辭和最多三行文字。** 詳細的補充說明不一定需要閱讀。
-   **使用完整的句子和結束標點符號。**

**命令連結群組標籤**

-   **請勿使用群組標籤。** 主要指示可作為命令連結的群組標籤。
