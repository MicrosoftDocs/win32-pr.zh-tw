# <a name="animations-and-transitions"></a>動畫和轉換

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

使用動畫和轉換的策略可以讓您的程式更容易瞭解、更順暢、更自然、更高的品質，而且更吸引人。 但是，不使用動畫和轉換的方式可能會讓您的程式雜亂，甚至令人討厭。

動畫會提供動作的外觀或隨時間變化的變更。 使用動畫來提供意見反應、預覽動作效果、顯示物件之間的關聯性、強調變更，或以視覺化方式說明工作。

![反白顯示按鍵的數位鍵台圖 ](images/vis-animations-image1.png)

Microsoft Windows 使用背景 flash 動畫來提供已按一下物件的意見反應。

轉換是在使用者介面 (UI) 狀態變更和物件操作時，用來保持使用者導向的動畫，並讓這些變更感覺很順暢，而不是 jarring。 良好的轉換感覺很自然，通常會提供使用者與真實世界物件互動的假像。

![顯示三種天氣小工具大小的螢幕擷取畫面。](images/vis-animations-image2.png)

Windows 桌面小工具在其簡潔和詳細資料狀態之間使用順暢的轉換。

一般而言，最佳的動畫和轉換會用來與非 verbally 的使用者通訊，並讓狀態變更更自然且更不明顯。 相較之下，最不有效的是，它們不會傳達任何事物，也不會引起不必要的注意。 動畫最適合用來做為次要的通訊形式。 他們應該傳達有用但不重要的資訊，而且可以存取，使用者應該能夠透過其他方式來判斷對等的資訊。

**注意：**[軟體品牌](exper-branding.md)、[音效](vis-sound.md)和 [協助工具](inter-accessibility.md)的相關指導方針會顯示在不同的文章中。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定，請考慮下列問題。

### <a name="animations"></a>動畫

適用下列條件嗎？

-   動畫會以視覺化方式傳達有用的內容，例如提供意見反應、顯示關聯性、原因和效果，或將注意力放在重要的變更。
-   觀看動畫並不重要。 您可以用另一種方式取得對等的資訊。 如果有下列情況，則使用者可能無法從動畫獲益：
    -   它們已關閉動畫。
    -   他們的注意力是在其他地方。
    -   它們在視覺上受損。
    -   動畫會被另一個視窗遮蔽。
    -   因為系統效能不足，所以不會播放動畫。
-   動畫並不會影響使用者的生產力。 任一：
    -   它會快速 (200 毫秒或更少) 。
    -   它不會干擾互動，也不會中斷。
    -   使用者仍然必須等待。
-   動畫不會影響使用者的流程。
    -   這可能是使用者的注意力，也就是在完成工作時很重要或很有用的地方，特別注意。
    -   它很容易忽略，不會造成干擾或討厭。
    -   它不會變成一定乏味。 即使在重複的觀看之後，使用者仍然會覺得適當且更有樂趣。

如果是的話，請考慮使用動畫。

### <a name="transitions"></a>轉換

是物件或場景變更狀態，而且會針對使用動畫以及下列任何條件執行上述所有條件？

-   狀態變更的概念是 disorienting、混淆或難以理解。
-   狀態變更以視覺化方式 jarring、缺乏持續性或平滑或閃爍;或會出現非自然、unpolished 或品質不良的情況，特別是當它牽涉到大型螢幕區域時。
-   使用轉換會讓狀態變更的顯示速度更快。
-   狀態變更值得特別注意。

如果是的話，請考慮使用轉換。

## <a name="design-concepts"></a>設計概念

動畫和轉換是以視覺方式傳達資訊的有效方式，否則需要文字來解釋或可能會錯過使用者。

**不正確：**

**正確：**

![以視覺化方式進行動畫通訊的圖表 ](images/vis-animations-image4.png)

使用動畫會以自然且不顯眼的方式來傳達相同的資訊。 您會看到上千次呢？

**動畫和轉換不需要注意成功。** 事實上，它們通常是用來避免使用者不需要察覺的程式機制。 許多成功的動畫都是自然的，使用者甚至不知道它們;而是使用者只會注意到他們的缺席。 出現的頻率增加了奧妙的需求，因此需要注意的不常發生事件也是真的值得注意的。

![變更為上一頁箭號的所有程式的螢幕擷取畫面 ](images/vis-animations-image5.png)

避免繪製注意力的 [開始] 功能表轉換。

除了讓您的程式更容易瞭解和操作， **設計完善的動畫和轉換是在您的程式中加入個人化、字元和樣式的絕佳方式。** 他們可以提供自然的真實世界風格，讓使用者體驗更有吸引力且更吸引人。

![顯示滑鼠停留如何影響按鈕色彩的圖 ](images/vis-animations-image6.png)

Windows 7 會在滑鼠停留時，根據目前的滑鼠位置和程式圖示色彩來醒目顯示工作列按鈕。 這種方法的視覺效果很吸引人，而且很微妙，傳達 humble 的特質。

**不過，動畫和轉換在有明確的用途時，是最有效且歡迎的。** 它們應該用來改善可用性、平滑度和流程，以及品質的認知，而不會大幅降低效能。

雖然某些類型的動畫是用來繪製使用者的注意力，但請確定注意的公司很好，而且有很好的需要中斷使用者的想法訓練。 人們的眼睛對移動很敏感，尤其是周邊動作。 當有閃爍的工作列按鈕或旋轉通知區域圖示時，使用者可能會很難專心。 **避免使用動畫來中斷或干擾使用者，或特別注意不保證使用者注意的事項。**

**不正確：**

![因沒有原因而醒目提示工作列按鈕的圖 ](images/vis-animations-image7.png)

除非使用者必須立即進行重要的動作，否則程式不應將工作列按鈕閃爍。 在此情況下，使用者唯一需要做的就是啟動程式。

**使用動畫和轉換，因為您的程式需要它們，而不只是因為您可以。** 針對協助工具，請勿使用動畫作為傳遞基本資訊的唯一方法。 請確定使用者可以用不同的方式取得對等的資訊。

### <a name="attributes-of-good-animations-and-transitions"></a>良好動畫和轉換的屬性

良好的動畫和轉換會在這些屬性之間取得適當的平衡：

-   **顯然有目的。** 有個不錯的動畫是因為它們必須是、是否傳達資訊、讓互動感覺真實，或是特別注意一些值得注意的事項。 而且有目的動畫是精確的;如果動畫顯示工作正在執行，這是因為工作已完成。

**不正確：**

![電池圖示和完整收費標籤的螢幕擷取畫面 ](images/vis-animations-image8.png)

在此範例中，動畫顯示已收取完全充電的電池費用。

-   **查看平滑和連續。** 良好的動畫藉由顯示關聯性並提供位置和內容的意義，順暢地移除場景或元素狀態變更之間的接縫。 持續性可協助使用者瞭解他們在哪裡的位置，以及如何取回其來源。

![三個工作列視窗預覽的螢幕擷取畫面 ](images/vis-animations-image9.png)

當使用者從某個程式移至另一個程式時，Windows 7 工作列視窗預覽 morphs 持續性。

-   **是真實的。** 良好的動畫可模擬物件的真實世界實體屬性和行為。 這可協助使用者預測及瞭解其互動的結果。 您不必完全建立真實世界的模型，但是如果您使用實際的動畫，就必須保持與真實世界一致。 使用者應該永遠不會驚訝或使結果混淆，尤其是用於直接操作的動畫。

![拖曳至新位置的工作列按鈕圖 ](images/vis-animations-image10.png)

在此範例中，Windows 7 工作列所使用的「移出」動畫會比靜態插入點更真實。

-   **是真實的。** 即使是在真實世界中找不到的物件，也可能會以不同但相關之物件的真實世界行為為基礎來自然顯示。 只有當關聯性明確地傳達預期用途和行為時，此比喻才適用。

![[已移動] 視窗背後引發效果的螢幕擷取畫面 ](images/vis-animations-image11.png)

在此範例中，Windows 7 所使用的視窗 "squeegee" 會因為它與半透明視窗在真實世界中的運作方式一致，而感到很真實。

-   **使用自然對應。** 自然對應為實體或文化。 例如，以文化特性為基礎的自然對應可能會從以西方文化特性（從左至右讀取）的事實開始。 因此，為了表達物件的時間序列，中間物件是最新的，左邊的物件是過去的物件，而右邊的物件則是未來的物件。 向前復原是由由左至右的移動所表示。

![media player 進度列的螢幕擷取畫面 ](images/vis-animations-image12.png)

在此範例中，Windows Media Player 控制項具有自然對應，因為播放會將位置從左至右移動到右邊。

-   **擁有個人化。** 選擇適當的動畫是在您的程式中加入特性、字元和樣式的絕佳方式。 他們可以讓使用者體驗更有吸引力且更吸引人。 雖然動畫的類型會決定它所傳達的內容，但是執行動畫的特定方式會顯示程式的特質。 良好的動畫可為您的程式（不論是嚴重或比較古怪，或是介於兩者之間的地方）提供正確的特質。

![創造性設計 zune 介面的螢幕擷取畫面 ](images/vis-animations-image13.png)

在此範例中，Zune 使用動畫文字和動態觀點有助於塑造其特質。

-   **外觀和操作迅速。** 良好的動畫不會藉由封鎖使用者的其他互動或強制使用者觀賞，來損害使用者的生產力。 無論您的程式動畫有多自然和吸引人，沒有人想要以獨佔方式等候它們。 良好的動畫也可以迅速回應，而不需要透過以軟登陸快速開始進行 jarring。 回應式動畫也可以快速地傳達其用途。 使用者不需要長時間觀賞動畫，只要找出它的作用或完成的時間。 針對直接操作，回應式動畫對於維護直接且吸引人的真實世界來說非常重要。 若要直接操作，物件的接觸點必須在整個操作中順暢地保持在指標底下。 任何延遲、不穩定的回應或失去的連絡人都會損毀直接操作的認知。

![觸碰觸控式螢幕的指標圖 ](images/vis-animations-image14.png)

在此範例中，觸控移動轉換會藉由將連絡人點保留在使用者的手指下進行操作，來進行回應。

-   **吸引最適當的關注程度。** 良好的動畫通常很微妙，只繪製完成其用途所需的注意。 如此一來，就不會造成困擾、討厭、過於複雜、過於久或重複。 在重複檢視之後，它們不會變成一定乏味。

![檔案名稱的淡化醒目提示螢幕擷取畫面 ](images/vis-animations-image15.png)

在此範例中，Windows 搜尋會暫時將注意力吸引到相符的搜尋文字，然後淡出。

-   **只有在真的特殊時，才會顯示特殊。** 頻率增加了奧妙的需求，因此常見的互動需要簡單的動畫，以簡單的方式傳達簡單的概念。 針對特殊、不常用的體驗保留特殊的複雜動畫。

![四個圓形變成 windows 標誌的螢幕擷取畫面 ](images/vis-animations-image16.png)

在此範例中，Windows 會在啟動時使用醒目提示的動畫，讓體驗變得更為特殊，但在其他地方則會不適當的動畫。

如果移除這些屬性的任何一項，您將會知道您已達到適當的平衡。

### <a name="creating-an-animation-vocabulary"></a>建立動畫詞彙

良好的動畫是關於有效的視覺效果通訊，而且一致性對其有效性而言很重要。 如果您使用特定的轉換，例如將場景從右邊推送至下一個場景，這應該是用於該用途的唯一轉換，而且不應該將轉換用於任何其他用途。 將不同意義指派給相同的動畫，會損害其通訊的能力。 藉由將特定動畫和轉換指派給特定的意義，您將建立動畫詞彙。

此問題適用于具有意義的動畫和轉換，而不是使用者可能指派意義的意義，或要到不其目的的轉換。 例如，淡化之類的動畫和特殊效果（例如溶解）沒有特殊意義，因此可以自由使用。

良好的詞彙會指派動畫來為物件的真實世界、實體行為建立模型。 如果您需要將動畫指派給沒有真實世界的物件或動作，請選擇一個動畫來顯示物件的實際行為。

![滑鼠停留如何讓 windows 標誌亮亮的螢幕擷取畫面 ](images/vis-animations-image17.png)

雖然 [開始] 功能表不是實際的物件，但在啟動時，其停留效果會像真實世界的物件一樣亮起。

詞彙中的每個動畫都必須明確相異。 只有在相關聯的動作同樣相關時，動畫才會具有類似的行為。 例如，移動轉換建議導覽，因此您可以使用不同方向的移動轉換來指出不同類型的導覽。

您會知道當使用者發現結果令人困惑、驚訝或非預期時，動畫和轉換無法正常通訊。 一般來說，最好的方式是讓單一用途比多個目的更好。

在理想的情況下，您的動畫詞彙在需要的程式的所有區域中都應該是完整的。 如果只有少數的互動有自然的動畫，則會將注意力吸引到那些不是的互動。

若要深入瞭解 Windows 動畫詞彙，請參閱本文的 [使用模式](#usage-patterns) 一節。

### <a name="designing-the-right-personality"></a>設計正確的風格

雖然動畫的型別決定了它的通訊，但是動畫執行的特定方式會與程式的特質相關，並強化其品牌。

您的程式特性應該反映其工作的本質和其使用者的特質，因此不是任意選擇。 相反地，設計完善的特質應該很好用;絕對不要嘗試強制執行。 特質應該對使用者建立情緒的連接。 需要考慮的一些因素：

-   工作 **：** 很重要或有趣;選擇性或必填。
-   **結果：** 嚴重或次要。
-   **成本：** 免費或購買;如果購買，則價格低廉或成本高昂。
-   **使用者焦點：** 相對較窄的目標使用者群組，或寬的一般物件。
-   **使用者環境：** Professional、隨便或 home。
-   **使用者年齡：** 年輕或更舊。
-   **使用頻率：** 頻繁或不頻繁。

這些因素的組合有助於為您的程式判斷適當的特質。 以下是常見程式類型的一些適當組合：

**生產力應用程式**

當然，生產力應用程式必須專注于生產力。 雖然有一些特殊的體驗可以用，但大部分其他的動畫都應該具有下列特性：

-   小
-   自然、實際
-   微妙、subdued
-   快速、有效率
-   寬鬆

**公用程式**

公用程式通常會短暫使用，因此使用動畫可能會更積極：

-   真實、說明、一目了然
-   保險箱
-   吸引人

**娛樂，遊戲**

因為這些計畫的目標是要參與和取悅使用者，所以動畫和轉換可能會以下列特性更積極地執行：

-   大量 (可能成為體驗不可或缺的一部分) 
-   人工、surreal
-   具影響力，活躍
-   情緒、有趣、比較古怪
-   精力充沛

建立情緒連線對娛樂程式來說很重要，因為這麼做可接受將某些規則折到最愛的程式。 例如，如果大部分的使用者不太可能使用經常使用的程式，動畫或轉換在百分之一次之後會變成一定乏味。

一般而言，較小、自然、subdued、有效率、但放寬的動畫和轉換是最安全的。 使用這些特性的轉換通常會採用最短的路徑，從一開始到結尾、快速開始、以不超過的方式結束。 此外，設計完善的轉換是設計成能在其將使用的整個距離範圍內順利運作。

### <a name="animation-performance"></a>動畫效能

設計動畫時，請確定它們不會影響使用者有效率地使用您的程式的能力。 一般而言，讓您的動畫速度夠慢以達成其用途，但速度夠快，不會干擾回應性、需求太多，或變成一定乏味。

**不正確：**

![從右至左開啟頁面的圖表 ](images/vis-animations-image18.png)

雖然此頁面會讓動畫有吸引人的真實世界，但它會花更長的時間來減少使用者的生產力。

簡短轉換 (200 毫秒或更少) 是特殊案例 (特別是當它們經常因延遲而無法) 時，是因為使用者會注意到他們必須等待分割的秒數。 如果有下列情況，使用者願意等候這樣的動畫：

-   觀察到的等候相當短暫 (200 毫秒或更少) 。
-   轉換讓互動感覺更流暢且自然。
-   轉換讓互動感覺更具回應性。
-   任何延遲都有助於讓使用者控制互動。

![拖曳至新位置的工作列按鈕圖 ](images/vis-animations-image19.png)

使用者將會接受工作列按鈕重新排列動畫的短暫延遲，因為它非常簡單，而且會讓互動感覺更自然。

動畫可能會對效能造成負面影響的三種方式：速度、回應性和認知。

為了加快速度，某些動畫的視覺效果 veneers 過度耗用 CPU 的工作，因此，您應該做的最後一件事，就是使用 CPU 密集動畫來加快這些工作的速度。 最需要大量 CPU 的動畫 ( 「繁重」動畫) 傾向于：

-   牽涉到許多元素獨立移動。
-   長時間或距離的播放。
-   牽涉到大量的螢幕空間。
-   需要大量的資源。

對效能影響較低的動畫：

-   牽涉到單一物件。
-   播放短時間或距離。
-   牽涉到少量的螢幕空間。
-   不會以數學方式密集。

為了確保效能良好，您應該只針對不需要大量 CPU 的工作使用繁重的動畫，而在任何地方都可以使用淺動畫。

針對回應性，大部分的動畫和轉換都應設計成可讓使用者在動畫執行時進行互動。 除非動畫是處理常式的一部分，否則請讓它獨立于使用者的主要互動，並允許使用者將它中斷。

動畫實際上可能不會對工作的效能造成負面影響，但使用者可能會有其功能的認知。 例如，請勿使用對緩慢、需要大量 CPU 的工作造成繁重的動畫，即使它不會損害效能，因為使用者可能會得出動畫是工作速度很慢的原因。 **如果看起來很慢，則會變得很慢，因此最好使用簡單、輕量又快速的動畫。** 針對需要大量 CPU 的工作，使用 snappy 開頭的動畫會有説明。

如果動畫和轉換的品質很差，就不會有任何價值，因為它們的品質很差，所以不會讓體驗變得更順暢。 為了維持其品質，動畫應設計成在有足夠的系統資源可用時正常地降級。 動畫可能會因需要較少資源的變化而降低 (例如較短的長度或較低的畫面播放速率) ，甚至完全不執行。 無論可用的資源為何，請確定動畫具有高品質，並看起來像是動畫，而不是軟體錯誤。

最後，如果使用者認為您的程式動畫和轉換會降低其生產力，則某些使用者很有可能會想要將它們關閉。 若要支援這項功能，請考慮關閉在 Windows 輕鬆存取中心中找到的 **所有不必要動畫** 的選項。

### <a name="attracting-the-right-level-of-attention"></a>吸引適當的關注等級

雖然只有某些類型的動畫和轉換是特別設計來吸引使用者的注意力，但其設計目的是要吸引適當程度的注意力來滿足其用途。 有哪些不同的方式可以吸引注意力，以及您要如何挑選正確的方法？

**動畫效果**

不同的動畫效果會吸引不同程度的注意。 下列清單摘要說明最常見的方法，從最重視的地方開始：

-   **快速閃爍。** 要求立即關注。 無論發生閃爍的位置為何，都可以中斷使用者的集中。
-   **適中閃爍。** 相同，但不會以較低的頻率要求較少的注意。
-   **彈跳。** 對周邊視覺和同等需求的本質很明顯。 使用者可能會注意到，但只有在持續時間很短時，才能繼續專注于其他位置。
-   **運動。** 在周邊視覺中很明顯，但不具需求。 不過，複雜或立體的動作會比簡單或2-d 的動作更吸引人。 使用者可能會注意到，但可以繼續專注于其他地方。
-   **適中的閃爍。** 很明顯，但不會干擾周邊視覺。 使用者可以繼續專注于其他地方。 可以脈衝亮度、色彩和大小。
-   **緩慢的閃爍/光亮。** 明顯但很微妙。 吸引比靜態效果更多，但使用者可能不會注意到動畫，除非他們已在尋找。
-   **褪色。** 更不明顯。 吸引比靜態效果更多，但使用者可能不會注意到動畫，除非他們已在尋找。
-   **靜態醒目提示/gleam。** 如果使用者選擇查看，但在其他地方不需要注意，則會很明顯。
-   **環境/自然。** 藉由自然的真實世界外觀，特意並不明顯。

若要為您的程式或功能判斷正確的方法，請考慮這些因素與您的功能案例有何關聯。

例如，假設您正在設計立即訊息程式，而有人剛剛傳送訊息給使用者。 這種情況需要使用者注意，在任何地方都應該很明顯，而且使用者通常會想要快速回應。 此案例建議適中閃爍動畫是不錯的選擇。 相反地，假設您想要通知使用者列印工作已完成。 使用者應該能夠繼續專心工作，並在其他地方工作，而且如果使用者未注意到，這是可接受的。 此案例建議您最好選擇「適中到緩慢的閃爍」或「發光」。

**有效期間**

注意動畫的適當持續時間取決於案例和所用動畫的特定類型。 動畫效果的需求愈多，時間就越短。 雖然很微妙的後果 (像是緩慢的閃爍) 可以無限期地播放，但應該只在1到3秒之間播放受要求的影響。 任何較長的風險會讓動畫變得更龐大且討厭。

![反白顯示工作列按鈕的螢幕擷取畫面 ](images/vis-animations-image22.png)

在 Windows 7 中，工作列只會閃爍一秒鐘。 任何時間都很麻煩。

**效果衰減**

您應該根據假設使用者沒有立即回應的假設來設計醒目提示的動畫，原因是他們忙於執行其他動作，而不想中斷。 因此，您的目標應該是在不要求的情況下吸引您的注意力。

若要在不需要時取得適當的平衡，請在一段時間內衰減效果的濃度。 例如，為了吸引您的注意，您可以讓效果一開始變強，但很快就會減緩效果。 如此一來，最吸引人的電源大多是由初始效果決定，但使用者的整體印象大多取決於其完成時間。

![示範減少閃光燈速率的螢幕擷取畫面 ](images/vis-animations-image23.png)

在 Windows 7 中，工作列的閃爍效果會在結束時變慢。

### <a name="what-about-powerpoint"></a>PowerPoint 呢？

Microsoft PowerPoint 轉換通常刻意違反這些指導方針，因為它們的設計目的是要特別注意投影片的轉換，並要求使用者等候。 此外，它們沒有任何特殊意義，因此它們不會在投影片變更的事實之外進行通訊。

PowerPoint 樣式的轉換在正確使用時，具有下列用途：

-   它們會強制展示者暫停，以將長簡報分成較小的區塊。
-   他們吸引觀眾關注簡報的變更，協助人們將他們的思維。
-   他們提供簡報節奏，讓它不會覺得單調或太龐大。
-   其樣式反映了展示者或材質的特質。

雖然這些是簡報的重要目標，但這類轉換會在大部分程式類型的 UI 中繪製不必要的資訊，並會快速一定乏味。

**下一行：** 請勿使用 PowerPoint 樣式轉換做為您程式的模型。

**如果您只進行六件事 .。。**

1.  使用動畫和轉換，讓您的程式更容易瞭解，而且更流暢且更吸引人。 它們應該具有明確的用途。 請勿使用動畫，因為您可以或對程式繪製不必要的注意力。
2.  定義動畫詞彙並一致地在整個程式中使用。 適當時，請使用 Windows 7 動畫詞彙。
3.  您可以使用動畫的特性來為您的程式特性提供個人化，並強化其品牌。
4.  使大部分的動畫變簡單、簡短和微妙。 請記住，動畫不需要注意成功。 如果動畫是適當且自然的，則使用者只會注意到它沒有。
5.  讓您的動畫更快速且迅速回應，並讓他們有輕量的感覺。 無論您的動畫是如何吸引的，沒有人想要在等候它們的情況下。 設計更重的動畫以順利降低效能。
6.  長期執行的設計。 如果動畫很討厭、混亂或一定乏味，請重新設計或移除動畫。

## <a name="guidelines"></a>指導方針

### <a name="effective-communication"></a>有效的通訊

-   **定義和使用動畫詞彙** ，以確保您的動畫和轉換具有一致的意義，並在整個程式中以一致的方式使用。 大部分詞彙都應該包含場景和物件外觀和消失的專案、導覽、基本互動 (滑鼠游標、選取、按一下) 、物件操作和互動 (移動、卸載、調整大小、滾動、移動、縮放、旋轉、篩選) ，以及吸引注意力。 一致的意義對於有效的通訊而言很重要。
-   **只要可行，請使用 Windows 動畫詞彙。** 雖然您的程式可能會有不同的物件和不同的需求，但一致性和熟悉的優點通常會超過不同的優點。 如果您程式的詞彙必須不同，請使用與 Windows 相同的基本動畫類型，但為您的程式提供適當的特質。
-   **請勿將特定意義指派給動畫詞彙中的泛型動畫和轉換。** 像是淡化和特殊效果（例如溶解）的泛型轉換沒有任何特殊意義 (除了出現或消失) 之外，也可以自由使用。

    **不正確：**

    ![其中一個對話方塊漸淡到另一個對話方塊的螢幕擷取畫面 ](images/vis-animations-image43.png)

    在此範例中，交叉淡化不正確地用來流覽至下一個專案。 由於交叉淡化沒有特殊意義，因此這項轉換不會提供內容。

-   **讓詞彙專案清楚相異。** 相關的動作可能會有類似的效果 (例如，放大和縮小應該會有反向轉換) ，但不相關的動作應該會有明顯的不同效果 (例如，縮放不應與旋轉) 混淆。
-   **讓真實世界的效果保持真實且一致。** 如果您使用實際的動畫和轉換，請讓體驗與真實世界保持一致。 使用者應該永遠不會對結果感到驚訝、混淆或誤導。 基於一致性，請不要混用比喻。
-   **為反向動畫提供反向動作。** 這麼做可符合使用者的期望，並簡化詞彙。 例如，如果在中滑動的窗格出現，請將其移除，而不是其他效果。
-   **讓動畫理解。** 使用者應該能夠快速瞭解動畫的用途。 您可以製作太小的動畫、太短 (小於50毫秒) ，或讓使用者無法理解其用途。 在這種情況下，請重新設計以使意義明確，或移除。

    **不正確：**

    ![刪除對話方塊時的動畫屏幕快照 ](images/vis-animations-image44.png)

    在此範例中，這種效果相當小，而且很微妙，少數使用者可以理解其用途。 更妥善地重新設計或移除。

### <a name="patterns"></a>模式

**停留意見反應**

-   **為了讓回應速度變快，請在進入或離開停留狀態的50毫秒內，努力播放動畫。**
-   **若要快速顯示，請將停留動畫的持續時間設為小於50毫秒。**
-   **使用淡入/淡出停留效果。** 這樣做會讓暫止效果與點擊和選取意見反應有明顯的差異。

**按一下 [意見反應]**

-   **若要顯示回應性，請在按下事件的50毫秒內努力播放動畫。** 按一下 [事件不需要按一下意見反應]。
-   **若要快速顯示，請讓點擊動畫的持續時間小於50毫秒。**
-   **使用背景閃爍或閃爍效果。** 這樣做會讓按一下效果明顯有別于暫止和選取意見反應。 因為按一下 [需要滑鼠停留]，所以請按一下 [意見反應]，將暫留的意見反應加入

**選取意見反應**

-   **若要顯示回應，請在選取或 deselection 的50毫秒內努力播放動畫。**
-   **若要快速顯示，請讓選取動畫的持續時間小於50毫秒。**
-   **使用淡入/淡出選取矩形效果。** 這樣做會讓選取專案清楚地有別于滑鼠停留，並按一下 [意見反應]。

**進度意見反應**

-   **當無法在第二個動作中執行動作時，請使用活動指標。** 這麼做會指出已收到命令。
-   **當工作需要五秒以上時，請使用進度列。** 如需詳細的指導方針，請參閱 [進度](progress-bars.md)列。
-   **使用進度回饋動畫來協助使用者將長時間執行之工作的效果視覺化。** 避免不必要的進度回饋動畫如果動畫未傳達任何有用的資訊，請改用進度列。
-   **清楚識別完成和失敗狀態。** 使用者必須能夠快速判斷這些最終狀態。
-   **當基礎工作未進行時，停止顯示進度。** 使用者必須能夠判斷進度是否未進行，並據以回應。

**吸引**

-   **使用 attractors 搭配擋板。** 除非資訊是緊急、重大，或可能影響使用者的立即行為，否則變更狀態 inconspicuously 並讓使用者自行探索變更通常是比較好的做法。 [解決分散注意力，而不](/windows/desktop/uxguide/how-to-design-desktop-ux)是可探索性。

    ![無線狀態圖示的螢幕擷取畫面 ](images/vis-animations-image45.png)

    在此範例中，無線網路通知區域圖示會使用動畫來解決重大問題，但可讓使用者自行探索弱式信號。

-   **選擇繪製正確程度的動畫。** Attractor 動畫應該只是為了滿足其目的而吸引自己的，但不再需要。 如果使用者必須立即採取行動，請選擇在使用者所尋找的地方都需要注意的效果。 在其他情況下，請參閱「 [吸引適當的注意](#attracting-the-right-level-of-attention) 」區段，以取得適當的注意、noticeability 和緊急組合。

    **不正確：**

    ![[曲別針 office 助理] 的螢幕擷取畫面 ](images/vis-animations-image46.png)

    Microsoft Office 助理會吸引自己的不必要注意力。

-   **如果使用者沒有回應，請不要重複動畫或使用連續動畫。** 相反地，假設使用者選擇不立即採取行動，但稍後可能會採取行動。 連續的動畫讓使用者難以專注于其他任何東西。

**關聯性動畫**

-   **使用關聯性動畫來顯示物件的來源或位置。**
-   **關聯性動畫的開頭或結尾必須是選取的物件。** 不要顯示使用者目前不會與之互動的物件之間的關聯性。 如果使用者注意到，他們會注意到的是干擾。

**圖例/預覽**

-   **您可以使用 [預覽] 來顯示命令的效果，而不需要使用者先執行此命令。** 藉由使用有用的預覽，您可以提升效率並輕鬆學習程式，並減少試用和錯誤的需求。
-   **使用具有明確轉譯的圖例和預覽。** 如果混淆，則沒有太大的價值。
-   **一次只播放一個圖例** ，以避免使用者太龐大。 如果有多個同時圖例，請使用滑鼠停留或 [播放] 按鈕讓使用者指出他們的興趣。
-   **如果是視窗或頁面的主要用途，則會自動播放圖例。** 否則，如果是選擇性的，請讓使用者在準備就緒時加以播放。
-   **以最佳速度播放動畫**：不太快，它們很難瞭解，但不太慢，因此監看起來很繁瑣。

**物件成長/縮小**

-   **請勿在調整大小期間剪輯內容。** 在新增內容之前先展開容器。 請在減少容器之前移除內容。

    **不正確：**

    ![截斷計算機的螢幕擷取畫面 ](images/vis-animations-image47.png)

    在此範例中，會在調整大小時裁剪內容。

**內容顯示/隱藏/變更**

-   **以靜態方式顯示重要資訊。** 使用者不需要透過漸進式洩漏來存取重要資訊。

**控制或 affordance 顯示/隱藏**

-   **當使用者將指標放在視窗或窗格中的任意位置，或在全螢幕上移動滑鼠時，顯示重要的控制項。** 使用者不需要搜尋這些控制項，因此請確定其探索。

    ![顯示滑鼠游標如何顯示控制項的圖 ](images/vis-animations-image48.png)

    在此範例中，每當指標在視窗上時，Windows Media Center 就會顯示其控制項。

-   **當使用者將指標放置在命令旁邊或附近時，顯示次要控制項或控制項 affordances。** 為了方便探索，請讓位置變得很明顯，並將目標設為大型。

    ![指標洩漏次要命令的螢幕擷取畫面 ](images/vis-animations-image49.png)

    在此範例中，當指標在右上角附近時，Windows Live Messenger 會顯示次要命令。

**場景轉換**

-   **使實體場景轉換與自然對應一致。** 人們會在西方文化中從左至右讀取，階層式圖表則會從上到下流動。 因此，向前快轉會以由左至右的移動來表示。 下列實體場景轉換具有自然對應：



    | 轉換                          | 意義                                     |
    |---------------------------|--------------------------------------|
    | 從左邊<br/>      | 在工作流程中移回<br/>    |
    | 從右邊<br/>     | 在工作流程中向前移動<br/> |
    | 從最上層<br/>       | 上移工作階層<br/>    |
    | 從底部<br/>    | 下移工作階層<br/>  |



 

-   **如果您的程式播放音效，請同時設計場景轉換和音訊轉換。** 例如，如果場景逐漸淡出，則任何音效也都應該逐漸淡出。 不要因為音效轉換而造成無縫的視覺轉換。 如需更音效的指導方針，請參閱 [音效](vis-sound.md)。

**直接操作**

-   **在互動 (（例如是讓) ）中使用實體手勢時，請將動畫設計成像是對手勢的自然回應。** 連結與轉換效果的互動原因。 提供動畫真實的實體特性，例如加速、減速、動力、阻力、權數、彈跳和旋轉。
-   **若要維持直接操作，請將物件在互動時的連絡人點保持在最順暢的範圍內。** 任何延遲、不穩定的回應或失去的連絡人都會損毀直接操作的認知。 在操作時，物件絕對不能消失。

**排序、篩選或重新排序轉換**

-   **針對簡單的變更，顯示整個轉換。** 使用者將能夠輕鬆地追蹤整個轉換。 簡單的變更包含四個或更少的專案。
-   **針對複雜的變更，請強調動作的結束，因為它會變慢，並讓眼睛填滿其餘部分。** 這麼做可讓運動感覺更具回應性且更有條理。

**效能轉換**

-   **考慮在兩個或三個階段中執行緩慢的轉換，讓它們更快速且互動式地顯示。** 適當時，請使用下列撰寫順序：
    -   外部畫面格
    -   背景
    -   初始內容 (在必要時使用暫時表示) 
    -    (的主要控制項，讓使用者可以立即互動) 
    -   次要控制項和任何剩餘的 UI 元素
    -   最終內容 (是否使用暫時表示) 使用淡化和投影片之類的轉換，讓組合以平滑、整齊、整齊的方式呈現。

![地圖與衛星相片和方格的螢幕擷取畫面 ](images/vis-animations-image50.png)

當您在「鳥瞰」視圖中進行滾動時，Bing 地圖服務會顯示暫時性方格背景。 這麼做可讓使用者在呈現最終內容之前，立即繼續滾動。

**特殊體驗動畫**

-   **重新考慮動畫啟動顯示畫面 (以及靜態啟動顯示畫面) 。** 啟動顯示畫面通常只會注意程式載入的時間長度，而它們很快就會吸引他們的歡迎。 如果啟動顯示畫面只會在使用者互動時才顯示，則每次有更好的替代方式，就是設計您的程式，讓使用者可以立即與其進行互動（即使仍在載入時）。
-   **如果動畫啟動顯示畫面需要三秒以上，請提供略過簡介命令。** 按一下啟動顯示畫面上的任何位置，也應該將其關閉。 或者，在初始期間之後使用動畫的簡短版本。

### <a name="performance"></a>效能

-   **請勿讓使用者等待您的程式動畫和轉換。** 請盡可能使用簡單的動畫和轉換 (小於200毫秒) 。 使用更快速的動畫 (100 毫秒) 以進行更頻繁的作業。 設計較長的動畫 (一秒以上，通常是進度的意見反應、圖例和特殊體驗模式) ，讓使用者在執行時可以繼續運作。
-   **設計長時間執行的動畫，讓使用者清楚能在動畫執行時進行互動。** 如果視覺提示建議他們無法繼續工作，使用者就不會嘗試繼續工作。

    ![狀態列中進度列的螢幕擷取畫面 ](images/vis-animations-image51.png)

    在此範例中，從 Windows Internet Explorer，狀態列中的低索引鍵進度列會建議使用者不必等待完成，就可以進行互動。

-   **使用輕量動畫來執行需要大量 CPU 的工作。** 這樣做可提供工作的完整處理能力。 此外，使用者也不會察覺到輕量動畫是工作耗用 CPU 的原因。
-   **不要在動畫或轉換期間顯示活動指標。** 這麼做將會損毀效果。 設計動畫和轉換，讓它們可以立即啟動。
-   **當系統資源不足時，設計動畫以妥善降級。** 動畫可能會因需要較少資源的變化而降低 (例如較短的長度或較低的畫面播放速率) ，甚至完全不執行。 無論可用的資源為何，請確定動畫具有高品質，並看起來像是動畫，而不是軟體錯誤。

    **不正確：**

    ![桌上型電腦上褪色程式框架的螢幕擷取畫面 ](images/vis-animations-image52.png)

    在此範例中，即使沒有足夠的系統資源可正常播放，還是會使用視窗還原轉換。 因此，凍結的框架似乎是錯誤。 如果沒有可用的資源，最好只在沒有轉換的情況下顯示視窗。

### <a name="animation-characteristics"></a>動畫特性

設計完善的動畫和轉換通常具有下列特性：

-   **短暫的持續時間。** 大部分的動畫應介於100到300毫秒之間，最好是1/6 秒 (167 毫秒) 或1/4 秒 (250 毫秒) 。  (特殊體驗和進度意見反應可能更長。 ) 使用更快速的動畫時間，以進行更頻繁的作業。 一般而言，較長的動畫需要更多時間才能完成、需要更多時間來瞭解，而且感覺很慢。
-   **反應。** 動畫應在起始事件或使用者動作的50毫秒內開始。 較長的開始時間不會有回應。
-   **加速/減速。** 若要查看自然，大部分動畫效果都需要在停止時加速啟動和減速。 若要深入瞭解，請設計動畫以快速開始。 若要顯示為已控制，請設計動畫，以在結尾有軟 landings。 雖然這適用于動畫效果，但也適用于建議移動的任何效果，例如縮放甚至淡化。

    ![圖表顯示一段時間的降低速度 ](images/vis-animations-image53.png)

    大部分的動畫都應該快速開始和虛尾結束，以提供回應式但可控制的感覺。

-   **運動。** 動畫拿動作特別需要加速和減速，因此除非動畫持續時間非常短，否則請勿使用線性動作。 在不 overshooting 的情況下，動作應該從開始到結束時取得短路徑。 不一定需要完整的移動路徑。 適當時，請強調動作的結束，因為它會變慢，並讓眼睛填滿其餘部分。 這麼做可讓運動感覺更具回應性且更有條理。 當您同時為數個物件的動作製作動畫時，請為它們提供稍微不同的路徑，而感覺更自然。
-   **畫面播放速率。** 大部分的動畫都應該使用每秒20個畫面格的畫面播放速率。 如果動畫適用于特殊體驗或與程式的主要用途有關，請考慮使用較高的每秒 24 30 畫面格速率來改善平滑度和真實感。
-   **規模。** 設計動畫，以在其整個範圍的預期使用方式中發揮效果。 例如，頁面轉換應設計成適用于所有頁面大小。
-   **人格。** 設計動畫以自然、subdued 且有效率，而非人工、比較古怪或慢。

### <a name="animated-text"></a>動畫文字

-   雖然您可以使用轉換來顯示文字，但 **不會持續為文字建立動畫。** 動畫文字通常會造成干擾，而且難以讀取靜態文字。 **異常：**
    -   您可以在傳統動畫的情況下建立文字的動畫，並提供可存取的替代方案。
    -   如果文字的目的主要是裝飾性的，您可能會將文字建立動畫。

![創造性設計 zune 介面的螢幕擷取畫面 ](images/vis-animations-image13.png)

在此範例中，Zune 會將文字動畫，但其目的主要是裝飾性。 如果使用者不小心閱讀文字，就不會發生問題。

### <a name="reducing-power-consumption"></a>減少耗電量

-   **設計動畫以減少耗電量。** 當設計正確時，動畫不應大幅增加耗電量。 減少耗電量：
    -   **當顯示器關閉時，停止動畫。** 顯示器可能基於省電的目的而關閉。
    -   **請勿使用不是由使用者啟動的長時間執行動畫。** 使用高解析度定期計時器的動畫可降低處理器電源管理的效率。 此外，當動畫完成時，請務必停用任何高解析度的週期性計時器。
    -   **當系統變成閒置狀態時，暫停所有的動畫。** 要變成閒置的使用者閒置期間，是由主控台中的電源選項所決定。

### <a name="accessibility"></a>協助工具選項

-   **請勿使用動畫作為傳達重要資訊的唯一方法。** 動畫應該傳達有用但不重要的資訊，因為它們無法供視力受損的使用者存取。
-   **請確定可透過其他方式取得對等的資訊，** 例如：

    -   **藉由檢查。** 使用者可以藉由查看與動畫相關的畫面或物件來判斷對等的資訊。
    -   **藉由簡單的互動。** 使用者可以藉由暫留、按一下或按兩下，來判斷對等的資訊。

    ![bing 首頁的螢幕擷取畫面（具有作用點） ](images/vis-animations-image54.png)

    Bing 首頁的初始動畫會顯示數個作用點。 使用者也可以藉由將游標移近上方來顯示作用點。

    請注意，「對等資訊」並不代表相同的資訊。 此資訊可能採用不同的格式，或需要進行簡單的推斷。

-   **適當時，在轉換期間，將物件的輸入焦點變更為已變更。** 這麼做可讓輔助技術偵測變更發生的位置。 但是，當使用者使用鍵盤時，請勿變更輸入焦點。
-   **請勿使用動畫或轉換來快速地閃爍或調整物件大小。** 閃爍和快速的螢幕變更可能會對具有強制障礙的人員和其他 neurological 失常造成問題。
-   **允許使用者關閉程式的動畫和轉換。** 若要支援這項功能，請遵循 Windows 的輕鬆存取中心中的 [關閉所有不必要的動畫] 選項。

    **開發人員：** 您可以使用 SystemParametersInfo API 來判斷是否已啟用動畫。

-   **假設使用者會關閉程式動畫的設計工作。** 請確定這樣做並不會大幅中斷工作流程。

如需更多協助工具的指導方針，請參閱 [協助工具](inter-accessibility.md)。

## <a name="documentation"></a>文件

-   請盡可能避免參考動畫。 相反地，請參考要進行動畫處理的物件，如有必要，請參考動畫的類型。
-   除了技術檔之外，請勿參考轉換。 相反地，請參考其最終或初始狀態中的物件。
-   如果使用者明確起始動畫，請使用動詞播放;否則，請使用適用于技術檔的動詞。

範例：

-   您會知道當專案的圖示開始跳動時，專案需要您的注意。
-   首先，請選取您要列印的相片 (請注意，相片會在選取範圍) 放大。
-   使用交叉淡化轉換，順暢地變更物件的狀態。
