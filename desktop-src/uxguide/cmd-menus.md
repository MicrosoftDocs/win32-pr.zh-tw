# <a name="menus-design-basics"></a>功能表 (設計基本概念) 

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

功能表是命令的階層式清單，或可供使用者在目前內容中使用的選項。

下拉式功能表是按滑鼠按一下或暫留時，視需要顯示的功能表。 它們通常會隱藏在視野之外，因此是節省螢幕空間的有效方式。 子功能表或階層式功能表是從功能表內依需求顯示的次要功能表。 它們是以子功能表標籤結尾的箭號表示。 功能表項目是功能表中的個別命令或選項。

功能表通常會從功能表列顯示，這是標示功能表分類的清單，通常位於視窗頂端附近。 相反地，當使用者以滑鼠右鍵按一下支援內容功能表的物件或視窗區域時，內容功能表就會下降。

![功能表列具有功能表和子功能表的螢幕擷取畫面 ](images/cmd-menus-image1.png)

顯示下拉式功能表和子功能表的一般功能表列。

> [!Note]  
> 與 [命令按鈕](ctrl-command-buttons.md)、 [工具列](cmd-toolbars.md)和 [鍵盤](inter-keyboard.md) 相關的指導方針會顯示在不同的文章中。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用時機，請考量下列問題：

### <a name="menu-bars"></a>功能表列

適用下列條件：

-   視窗是否為主視窗？
-   有許多功能表項目嗎？
-   有許多功能表類別目錄嗎？
-   大部分的功能表項目都適用于整個程式和主視窗嗎？
-   功能表是否需要適用于所有使用者？

如果是的話，請考慮使用功能表列。

### <a name="toolbar-menus"></a>工具列功能表

適用下列條件：

-   視窗是否為主視窗？
-   視窗是否有工具列？
-   有幾個功能表類別目錄嗎？
-   功能表是否需要適用于所有使用者？

如果是的話，請考慮使用工具列功能表，而不是功能表列。

### <a name="tab-menus"></a>索引標籤功能表

適用下列條件：

-   視窗是否為主視窗？
-   視窗是否有索引標籤，其中每個索引標籤會用於一組專用的工作 (而不是使用索引標籤來顯示不同的視圖) ？
-   有一個適用于每個索引標籤的功能表類別目錄嗎？
-   是否有許多命令和選項，但每個索引標籤只會有一小部分？

如果是的話，請考慮使用索引標籤功能表，而不是功能表列。

### <a name="context-menu"></a>捷徑功能表

適用下列條件：

-   是否有一組適用于所選物件或視窗區域的內容相關命令和選項？
-   這些功能表項目是否重複？
-   目標使用者是否熟悉快顯功能表？

如果是的話，請考慮為需要它們的物件和視窗區域提供快顯功能表。

針對以瀏覽器為基礎的程式，工作窗格功能表是更常見的內容命令解決方案。 目前，使用者預期以瀏覽器為基礎的程式中的內容功能表為泛型和無用。

### <a name="task-pane-menu"></a>工作窗格功能表

適用下列條件：

-   視窗是否為主視窗？
-   是否有一組適用于所選物件或程式模式的內容相關命令和選項？
-   有幾個功能表類別目錄嗎？
-   功能表是否需要適用于所有使用者？

如果是的話，請考慮使用工作窗格功能表，而不是操作功能表。

## <a name="design-concepts"></a>設計概念

有效的功能表，可提升絕佳的使用者體驗：

-   使用符合您的程式類型、視窗類型、命令使用方式和目標使用者的命令呈現。
-   妥善地組織，並在適當時使用標準功能表組織。
-   有效使用功能表列、工具列和快顯功能表。
-   有效地使用圖示。
-   有效地使用存取金鑰和快速鍵。

**如果您只做一件事 .。。**

選擇符合您的程式類型、視窗類型、命令使用方式和目標使用者的命令呈現。

## <a name="guidelines"></a>指導方針

### <a name="general"></a>一般

-   **功能表列以外的所有功能表模式都需要下拉式箭號，以指示下拉式功能表是否存在。** 功能表列中的功能表不存在，而是在其他模式下。
-   **請勿動態變更功能表項目名稱。** 這麼做會造成混淆和非預期的情況。 例如，在選取時，請勿將直向模式選項變更為橫向模式。 若為模式，請改為使用 [專案符號和複選](#bullets-and-checkmarks) 標記。
    -   **例外狀況：** 您可以動態變更以物件名稱為基礎的功能表項目名稱。 例如，最近使用的檔案或視窗名稱清單可以是動態的。

### <a name="menu-bars"></a>功能表列

-   **考慮刪除具有三個或更少功能表分類的功能表列。** 如果只有幾個命令，則偏好較輕的替代專案（例如工具列功能表），或更多直接替代專案，例如命令按鈕和連結。
-   **沒有10個以上的功能表類別。** 太多功能表分類太大，因此不容易使用功能表列。
-   如果工具列或直接命令提供大部分使用者所需的幾乎所有命令，**請考慮隱藏功能表列**。 允許使用者使用工具列功能表中的功能表列核取記號選項來顯示或隱藏。

![選取功能表列的選項清單螢幕擷取畫面 ](images/cmd-menus-image9.png)

在此範例中，Windows Internet Explorer 提供功能表列選項。

如需詳細資訊，請參閱 [隱藏功能表列](#hiding-menu-bars)。

### <a name="hiding-menu-bars"></a>隱藏功能表列

一般而言，工具列可與功能表列一起運作，因為兩者都可讓每個使用者專注于其優勢，而不會受到危害。

-   如果您的工具列設計有多餘的功能表列，預設會隱藏功能表列。
-   隱藏功能表列，而不是完全移除功能表列，因為鍵盤使用者更能存取功能表列。
-   若要還原功能表列，請在 [主要工具列] 的 [視圖] (中提供功能表列核取記號選項) 或次要工具列) 功能表類別的 [工具] (。 如需詳細資訊，請參閱 [標準功能表和分割按鈕](cmd-toolbars.md)。

### <a name="menu-categories"></a>功能表分類

-   **選擇 [單一單字名稱] 作為功能表類別目錄。** 使用多個單字讓類別之間的分隔變得令人困惑。
-   **針對建立或查看檔的程式，請使用 [** 檔案]、[編輯]、[視圖]、[工具] 和 [說明] 等標準功能表類別。 這麼做可讓一般功能表項目成為可預測且更容易尋找的專案。
-   針對其他類型的程式，請考慮根據程式的用途以及使用者對其工作和目標的考慮，將 **命令和選項群組織成更有用的自然類別**。 如果您的程式不適合使用標準功能表組織，請不要覺得有義務。
-   **如果您選擇使用非標準的功能表類別目錄，您必須選擇正確的類別名稱。** 如需詳細資訊，請參閱 [標籤](#labels) 區段。
-   **偏好一般分類的工作導向功能表類別。** 工作導向類別可讓您更輕鬆地尋找功能表項目。

![具有翻錄、燒錄和同步的功能表列螢幕擷取畫面 ](images/cmd-menus-image10.png)

在此範例中，Windows Media Player 使用工作導向的功能表類別。

-   **避免只有一或兩個功能表項目的功能表分類。** 如果有的話，請使用子功能表來合併其他功能表類別。
-   **請考慮只在下列情況中，將相同的功能表項目放在多個類別中：**
    -   功能表項目以邏輯方式隸屬于多個功能表類別。
    -   您有資料顯示使用者在單一功能表類別目錄中找不到該專案的問題。
    -   您有多個類別中的一或兩個無人參與的功能表項目。
-   **請勿在多個類別中放入使用相同名稱的不同功能表項目。** 例如，在多個類別中沒有不同的選項功能表項目。
    -   **例外狀況：** 索引標籤功能表模式在每個索引標籤功能表上可能會有不同的選項和 [說明] 功能表項目。

![具有重複功能表項目之索引標籤功能表的螢幕擷取畫面 ](images/cmd-menus-image11.png)

在此範例中，Windows Media Player 會在每個索引標籤功能表上提供選項和 [說明] 功能表項目。

### <a name="menu-item-organization-and-order"></a>功能表項目組織和訂單

-   **將功能表項目組織成具有七個或較少之強相關專案的群組。** 在此情況下，子功能表會計算為父功能表中的單一功能表項目。
-   **請勿在功能表的單一層級中放入超過25個專案** (不會) 的子功能表。
-   **在功能表內的群組之間放置分隔符號。** 分隔符號是橫跨功能表寬度的單一行。
-   **在功能表中，將群組放在其邏輯順序中。** 如果沒有邏輯順序，請先放置最常使用的群組。
-   **在群組中，將專案放在其邏輯順序中。** 如果沒有邏輯順序，請先放置最常使用的專案。 將數值專案 (例如縮放百分比) 以數位順序表示。

### <a name="submenus"></a>子功能表

-   **避免使用不必要的子功能表。** 子功能表需要更多實體精力才能使用，而且通常會讓功能表項目更難找到。
-   **請勿將常用的功能表項目放在子功能表中。** 這麼做會讓這些命令的使用效率不佳。 不過，您可以將常用的命令放在子功能表中，如果通常會直接存取，例如使用工具列。
-   **如果有下列情況，請考慮使用子功能表：**
    -   這麼做會簡化上層功能表，因為它有許多專案 (20 個以上) ，或子功能表是超過七個專案的群組之一。
    -   子功能表中的專案所使用的頻率，會比在父功能表中的專案少。
    -   子功能表會有三個或多個專案。
    -   有三個或多個以相同字組開頭的命令。 在此情況下，請使用該單字作為子功能表標籤。

![具有四個子功能表項目目的 [新增] 功能表的螢幕擷取畫面 ](images/cmd-menus-image12.png)

在此範例中，新的子功能表會取代新的電子郵件訊息、新的新聞訊息、新資料夾和新連絡人的個別命令。

-   **最多有三個層級的功能表使用。** 也就是說，您可以有主要功能表和最多兩層的子功能表。 兩個層級的子功能表應該很罕見。

### <a name="presentation"></a>簡報

-   **停用未套用至目前內容的功能表項目，** 而不是將它們移除。 這樣做會讓功能表列內容穩定且更容易尋找。 **異常：**
    -   針對內容功能表分類，請 **移除，而不要停用未套用至目前內容的內容功能表項目。** 當功能表分類只針對特定模式顯示時（例如選取特定物件類型時），就會顯示為內容。 如需詳細資訊，請參閱移除操作功能表的 [與停](#context-menus) 用指南。
    -   如果決定何時應該停用功能表項目會造成明顯的效能問題，請讓功能表項目保持作用中狀態，並在必要時將其選取專案產生錯誤訊息。

### <a name="tab-menus"></a>索引標籤功能表

-   **每個索引標籤功能表都可以有內容特定的選項和 [說明] 功能表項目。** 這與其他所有功能表模式相反。 每個索引標籤都是用於一組專用的工作，因此任何跨索引標籤功能表的冗余都不會造成混淆。

### <a name="context-menus"></a>操作功能表

-   **只針對內容相關命令和選項使用操作功能表。** 功能表項目只能套用至選取的 (，或) 物件或視窗區域（而不是整個程式）上按一下。
-   **不要讓命令只能透過內容功能表使用。** 如同快速鍵，操作功能表是執行命令及選擇選項的替代方法。 例如，[屬性] 命令也可以透過功能表列或 Alt + Enter 存取金鑰來使用。
-   **提供所有物件和視窗區域的內容功能表，這些物件和視窗區域** 都可受益于一組小型的內容相關命令和選項。 許多使用者定期以滑鼠右鍵按一下，並預期會在任何位置找到內容功能表。
-   **針對以所有使用者為目標的內容功能表，請考慮使用功能表下拉箭號按鈕。** 內容功能表通常適用于以 advanced users 為目標的命令和選項。 不過，您可以在內容功能表是最佳功能表選項，且您需要將目標設為所有使用者的情況下，使用功能表下拉式按鈕。

![具有下拉式功能表按鈕的相片螢幕擷取畫面 ](images/cmd-menus-image7.png)

在此範例中，會使用功能表下拉式按鈕來顯示內容功能表。

**功能表項目組織和訂單**

-   **將功能表項目組織成具有七個或較少之強相關專案的群組。**
-   **避免使用子功能表** ，讓內容功能表保持簡單、直接且有效率。
-   **請勿在內容功能表中放入15個以上的專案。**
-   **在功能表內的群組之間放置分隔符號。** 分隔符號是橫跨功能表寬度的單一行。
-   **使用下列順序來顯示功能表項目：**

### <a name="bullets-and-checkmarks"></a>專案符號和核取記號

-   **選項的功能表項目可以使用專案符號和核取記號。** 命令可能不是。
-   **使用專案符號，從一組較少的互斥選項中選擇一個選項。** 群組中應該一律至少有兩個專案符號。 如需詳細資訊，請參閱 [選項按鈕](ctrl-radio-buttons.md)。
-   **使用核取記號來切換獨立設定的開啟或關閉。** 如果選取和清除的狀態不清楚且明確地相反，請改為使用一組專案符號。 如需詳細資訊，請參閱 [核取方塊](ctrl-check-boxes.md)。
-   **若為混合的核取記號狀態，則顯示沒有核取記號的功能表項目。** 混合狀態用於多個選取範圍，表示已針對部分（而非全部）物件設定選項，因此每個個別物件都有選取或清除的狀態。 混合狀態不會當做個別專案的第三個狀態使用。
-   **在相關的核取記號或專案符號集之間放置分隔符號。** 分隔符號是橫跨功能表寬度的單一行。

-   **針對動態功能表項目 (例如) 最近使用過的檔案，請以數位方式指派存取金鑰。**

-   **讓存取金鑰易於尋找：**
    -   針對最常使用的功能表項目，請在標籤的第一個或第二個單字開頭選擇字元，最好是第一個字元。

-   **偏好使用特殊的輔音或母音，** 例如 "x" In "Exit"。
-   **避免使用使底線難以查看的字元，例如，** 從最有問題的 (到最不有問題的) ：
    -   只有一個圖元寬度的字母，例如 i 和 l。
    -   具有下行的字母，例如 g、j、p、q 和 y。

### <a name="shortcut-keys"></a>快速鍵

-   **指派快速鍵給最常使用的功能表項目。** 不常使用的功能表項目不需要快速鍵，因為使用者可以改用存取金鑰。
-   **不要讓快速鍵成為執行工作的唯一方法。** 使用者也應該可以使用滑鼠或鍵盤搭配 Tab 鍵、箭號和便捷鍵。
-   **針對已知的快速鍵，請使用標準指派。**
-   **請勿將不同意義指派給已知快速鍵。** 因為它們是記下，所以已知快速鍵的不一致意義很令人沮喪，而且容易出錯。 如需 Windows 程式所使用之知名快速鍵的詳細說明，請參閱 Windows 鍵盤快捷方式機碼。
-   **請勿嘗試指派全系統的程式快速鍵。** 只有當您的程式具有輸入焦點時，程式的快速鍵才會生效。
-   **記錄所有快速鍵。** 這麼做可協助使用者瞭解快速鍵的指派。
    -   **例外狀況：** 不要在內容功能表中顯示快速鍵指派。 內容功能表不會顯示快捷方式索引鍵指派，因為它們已針對效率進行優化。
-   **針對非標準的金鑰指派：**
    -   **選擇沒有標準指派的快速鍵。** 絕不重新指派標準快速鍵。
    -   **以一致的方式在整個程式中使用非標準的金鑰指派。** 請勿在不同的視窗中指派不同的意義。
    -   **可能的話，請選擇 [助憶鍵指派]，** 特別是在常用的命令。
    -   **針對具有小規模效果的命令** （例如適用于所選物件的命令），使用函式金鑰。 例如，F2 會重新命名選取的專案。
    -   針對具有大規模效果的命令（例如適用于整份檔的命令 **），使用 Ctrl 鍵組合**。 例如，Ctrl + S 會儲存目前的檔。
    -   **針對擴充或補充標準快速鍵之動作的命令，請使用 Shift 鍵組合。** 例如，Alt + Tab 快速鍵會透過開啟的主視窗進行迴圈，而 Alt + Shift + Tab 鍵則會以反向順序進行迴圈。 同樣地，F1 會顯示說明，而 Shift + F1 則會顯示即時線上說明。
    -   **請勿使用下列字元作為快速鍵：** @ $ {} \[ \] \\  ~  \| ^ '  < >。 這些字元需要跨語言的不同按鍵組合，或是地區設定專屬的。
    -   **請勿使用 Ctrl + Alt 組合，** 因為 Windows 會將某些語言版本中的此組合解讀為 AltGR 索引鍵，以產生英數位元。
-   **如果您的程式指派許多快速鍵，請提供自訂指派的能力。** 這麼做可讓使用者重新指派衝突的快速鍵，並從其他產品遷移。 大部分的程式都不會指派足夠的快速鍵來需要這項功能。

如需更多指導方針和標準快速鍵指派，請參閱 [鍵盤](inter-keyboard.md)。

### <a name="using-ellipses"></a>使用省略號

雖然功能表命令可用於立即動作，但可能需要更多的資訊來執行此動作。 **指出需要額外資訊的命令 (包括確認) ，方法是在標籤結尾加上省略號。**

![[列印命令和列印] 對話方塊的螢幕擷取畫面 ](images/cmd-menus-image15.png)

在此範例中，Print .。。命令會顯示 [列印] 對話方塊，以收集詳細資訊。

**適當地使用省略號很重要，這表示使用者可以在執行動作之前做出進一步的選擇，或甚至完全取消動作。** 省略號提供的視覺提示可讓使用者探索您的軟體，而不需要擔心。

這並不表示每當動作只在執行動作需要額外的資訊時，才會 **顯示另一個視窗時，您應該使用省略號**。 例如，[關於]、[Advanced]、[說明]、[選項]、[屬性] 和 [設定] 的命令必須在按下時顯示另一個視窗，但不需要使用者提供其他資訊。 因此不需要省略號。

**如果有不明確的 (例如，命令標籤缺少動詞) ，請根據最可能的使用者動作來決定。** 如果單純地查看視窗是常見的動作，請不要使用省略號。

**正確：**

其他色彩 .。。

版本資訊

在第一個範例中，使用者很可能會選擇色彩，因此使用省略號是正確的。 在第二個範例中，使用者很可能會看到版本資訊，因此不需要省略號。

> [!Note]  
> 判斷功能表命令是否需要省略號時，請不要使用將 [許可權提升](winenv-uac.md) 為因素的需求。

 

提高許可權不是執行命令 (所需的資訊，而是用於許可權) ，而提高許可權的需求則會以安全性保護盾表示。

-   若為非標準的分類名稱，請 **使用可清楚且準確地描述功能表內容的單一特定單字。** 雖然這些名稱不一定要用來描述功能表中的所有專案，但它們應該是可預測的，如此一來，使用者在功能表中找到的內容就不會驚訝。

-   **使用特定的動詞命令。** 避免泛型、無用動詞，例如變更和管理。
-   **針對適用于單一物件的命令使用單數名詞**，否則請使用複數名詞。

-   可能的話，請使用粗體文字來格式化標籤。 否則，請只在必要時才將標籤放在引號中，以避免混淆。
