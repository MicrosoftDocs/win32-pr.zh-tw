# <a name="ux-checklist-for-desktop-applications"></a>桌面應用程式的 UX 檢查清單

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

以下是 UX 指南中一些重要指導方針的集合。 您可以使用此選項作為檢查清單，以確保您的程式使用者介面能正確取得這些重要專案。

## <a name="windows"></a>Windows

-   **支援最小的 Windows 有效解析度（800x600 圖元）。** 對於必須在安全模式下運作的重要使用者介面 (Ui) ，支援 [有效](glossary.md) 的640x480 圖元解析度。 請務必考慮工作列所使用的空間，其方式是針對顯示在工作列上的 windows 保留48垂直 [相對圖元](glossary.md) 。
-   **優化可調整大小的視窗版面配置，以有效解決1024圖元。** 以仍可正常運作的方式，自動調整這些視窗的大小以取得較低的螢幕解析度。
-   **請務必以每英寸96點測試您的 windows (DPI)  (為800x600 圖元) 、120 DPI (1024x768 圖元) ，以及 144 DPI (1200x900 圖元) 模式。** 檢查是否有版面配置問題，例如剪輯控制項、文字和視窗的裁剪，以及縮放圖示和點陣圖。
-   **針對具有觸控和行動使用案例的程式，請針對 120 DPI 進行優化。** 高 DPI 螢幕目前在觸控和行動電腦上很普遍。
-   **如果視窗是擁有的視窗，一開始會將它「置中」顯示在擁有者視窗的頂端。** 絕不會在下方。 針對後續顯示，請考慮將它顯示在其最後一個位置 (相對於擁有者視窗) 如果這麼做，可能會比較方便。
-   **如果視窗是內容相關的，請一律將它顯示在其啟始物件附近。** 不過，請將它放在 (最好的位移和右邊的) ，如此視窗才不會涵蓋物件。

## <a name="layout"></a>Layout

-   **調整視窗內的控制項和窗格，以符合其一般內容。** 避免截斷的文字及其關聯的省略號。 使用者永遠都不需要與視窗互動來查看其一般內容—保留調整大小，並針對異常大型內容進行滾動。 特別檢查：
    -   **控制項大小。** 將控制項大小調整為一般內容，如有必要，讓控制項更寬、更高或多行。 調整控制項的大小，以消除或減少具有大量可用空間的視窗滾動。 此外，應該永遠不會有很多的標籤，或 windows 中有足夠可用空間的截斷文字。 不過，若要讓文字更容易閱讀，請考慮將行寬度限制為65個字元。
    -   **資料行寬度。** 請確定清單檢視列具有適當的預設值、最小值和最大大小。 選擇 [清單視圖] 預設資料行寬度（不會產生截斷的文字），特別是在清單視圖中有可用的空間時。
    -   **版面配置餘額。** 視窗的版面配置應該大致平衡。 如果配置的外觀很大，請考慮讓控制項更寬，然後將一些控制項移到右側。
    -   **版面配置調整大小。** 當視窗可調整大小且資料遭到截斷時，請確定較大的視窗大小會顯示更多資料。 當資料遭截斷時，使用者會預期要調整大小的視窗，以顯示詳細資訊。
-   **設定最小的視窗大小（如果大小低於此值，內容就無法再使用）。** 針對可調整大小的控制項，請將可調整大小下限的元素大小設定為其最小的功能大小，例如清單視圖中的最小功能性

## <a name="text"></a>Text

-   **您可以使用一般的對話詞彙。** 專注于使用者目標，而不是技術。 如果您要說明複雜的技術概念或動作，這項功能就特別有效。 想像一下，您想要查看使用者的肩，並說明如何完成工作。
-   **有禮貌的、可獲得的和鼓勵。** 使用者絕對不應該覺得 condescended、應該過度或 intimidated。
-   **移除多餘的文字。** 在視窗標題、主要指示、補充指示、內容區域、命令連結和認可按鈕中尋找重複的文字。 一般而言，請將全文檢索保留在主要指示和互動式控制項中，並從其他地方移除任何多餘的內容。
-   **針對標題使用標題樣式的大小寫，並針對所有其他 UI 元素使用句子樣式的大小寫。** 這麼做會更適合 Windows 的聲音。
    -   **例外狀況：** 針對繼承應用程式，如果有必要，您可以針對命令按鈕、功能表和資料行標題使用標題樣式的大小寫，以避免混用大小寫樣式。
-   **對於功能和技術名稱而言，請務必保守。** 一般來說， (使用標題樣式的大小寫) ，則只能使用大寫的主要元件。
-   **若為功能和技術名稱，請在大寫中保持一致。** 如果名稱在 UI 畫面上出現一次以上，則應該一律以相同的方式來顯示。 同樣地，在程式的所有 UI 畫面中，名稱應該一致地呈現。
-   **不要將一般使用者介面專案的名稱大寫，** 例如工具列、功能表、捲軸、按鈕和圖示。
    -   **例外狀況：** Address bar、Links bar、功能區。
-   **所有大寫字母都不要使用鍵盤按鍵。** 相反地，請遵循標準鍵盤所使用的大小寫，或者，如果鍵盤上未標示按鍵，則為小寫。
-   **省略號表示不完整性。** 使用 UI 文字中的省略號，如下所示：
    -   **命令。** 指出命令需要額外的資訊。 當動作顯示另一個視窗時，請不要使用省略號，只有在需要額外的資訊時才使用。 隱含動詞用來顯示其他視窗的命令不會採用省略號，例如 Advanced、Help、Options、Properties 或 Settings。
    -   **資料。** 指出文字已被截斷。
    -   **標籤。** 指出工作正在進行中 (例如「正在搜尋 ...」 ) 。

**秘訣：** 在具有未使用空間的視窗或頁面中截斷文字表示版面配置不良，或預設視窗大小太小。 致力於消除或減少截斷文字數量的版面配置和預設視窗大小。 如需詳細資訊，請參閱[配置](vis-layout.md)。

-   **請勿使用不是連結的藍色文字，因為使用者可能會假設它是連結。** 使用粗體或灰色陰影，您也可以使用彩色文字。
-   **請謹慎使用粗體，以吸引使用者必須閱讀的文字。**
-   **您可以使用主要指示，精確地說明使用者在指定視窗或頁面中應執行的動作。** 良好的主要指示傳達使用者的目標，而不是只著重在操作 UI。
-   **以命令式方向或特定問題的形式表達主要指令。**
-   **請勿在控制項標籤的結尾或主要指示之間放置句號。**
-   **在句子之間使用一個空格。** 不是二。

## <a name="controls"></a>控制項

-   **一般**
    -   **為每個控制項或一組控制項加上標籤。異常：**
        -   您可以使用提示來標記文字方塊和下拉式清單。
        -   次級控制項使用其相關聯控制項的標籤。 微調控制項一律為從屬控制項。
    -   **針對所有控制項，請選取最安全的 (，以防止遺失資料或系統存取) ，預設為最安全的值。** 如果安全性和安全性不是因素，請選取最有可能或方便的值。
    -   **偏好受限制的控制項。** 盡可能使用受限制的控制項（例如清單和滑杆），而不是不受限制的控制項（例如文字方塊），以減少文字輸入的需要。
    -   **重新考慮停用的控制項。** 停用的控制項可能很難使用，因為使用者實際上必須推算停用的原因。 當使用者預期要套用控制項時，請停用它，他們可以輕鬆地推算控制項停用的原因。 如果沒有任何方法可讓使用者啟用該控制項，或不希望它套用，或讓它保持啟用狀態，但在使用錯誤訊息時提供錯誤訊息，請將它移除。
        -   **秘訣：** 如果您不確定是否應該停用控制項或提供錯誤訊息，請從撰寫您可能提供的錯誤訊息開始。 如果錯誤訊息包含以使用者無法快速推算的實用資訊，請讓控制項保持啟用並提供錯誤。 否則，請停用控制項。
-   **命令按鈕**
    -   **偏好泛型標籤上的特定標籤。** 在理想的情況下，使用者不需要讀取任何其他人來瞭解標籤。 使用者比靜態文字更可能讀取命令按鈕標籤。
        -   **例外狀況：** 如果取消的意義明確，請不要重新命名 [取消] 按鈕。 使用者不應必須讀取所有按鈕，以判斷哪一個按鈕取消動作。 但是，如果不清楚正在取消的動作（例如，有數個暫止動作時），請重新命名 [取消]。
    -   **提出問題時，請使用符合問題的標籤。** 例如，提供 [是]，且沒有任何回應為 [是] 或 [否]。
    -   **請勿在不是屬性工作表或控制台專案的對話方塊中使用 [套用] 按鈕。** [套用] 按鈕表示套用暫止的變更，但讓視窗保持開啟。 這麼做可讓使用者在關閉視窗之前評估變更。 不過，只有屬性工作表和 [控制台] 專案具有這項需求。
    -   如果取消將環境傳回至先前的狀態，請將按鈕標示為取消， (不會有副作用) ;否則，請將按鈕標示為關閉 (如果作業已完成) ，或在作業進行中時停止 () 表示它會將目前變更的狀態保持不變。
-   **命令連結**
    -   **一律在兩個或以上的集合中顯示命令連結。** 邏輯上，沒有理由要求只有一個答案的問題。
    -   **提供明確的 [取消] 按鈕。** 請勿將命令連結用於此用途。 使用者通常會發現他們不想要執行工作。 使用命令連結來取消會要求使用者謹慎地讀取所有命令連結，以判斷哪一種方法取消。 具有明確的 [取消] 按鈕，可讓使用者有效率地取消工作。
    -   **如果提供明確的 [取消] 按鈕，則會保留單一命令連結，同時提供 [取消] 和 [取消] 按鈕的命令連結。** 這麼做可讓使用者清楚知道使用者有選擇。 以與第一個回應不同的方式來說明此命令連結，而不只是「取消」或某些變化。
-   **[不要再顯示 \<item\> ] 核取方塊**
    -   **\<item\>如果沒有更好的替代方案，請考慮使用 [不要再顯示] 選項來允許使用者隱藏週期性對話方塊。** 最好的方式是在使用者真的需要時顯示對話方塊，或直接將它排除（如果沒有的話）。
    -   **取代 \<item\> 為特定的專案。** 例如，不要再顯示此提醒。 在一般情況下參考對話方塊時，請使用 [不要再顯示此訊息]。
    -   藉由在選項下新增下列句子，**清楚地指出使用者輸入將用於未來預設值**：未來將會使用您的選取專案。
    -   **預設不會選取此選項。如果對話方塊真的只應顯示一次，請在不詢問的情況下進行。** 請勿使用這個選項來干擾使用者，請確定預設行為並不討厭。
    -   **如果使用者選取選項，然後按一下 [取消]，此選項就會生效。** 這項設定是中繼選項，因此它不會遵循不留下副作用的標準取消行為。 請注意，如果使用者不想要在未來看到對話方塊，很有可能會想要取消它。
-   **連結**
    -   **請勿指派**[存取金鑰](glossary.md)。 使用 Tab 鍵存取連結。
    -   **請勿在連結文字中新增「按一下」或「按一下這裡」。** 這並不是必要的，因為連結表示按一下。
-   **工具提示**
    -   **使用工具提示來提供未標記控制項的標籤。** 您不需要為加上標籤的控制項提供工具提示，只是為了達成一致性。
    -   **如果這樣做很有用，工具提示也可以為標示的工具列按鈕提供更多詳細資料。** 請勿只重複或提供標籤中已有內容的冗長重述。
    -   **避免涵蓋使用者即將查看或互動的物件。** 一律將提示放置在物件的側邊，即使這需要在指標與提示之間區隔。 只要物件與其秘訣之間的關聯性很清楚，某些分隔就不會有問題。
        -   **例外狀況：** 清單和樹狀目錄中使用的完整名稱工具提示。
    -   **若為專案集合，請避免涵蓋使用者可能會看到或互動的下一個物件。** 針對水準排列的專案，請避免將秘訣放在右邊;若為垂直排列的專案，請避免在下方放置提示。
-   **漸進式揭露**
    -   **使用 [更多]/[漸進式洩漏] 按鈕來隱藏使用者通常不需要的 advanced 或罕見選項、命令和詳細資料。** 請勿隱藏常用的專案，因為使用者可能找不到這些專案。 但請確定任何隱藏的值都有值。
    -   如果介面一律顯示一些選項、命令或詳細資料，請使用下列標籤配對：
        -   **更多/較少的選項。** 用於選項或混合使用選項、命令和詳細資料。
        -   **更多/較少的命令。** 僅用於命令。
        -   **更多/較少的詳細資料。** 僅供資訊使用。
        -   **更多/較少 \<object name\> 。** 針對其他物件類型（例如資料夾）使用。
    -   否則：
        -   **顯示/隱藏選項。** 用於選項或混合使用選項、命令和詳細資料。
        -   **顯示/隱藏命令。** 僅用於命令。
        -   **顯示/隱藏詳細資料。** 僅供資訊使用。
        -   **顯示/隱藏 \<object name\> 。** 針對其他物件類型（例如資料夾）使用。
-   **進度列**
    -   **針對需要限制時間的作業使用確定進度列，** 即使該時間量無法準確預測也一樣。 不定的進度列會顯示正在進行的進度，但不提供任何其他資訊。 請勿只根據可能缺少的精確度來選擇不定的進度列。
    -   **如果您可以正確地完成，請提供剩餘的預估時間。** 精確度較高的估計值會很有用，但從標示或跳動開始的估計值卻很有用。 您可能需要先執行一些處理，才能提供精確的估計值。 如果是這種情況，請不要在此初始期間顯示可能不准確的估計值。
    -   **請勿重新開機進度。** 如果進度列在重新開機時遺失其值 (可能是因為作業中的步驟完成) ，因為使用者無法得知作業何時會完成。 相反地，請讓作業中的所有步驟共用進度的一部分，並讓進度列移至完成一次。
    -   **提供有用的進度詳細資料。** 提供其他的進度資訊，但僅限使用者可以使用它來執行某些動作。 請確定文字顯示的時間夠長，讓使用者能夠讀取該文字。
    -   **請勿將進度列與忙碌指標合併。** 請使用其中一種，但不能同時使用兩者。
-   **提示**
    -   **當螢幕空間位於不需要使用標籤或指示的 premium （例如工具列）時，請使用提示** 。
    -   **提示的主要目的是要以精簡的方式識別文字方塊或下拉式方塊的用途。** 使用控制項時，它不一定是使用者需要查看的重要資訊。
    -   **提示文字不能與真正的文字混淆。** 若要這樣做：
        -   以斜體灰色繪製提示文字，並以羅馬黑色呈現實際的輸入文字。
        -   當使用者在文字方塊中按一下或 tab 鍵時，提示文字應該無法編輯，而且應該會消失。
            -   **例外狀況：** 如果文字方塊具有預設輸入焦點，則會顯示提示，並在使用者開始鍵入之後消失。
    -   請勿使用結尾標點符號或省略號。
-   **通知**
    -   **使用與目前使用者活動無關的事件通知，不需要立即使用者動作，使用者可以自由地忽略。**
    -   **不要濫用通知：**
        -   **只有在需要時，才使用通知。** 當您顯示通知時，可能會中斷使用者或甚至討厭使用者。 請確定中斷的對齊。
        -   **針對不需要立即使用者動作的非重大事件或情況，請使用通知。** 對於需要立即執行使用者動作的重大事件或情況，請使用替代的 UI 元素 (例如) 的強制回應對話方塊。
        -   **請勿使用功能公告的通知！**

## <a name="keyboard"></a>鍵盤

-   **將初始輸入焦點指派給使用者最有可能與第** 一個互動式控制項互動的控制項。 如果第一個互動式控制項不是理想的選擇，請考慮變更視窗的版面配置。
-   **將定位停駐點指派給所有互動式控制項，包括唯讀編輯方塊。異常：**
    -   以單一控制項（例如選項按鈕）行為的相關控制項群組。 這類群組會有一個定位停駐點。
    -   適當地包含群組，讓箭號按鍵在群組內向前和向後迴圈，並保留在群組內。
-   **定位順序應從左至右、從上到下流動。** 一般而言，定位順序應遵循讀取順序。 請考慮將一般使用的控制項放在定位順序中，藉此建立例外狀況。 索引標籤應該在兩個方向的所有索引標籤都不會停止的情況下迴圈。 在群組中，定位順序應以順序排列，而不會發生例外狀況。
-   在 **索引標籤停止內，箭號按鍵順序應從左至右流動**，而不會發生例外狀況。 箭號按鍵應該會迴圈執行雙向的所有專案，而不會停止。
-   **依下列順序顯示認可按鈕：**
    -   確定/ \[ \] /Yes
    -   \[不要 \] /No
    -   取消
    -   套用 (（如果有的話）) 

\[它是什麼 \] ，而不 \[ \] 是做為主要指示的特定回應。

-   **請勿將便捷鍵與快速鍵混淆。** 雖然這兩個存取金鑰和快速鍵都提供鍵盤存取權，但它們具有不同的用途和指導方針。
-   **請盡可能將唯一的存取金鑰指派給所有互動式控制項或其標籤。** [唯讀文字方塊](ctrl-text-boxes.md) 是互動式控制項 (因為使用者可以將文字滾動並複製) ，因此可從存取金鑰獲益。 **請勿將存取金鑰指派給：**
    -   [確定]、[取消] 和 [關閉] 按鈕。 輸入和 Esc 可用於其存取金鑰。 不過，請一律將便捷鍵指派給表示 [確定] 或 [取消]，但具有不同標籤的控制項。
-   **指派快速鍵給最常使用的命令。** 不常使用的程式和功能不需要快速鍵，因為使用者可以改用存取金鑰。
-   **不要讓快速鍵成為執行工作的唯一方法。** 使用者也應該可以使用滑鼠或鍵盤搭配 Tab 鍵、箭號和便捷鍵。
-   **請勿將不同意義指派給已知快速鍵。** 因為它們是記下，所以已知快速鍵的不一致意義很令人沮喪，而且容易出錯。
-   **請勿嘗試指派全系統的程式快速鍵。** 只有當您的程式具有輸入焦點時，程式的快速鍵才會生效。

## <a name="mouse"></a>滑鼠

-   **永遠不需要使用者按一下物件，以判斷是否可按一下物件。** 使用者必須能夠單獨判斷 clickability 的視覺化檢查。
    -   主要 UI (（例如認可按鈕）) 必須有靜態按一下 affordance。 使用者不應該停留以探索主要 UI。
    -   次要 UI (（例如次要命令或漸進式洩漏控制項）) 可以在滑鼠停留時顯示其 click affordance。
    -   文字連結應以靜態方式建議連結文字，然後在滑鼠停留時顯示其 click affordance (底線或其他簡報變更，並將指標) 。
    -   圖形連結只會在滑鼠停留時顯示手形指標。
-   **使用手形 (或 [連結選取] ) 指標僅適用于文字和圖形連結。** 否則，使用者必須按一下物件，以判斷它們是否為連結。

## <a name="dialog-boxes"></a>對話方塊

-   **強制回應對話方塊需要互動，因此請將其用於使用者必須回應的專案，才能繼續工作。** 請確定中斷的排列，例如需要完成的重要或不頻繁的一次性工作。 否則，請考慮非模式替代專案。
-   **非強制回應對話方塊不需要互動，因此當使用者需要在對話方塊與擁有者視窗之間切換時，請使用這些對話方塊。** 它們最適合用於經常、重複或進行中的工作。 不過，功能區、工具列和選擇區視窗通常是較佳的替代方案。

## <a name="property-sheets"></a>屬性工作表

-   **請確定屬性是必要的。** 請勿使用不必要的屬性來雜亂屬性頁，以避免進行硬性設計決策。
-   **以使用者目標為依據，而不是以技術呈現屬性。** 因為屬性會設定特定的技術，並不表示您必須根據該技術來呈現該屬性。
    -   如果您必須以技術來呈現設定 (可能是因為您的使用者辨識技術的名稱) ，請包含使用者權益的簡短描述。
-   **使用有意義的特定索引標籤標籤。** 避免可套用至任何索引標籤的一般索引標籤標籤，例如 [一般]、[Advanced] 或 [設定]。
-   **避免一般頁面。** 您不需要有 [一般] 頁面。 只有在下列情況時，才使用一般頁面：
    -   這些屬性適用于數個工作，對大部分的使用者而言都有意義。 請勿在 [一般] 頁面上放置特製化或 advanced 屬性，但您可以透過 [一般] 頁面上的 [命令] 按鈕來存取它們。
    -   屬性不符合更明確的類別。 如果有的話，請改為使用該頁面的名稱。
-   **避免 [Advanced] 頁面。** 只有在下列情況時，才使用 Advanced 頁面：
    -   這些屬性會套用至不常見的工作，而且主要是對 advanced 使用者有意義的。
    -   屬性不符合更明確的類別。 如果有的話，請改為使用該頁面的名稱。
-   **如果屬性視窗只有一個索引標籤，而且無法擴充，請不要使用索引標籤。** 請改用 [確定]、[取消] 和 [選擇性套用] 按鈕的一般對話方塊。 不過，可由協力廠商擴充的可延伸屬性 windows () 必須使用索引標籤。

## <a name="wizards"></a>精靈

-   **先考慮輕量替代方案，例如對話方塊、工作窗格或單一頁面。** 這是很大的 UI，最適合用於多步驟、不常執行的工作。 您不需要使用 [使用嚮導]，您可以在任何 UI 中提供有用的資訊和協助。
-   **只有在前進到下一個頁面而不需承諾用量時，才使用下一個。** 當您按一下 [上一步] 或 [取消] 時，前進到下一個頁面會被視為承諾量。
-   **請使用 Back 來更正錯誤。** 除了更正錯誤，使用者不需要按 [上一步]，就能在工作中進行進度。
-   **當使用者認可工作時，請使用對主要指示的特定回應的 [認可] 按鈕 (例如，[列印]、[連接] 或 [啟動]) 。** 請勿使用一般標籤（如下 (所示），這不表示承諾用量) 或完成 (不是認可工作的特定) 。 這些認可按鈕上的標籤應該有意義。 一律使用動詞來啟動認可按鈕標籤。 **異常：**
    -   當特定回應仍為一般時，請使用 [完成]，例如 [儲存]、[選取]、[選擇] 或 [取得]。
    -   使用 [完成] 來變更特定設定或設定集合。
-   **只針對選項使用命令連結，而非承諾用量。** 特定的認可按鈕表示承諾用量優於 wizard 中的命令連結。
-   **使用命令連結時，隱藏 [下一步] 按鈕，但保留 [取消] 按鈕。**
-   **針對 Follow-Up 和完成頁面使用 Close。** 請勿使用 [取消]，因為關閉視窗並不會放棄目前為止所做的任何變更或動作。 請勿使用完成，因為它不是命令式動詞。
-   **請勿在 [wizard 名稱] 中使用 [wizard]。** 例如，使用 [連線到網路]，而不是 [網路設定向導]。 不過，您可以將 [瀏覽器] 稱為 [嚮導]。 例如：「如果您是第一次設定網路，可以使用 [連線到網路嚮導] 取得協助。
-   **透過導覽保留使用者的選取專案。** 例如，如果使用者進行變更，請按 [上一步]，然後再按一下這些變更。 除非使用者明確選擇清除變更，否則不會預期需要重新輸入變更。

## <a name="wizard-pages"></a>精靈頁面

-   **專注于有效率的決策制定。** 減少要專注于 essentials 的頁面數目。 合併相關的頁面，並從主要流程取出選用的頁面。 讓使用者透過您的 wizard 完全按下 [下一步] 看起來似乎是很好的體驗，但是如果使用者永遠不需要變更預設值，可能就不需要這些頁面。
-   **請勿使用 [歡迎使用] 頁面，請盡可能讓第一頁運作。** 只有在下列情況時，才使用選擇性的開始使用頁面：
    -   Wizard 具有成功完成嚮導所需的必要條件。
    -   使用者可能無法根據其第一個選擇頁面瞭解嚮導的目的，而且沒有空間可供進一步說明。
    -   開始使用頁面的主要指示是「在您開始之前：」。
-   **在要求使用者進行選擇的頁面上，針對最有可能的情況進行優化。** 這些類型的頁面應該會顯示實際的選項，而不只是指示。
    -   如果您未使用開始使用頁面，請在第一頁選擇的頂端說明 wizard 的用途。
-   **當使用者認可工作時，請使用 Commit 頁面讓它清楚。** [認可] 頁面通常是最後一頁的選項，而 [下一步] 按鈕會重新標記以指出正在認可的工作。
    -   請勿使用僅摘要使用者先前的選取專案的摘要頁面，除非工作具有風險 (涉及安全性，或遺失時間或金錢) 或使用者可能不了解其選取專案，且需要加以檢查。
-   **請勿使用不執行任何動作的恭喜頁面，而是結束 wizard。** 如果使用者清楚看得到的結果，只要在最終的 [認可] 按鈕上關閉嚮導即可。
    -   當有使用者可能會做後續處理的相關工作時，請使用 Follow-Up 頁面。 避免常見的後續工作，例如「傳送電子郵件訊息」。
    -   只有當結果不可見，而且沒有更好的方法可以提供工作完成的意見反應時，才使用完成頁面。
    -   具有進度頁面的嚮導必須使用完成頁面或 Follow-Up 頁面來指出工作完成。 針對長時間執行的工作，請在 [認可] 頁面上關閉嚮導，並使用通知來提供意見反應。

## <a name="error-messages"></a>錯誤訊息

-   **當使用者不可能執行動作或變更其行為做為訊息的結果時，請不要提供錯誤訊息** 。 如果沒有使用者可以採取的動作，或如果問題不重要，請隱藏錯誤訊息。
-   **請盡可能提出解決方案，讓使用者可以修正問題。** 不過，請確定建議的解決方案可能會解決此問題。 不要藉由建議可能但未必的解決方案，來浪費使用者的時間。
-   **是特定的。** 避免模糊的用語，例如語法錯誤和不合法的作業。 提供相關物件的特定名稱、位置和值。
-   **請勿使用意思暗示指責使用者或隱含使用者錯誤的片語。** 避免在片語中使用您和。 當使用中的語音通常是慣用的時候，請在使用者為主旨時使用被動聲音，如果使用的是使用中的聲音，可能會覺得應該過度錯誤。
-   **請勿針對錯誤訊息使用 [確定]。** 使用者不會看到錯誤，因為沒問題。 如果錯誤訊息沒有直接動作，請改用 Close。
-   **請勿使用下列文字：**
    -   錯誤，失敗 (改為使用問題) 
    -   無法 (使用無法改為) 
    -   不合法、無效、錯誤 (使用不正確或無效) 
    -   Abort、kill、terminate (使用 stop 改) 
    -   嚴重的嚴重 (請改用嚴重的) 

這些條款是不必要的，與 Windows 的鼓勵語氣相反。 相反 [地，如果正確使用](vis-std-icons.md)錯誤圖示，就會充分傳達發生問題。

-   **不要伴隨著音效效果的錯誤訊息。** 這麼做是 jarring 且不必要的。

## <a name="warning-messages"></a>警告訊息

-   **警告描述可能會在未來發生問題的狀況。** 警告不是錯誤或問題，因此請勿將常式問題以警告的形式加以述。
-   **當使用者不可能執行動作或變更其行為做為訊息的結果時，請勿提供警告訊息。** 如果沒有使用者可以採取的動作，或如果條件不重要，則隱藏警告訊息。

## <a name="confirmations"></a>確認

-   **請勿使用不必要的確認。** 只有在下列情況才使用確認：
    -   有很明確的原因不是繼續進行，而且有時候使用者可能不會這麼做。
    -   動作有顯著的後果或無法輕易復原。
    -   動作會造成使用者可能不知道的後果。
    -   繼續進行動作需要使用者選擇沒有適當的預設值。
    -   由於目前的內容，使用者可能已執行錯誤的動作。
-   **片語確認為是或沒有問題，並提供是或無答案。** 與其他類型的對話方塊不同的是，確認是設計來防止使用者做出快速的決策。 如果使用者未考慮到其回應，確認沒有任何值。

## <a name="icons"></a>圖示

-   **所有圖示都應該遵守** [Aero 樣式的圖示指導方針](vis-icons.md)。 取代所有 Windows XP 樣式的圖示。
-   **根據其訊息類型選擇標準圖示，而不是基礎問題的嚴重性：**
    -   **錯誤。** 發生的錯誤或問題。
    -   **警告。** 可能會在未來發生問題的狀況。
    -   **資訊。** 有用的資訊。

如果問題結合了不同的訊息類型，請將焦點放在使用者需要採取行動的最重要層面。

-   **圖示必須一律符合主要指示或其他對應的文字。**
-   **非嚴重的使用者輸入問題不需要錯誤圖示。** 不過，就地錯誤需要圖示，因為這類內容相關的意見反應可能太容易忽略。
-   **請勿使用警告圖示來「柔化」非重大錯誤。** 錯誤不是警告;請改為套用錯誤圖示的指導方針。
-   **針對問題對話方塊，請只針對具有顯著後果的問題使用警告圖示。** 請勿使用警告圖示進行例行問題。

## <a name="help"></a>Help

-   **連結到特定的相關說明主題。** 請勿連結至說明首頁、目錄、搜尋結果清單，或只連結至其他頁面的頁面。 請避免連結到結構化為大型常見問題清單的頁面，因為它會強制使用者搜尋符合他們所選連結的頁面。 請勿連結至與手邊無關的特定說明主題。 永遠不會連結到空白頁面。
-   **請勿將說明連結放在每個視窗或頁面上以達到一致性。** 在一個地方提供說明連結並不表示您必須在所有地方提供它們。
-   **如果可能的話，片語說明會根據說明內容回答的主要問題來連結文字。** 請勿使用「深入瞭解」或「取得此措辭的協助」。
-   **使用連結文字的整個說明連結，** 而不只是關鍵字。
-   **使用完整句子。**
-   **請勿使用結尾標點符號或省略號，但問號除外。**
-   **如果說明內容已上線，請在連結文字中將其清除。** 這樣做有助於讓連結的結果成為可預測的結果。
