# <a name="help"></a>Help

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

使用「說明」做為次要機制，以協助使用者完成及更瞭解主要機制成為 UI 本身的工作。 套用這些指導方針，讓內容真正實用且容易找到。

說明系統是由各種類型的內容所組成，其設計目的是要協助使用者無法完成工作、想要更詳細地瞭解概念，或需要比 UI 提供更多的技術詳細資料。

在本文中，我們將說明做為 UI 的次要資料庫。 UI 是主要的，因為這是使用者第一次嘗試解決其問題的地方。 只有在無法使用 UI 完成工作時，他們才會洽詢說明系統。

![[windows 說明及支援] 頁面的螢幕擷取畫面 ](images/winenv-help-image1.png)

[Windows 說明及支援] 首頁，可從 [開始] 功能表取得。

**注意：** 與 [樣式和語氣](text-style-tone.md) 相關的指導方針會在個別的文章中顯示。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用時機，請考量下列問題：

-   **您的目標使用者的動機為何？** 更具動機的是探索您程式的功能，並成為中繼或甚至是資深的使用者，而他們越願意透過諮詢說明主題來研究問題的答案。
-   **您是否使用說明來修正不正確的 UI？** 您的 UI 愈好，越少的使用者就會尋求其他協助。 如果您的程式有很清楚、實用的主要 UI (（例如，不含術語的錯誤訊息、正確撰寫的嚮導，以及明確的對話方塊）) ，您可能完全不需要次要說明系統。
-   **您的程式相當簡單嗎？** 如果是的話，請考慮將所有必要的協助內容併入您的主要 UI 介面中。 使用者更可能在執行複雜工作的程式中尋求其他協助。
-   **您的應用程式適用于開發人員、IT 專業人員或其他軟體專家嗎？** 這些使用者傾向于提供程式設計語言慣例的參考說明，以及熟悉功能的深入概念說明。

## <a name="design-concepts"></a>設計概念

如果您決定在程式中包含說明，請將它整合到您的整體設計中。 說明介面應該簡單、有效率且相關;它應該可以讓使用者輕鬆地取得協助，然後返回其工作。 以使用者的時間來思考您的說明系統：先將基本協助直接併入您的 UI，然後在更詳細的說明中建立清楚且一致的進入點，以最小化中斷。

Windows 協助是根據這些原則所設計。 以下是 Windows 說明使用者體驗的一些設計變更：

-   有更多可探索的進入點可協助您從主要 UI (特別是 UI 介面（例如對話方塊、錯誤訊息和) 的次要資料庫）的新說明連結。 說明連結會帶您直接前往 [說明] 中的相關主題。
-   在大部分主控台中樞頁面的右上角，以及 shell 資料夾都有提供 [說明] 按鈕圖示。
-   使用者可以在線上時，選擇取得 Windows 線上說明及支援的最新說明內容。
-   說明主題現在是以工作為基礎，而不是以功能為主，讓使用者可以快速且有效率地完成其工作。
-   說明主題現在主要是以已知的最上層使用者案例為基礎。
-   使用真實世界的語言，說明主題具有更寬鬆且非正式的 [語氣](text-style-tone.md)。
-   說明主題的設計目的是為了有效掃描，因為使用者很少會讀取單字的單字內容。

### <a name="an-analogy-for-help"></a>說明的類比

若要更深入地瞭解如何設計您的說明系統，請考慮與日常生活的對比。 您會在不熟悉的城市中遺失。 您該怎麼辦？ 許多會回應如下：

-   取得導向;尋找地標、街道號 (名稱和指向) 位置的指標。
-   尋找地圖。
-   最後，做為最後的手段，請要求指示或打電話給 friend。

城市的「介面」設計會影響您的協助需求。 有條理的街道、明確方向 (醫院、機場、博物館和郵局) 的指標，以及清楚的地理功能或大樓等清楚的地標，可協助您找到您的方式。

您會在最後手段時要求協助。 這表示城市的「介面」因設計不佳而造成混淆。 您較有可能在具有特定標籤的位置中要求協助，以建議效益。 例如，您比較有可能在標示為「方向」或「資訊」的位置，在像城市廳一樣的地方提出協助，即使是城市廳的任何人都能提供指引。

當您尋求協助時，很可能會感到挫折，而只想要取得您想要的目的地。 您可能不打算花時間來導覽城市或學習其歷程記錄。 此外，您的動機取決於工作的重要性。 如果您嘗試找出旅館的房間，您將會採取任何動作。 但是，如果您的目標是要找出稍微重要的一點，您很可能只會在適當的努力之後放棄。

在真正的空間中找出您的所有層面，都對應到使用者在您的程式虛擬空間中的一般方式。 在主要 UI 之外搜尋說明的本質是 disorienting;您最好透過設計完善的 UI 和智慧型的「街道符號」來緩和這類體驗，以將使用者導向所需的答案。

### <a name="designing-ui-so-that-help-is-unnecessary"></a>設計 UI 讓您不需要協助

請先嘗試以下列方式提供不必要的協助：

-   讓您輕鬆探索和執行一般工作。
-   提供明確的 [主要指示](text-ui.md)。
-   提供明確、精確的控制項標籤，其為目標和工作導向。
-   在需要時提供補充指示和說明。
-   使用受限制為有效選擇的控制項、提供適當的預設值、處理所有輸入格式，以及防止錯誤，來預測肇因問題。
-   撰寫錯誤訊息，為使用者提供清楚的解決方案或動作。
-   避免混淆的 UI 設計，例如流程較差的工作，或使用已停用的控制項，因此沒有明顯的原因。
-   及早在開發週期中使用寫入器和編輯器，在整個程式中建立高品質且一致的 [UI 文字](text-ui.md) 。

使用者應該不需要移到其他地方來瞭解如何使用您的 UI。 將基本資訊直接新增至主要 UI，而不是強迫使用者立即內容和 [說明] 窗格中的使用者。 如果重要資訊只存在於說明主題中，使用者很有可能不會看到它。 如需選擇性且更詳細的資訊，請使用主要 UI 的 [說明] 連結，以取得補充協助的相關說明主題。

### <a name="considering-user-motivation"></a>考慮使用者動機

針對大部分的使用者，速度和效率都是良好程式的最重要優點。 使用者想要完成工作。 一般而言，他們對於學習程式和技術並不感興趣，其耐心等候的時間只是因為該計畫提供自己的興趣，並解決手邊的問題。

將您的說明系統設計成符合使用者的動機。 例如，假設有一個 strolled 的使用者，最多可達一間的 kiosk。 如果她無法快速地執行此工作，則可能只是放棄並離開。 她不太可能花時間使用說明。 或者，高度動機的使用者對於研究您的說明系統以取得答案所花的時間會有更高的承受度。 例如，必須平衡書籍的商務使用者可能願意查閱說明內容，以充分利用這個新的會計應用程式。

### <a name="writing-content-for-scanning"></a>寫入內容以進行掃描

撰寫說明主題，瞭解這些主題將會掃描是否有特定的資訊，而非單字的讀取字。 簡潔地撰寫、快速地取得點數，並提供使用者可採取行動的資訊。

-   以一致的格式撰寫使用編號步驟的「使用說明」主題，讓使用者辨識他們正在取得程式性協助。
-   撰寫參考主題，並使用資料表（例如）來呈現 UI 選項或語言語法，以方便掃描。
-   撰寫以子標題邏輯方式組織的概念主題，讓使用者可以略過較不感興趣的整個區段。

在所有說明內容中，您可以更輕鬆地掃描專案符號清單，而非文字的標準段落區塊;但請謹慎使用專案符號清單，而不是未經組織材質的 crutch。

### <a name="creating-content-that-matters"></a>建立重要內容

假設沒有任何協助系統可預期每個使用者可能有的每個問題，請將大部分的內容放在最重要的案例中，以供您的目標使用者使用。 例如，在其他工作中，有效搜尋及如何建立網路連線 () 可能會有高度搜尋的主題。 此外，請將焦點放在您最常使用的使用者案例中，而不是為自己的工作記錄一項功能或技術。

**秘訣：** 技術支援是協助內容的絕佳來源。 技術支援人員通常會記錄有關使用者嘗試 (和無法完成) 的特定程式或工作的常見問題記錄。

不需要為 UI 中的每個功能提供協助。 **無用可協助您建立所有專案的說明。** 如果 UI 的設計是妥善的，則這些說明主題大部分都不會有説明。他們只會重新聲明明顯的。

如果有一種以上的方法可以執行工作，則在大部分情況下，您可以只記錄缺乏經驗的使用者所使用的最常見方式。 這種情況的例外狀況包括協助工具考慮 (記載滑鼠動作的鍵盤對應專案，例如) 和平臺考慮 (針對 tablet 外型規格進行記錄，例如，或針對命令列可以用來取代圖形使用者介面) 的伺服器環境考慮。

請記住，使用者通常不會覺得它們所遇到的問題與您一樣。 例如，使用者可能會發現，將本身視為「帳戶」很奇怪。 請務必設計您的搜尋和編制索引功能，然後再考慮可能的術語變化和同義字。

不過，在主要 UI 和說明系統之間，如果不相同，則詞彙應該非常類似。 當說明系統語言與其在螢幕上看到的內容不緊密關聯時，使用者可能會感到困惑。

### <a name="writing-compelling-help-link-text"></a>撰寫吸引人的說明連結文字

當您從主要 UI 連結到說明主題時，請務必撰寫吸引人的說明連結文字。 清除和特定語言激勵信賴度。 使用者傾向于將一般說明連結 (具有「說明」一字或「深入瞭解」的按鈕) 將不會產生正確的資訊，而不需要投入大量的時間。

**如果您只進行五件事 .。。**

1.  設計您的 UI，讓使用者不需要協助。
2.  藉由將主要問題的內容放在目標使用者的最上層案例中，讓您的協助更加實用。
3.  顯示說明內容，以便輕鬆地進行掃描。
4.  瞭解您不需要為 UI 中的每一項功能提供協助。
5.  讓協助進入點可供探索且吸引人。

## <a name="guidelines"></a>指導方針

### <a name="entry-points"></a>進入點

-   **連結到特定的相關說明主題。** 請勿連結至說明首頁、目錄、搜尋結果清單，或只連結至其他頁面的頁面。 請避免連結到結構化為大型常見問題清單的頁面，因為它會強制使用者搜尋符合他們所選連結的頁面。 請勿連結至與手邊無關的特定說明主題。 永遠不會連結到空白頁面。
-   **請勿將說明連結放在每個視窗或頁面上以達到一致性。** 在一個地方提供說明連結並不表示您必須在所有地方提供它們。
-   **使用對話方塊、錯誤訊息、嚮導和屬性工作表的 [說明] 連結。** 如果說明連結適用于特定控制項，請將其放在其下靠左對齊。 如果 [說明] 連結套用至整個視窗，請將它放在視窗內容區域的左下角。

    ![具有群組方塊的屬性工作表螢幕擷取畫面 ](images/winenv-help-image5.png)

    在此範例中，第二個說明連結會套用至控制項群組。

    ![[屬性工作表] 和 [說明] 連結的螢幕擷取畫面 ](images/winenv-help-image6.png)

    在此範例中，[說明] 連結會套用至整個視窗。

-   **您可以使用 [說明] 連結，而不是一般文字參考，以便在技術上可行時使用**

    **正確：**

    如何修復磁片錯誤？

    **不正確：**

    如需修復磁片錯誤的詳細資訊，請參閱說明及支援。

-   **針對 [控制台] 專案的中樞頁面，使用 [說明] 按鈕和 [說明] 圖示。** 將它放置在右上角。 這些按鈕沒有標籤，但是具有可讀取說明的工具提示。

    ![具有 [說明] 按鈕之控制台專案的螢幕擷取畫面 ](images/winenv-help-image7.png)

    具有 [說明] 按鈕的 [控制台] 專案。

-   **F1 說明是選擇性的。** 使用者習慣于透過按 F1 鍵（在標準鍵盤上標示為「說明」）來尋找與螢幕上 UI 立即內容相關的說明資訊。 如果您的使用方式研究顯示使用者預期會找到它，或您的程式 UI 很複雜，可受益于內容相關的協助，您可以包含 F1 說明。
-   **具有功能表列的程式可以有 [說明] 功能表分類。** 如需說明功能表的指導方針，請參閱 [功能表](cmd-menus.md)。

    ![從功能表列存取 [說明] 的螢幕擷取畫面 ](images/winenv-help-image8.png)

    在此範例中，Windows 繪圖配件有一個 [說明] 功能表類別。

-   **針對鍵盤協助工具，提供 [說明] 按鈕和連結的定位停駐點。**
-   [說明] 按鈕和連結行為應如下所示： [說明] 窗格隨即開啟，並顯示專用的說明主題。叫用 [說明] 窗格的 UI 應該保持開啟狀態，以保留內容相關體驗。
-   **請勿使用下列過時的說明進入點樣式：「深入瞭解」或「深入瞭解 ...」標題列上的連結、一般說明按鈕，以及即時線上說明按鈕。** 雖然過去曾使用過這些功能，但可用性研究已判定使用者通常會忽略這些專案。 請改用特定說明主題的連結。 如需撰寫良好說明連結的指導方針，請參閱說明 [連結](#text)。

    **不正確：**

    ![具有 [深入瞭解] 連結之對話方塊的螢幕擷取畫面 ](images/winenv-help-image9.png)

    請勿使用「深入瞭解」或「深入瞭解 ...」連結。

    **不正確：**

    請勿使用標題列上的即時線上說明按鈕。

### <a name="content"></a>Content

-   **請勿建立明顯的內容。** 重複主要 UI 中的 [說明] 主題不會增加價值。
-   **請勿建立使用者無法以某種方式採取行動的內容。**
    -   **例外狀況：** 有些概念內容會提供重要的背景資訊，而不一定會導致使用者的動作。
-   **避免發生問題的模糊解決問題。** 例如，「洽詢您的系統管理員」或「重新安裝應用程式」通常會讓使用者感到不快。
    -   **例外狀況：** 如果這是唯一實用的解決方案，建議您聯繫系統管理員，系統管理員則會預期會與問題聯繫。
-   **避免可解決極不太可能使用者案例的內容。** 針對您預期的一般使用方式，開發您的主要說明內容。請注意預期使用方式的重要例外狀況，但會將此內容視為較低的優先順序。
-   **收集您的使用者意見反應，瞭解您的說明主題有多有用。** 允許使用者對個別主題進行評分。 在您的檔上進行 [可用性研究](glossary.md) ，以確定涉及品質和內容可搜尋性的問題。
    -   **秘訣：** 使用者意見反應也是產生更多以工作為基礎之內容的好方法，著重于使用者實際使用您的程式，而不是以功能為基礎的內容，只著重于技術的描述。
-   **提供多種方式來存取您的內容。** 目錄、索引和 [搜尋](ctrl-search-boxes.md) 機制是改善探索能力的三種最常見方法。
-   **如果有一種以上的方法可以執行工作，則在大部分情況下，您可以只記錄缺乏經驗的使用者所使用的最常見方式。**

## <a name="text"></a>Text

**說明連結**

-   提供與說明主題內容相關的特定資訊，視需要使用盡可能精確的文字。 使用者通常會忽略一般說明連結。 請確定連結的結果是可預測的使用者，結果不應驚訝。

    -   **例外狀況：** 您可以使用 [詳細資訊] 來補充直接在 UI 中的指示，特別是如果在 [說明] 連結中提供特定資訊會導致不必要的重複，或讓連結較不吸引人。

    **不正確：**

    強式密碼具有至少六個混合的大寫字母、數位及符號。 什麼是強式密碼？

    **正確：**

    強式密碼具有至少六個混合的大寫字母、數位及符號。 詳細資訊

    在不正確的範例中，說明連結是重複的。 它會詢問已回答的問題。

-   在可能的情況下，請在說明內容所回答的主要問題方面，片語說明連結文字。 請勿使用「深入瞭解」、「告訴我更多資訊」或「取得此措辭的協助」。

    **不正確：**

    深入瞭解如何新增例外狀況

    **正確：**

    允許例外狀況的風險有哪些？

    如何? 新增例外狀況？

    在正確的範例中，連結的片語是說明主題所回答的主要問題。

-   如果您可以簡潔地摘要最相關的資訊，請將摘要直接放在 UI 中，而不是使用說明連結。 不過，您可以使用 [說明] 連結來提供補充資訊。

    **不正確：**

    ![[強式密碼] 連結的螢幕擷取畫面？ ](images/winenv-help-image14.png)

    **正確：**

    ![有關密碼的補充文字螢幕擷取畫面 ](images/winenv-help-image15.png)

    **較佳：**

    ![具有詳細資訊連結的文字螢幕擷取畫面 ](images/winenv-help-image16.png)

    正確的範例會簡潔地摘要說明資訊，大幅改善使用者將讀取此資訊的可能性。 更好的範例會提供 [說明] 連結，以取得此複雜主題的詳細資訊。

-   清楚指出協助的片語說明連結。 說明連結絕對不能像動作連結一樣閱讀。
-   使用連結文字的整個說明連結，而不只是關鍵字。

    **正確：**

    允許例外狀況的風險有哪些？

    **不正確：**

    允許例外狀況的風險有哪些？

    在正確的範例中，連結文字會使用整個說明連結句子。

    -   **例外狀況：** 外部網站的 [說明] 連結應該只使用網站或頁面的名稱做為連結。 任何導入網站名稱的文字都不需要包含在連結本身。

-   說明連結不需要完全符合說明主題標題，但是兩者之間應該有很強且明顯的連接。 基於這個原因，設計成對的連結和標題。

    **正確：**

    如何改進這項功能的效能？  (連結文字) 

    設定這項功能以獲得最佳效能 (主題標題) 

    **不正確：**

    如何改進這項功能的效能？  (連結文字) 

    本功能的完整總覽 (主題標題) 

    在不正確的範例中，[說明] 主題標題的範圍明顯不同于 [說明] 連結文字的範圍，而且可能會 disorienting。

-   如果說明內容已上線，請在連結文字中將其清除。 這樣做有助於讓連結的結果成為可預測的結果。

    **正確：**

    如需其他格式和工具，請移至 Microsoft 網站。

    **不正確：**

    我可以在哪裡找到其他格式與工具？

-   使用完整句子。
-   請勿使用結尾標點符號，但問號除外。
-   請勿使用省略號來取得說明連結或命令。

**說明內容**

-   使用粗體將 UI 元素格式化，以方便識別。 這特別適用于程式說明主題，可讓使用者透過程式進行掃描，並快速查看相關的 UI 元素。
-   使用斜體格式化字幕。 這適用于從簡短文字說明中獲益的資料表、美工圖案、螢幕擷取畫面和其他圖形元素。
-   如需說明，請參閱 [說明]。 一般而言，除非您實際參考網站上的內容，否則請不要使用「線上說明」片語。
