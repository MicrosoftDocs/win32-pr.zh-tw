# <a name="wizards"></a>精靈

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

比較古怪名稱雖然很棒，但卻不是一種特殊形式的使用者介面，而且只有特定範圍的公用程式。

您可以使用嚮導來執行多步驟工作。

![顯示 [新增印表機] 的螢幕擷取畫面，其中顯示 [您要安裝哪一種印表機？] 提示。](images/win-wizards-image1.png)

![顯示搜尋可用印表機時 [新增印表機] 的螢幕擷取畫面。](images/win-wizards-image2.png)

![[新增印表機嚮導] 的螢幕擷取畫面 ](images/win-wizards-image3.png)

Wizard 的多個步驟會以一系列的頁面呈現。

嚮導通常包含下列類型的頁面：

-   選擇頁面可用來收集資訊，並允許使用者進行選擇。
-   [認可] 頁面是用來執行無法藉由按一下 [上一步] 或 [取消] 來復原的動作。
-   進度頁面可用來顯示冗長作業的進度。

新式的 wizard design 會提高效率，使進度頁面可選擇性用於較短的作業，而且通常會在一開始和結束時使用傳統的 [ [歡迎使用] 頁面](glossary.md) 和 [恭喜頁面](glossary.md) 關注。

所有的 wizard 頁面都有這些元件：

-   用來識別 wizard 名稱的標題列，左上角的 [上一頁] 按鈕，以及選用的最小化/最大化和還原按鈕的 [關閉] 按鈕。 請注意，標題列也包含圖示來識別工作列上的圖示。
-   使用頁面來說明使用者目標的主要指示。
-   具有選擇性文字及其他可能控制項的內容區域。
-   至少具有一個認可按鈕的命令區域，可認可至工作，或繼續進行下一個步驟。

雖然 wizard 有多個步驟，但這些步驟全都必須從使用者的觀點來新增至單一工作。 這是「單一 wizard，一個工作」的基本 wizard 設計原則。

因此，在本文中，工作是 wizard 的基本功能 (例如，安裝程式的工作就是安裝) 的程式。 子工作是較大工作的層面 (例如，安裝程式的子工作可能是設定要安裝) 的程式。 最後，每個 wizard 頁面都會被視為指定子工作或工作 (中的一個步驟，例如，設定程式) 可能有兩個或三個步驟。

**注意：** 與 [安裝程式](exper-setup.md)、 [對話方塊](win-dialog-box.md)和 [進度](progress-bars.md) 列相關的指導方針會顯示在個別文章中。

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

您可以針對任何需要多個輸入步驟的工作使用 wizard。 不過，有效的嚮導有額外的需求：

-   **Wizard 會執行單一、不可部分完成的工作嗎？** 請勿使用非單一工作的互動 (整個程式絕對不應該是 wizard，除非它會) 執行單一工作。 請勿使用 [嚮導] 來結合獨立工作或主要不相關的步驟。
-   **所需的問題數目是否可減少？可接受的預設值是否適用于大部分的情況，或可在稍後視需要調整？因此，可以減少的頁面數目嗎？** 若是如此，請嘗試簡化工作，讓它可以在單一頁面上顯示 (例如對話方塊) ，或是完全不需要輸入， (讓工作直接) 執行。
-   **必須依序提供所需的問題嗎？有幾個可能但有選擇性的問題嗎？** 如果是的話，請考慮使用對話方塊或索引標籤式對話方塊。

    **正確：**

    ![[列印選項] 對話方塊的螢幕擷取畫面 ](images/win-wizards-image4.png)

    [Microsoft PowerPoint 列印選項] 對話方塊包含許多使用者輸入選項，因此您可以在嚮導中加以呈現。 但是，不需要順序提供它們，因此對話方塊是較佳的選擇。

使用者介面的格式相當繁重;如果有適合的輕量解決方案可用，請使用！

## <a name="design-concepts"></a>設計概念

### <a name="overuse-of-wizards"></a>過度利用嚮導

在過去，與一般 UI 不同之處在于，其設計目的是為了協助使用者執行特別複雜的工作 (與位於不同位置的步驟) ，而且通常會有內建的智慧功能可協助使用者成功。 目前，所有的 UI 都應設計成盡可能簡化工作，因此不需要特殊的 UI 就能達到這個目的。

但是，這種方式可以確保這些是特殊的 UI，因為它們稱為「嚮導」 (更容易使用，例如「對話」和「屬性視窗」 ) 。 相反地，最好是將它們視為多步驟的工作，而不是特別注意這一點。

建立嚮導之前，請考慮使用者是否真的必須從程式的主要流程中斷。 可能會有較淺、內嵌的內容解決方案，最終對使用者感到更有説明且更有效率。 例如，程式中設計錯誤的功能，並不保證可解釋和簡化程式。它會保證功能本身的重新設計。 您不應該使用嚮導做為寬線輔助來修正程式的更基本問題。

### <a name="wizards-do-have-appropriate-functions"></a>嚮導有適當的功能

嚮導是簡化使用者體驗的其中一個關鍵。 它們可讓您執行複雜的作業，例如程式的設定，並將其分成一系列簡單的步驟。 在程式的每個階段，您可以提供所需專案的說明，並顯示可讓使用者進行選取並輸入文字的控制項。

某些類型的多重步驟工作本身就是 wizard 表單。 例如，在 Windows 中，有數個嚮導牽涉到連線功能 (至網際網路或公司網路，或) 的周邊裝置，例如印表機和傳真電腦。

![[連接嚮導] 的螢幕擷取畫面 ](images/win-wizards-image5.png)

連線到網路是 Windows 適用于 wizard 的一般工作。

在此，wizard 的功能是在已知和穩定 (現成的作業系統) ，以及電話公司或網際網路服務提供者) 有未知和變數 (的連線能力。 計算生態系統的複雜性有很大的説明，因為真的很有説明，可以使用嚮導來降低複雜度。

其他可做為 Windows 的工作的工作類型，包括 (的高階功能，例如語音和手寫辨識) 以及豐富的媒體體驗 (例如，設定用來製作及發佈電影的選項) 。 您也可以針對更基本的多步驟工作（例如疑難排解）來部署嚮導。 簡單地說，如果不同的使用者可能想要以各式各樣不同的方式來體驗您的程式，這可能表示需要 wizard 和其容量適用于多個使用者輸入點。

在您的程式中，您需要事先進行一些設計階段，以判斷您的 wizard 所提供的功能，以及該函式是否真的會增加嚮導的部署層級。

### <a name="wizard-length"></a>Wizard 長度

在頁面和選項的數目和組織中，自然會出現設計問題。 例如：

-   是否有最佳的 wizard 頁面數目？ 或至少需要最適合的範圍？
-   Wizard 應該簡潔又簡單，讓使用者能夠儘快完成嗎？
-   是否有更多需要較少選擇的頁面？ 或較少的頁面有更多複雜性？ 哪些設計被視為更容易使用？
-   您可以套用 UI 慣例（例如索引標籤式頁面）來設計更快速的 wizard 體驗嗎？

Microsoft 用來建議將三頁或更少的程式設計為簡單的 wizard，以及四個或更多頁面的檢查，都使用 advanced wizard 設計 (查看 1999) 的[Windows 使用者體驗](/previous-versions/ms997609(v=msdn.10))指導方針。 但是，目前的 wizard 設計標準會產生簡單和表單之間的其中一個主要差異 (使用 [歡迎] 和 [恭喜] 頁面) ，因此這些類別現在感覺不出，而決定設計選擇的頁面數目似乎是任意的。

您的 wizard 應為工作所需的完整或短時間;其長度沒有固定的指導方針。 單頁的 wizard 應該真的以對話方塊的形式呈現，因此兩頁可能是 wizard 最簡潔的形式。

**正確：**

![[建立光碟] 對話方塊的螢幕擷取畫面 ](images/win-wizards-image6.png)

這項工作有很少的選項，可讓您以 wizard 的形式呈現。 對話方塊是此使用者介面的適當表單。

在色譜的另一端，如果您有一個包含多個決策點和分支的 wizard，而且經常會導致使用者無法追蹤其導覽路徑，則您已超過實際的限制，而且應該減少嚮導的長度。 或者，您也可以將嚮導分成數個不同的工作。

當您決定最適合您的 wizard 的長度時，請特別注意您的目標使用者。 終端使用者的程式（例如家用取用者和辦公室工作者）通常會使用嚮導來隱藏複雜度;您可以使用乾淨、簡單的頁面設計和預先選取的預設值，盡可能縮短這些嚮導的選項。 相較之下，適用于 IT 專業人員的伺服器嚮導或程式通常會變得更複雜。 這組目標使用者有更高的容錯能力，可進行設定決策，而且如果隱藏太多複雜性，事實上可能會變成可疑。

如果有本質上的 wizard 簡化了複雜的工作，就應該對技術複雜的物件進行相對較少的動作，而且對新手的使用者群來說相當積極。

**正確：**

![顯示語言 wizard 的螢幕擷取畫面 ](images/win-wizards-image7.png)

這個 wizard 頁面的設計是為了讓使用者更容易使用，因為它會將可能很複雜的主旨降至簡單的邏輯二進位選擇：安裝或卸載。

**正確：**

![顯示 [SQL Server 安裝程式] 的 [功能選取] 頁面的螢幕擷取畫面。](images/win-wizards-image8.png)

在 Microsoft SQL Server 2008 的安裝程式中，頁面設計更忙碌，而許多選擇需要更多考慮，但是目標物件是資料庫管理員，需要嚴格控制特徵選取。

最後，請注意特定工作可能執行的頻率。 不頻繁的工作可能會部署較長的 wizard，而經常的工作則應該優先採用簡潔的作業。

### <a name="branching"></a>分支

針對較長的嚮導，您可能需要建立工作流程的分支，其中的分頁順序可能會根據提供的使用者輸入「上游」而有所不同。 分支本質上是 dislocating 的使用者，因此您必須設計使用者體驗以傳達穩定性。 建議您在整個 wizard 中不超過兩個會造成分支的決策點，而且單一分支內不會有一個以上的嵌套分支。

如需在分支嚮導內建立穩定使用者體驗的指導方針，請參閱本文的指導方針一節中的 [分支](#branching) 。

### <a name="providing-a-navigation-guide"></a>提供導覽指南

當工作中有許多步驟，而且使用者可能會遺失其在順序中的位置，或只想要知道需要花費多少時間才能完成時，流覽指南會很有用。

導覽指南通常會顯示為 wizard 的一或多個頁面清單，並在每個頁面左側的資料行或窗格中看起來像是目錄。 雖然清單會在整個 wizard 中保持一致 (相同的頁面清單會出現在每個頁面上) ，但是有一些視覺方法可用來指出使用者目前在序列中的位置 (例如，使用粗體來區分使用中的頁面或區段) 。

導覽指南可以是連續或非連續的。 順序類型會顯示過去的頁面以及已知的未來頁面。 如果這些步驟都是已知的，且頁面是相依的，您就可以在步驟而非頁面上提供未來。 然後，您可以在頁面被視為已知時，以動態方式填入頁面。 因為導覽順序是固定的，所以流覽指南不是互動式的。

非連續的流覽指南是互動式的，因此使用者可以直接重新流覽先前看到的頁面。 它們也可以跳過設計為選用之頁面的導覽順序。 選用頁面必須有大部分情況下可接受的預設值。 使用這種類型的指南：

-   您一律可以直接看到先前已查看的頁面。
-   如果有必要條件，可能無法查看未來的頁面。
-   可以造訪的頁面應該與無法 (例如，使用作用中或已停用) 的連結，以及必要或選擇性的頁面，來區別。

在此案例中，使用者可能會混淆 [上一頁] 按鈕的意義。 是否按下 [上一步] 會引導您前往導覽指南中的上一頁或區段，或查看最後一頁或區段？ 因為 Windows 的 wizard 現在會將 [上一頁] 按鈕放在 wizard 頁面的左上角，而不是在右下角加上其他認可按鈕，所以使用者會將其視為在 Web 上的後端功能。 因此，最好的解決方法是讓您的 [上一頁] 按鈕 Web 導覽， (按一下 [上一步] 應該會導致) 的最後一個頁面或區段，並使用瀏覽器導覽指南進行順序導覽。

### <a name="page-integrity"></a>頁面完整性

Wizard design 不僅牽涉到與整個工作流程有關的決策，例如如何處理導覽和分支體驗，以及與組成 wizard 的個別頁面相關的決策。 **設計良好的 wizard 頁面最重要的原則是完整性：頁面的內容應該在一起。**

如果每個頁面在概念上都在一起停止回應，則只會處理整體工作的一個部分，因此可大幅提高使用 Wizard 頁面的效用。 [主要指示](glossary.md)是達成此目的的主要方式。 清楚地找出頁面的目標或用途。 [補充指示](glossary.md)和頁面上的任何控制項都直接與主要指令相關。 雖然 wizard 頁面應該會為使用者提供一些需要考慮的選項，但是這樣做並不會覺得正常運作，因為它會以網頁本身的完整性為焦點。

可惜的是，wizard designer 通常會錯誤使用者快速按一下 [下一步] 按鈕，以證明其頁面的可用性、簡單性和完整性。 最後的 wizard 體驗並不是接下來，接下來，下一步，完成。 雖然這種體驗建議您最好選擇預設值，但它也會建議您不要因為所有選項都是選擇性的選項，所以不是必要的。

就視覺效果和文字而言，請將這些專案向下削減至 bare essentials。 您可以 (「burrito wizard」 ) 或使用索引標籤來呈現複雜的輸入需求，以避免在單一頁面上組合多個子工作。 單一頁面應該涵蓋整個 wizard 工作的單一子工作。

**不正確：**

![sql server 安裝程式的螢幕擷取畫面 ](images/win-wizards-image9.png)

有三個索引標籤需要相當密集的使用者輸入，此嚮導頁面會嘗試完成太多工作。

在大多數情況下，請在整個 wizard 中維護每個頁面的大小，以促進一致的外觀與風格。 雖然 Windows 的嚮導允許可調整大小的頁面，因此頁面的大小會符合內容數量，但只會使用此選項。

最後，在序列中維護每個 wizard 頁面的結構化元素。 例如，不要將 [上一頁] 按鈕從左上角移回頁面的 [認可按鈕] 區域。 此層級的版面配置一致性可協助使用者在嚮導內感覺穩定。 將此視為頁面視覺效果完整性的基準。

### <a name="finding-the-right-level-of-communication"></a>尋找正確的通訊層級

使用者具有低度的能力，可在螢幕上讀取大量的文字，而在 UI 介面中更少的範圍，其快速目的是要透過工作來移動 expeditiously。

嚮導的傾向是過度溝通。 它們佔用了大量的螢幕空間，這似乎會鼓勵磁片磁碟機填滿空間。 這就像是帕金森氏的定律變化一樣： UI 文字會展開以填滿可用的空間。

這項功能的其中一個原因是冗余。 由於早期 wizard 設計中使用的範本，相同的語言可能會出現在頁面上的多個位置，例如標題列、標題、內文文字、控制項標籤等等。

值得一提的是，雇用專業版的編輯器來剪除您的 wizard text ruthlessly。 在個別頁面上消除不必要的問題和選項，並將整個頁面中的整個頁面全部刪除為整個 (例如，傳統的歡迎和恭喜頁面) 。 使用您的目標物件用來描述工作的語言，而不是您或您的小組在內部使用的技術或功能的術語，以簡明撰寫的主要指示取得頁面的重點。 這種以使用者為中心的方法對於改善程式的嚮導的通訊相當重要。

請特別注意您的 wizard 語氣：有時候程式最持久的印象，就是您所說的結果不是您所說的結果，而是您說的結果！ 在 [使用者] 中，使用者可以使用易記的對話式語氣，在程式要求輸入時，可自由使用第二人稱的代詞 ( 「您」 ) 。 如需詳細的指導方針，請參閱 [樣式和語氣](text-style-tone.md)。

在 [wizard] 頁面上減少 [字數統計] 通常是貢獻值得讚賞的，但請注意不要太久。 如果工作很重要，且需要嚮導，則使用者會有足夠的資訊來做出明智的選擇。 下列範例顯示如何在不犧牲意義的情況下壓縮 wizard 文字。

**之前：**

![cleartype wizard、draft 的螢幕擷取畫面 ](images/win-wizards-image10.png)

**之後：**

![cleartype wizard 的螢幕擷取畫面，已修改 ](images/win-wizards-image11.png)

此 wizard 頁面的編輯版本提供以工作為導向的主要指示，移除主要指令下不必要的說明段落，並修訂核取方塊標籤以闡明核取方塊的用途。

**如果您只進行三件事 .。。**

1. 使用適當的 UI 對應您嘗試完成的工作;當您認為需要從使用者收集大量的輸入時，不只是預設為 wizard。

2. 仔細考慮一下您的 wizard 的長度和結構;偏好簡短的非分支嚮導，盡可能保持簡單的體驗，讓使用者可以回到其主要工作或對您的程式感興趣。

3. 確定您的 wizard 中每個頁面的完整性：頁面的內容應該清楚地在一起。

## <a name="guidelines"></a>指導方針

### <a name="general"></a>一般

-   **先考慮輕量替代方案，例如對話方塊、工作窗格或單一頁面。** 您不需要使用 [使用嚮導]，您可以在任何 UI 中提供有用的資訊和協助。
-   **使用多步驟工作的嚮導。** 使用多重頁面對話方塊進行單一步驟的工作，並提供意見反應。 如需詳細指導方針，請參閱 [對話方塊](win-dialog-box.md)。

    **正確：**

    ![[診斷] 對話方塊的螢幕擷取畫面 ](images/win-wizards-image12.png)

    ![診斷對話意見反應的螢幕擷取畫面 ](images/win-wizards-image13.png)

    在此範例中，Windows 網路診斷是由進度和結果頁面所組成。 因為工作只是一個步驟，所以不需要使用者在嚮導中需要的導覽按鈕。 它實際上是以多頁對話方塊的形式呈現。

### <a name="window-size"></a>視窗大小

-   **選擇可顯示所有 wizard 頁面，而不需要垂直或水準頁面滾動的視窗大小。** 雖然頁面上的控制項可能需要滾動，但是 wizard pages 本身也不能。
-   **Windows 大小夠大，足以輕鬆地執行工作。** 頁面配置不應為 cramped，或要求使用者必須過度滾動或調整大小。
-   **但是不要讓 windows 過大。** 較大的視窗讓工作感覺更複雜，且需要額外移動互動。
-   **使用可調整大小的視窗，讓您可以從更多螢幕空間獲益但不需要它。** 指派適當的最小大小。 當頁面需要與可調整大小的內容（例如大型清單視圖）進行互動時，可調整大小的視窗

    **正確：**

    ![visual studio 安裝程式的螢幕擷取畫面，部分清單 ](images/win-wizards-image14.png)

    **較佳：**

    ![visual studio 安裝程式的螢幕擷取畫面，完整清單 ](images/win-wizards-image15.png)

    在此範例中，調整視窗的大小可協助使用者查看完整清單。

-   **請考慮使用動態調整大小的嚮導，其頁面大小會隨著其內容的需要變更。** 這麼做可讓 wizard 容納具有廣泛內容的頁面配置。
-   **如果使用者可能會在其 wizard 的體驗中察覺到變更，則偏好動態調整大小。** 視覺穩定性通常會優先于內容的住宿。 大部分的嚮導應該採用標準的靜態視窗大小，並且在特殊情況下保留動態調整大小。

### <a name="wizard-length"></a>Wizard 長度

-   **讓您的 wizard 盡可能簡潔並簡化。** 消除不必要的選項和問題，並使用智慧型預設值來減少使用者輸入所需的頁面數目。
    -   **例外狀況：** IT 專業人員和其他技術使用者對於較長的嚮導和詳細的輸入需求具有較高的承受度。
-   **讓您的 wizard 至少有兩個頁面。** 您應改以對話方塊的方式重新設計單頁的 wizard。
-   **請勿只藉由增加每個頁面的複雜度來減少嚮導的頁面計數。** 例如，包含需要使用者輸入的三個索引標籤的 wizard 頁面應該重新設計成三個不同的頁面。
-   **不要增加嚮導的頁面計數，方法是讓每個頁面都變成簡單，讓使用者 mindlessly 按下 [下一步]。** 這是常見的 wizard 設計缺陷。 如果 wizard 頁面不需要進行某種程度的考慮，則可能完全不需要在嚮導中。

### <a name="branching"></a>分支

-   **優先使用非分支的 wizard 設計，而不是分支。** 非分支的嚮導通常更簡單、更短且容易流覽。 分支嚮導讓使用者更難以判斷工作中有多少步驟，以及它們在順序中的位置。
-   **如果您必須分支，請使用下列其中一種技術協助使用者自行定位：**
    -   **列舉頁面。** 常見的技巧是在每個頁面上的順序中指出使用者的位置，例如使用片語步驟 X of Y。請確定端點 (Y) 穩定。 如果它改變了價值，就破壞使用者的信心。
    -   **包含子步驟的概念** (例如 6) 的步驟2a。
    -   **建立與頁面無關的步驟，其中每個步驟都可能牽涉到數個頁面。** 例如，旅遊服務可能會根據業界的良好電子商務慣例來採用 wizard 組織。

        **正確：**

        ![以步驟為基礎的 wizard 組織螢幕擷取畫面 ](images/win-wizards-image16.png)

        邏輯標籤可為分支嚮導的使用者提供適當的方向。

    -   **將選擇性步驟視為在列舉順序中的持續性。** 例如，如果分支只是略過一些選擇性的步驟，也只會略過意見反應中的步驟，而不是重新編號。 因此，如果使用者選擇第2頁，而導致頁面3和4變成選用，則顯示步驟1、2、5和6。 請勿將步驟5和6重新編號。
    -   **如果 wizard 採用單一分支，且分支在工作初期發生，請在該點開始順序，然後直接使用非分支方法。** 亦即，從分支的點開始，順序到分支結尾的進度。

-   **如果您必須進行分支，請將分支數目限制在單一 wizard 中的一個或兩個。** 請勿在分支內包含一個以上的分支， (「嵌套」分支) 。

### <a name="commit-buttons"></a>認可按鈕

-   **當使用者認可工作時，請使用對主要指示的特定回應** 的 [認可] 按鈕 (例如，[列印]、[連線] 或 [啟動) ]。 請勿使用一般標籤（如下 (所示），這不表示承諾用量) 或完成 (不是認可工作的特定) 。 這些認可按鈕上的標籤應該有意義。 一律使用動詞來啟動認可按鈕標籤。 **異常：**
    -   當特定回應仍為一般時，請使用 [完成]，例如 [儲存]、[選取]、[選擇] 或 [取得]。
    -   使用 [完成] 來變更特定設定或設定集合。
-   **單一的 wizard 可以有多個認可點，但最好是一個重點。**
-   **如有必要，您可以重新命名或隱藏頁面上的認可按鈕。** 這種彈性是新的 wizard 設計在舊版的 wizard 中未提供的 Windows 中的一項優點。 請注意，隱藏認可按鈕與停用的不同。
-   **避免停用正面的認可按鈕。** 否則使用者必須推算認可按鈕停用的原因。 最好讓認可按鈕保持啟用狀態，並在發生問題時提供實用的錯誤訊息。 只有在執行此動作的原因很明顯且明確時，才可接受停用按鈕。
-   **不要混淆導覽按鈕 (下一步和上一頁) 與認可按鈕。** 下一步是在 wizard 中進行，而不需承諾用量;在下一個頁面上應該一律可以使用 back，然後按一下 [上一步]，就可以復原上一個 [下一步] 按鈕的效果。 如果無法這麼做，使用者會做出承諾，並透過 [認可] 按鈕上的特定標籤來表示。 如需 [下一步] 和 [上一頁] 按鈕的詳細指導方針，請參閱 [導覽](#providing-a-navigation-guide)。

### <a name="cancel-buttons"></a>取消按鈕

-   **請勿要求使用者確認他們是否真的想要取消。** 這樣做可能會很麻煩。 **異常：**
    -   此動作會產生顯著的後果，如果不正確，則無法輕易修復。
    -   此動作可能會導致使用者的時間或工作大幅遺失。
    -   動作與其他動作明顯不一致。
-   **允許使用者重新開機嚮導，以防錯誤地取消。**
-   **請勿停用 [取消] 按鈕。異常：**
    -   如果取消很有害，可能是在獨立的嚮導中執行工作時的情況。
    -   如果無法取消，可能是因為 wizard 沒有所有步驟的控制權。

### <a name="close-buttons"></a>關閉按鈕

-   **針對 Follow-Up 和完成頁面使用 Close。** 請勿使用 [取消]，因為關閉視窗並不會放棄目前為止所做的任何變更或動作。 請勿使用完成，因為它不是命令式動詞。
-   **工作執行完成後，[取消] 應該會變成) 的獨立嚮導 (關閉。** 關閉的效果只是關閉視窗。

### <a name="other-controls"></a>其他控制項

-   **只針對選項使用命令連結，而非承諾用量。** 特定的認可按鈕表示承諾用量優於 wizard 中的命令連結。
-   **使用命令連結時，隱藏 [下一步] 按鈕，但保留 [取消] 按鈕。**

### <a name="using-pages-vs-dialog-boxes-or-inline-ui"></a>使用頁面 (vs. 對話方塊或內嵌 UI) 

-   **一般情況下，偏好頁面至對話方塊。** 使用者希望有以頁面為基礎的嚮導。
-   **使用對話方塊來協助完成頁面，** 例如使用物件選擇器和瀏覽器。
-   **使用對話方塊可提供適用于整個頁面的錯誤訊息，以及按一下 [認可] 按鈕所產生的結果。**
-   **針對簡單的動態行為** （例如漸進式洩漏和內容 UI）使用內嵌呈現。
-   **針對適用于特定控制項的錯誤訊息，請使用內嵌呈現。**

### <a name="wizard-pages"></a>精靈頁面

-   **專注于有效率的決策制定。** 減少要專注于 essentials 的頁面數目。 合併相關的頁面，並從主要流程取出選用的頁面。 讓使用者透過您的 wizard 完全按下 [下一步] 看起來似乎是很好的體驗，但是如果使用者永遠不需要變更預設值，可能就不需要這些頁面。
-   **將每個頁面設計成具有單一用途和視覺化一致性。** 如需詳細資訊，請參閱 [頁面完整性](#page-integrity)。
-   **請勿使用 [歡迎使用] 頁面，請盡可能讓第一頁運作。** 只有在下列情況時，才使用選擇性的開始使用頁面：

    -   Wizard 具有成功完成嚮導所需的必要條件。
    -   使用者可能無法根據其第一個選擇頁面瞭解嚮導的目的，而且沒有空間可供進一步說明。
    -   開始使用頁面的主要指示是「在您開始之前：」。

    **不正確：**

    ![mappoint 設定 [歡迎使用] 頁面的螢幕擷取畫面 ](images/win-wizards-image17.png)

-   新式的嚮導會選擇功能優先的第一頁。 這裡沒有要做的事，但按 [下一步]。 為何要在寶貴的時間強制使用者支付此 token 稅？
-   **在要求使用者進行選擇的頁面上，針對最有可能的情況進行優化。** 這些類型的頁面應該會顯示實際的選項，而不只是指示。
    -   如果您未使用開始使用頁面，請在第一頁選擇的頂端說明 wizard 的用途。
-   **當使用者認可工作時，請使用 Commit 頁面讓它清楚。** [認可] 頁面通常是最後一頁的選項，而 [下一步] 按鈕會重新標記以指出正在認可的工作。
    -   請勿使用僅摘要使用者先前選取專案的摘要頁面，除非工作具有風險 (涉及安全性，或) 遺失時間或金錢，或是使用者很有可能需要檢查其選擇。
-   **使用進度頁面來顯示冗長作業的狀態。** 成功完成時，[進度] 頁面應該會自動前進到下一個步驟。 只有當使用者需要查看問題時，它才會留在 [進度] 頁面上。 按一下 [回到進度] 頁面不會有任何副作用。
    -   **使用單一確定進度列。** 遵循 [確定進度列指導方針](progress-bars.md)，包括：
        -   明確指出完成。 除非作業已完成，否則不要讓進度列移至100%。
        -   請勿重新開機進度。 如果進度列在重新開機時遺失其值 (可能是因為作業中的步驟完成) ，因為使用者無法得知作業何時會完成。 相反地，請讓作業中的所有步驟共用進度的一部分，並讓進度列移至完成一次。
    -   **提供進度列上方目前步驟的簡短描述。** 若為快速作業，則不需要這類文字;單獨的進度列即已足夠。 對於需要分鐘或更長時間的作業來說，文字可能很有説明。
        -   使用句子片段，通常以動詞開頭，並以省略號結尾。 範例：正在複製檔案 ...，正在安裝必要元件 .。。
        -   將文字放在橫條的上方，而不是下方。
        -   **不正確：**
        -   ![進度列的螢幕擷取畫面，其中包含下列文字 ](images/win-wizards-image18.png)
        -   在此範例中，解說文字應該會出現在進度列的上方。
        -   避免使用不必要的詳細資料來雜亂進度頁面。 本頁面不適用於技術支援;適用于使用者。
        -   **不正確：**
        -   ![進度列具有太多詳細資料的螢幕擷取畫面 ](images/win-wizards-image19.png)
        -   在此範例中，像是 Guid 等技術詳細資料對使用者沒有意義。
-   **請勿使用不執行任何動作的恭喜頁面，而是結束 wizard。** 如果使用者清楚看得到的結果，只要在最終的 [認可] 按鈕上關閉嚮導即可。
    -   當有使用者可能執行的相關工作時，請使用 Follow-Up 頁面，如下所示。 避免常見的後續工作，例如「傳送電子郵件訊息」。
    -   只有當結果不可見，而且沒有更好的方法可以提供工作完成的意見反應時，才使用完成頁面。
    -   具有進度頁面的嚮導必須使用完成頁面或 Follow-Up 頁面來指出工作完成。 針對長時間執行的工作，請在 [認可] 頁面上關閉嚮導，並使用通知來提供意見反應。
-   **只有當輸入很複雜，而且使用者需要檢查時，才會使用摘要頁面，如果工作牽涉到嚴重的風險 (例如財務轉換) ，或如果嚮導會根據使用者輸入來採取行動，而不 (是透過透明度) 來建立信任的使用者輸入。** 通常，摘要頁面不會符合此相關性列，而且可以省略。
-   **如果因為無法復原的問題而無法完成嚮導，請使用錯誤頁面。** 在此頁面上，以清楚的語言說明問題，而不是使用者所能瞭解的技術術語。 也提供使用者可採取來解決問題的實際步驟。 如需詳細指導方針，請參閱 [錯誤訊息](mess-error.md)。
    -   **例外狀況：** 如果嚮導完成時有可能發生復原的問題，請將問題顯示為額外的工作而非錯誤。 使用正面、成功導向、鼓勵語言，而不是錯誤、失敗或問題等詞彙。 請勿使用錯誤圖示。

### <a name="navigation"></a>導覽

-   **只有在前進到下一個頁面而不需承諾用量時，才使用下一個。** 當您按一下 [上一步] 或 [取消] 時，前進到下一個頁面會被視為承諾量。
-   **請使用 Back 來更正錯誤。** 除了更正錯誤，使用者不需要按 [上一步]，就能在工作中進行進度。
-   **透過導覽保留使用者的選取專案。** 例如，如果使用者進行變更，請按 [上一步]，然後再按 [下一步]，應該保留這些變更。 除非使用者明確選擇清除變更，否則不會預期需要重新輸入變更。
-   **除非重複步驟是有害的，否則請勿停用 [上一頁] 按鈕。**
-   **允許使用者流覽或修改下列導覽案例中的選項：**
    -   使用者提供輸入、按一下 [認可] 按鈕、按 [上一步] 以檢查先前的變更，不會變更任何變更，然後再次按一下 [認可] 按鈕。 一般來說，這應該是可行的，而第二個認可應該只前進到下一個頁面 (因為工作已經) 完成。
    -   使用者提供輸入、按一下 [認可] 按鈕、按 [上一步] 以檢查先前的變更、變更某個內容，然後再按一下 [認可] 按鈕。 一般來說，這應該是可行的，而第二個認可應該使用變更的輸入來重做工作， (取代或復原第一個) 的效果。

### <a name="help"></a>說明

-   **[設計嚮導] 頁面提供足夠的資訊，因此不需要參考程式說明中的檔。** Wizard 已經讓使用者離開自己想要的使用者，直接與程式互動;要求使用者尋求外部說明可將其從這個狀態進一步移除。 說明應該是例外狀況，而不是規則。
-   **如果您必須提供協助的存取點，請在命令區域) 上方 (頁面內容區域左下角的連結。** 此連結應該是短暫的，而且通常會以使用者最可能想要回答的問題形式片語。
-   **正確：**
-   ![具有說明連結之 wizard 頁面的螢幕擷取畫面 ](images/win-wizards-image5.png)
-   此連結可協助您適當，因為像這樣的基本背景資訊會讓 wizard 頁面過於雜亂。

## <a name="text"></a>Text

### <a name="general"></a>一般

-   使用您和您的來參考使用者和使用者的電腦、檔、設定等等。 請勿使用第一個人 (I，我的) 指的是電腦或 wizard。 不過，在使用者選取的選項中使用第一個人是可接受的。 **範例： [僅我使用** ] 核取方塊。
-   每個 wizard 頁面都必須有 [主要的指示](glossary.md)。

### <a name="titles"></a>標題

-   將嚮導的名稱放在標題列中。 使用 [標題樣式的大小寫](glossary.md)。
-   標題不應包含標點符號，除了具有問號的標題。
-   請勿在 wizard 標題中包含 word Wizard。 例如，使用網路的連線，而不是網路安裝精靈。

### <a name="buttons"></a>按鈕

-   請勿在 [上一頁] 按鈕上包含文字。 使用箭號字元代替未標記。
-   在 [下一步] 按鈕上包含文字。 請勿使用字元 (例如，> 或 >>) 以及下一個字。
-   使用適當的特定認可按鈕標籤，並回應主要指令。 在理想的情況下，使用者不需要讀取任何其他人來瞭解標籤。 使用者比靜態文字更可能讀取命令按鈕標籤。
-   可能的話，請不要使用 [認可] 按鈕標籤的 [完成] 這個字，因為通常會有更好的更明確的 [認可] 按鈕：
    -   如果按一下 [對工作的認可] 按鈕 (所以尚未執行此工作) ，請使用以動詞命令開頭的特定標籤 (範例： [列印]、[連線]、[開始]) 。
    -   如果已在嚮導內執行此工作，請改用 [關閉]。

        **異常：**

        -   您可以在特定標籤仍為一般時使用 [完成]，例如 [儲存]、[選取]、[選擇] 或 [取得]。
        -   當工作牽涉到變更設定或設定集合時，您可以使用 [完成]。

-   使用動詞啟動認可按鈕標籤。 例外狀況是 [確定]、[是] 和 [否]。
-   使用 [句子樣式的大小寫](glossary.md)。
-   請勿使用結束標點符號。
