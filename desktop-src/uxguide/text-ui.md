# <a name="user-interface-text"></a>消費者介面文字

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

使用者介面文字會出現在 UI 表面上。 此文字包含控制項標籤和靜態文字：

-   控制項標籤可識別控制項，並且直接放在控制項的旁邊或旁邊。
-   靜態文字（因為它不是互動式控制項的一部分）會為使用者提供詳細的指示或說明，讓他們能夠做出明智的決策。

**注意：**[樣式和語氣、字型](text-style-tone.md)[和](vis-fonts.md)[通用控制項](controls.md)標籤的相關指導方針會顯示在不同的文章中。

## <a name="design-concepts"></a>設計概念

軟體發展人員通常會將文字視為產品檔和技術支援。 「首先我們要撰寫程式碼，然後雇用某人協助我們解釋我們所開發的內容。」 但事實上，在程式中，我們會在程式中稍早撰寫重要的文字，因為 UI 是在設計和撰寫程式碼時。 在所有的情況下，這種文字會比其他類型的技術撰寫更頻繁、更多人看到。

**理解文字對有效的 UI 而言很重要。** Professional 的編寫器和編輯器應該與軟體發展人員一起使用 UI 文字，做為設計流程不可或缺的一部分。 讓它們提早處理文字，因為文字問題通常會顯示設計問題。 如果您的小組在說明設計方面遇到問題，通常這是設計，而不是說明，需要改善。

### <a name="a-design-model-for-ui-text"></a>UI 文字的設計模型

當您考慮 UI 文字及其在 UI 表面上的位置時，請考慮下列事實：

-   在焦點的情況下，大聲閱讀的讀者會以左至右、由上到下的順序， (西方文化特性) 。
-   使用軟體時，使用者不會在 UI 本身的工作中可以沉浸。 因此，使用者不會讀取其掃描的 UI 文字。
-   當您掃描視窗時，使用者可能會看到文字正在進行篩選，而實際上是正在進行篩選。 除非他們觀察到的需求，否則通常不會真正理解 UI 文字。
-   在視窗中，不同的 UI 元素會收到不同的注意層級。 使用者通常會先讀取控制項標籤，特別是與完成工作相關的程式。 相較之下，使用者通常會在他們認為需要時才讀取靜態文字。

若為一般設計模型，請不要假設使用者謹慎地以左至右、由上到下的順序讀取文字。 相反地，假設使用者從快速掃描整個視窗開始，然後以大約下列順序讀取 UI 文字：

-   中心內的互動式控制項
-   [認可按鈕](glossary.md)
-   在其他地方找到互動式控制項
-   主要指示
-   補充說明
-   視窗標題
-   主要主體中的其他靜態文字
-   註腳

您也應該假設一旦使用者決定要做什麼之後，他們就會立即停止讀取並進行。

### <a name="eliminate-redundancy"></a>消除冗余

多餘的文字不僅需要寶貴的螢幕空間，還可削弱您嘗試傳達的重要想法或動作的有效性。 它也會浪費讀者的時間，而在掃描的情況下則會有更多的內容。 **Windows 致力於說明使用者需要做什麼才能順利完成。**

請檢查每個視窗，並消除重複的單字和語句（不論是在控制項內或跨控制項）。 請勿在必要的地方明確地明確地顯示文字，但不是多餘的，也不會說明明顯的。

### <a name="avoid-over-communication"></a>避免過度通訊

即使文字不是多餘的，也可能太冗長，以說明每個詳細資料。 **太多文字不太方便閱讀，很可能會略過它諷刺是，而不是比其他更少的通訊。** 在 UI 文字中，明確地傳達基本資訊。 如果某些使用者或某些案例需要詳細資訊，請提供更詳細說明 [內容](winenv-help.md)的連結，或可能的詞彙專案以進行詞彙的澄清。

**不正確：**

![具有6個段落之對話方塊的螢幕擷取畫面](images/text-ui-image6.png)

在此範例中，有太多文字可以輕易地進行掃描。 雖然設計工具並不會用到，但使用者很可能會按 [下一步]，而不需要讀取任何文字。

若要避免不鼓勵閱讀的文字，請製作您的文字來建立每個字數的字數。 未加入減法，所以請使用簡單簡潔的文字。

### <a name="use-the-inverted-pyramid"></a>使用反向金字塔

學術撰寫通常是使用「金字塔」結構樣式來奠定事實的基礎、使用這些事實，然後建立結論形成類似金字塔的結構。 相較之下，新聞工作者會使用「反向金字塔」樣式，開頭為「重點」的基本概念，讀者必須要有這些結果。 然後，它會更詳細地填滿讀者可能會想要掃描的詳細資料。 這種樣式的優點是它是直接的，並且可讓讀者在選擇的任何時間點停止讀取，並且仍瞭解重要資訊。

您應該將反向金字塔結構套用至 UI 文字。 取得重要資訊的重點，讓使用者隨時都能隨時停止閱讀，並使用說明連結來呈現金字塔的其餘部分。

![加入 windows 程式之訊息的螢幕擷取畫面 ](images/text-ui-image7.png)

在此範例中，基本資訊是在查詢主要指示文字時，其他有用的資訊則是在補充指示中，而您可以按一下 [說明] 連結來取得詳細資料。

**如果您只進行五件事 .。。**

1.  及早處理文字，因為文字問題通常會顯示設計問題。
2.  設計要掃描的文字。
3.  消除重複的文字。
4.  使用簡單易懂的文字;不要進行通訊。
5.  如有必要，請提供說明內容的連結，以取得更詳細的資訊。

## <a name="guidelines"></a>指導方針

### <a name="general"></a>一般

-   **移除多餘的文字。** 在視窗標題、主要指示、補充指示、內容區域、命令連結和認可按鈕中尋找重複的文字。 一般而言，請將全文檢索保留在主要指示和互動式控制項中，並從其他地方移除任何多餘的內容。
-   **避免 UI 文字的大型區塊。** 執行這項操作的方法包括：
    -   將文字區塊化成較短的句子和段落。
    -   必要時，提供實用但不重要的資訊說明 [連結](winenv-help.md) 。
-   **選擇明確通訊的物件名稱和標籤，並區分物件的用途。** 使用者不應該需要找出物件真正的意義，或與其他物件有何不同。

-   **如果您想要確保使用者讀取與某個動作相關的特定文字，請將它放在互動式控制項上。**

### <a name="other-text-characteristics"></a>其他文字特性

**粗體字**

-   **請謹慎使用粗體，以吸引使用者必須閱讀的文字。** 例如，當使用者掃描選項按鈕選項清單時，可能會很樂意看到以粗體顯示的標籤，而是從新增每個選項相關補充資訊的文字中取出。 請注意，使用太多粗體會減少其影響。
-   **使用加上標籤的資料時，請使用粗體來強調對資料整體來說較重要的資料。**
    -   大部分的一般資料 (，其中的資料在沒有標籤的情況下沒有任何意義，如同數位或日期) 一樣，請使用粗體標籤和一般資料，讓使用者可以更輕鬆地掃描及瞭解資料類型。
    -   大部分的自我理解資料都是使用純文字標籤和粗體資料，讓使用者可以專注于資料本身。
    -   或者，您可以使用暗灰色文字來消除不重要的資訊，而不是使用粗體來強調更重要的資訊。

        ![windows explorer 縮圖視圖的螢幕擷取畫面 ](images/text-ui-image25.png)

        在此範例中，您不會使用粗體強調資料，而是使用深灰色來反白顯示標籤。

-   **並非所有字型都支援粗體，因此在瞭解文字時絕對不重要。**

**斜體**

-   用來真正參考文字。 請勿將引號用於此用途。

    **正確：**

    檔和檔案等詞彙通常會交替使用。

-   用於[文字方塊](ctrl-text-boxes.md)中的[提示](glossary.md)和[可編輯的下拉式清單](/windows/desktop/uxguide/ctrl-drop)。

    ![[搜尋] 文字方塊的螢幕擷取畫面 ](images/text-ui-image26.png)

    在此範例中，[搜尋] 方塊中的提示會格式化為斜體文字。

-   請謹慎使用以強調特定文字，以協助理解。
-   **並非所有字型都支援斜體，因此在瞭解文字時絕對不重要。**

**粗體斜體**

-   請勿在 UI 文字中使用。

**Underline**

-   請勿使用，但連結除外。
-   請勿使用強調。 改用斜體。

### <a name="punctuation"></a>標點符號

**句點**

-   **請勿放置於控制項標籤、主要指示或說明連結的結尾。**
-   放在補充指示的結尾、補充說明，或形成完整句子的任何其他靜態文字。

**問號**

-   **放在所有問題的結尾。** 與句號不同的是，所有類型的文字都會使用問號。

**驚嘆號**

-   在商務應用程式中，請避免。
    -   **例外狀況：** 在下載完成的內容中，有時會使用驚嘆號 ( 「完成！」) ，並 ( 「新的！」中呼叫 Web 內容) 。

**逗號**

-   在三個或多個專案的清單中，一律在清單中的下一個到最後一個專案後面加上逗號。

**冒號**

-   **在外部控制標籤的結尾使用冒號。** 這對存取範圍特別重要，因為有些輔助技術會尋找用來識別控制項標籤的冒號。
-   使用冒號來引進專案清單。

**橢圓**

-   **省略號表示不完整性。** 使用 UI 文字中的省略號，如下所示：
    -   **命令：** 指出命令需要額外的資訊。 當動作只在需要額外的資訊時顯示另一個視窗時，請勿使用省略號。 如需詳細資訊，請參閱 [命令按鈕](ctrl-command-buttons.md)。
    -   **資料：** 指出文字已被截斷。
    -   **標籤：** 指出工作正在進行中 (例如「正在搜尋 ...」 ) 。

        **秘訣：** 在具有未使用空間的視窗或頁面中截斷文字表示版面配置不良，或預設視窗大小太小。 致力於消除或減少截斷文字數量的版面配置和預設視窗大小。 如需詳細資訊，請參閱[配置](vis-layout.md)。

-   **不要讓橢圓形成為互動。** 若要顯示截斷的文字，請讓使用者調整控制項大小以查看更多文字，或改用 [漸進式洩漏控制項](ctrl-progressive-disclosure-controls.md) 。

**引號和單引號**

-   若要逐字參考文字，請使用斜體格式，而非引號。
-   只有在需要時才將視窗標題和控制項標籤放在引號中，以避免混淆，您也無法改為使用粗體來格式化。
-   如果是引號，則偏好使用雙引號 ( "" ) ;請避免用單引號括住。

    **正確：**

    確定要刪除「Sparky 的貓資料夾」嗎？

    **不正確：**

    確定要刪除 ' Sparky ' 的貓資料夾 ' 嗎？

### <a name="capitalization"></a>大寫

-   **針對標題使用標題樣式的大小寫，針對所有其他 UI 元素使用句子樣式的大小寫。** 這麼做會更適合 Windows 的[色調](text-style-tone.md)。

    -   **例外狀況：** 針對繼承應用程式，如果有必要，您可以針對命令按鈕、功能表和資料行標題使用標題樣式的大小寫，以避免混用大小寫樣式。

        ![一般屬性工作表的螢幕擷取畫面 ](images/text-ui-image27.png)

    此一般範例會針對屬性工作表顯示正確的大小寫和標點符號。

    ![[一般] 對話方塊的螢幕擷取畫面 ](images/text-ui-image28.png)

    此一般範例會針對對話方塊顯示正確的大小寫和標點符號。

-   **對於功能和技術名稱而言，請務必保守。** 一般來說， (使用標題樣式的大小寫) ，則只能使用大寫的主要元件。

    **正確：**

    Analysis Services、cube、維度

-   **若為功能和技術名稱，請在大寫中保持一致。** 如果名稱在 UI 畫面上出現一次以上，則應該一律以相同的方式來顯示。 同樣地，在程式的所有 UI 畫面中，名稱應該一致地呈現。
-   不要將一般使用者介面專案的名稱大寫，例如工具列、功能表、捲軸、按鈕和圖示。
    -   **例外狀況：** 網址列、連結列。
-   所有大寫字母都不要使用鍵盤按鍵。 相反地，請遵循標準鍵盤所使用的大小寫，或者，如果鍵盤上未標示按鍵，則為小寫。

### <a name="globalization-and-localization"></a>全球化和當地語系化

全球化表示建立可在任何國家、地區或文化特性中使用的檔或產品。 當地語系化指的是調整檔或產品，以用於非來源國家/地區以外的地區設定。 撰寫 UI 文字時，請考慮全球化和當地語系化。 您的程式可能會轉譯成其他語言，而且與您自己的文化特性不同。

-   針對具有變數內容 (例如清單查看和樹狀檢視) 的控制項，請 **選擇適用于最長有效資料的寬度。**
-   **在 UI 介面中包含足夠的空間來提供額外的 30%** (高達200% 的文字，以縮短文字 (的文字) ，而非將當地語系化的數位) 。 從一種語言翻譯成另一種語言，通常會變更文字的行長度。
-   請勿在執行時間從子字串撰寫字串。 相反地，請使用完整的句子，讓翻譯工具不會有任何混淆。
-   **請勿使用從屬控制項、它所包含的值，或其單位標籤來建立句子或片語。** 這種設計不可當地語系化，因為句子結構會隨著語言而有所不同。

    **不正確：**

    ![核取方塊標籤內文字方塊的螢幕擷取畫面 ](images/text-ui-image32.png)

    **正確：**

    ![核取方塊標籤之後文字方塊的螢幕擷取畫面 ](images/text-ui-image33.png)

    在不正確的範例中，文字方塊放在核取方塊標籤內。

-   請勿讓句子的一部分成為連結，因為翻譯時，該文字可能不會保持在一起。 因此，連結文字應自行形成完整的句子。
    -   **例外狀況：** 詞彙連結可以內嵌方式插入，作為句子的一部分。

如需詳細資訊，請參閱「 [移至全球開發人員中心](https://msdn.microsoft.com/goglobal/)」。

### <a name="title-bar-text"></a>標題列文字

-   根據視窗的類型選擇標題列文字：
    -   以 **檔為主的最上層程式視窗：** 使用「檔案名稱程式名稱」格式。 首先會顯示檔案名稱，以提供以檔為中心的風格。
    -   **不是以檔為主的最上層程式視窗：** 只顯示程式名稱。
    -   **對話方塊：** 顯示對話方塊的命令、功能或程式。 請勿使用標題來說明此對話方塊的目的，也就是主要指示的用途。 如需詳細指導方針，請參閱 [對話方塊](win-dialog-box.md)。
    -   **嚮導：** 顯示嚮導名稱。 請注意，「wizard」這個字不應該包含在嚮導名稱中。 如需詳細的指導方針，請參閱 [嚮導](win-wizards.md)。
-   **針對最上層的程式視窗，如果標題列標題和圖示在視窗頂端附近顯示，您可以隱藏標題列標題和圖示來避免重複。** 不過，您仍然必須在內部設定適當的標題，以供 Windows 使用。
-   **在對話方塊中，請勿在標題中包含 "dialog" 或 "進度" 單字。** 這些概念是隱含的，而將這些字組保持不變，可讓使用者更容易掃描標題。

### <a name="main-instructions"></a>主要指示

-   **您可以使用主要指示，精確地說明使用者在指定視窗或頁面中應執行的動作。** 良好的主要指示傳達使用者的目標，而不是只著重在操作 UI。
-   **以命令式方向或特定問題的形式表達主要指令。**

    **不正確：**

    ![程式名稱的螢幕擷取畫面，作為主要指示 ](images/text-ui-image34.png)

    在此範例中，main 指令只會指出程式的名稱;它不會明確邀請使用者採取動作。

    **例外狀況：** 錯誤訊息、警告訊息和確認可能會在其主要指示中使用不同的句子結構。

-   **盡可能使用特定動詞。** 特定動詞 (範例： [連接]、[儲存]、[安裝) 比一般使用者更有意義， (範例：設定、管理、設定) 。
    -   針對 [控制台] 頁面和 [wizard] 頁面，如果您無法使用特定的動詞，您可能會想要完全省略動詞。

        **可以接受：**

        輸入您的地區設定、地區和語言

        **較佳：**

        地區設定、地區和語言

    -   針對對話方塊（例如錯誤訊息和警告），請勿省略動詞。

-   如果新增它只會在 UI 的內容中重複或明顯，請不要覺得不必使用主要的指令文字。

    ![[另存新檔] 對話方塊的螢幕擷取畫面 ](images/text-ui-image35.png)

    在此範例中，UI 的內容已經非常清楚;不需要加入主要的指令文字。

-   **請務必只使用單一的完整句子。** 削減主要指示以瞭解基本資訊。 如果您必須進一步說明，請考慮使用補充指令。
-   使用句型大寫。
-   **如果指令是語句，則不包含最後句號。** 如果指令是問題，請包含最後一個問號。
-   在 **[進度] 對話方塊中，使用現在片語來簡短說明進行中的作業，** 並以省略號結尾。 範例：「列印您的圖片 ...」
-   **秘訣：** 您可以在說明要如何使用視窗或頁面時，藉由想像您對朋友的看法來評估主要指示。 如果回應主要指示的非自然、無用或麻煩，請重新修改指令。

如需詳細資訊，請參閱特定 UI 元件指導方針中的「主要指令」一節。

### <a name="supplemental-instructions"></a>補充指示

-   必要時，請 **使用補充指示來呈現有助於瞭解或使用視窗或頁面的其他資訊，** 例如：
    -   提供內容來說明視窗在程式或系統起始時，顯示的原因。
    -   協助使用者決定如何處理主要指示的合格資訊。
    -   定義重要術語。
-   **如果不需要，請不要使用補充指令。** 如果您可以簡潔地進行，則偏好使用主要指示傳達所有專案。
-   **請勿以稍微不同的用語重複主要指令。** 相反地，如果沒有其他專案要加入，請省略補充指令。
-   使用完整句子和句子樣式的大小寫。

-   **偏好泛型標籤上的特定標籤。** 在理想的情況下，使用者不需要讀取任何其他人來瞭解標籤。

    -   **例外狀況：** 已廣泛使用 [不要再顯示此訊息] 核取方塊標籤。

### <a name="supplemental-explanations"></a>補充說明

    在此範例中，選項按鈕標籤為粗體，可讓您更輕鬆地進行掃描。

-   **將補充說明新增至群組中的一個控制項，並不表示您必須為群組中的所有其他控制項提供說明。** 如果您只有在必要時才可以和使用說明，請在標籤中提供相關資訊。 沒有只重新聲明一致性標籤的補充說明。

    ![三個選項按鈕的螢幕擷取畫面 ](images/text-ui-image46.png)

    在此範例中，群組中的兩個控制項包含補充說明，但第三個則否。

-   如果有補充說明，請在命令連結之後，以第二個人撰寫補充文字。

    **範例：** 命令連結：建立無線網路設定並儲存至 USB 快閃磁片磁碟機

    補充說明：這會建立您可以使用 USB 快閃磁片磁碟機傳送至路由器的設定。 只有當您有支援 USB 快閃磁片磁碟機設定的無線路由器時，才需要這麼做。

-   使用完整的句子和結束標點符號。
