# <a name="layout"></a>Layout

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

版面配置是視窗或頁面中內容的大小調整、間距和位置。 有效的版面配置對於協助使用者找到所需的內容，以及讓外觀以視覺效果吸引人來說非常重要。 有效的版面配置可讓使用者立即瞭解的設計，以及讓使用者感覺 puzzled 和不知所措的不同。

**注意：** 與 [視窗管理](win-window-mgt.md) 相關的指導方針會在個別的文章中顯示。 建議的特定控制項大小調整和間距會顯示在其各自的指導文章中。

## <a name="design-concepts"></a>設計概念

### <a name="visual-hierarchy"></a>視覺化階層

當視窗或頁面的外觀指出其元素的關聯性和優先權時，它會有一個清楚的視覺階層。 如果沒有視覺化階層，使用者就必須自行瞭解這些關聯性和優先權。

視覺化階層是藉由 skillfully 結合下列屬性來達成：

-   **重點。** 版面配置會指出使用者需要先查看的位置。
-   **流。** 您可以透過介面中的清楚路徑，以流暢且自然的方式處理視覺流程，並依適當的順序尋找使用者介面 (UI) 元素。
-   **分組。** 邏輯相關的 UI 元素具有明確的視覺化關聯性。 相關專案會群組在一起;不相關的專案是分開的。
-   **重點。** UI 元素會根據其相對重要性強調。
-   **對準。** UI 元素有協調的放置，因此很容易掃描並依序顯示。

此外，有效的版面配置具有下列屬性：

-   **裝置獨立性。** 無論字型的字型或大小、每英寸 (DPI) 、顯示器或圖形介面卡，版面配置都會顯示為預期。
-   **易於掃描。** 使用者可以找到他們正在尋找的內容。
-   **效率。** 需要很大的 UI 元素，以及小型工作的元素很小。
-   **大小.** 如果有説明，視窗可以調整大小，而且其內容版面配置有效，而不論介面的大小或小。
-   **平衡。** 內容會平均分散在表面上。
-   **Visual 簡潔。** 您認為配置的情況不如需要的複雜。 使用者不會覺得版面配置的外觀。
-   **一致性。** 類似的視窗或頁面使用類似的版面配置，因此使用者一律會感覺導向。

雖然調整大小、間距和位置是簡單的概念，但使用版面配置的挑戰是為了達成這些屬性的適當組合。

在 Windows 中，會使用與裝置無關的計量（例如對話單位 (Dlu) 和相對圖元）來傳達版面配置。

### <a name="a-design-model-for-reading"></a>用於讀取的設計模型

**使用者可以選擇內容外觀和組織所讀取的內容。** 若要建立有效的版面配置，您必須瞭解使用者通常會看到哪些內容，以及原因。

您可以使用此設計模型來進行版面配置決策，以進行讀取：

-   人們以左至右、由上到下的順序讀取 (西方文化特性) 。
-   讀取的模式有兩種：沉浸式閱讀和掃描。 沉浸式閱讀的目標是要理解。

    ![等量朗讀模式的紅色箭號圖 ](images/vis-layout-image1.png)

    此圖表會將沉浸式閱讀模型。

    相反地，掃描的目標是找出事物。 整體掃描路徑看起來像這樣：

    ![對角線閱讀模式的紅色箭號圖 ](images/vis-layout-image2.png)

    此圖表會將掃描模型。

    ![向下和弧形模式中的紅色箭號圖 ](images/vis-layout-image3.png)

    如果頁面的左邊緣有文字正在執行，使用者會先掃描左邊緣。

-   使用軟體時，使用者不會在 UI 本身的工作中可以沉浸。 因此，使用者通常不會讀取其掃描的 UI 文字。 然後，他們只會在他們認為需要的時候，全面讀取文字的部分。
-   使用者傾向于略過頁面左側或右側的導覽窗格。 使用者辨識了它們，但只在想要流覽的時候查看流覽窗格。
-   使用者傾向于略過大型未格式化的文字區塊，而不需要完全讀取。

    ![顯示掃描文字之箭號的文字圖表 ](images/vis-layout-image4.png)

    使用者通常會在掃描時跳過大型的文字和流覽窗格。

-   所有專案都相等，使用者會先查看視窗的左上角，在頁面上掃描，然後在右下角結束掃描。 它們通常會忽略左下角。

    ![頁面和左上至右下箭號的圖表 ](images/vis-layout-image5.png)

    所有專案都相等，使用者將依下列順序讀取這些數位：1、2、4和3。

-   但是在互動式 UI 中，並非所有專案都相等，因此不同的 UI 元素會收到不同的注意層級。 使用者傾向于查看互動式控制項，尤其是視窗左上角的控制項和醒目顯示的文字。

![具有清晰和模糊文字的畫面圖 ](images/vis-layout-image6.png)

使用者將焦點放在主要的互動式控制項和重要的主要指示，並只在需要時才查看其他專案。

-   使用者傾向于讀取互動式控制項標籤，特別是與完成工作相關的程式。 相較之下，使用者通常會在他們認為需要時才讀取靜態文字。
-   出現的專案會有不同的吸引。 粗體文字和大型文字會從一般文字呈現出來。 色彩或彩色背景上的 UI 專案會顯得不上。具有圖示的專案會從沒有圖示的專案中移除。
-   除非使用者有原因，否則不會進行滾動。 如果 [超過折](glossary.md) 迭的內容沒有提供要滾動的原因，它們就不會出現。
-   一旦使用者決定要做什麼之後，他們會立即停止掃描並進行。
-   因為使用者在認為已完成時才會停止掃描，所以通常會忽略超出完成點的任何內容。

![鍵盤選項的螢幕擷取畫面 ](images/vis-layout-image7.png)

使用者認為完成時，會停止掃描。

當然，這個一般模型會有一些例外狀況。 眼睛追蹤裝置表示實際使用者的行為相當不穩定。 此模型的目標是要協助您做出良好的決策和取捨，而不是精確地建立使用者行為的模型。 但是，當您閱讀這份清單時，希望您也已辨識過許多自己的閱讀模式。

### <a name="designing-for-scanning"></a>設計掃描

**使用者不會讀取、掃描，因此您應該設計 UI 介面以進行掃描。** 不要假設使用者會以由左至右、由上到下的順序來讀取文字，而是會查看 UI 元素以吸引他們的注意。

若要設計掃描：

-   假設使用者一開始先快速掃描整個視窗，然後以大致順序讀取 UI 元素：
    -   中心內的互動式控制項
    -   認可按鈕
    -   在其他地方找到互動式控制項
    -   主要指示
    -   補充說明
    -   顯示警告圖示的文字
    -   視窗標題
    -   主要主體中的其他靜態文字
    -   註腳
-   將在左上角或左上角起始工作的 UI 元素放在上方。
-   將完成工作的 UI 元素放在右下角。
-   可能的話，請在互動式控制項上放置重要的文字，而非靜態文字。
-   避免將重要的資訊放在左下角或長滾動控制項或頁面的底部。
-   不要呈現大型的文字區塊。 消除不必要的文字。 使用 [反向金字塔](text-ui.md) 呈現樣式。
-   如果您採取一些動作來吸引使用者的注意，請務必注意這一點。

可能的話，請使用此模型，而不是將它對抗;但有時候您需要強調或反強調特定的 UI 元素。

強調主要 UI 元素：

-   將主要 UI 元素放在 [掃描路徑](glossary.md)中。
-   將任何 UI 置於左上角或左上角，以起始工作。
-   將認可按鈕放在右下角。
-   將其餘的主要 UI 放在中央。
-   使用吸引人注意的控制項，例如命令按鈕、命令連結和圖示。
-   使用顯著文字，包括大型文字和粗體文字。
-   Put text 使用者必須在互動式控制項中或透過圖示或 [橫幅](vis-graphic.md)來讀取。
-   在淺色背景使用深色文字。
-   以空間括住元素。
-   不需要任何互動，例如指標或暫留，即可看到您要強調的元素。

    ![windows 啟用選項的螢幕擷取畫面 ](images/vis-layout-image8.png)

    這個範例示範許多強調主要 UI 元素的方式。

若要取消強調次要 UI 元素：

-   將次要 UI 元素放在掃描路徑之外。
-   讓使用者通常不需要在視窗左下角或底部看到任何使用者。
-   使用不會吸引注意的控制項，例如工作連結，而不是命令按鈕。
-   使用標準或灰色文字。
-   在深色背景上使用淺文字。 暗灰色或藍色背景上的白色文字效果很好。
-   以最短的空間括住元素。
-   請考慮使用 [漸進式洩漏](ctrl-progressive-disclosure-controls.md) 來隱藏次要 UI 元素。

    ![大型和小型介面元素的螢幕擷取畫面 ](images/vis-layout-image9.png)

    此範例示範許多方法來消除次要 UI 元素的強調。

### <a name="using-screen-space-effectively"></a>有效使用螢幕空間

若要有效率地使用螢幕空間，您必須平衡數個因素：使用太多空間，而視窗的負荷很高且浪費，甚至很難根據 [Fitts 的法律](inter-mouse.md)來使用。

**不正確：**

![顯示太多空白字元的螢幕擷取畫面 ](images/vis-layout-image10.png)

在此範例中，視窗對其內容而言太大。

另一方面，使用的空間太少，而且視窗覺得 cramped、不舒服和很嚇人，而且如果需要滾動和其他操作，就很難使用。

**不正確：**

![具有太多控制項的螢幕擷取畫面 ](images/vis-layout-image11.png)

在此範例中，視窗對其內容而言太小。

雖然重要的 UI 必須符合最小支援的 [有效解析度](glossary.md)，但請不要假設使用螢幕空間，這表示 windows 應該盡可能地越小。 **有效的版面配置與開放空間有關，且不會嘗試將所有專案 cram 到可能的最小空間。** 新式顯示器有明顯的螢幕空間，如果可以的話，也可以有效地使用此空間。 因此，在使用太多螢幕空間，而不是太少的情況下，錯誤。 這樣做可讓您的 windows 感覺更輕且更平易近人。

您知道在下列情況中，配置會有效地使用螢幕空間：

-   視窗、視窗窗格和控制項不需要調整大小就能使用。 如果使用者的第一件事是調整視窗、窗格或控制項的大小，其大小就會錯誤。
-   資料不會被截斷。 清單視圖和樹狀檢視中的大部分資料沒有省略號，除非資料長度很大，否則不會裁剪其他控制項中的資料。 必須讀取才能執行工作的資料不應截斷。
-   視窗和控制項會適當地調整大小，以消除不必要的滾動。 有少數的水準捲軸和不需要的垂直捲動條。
-   控制項大多使用其標準大小。 例如，藉由在表面上只使用一或兩個命令按鈕寬度，來減少控制項大小的數目。
-   UI 介面已平衡。 沒有大量未使用的螢幕區域。

選擇夠大的視窗大小來滿足其用途。  (如果視窗是可調整大小的，則此目標會套用至其預設大小。 ) **截斷的資料或捲軸的組合，且有足夠的可用螢幕空間，就是無效配置的清楚符號。**

### <a name="control-sizing"></a>控制項大小調整

通常，使用螢幕空間的第一個步驟是決定各種 UI 元素的適當大小。 請參閱 [控制項大小調整表](#recommended-sizing-and-spacing) ，以及特定控制項指導方針文章中的建議大小。

Fitts 的定律指出，目標越小，以滑鼠取得它所需的時間就愈長。 此外，針對使用 Windows 平板電腦和觸控技術的電腦，「滑鼠」實際上可能是畫筆或使用者的手指，因此在決定小型控制項的大小時，您應該考慮替代的輸入裝置。 **16x16 相對圖元的控制項大小是適用于任何輸入裝置的最小大小。** 相較之下，標準15x9 相對圖元的微調控制項按鈕太小，無法供畫筆有效使用。

### <a name="spacing"></a>間距

提供大量的 (但) 空間不足，讓版面配置感覺更加舒適且更容易剖析。 有效空間不是未使用的空間，它扮演的重要角色是改善使用者掃描的能力，同時也增加了設計的視覺效果。 如需指導方針，請參閱 [間距表](#recommended-sizing-and-spacing)。

針對使用 Windows 平板電腦和觸控技術的電腦，「滑鼠」可能會實際是畫筆或使用者的手指。 使用畫筆或手指做為指標裝置時，鎖定目標會更困難，因而導致使用者在預期的目標之外進行點擊。 當互動式控制項放在一起時，但實際上並不觸及時，使用者可以按一下控制項之間的非使用中空間。 由於按一下非使用中空間沒有結果或視覺效果的意見反應，因此使用者通常不確定發生什麼錯誤。 如果小控制項的間距太大，則使用者需要利用精確的精確度來避免點擊錯誤的物件。 **若要解決這些問題，互動式控制項的目的地區域應該觸及或至少有3個 Dlu (5 個相對圖元之間的空間) 。**

您知道配置在下列情況中有良好的間距：

-   整體來說，UI 介面感覺很舒適，也不會覺得 cramped。
-   空間會以一致且平衡的空間顯示。
-   相關的專案會彼此封閉，不相關的元素則相對較遠。
-   要一起的控制項（例如工具列按鈕）之間沒有任何可用的空間。

### <a name="resizable-windows"></a>可調整大小的視窗

可調整大小的視窗也是有效使用螢幕空間的因素。 有些 windows 是由固定內容所組成，且不會因為可調整大小而受益，但是具有可調整大小內容的視窗應該可以調整 當然，使用者調整視窗大小的原因是要採用額外的螢幕空間，以便將內容提供給需要的 UI 元素更多空間來擴充。 具有動態內容、檔、影像、清單和樹狀結構的 windows，可從可調整大小的視窗中獲益。

![調整控制項取得捲軸大小的螢幕擷取畫面 ](images/vis-layout-image12.png)

在此範例中，調整視窗大小會調整清單視圖控制項的大小。

話雖如此，windows 也可以延伸到太寬。 例如，當內容寬度超過600的相對圖元時，許多控制台頁面都會變得很困難。 在此情況下，最好不要將內容區域的大小調整到超過這個最大寬度，或變更內容的原點，因為視窗的大小變大。 相反地，請保留最大寬度和固定左上角。

當行長度增加時，文字就會變得難以閱讀。 針對文字檔，請考慮使用最大的行長度80個字元，讓文字更容易閱讀。  (字元包含字母、標點符號和空格。 ) 

**不正確：**

![寬訊息方塊的螢幕擷取畫面，包含長文字 ](images/vis-layout-image13.png)

在此範例中，長文字長度會造成難以閱讀。

最後，可調整大小的視窗也需要使用較小的螢幕空間，方法是讓可調整大小的內容變得較小，並藉由從 UI 元素移除空間，而不需要它。 在某個時間點，視窗或其 UI 專案變得太小而無法使用，因此應該將大小下限指派給這些專案，或應該完全移除部分元素。

![具有高度、侵入式功能區之視窗的螢幕擷取畫面 ](images/vis-layout-image14.png)

![沒有功能區之視窗的螢幕擷取畫面 ](images/vis-layout-image15.png)

在此範例中，窗格有最小的大小。

有些程式受益于使用完全不同的簡報，讓內容可以使用較小的大小。

![中央媒體播放機按鈕的螢幕擷取畫面 ](images/vis-layout-image16.png)

在此範例中，Windows Media Player 會在視窗對標準格式變得太小時變更其格式。

### <a name="focus"></a>焦點

如果有一個明顯的位置要先查看，則配置具有焦點。 將焦點放在向使用者顯示開始掃描視窗或頁面的位置非常重要。 如果沒有清楚的焦點，使用者的眼睛將會穿梭 aimlessly。 焦點應該是使用者需要快速尋找和瞭解的重要內容，而且應該要有最大的視覺效果。 左上角是大多數視窗的自然焦點。

應該只有一個焦點。 就像在現實生活中一樣，只要一次就能專注于一件事，使用者就無法同時專注在多個地方。

若要讓 UI 元素成為焦點，您可以透過下列方式提供視覺效果強調：

-   將它放在表面的左上角或左上角部分。
-   使用重要且立即理解的互動式控制項。
-   使用顯著的文字，例如主要指令。
-   提供控制項預設選取和初始輸入焦點。
-   將控制項放在不同的彩色背景。

請考慮 Windows Search。 Windows Search 的焦點應該是 [搜尋] 方塊，因為它是工作的起點。 不過，它位於右上角，與標準搜尋方塊位置一致。 搜尋方塊具有輸入焦點，但在掃描路徑中會有其位置，而該線索本身並不足夠。

若要解決這個問題，視窗的中央部分會有明顯的指示，以將使用者導向至正確的位置。

### <a name="flow"></a>流程

當使用者透過其介面以明確的路徑進行逐步引導時，版面配置具有流程，並依適當的使用順序尋找 UI 元素。 使用者識別焦點之後，他們需要決定如何完成工作。 UI 元素的位置會傳達其關聯性，而且應該鏡像步驟來執行工作。 一般來說，這表示工作的步驟應該以從左至右、由上到下的順序， (在西歐文化特性) 。

您知道配置在下列情況中有良好的流程：

-   UI 元素的位置會反映使用者執行工作所需的步驟。
-   初始工作的 UI 元素位於左上角或左上角。
-   完成工作的 UI 元素位於右下角。
-   相關的 UI 元素會一起;不相關的元素是分開的。
-   必要步驟位於主要流程中。
-   選擇性步驟不在主要流程之外，可能會使用適當的背景或漸進式洩漏來將其反白。
-   常用的元素會出現在掃描路徑中不常使用的元素之前。
-   使用者一律知道接下來該怎麼做。 工作流程中沒有未預期的跳躍或中斷。

**不正確：**

![混淆對話方塊版面配置的螢幕擷取畫面 ](images/vis-layout-image20.png)

在此範例中，使用者不知道接下來該怎麼做。 工作流程中有未預期的跳躍和中斷。

**正確：**

![經過妥善規劃之對話方塊的螢幕擷取畫面 ](images/vis-layout-image21.png)

在此範例中，UI 元素的呈現會反映執行工作的步驟。

### <a name="grouping"></a>群組

當邏輯上相關的 UI 元素具有明確的視覺化關聯性時，配置會進行群組。 群組很重要，因為使用者更容易瞭解並專注于一組與專案個別的相關專案。 群組可讓版面配置更簡單且更容易剖析。

您可以利用下列方式來顯示群組 (增加 heaviness) ：

-   **佈局。** 您可以將相關的控制項彼此相鄰分組，並在不相關的控制項之間放置額外的間距。

    ![四個圖示的圖，其中顯示四個工作群組 ](images/vis-layout-image22.png)

    在此範例中，只會使用版面配置來顯示控制項關聯性。

-   **分隔符號。** 分隔符號是統一一組控制項的水準或垂直線。 分隔符號提供更簡單、更清楚的外觀。 但是，與群組方塊不同的是，當它們橫跨整個表面時，效果最好。

    ![三個圖示和三個分隔符號的螢幕擷取畫面 ](images/vis-layout-image23.png)

    在此範例中，標示的分隔符號會用來顯示控制項關聯性。

-   **集成.** 匯總工具是在強相關控制項之間建立視覺關聯性的圖形。

    ![橢圓線條內控制項的螢幕擷取畫面 ](images/vis-layout-image24.png)

    在此範例中，界限匯總工具是用來強調控制項之間的關聯性，讓它們感覺像是單一控制項，而不是八個。

-   **群組方塊。** 群組方塊是圍繞一組相關控制項的標記矩形框架。

    ![矩形框線中核取方塊的螢幕擷取畫面 ](images/vis-layout-image25.png)

    在此範例中，群組方塊會圍住一組相關的控制項，並為其加上標籤。

-   **背景。** 您可以使用背景來強調或反強調不同類型的內容。

    ![[控制台] 左側的螢幕擷取畫面 ](images/vis-layout-image26.png)

    在此範例中，[控制台] 工作窗格可用來將相關的工作和控制台專案分組。

    為了避免視覺效果混亂，執行此作業的最小加權群組是最好的選擇。 如需詳細資訊，請參閱[群組方塊](ctrl-group-boxes.md) [、索引標籤、](ctrl-tabs.md)[分隔符號和背景](vis-graphic.md)。

無論群組樣式為何，您都可以使用縮排來顯示群組內控制項的關聯性。 彼此相等的控制項應該靠左對齊，而相依的控制項則會縮排12個 Dlu 或18個相對圖元。

![三個縮排控制項層級的螢幕擷取畫面 ](images/vis-layout-image27.png)

相依控制項是縮排12個 DLU 或18個相對圖元，其設計是核取方塊和選項按鈕與其標籤之間的距離。

您知道配置在下列情況中有良好的分組：

-   視窗或頁面最多有7個群組。
-   每個群組的用途都很明顯。
-   每個群組內的控制項關聯性很明顯，尤其是控制相依性。
-   群組可簡化內容，而不是讓它變得更複雜。

### <a name="alignment"></a>對齊

對齊是 UI 元素的協調位置。 對齊很重要，因為它可讓您更輕鬆地掃描內容，並影響使用者對視覺複雜度的認知。

在決定對齊時，有幾個要考慮的目標：

-   **簡化水準掃描。** 使用者可以水準讀取並尋找彼此旁的相關專案，而不會有任何麻煩的間隙。
-   **簡化垂直掃描。** 使用者可以掃描相關專案的資料行，並立即找出他們要尋找的專案，並以最基本的水準眼睛移動。
-   **極低的視覺效果複雜度。** 如果使用者有不必要的垂直對齊格線，則會察覺版面配置的視覺效果複雜。

### <a name="horizontal-alignment"></a>水平對齊

**靠左對齊**

因為是由左至右的讀取順序，所以最適合大部分的內容使用左對齊。 靠左對齊可讓您輕鬆地垂直掃描單欄式資料。

**靠右對齊**

靠右對齊是數值資料的最佳選擇，特別是 [數值資料的資料行](ctrl-text-boxes.md)。 靠右對齊也適用于 [認可按鈕](glossary.md) ，以及對齊右視窗邊緣的控制項。

![advanced search 向下箭號按鈕的螢幕擷取畫面 ](images/vis-layout-image28.png)

在此範例中，「高階搜尋漸進式洩漏」控制項會靠右對齊，因為它是針對右邊的視窗邊緣所放置。

**置中對齊**

置中對齊是最好的，適用于靠左或靠右對齊不適當或顯示不平衡的情況。

![中央媒體播放機控制項的螢幕擷取畫面 ](images/vis-layout-image29.png)

在此範例中，media player 控制項的中心為提供平衡的外觀。

不要將視窗內容置中填滿空間。

**不正確：**

![具有太多泛空白字元之視窗的螢幕擷取畫面 ](images/vis-layout-image30.png)

在此範例中，內容在可調整大小的視窗中不正確置中，以填滿空間。

### <a name="vertical-alignment"></a>垂直對齊

**元素頂端**

因為是由上到下的讀取順序，所以最高的對齊方式適用于大部分內容。 最上層對齊讓 UI 元素能以水準方式進行快速掃描。

**文字基準**

垂直對齊具有文字的控制項時，對齊文字基準以提供順暢的水準讀取流程。

**正確：**

![對齊按鈕和標籤文字的螢幕擷取畫面 ](images/vis-layout-image31.png)

**不正確：**

![按鈕和標籤文字的螢幕擷取畫面未對齊 ](images/vis-layout-image32.png)

在正確的範例中，控制項及其標籤會根據其文字基準垂直對齊。

您知道配置在下列情況中有良好的對齊方式：

-   您可以輕鬆地水準和垂直掃描。
-   它有一個簡單的視覺外觀。

### <a name="label-alignment"></a>標籤對齊

一般對齊規則適用于控制項標籤，但這是值得特別注意的一般問題。 標籤對齊具有下列目標：

-   簡化垂直掃描以尋找正確的控制項。
-   簡化水準掃描以將標籤與其控制項產生關聯。
-   輕鬆進行當地語系化，處理跨語言的長度差異。
-   適用于混合不同的標籤長度。
-   可有效率地使用可用的空間，同時避免截斷的文字。

整體目標是要減少尋找使用者可能尋找的視覺移動量，但控制項的本質和使用者所要尋找的內容，取決於內容。

有四個常見的標籤位置和對齊樣式，各有其優點：

-   控制項上方靠左對齊的標籤
-   控制項左邊的靠左對齊標籤
-   靠左對齊的標籤位於控制項左方，左邊的控制項靠左對齊
-   控制項左邊的靠右對齊標籤

**控制項上方靠左對齊的標籤**

這是最容易當地語系化的樣式，因為版面配置不會相依于標籤的長度，但會採用最多的垂直空間。

![清單中有兩個標籤資料行的控制項 ](images/vis-layout-image33.png)

此樣式會採用最垂直的空間，但最簡單的方式是當地語系化。 這是最適合用來標記互動式控制項的選項。

最佳使用時機：

-   標記的控制項是互動式的 (而不只是文字) 。
-   UI 將會當地語系化。 這種樣式通常會為標籤長度的兩倍或甚至三倍的空間。
-   UI 使用固定的版面配置技術 (例如 Win32) 。
-   有十個或更少的控制項。 使用更多控制項，標籤很難掃描。
-   有足夠的垂直空間可容納標籤。
-   版面配置必須是自由格式，而不只是資料行。

**控制項左邊的靠左對齊標籤**

此樣式是垂直掃描的最簡單方式，而且當標籤的長度很大時，也很適合用來將標籤與控制項建立關聯。 如有必要，此樣式可以使用多行標籤。

![包含四個標籤的標籤資料行的清單 ](images/vis-layout-image34.png)

這種樣式的效果很好。 不過，有兩個數據行，但視覺效果看起來像是四個讓資料顯得更複雜的資料。

最佳使用時機：

-   使用者可能會垂直掃描以尋找特定標籤。
-   使用者不太可能從左至右、由上到下的方式讀取標籤和控制項。
-   有足夠的水準空間可容納標籤。
-   標籤的長度會有很大的差異。
-   有許多控制項，例如表單。
-   有幾個資料行。 標籤和控制項會以視覺化方式顯示為兩個個別的資料行。

**靠左對齊的標籤位於控制項左方，左邊的控制項靠左對齊**

這種樣式可讓您輕鬆地垂直掃描標籤，以及水準掃描標籤和控制項，而且非常節省空間。但是更難垂直掃描控制項。 控制項會靠右對齊，以充分利用可用的空間。

![具有不完全控制項的兩個標籤資料行清單 ](images/vis-layout-image35.png)

這種樣式是精簡且容易閱讀的，但很難垂直掃描控制項。

最佳使用時機：

-   UI 使用變數配置技術 (例如 Windows Presentation Foundation) 。
-   使用者可能會垂直掃描以尋找特定標籤。
-   使用者很可能會以由左至右、由上到下的方式讀取標籤和控制項。
-   使用者不太可能會以垂直方式掃描控制項。
-   控制項文字的長度會有所不同，而且如果使用其他樣式，可能會被截斷。
-   控制項是唯讀的，例如唯讀文字方塊。 針對其他控制項，此對齊方式看起來會草率。 不過，按一下時，控制項可能會變成可編輯。
-   有許多資料行，但資料行中有少數控制項。

**控制項左邊的靠右對齊標籤**

這是水準讀取的最簡單方式，可將標籤與控制項建立關聯，但很難垂直掃描標籤，而且在 labelsList 具有縮排標籤和控制項的長度很大時，不會正常運作。

![具有縮排標籤和控制項的清單 ](images/vis-layout-image36.png)

這種樣式可讓您輕鬆地垂直掃描控制項，但很難垂直掃描標籤。

最佳使用時機：

-   使用者很可能會以由左至右、由上到下的方式讀取標籤和控制項。
-   使用者可能無法垂直掃描以尋找特定標籤，可能是因為：
    -   有幾個控制項。
    -   標籤很已知。
    -   控制項大多很容易理解，而且很少空白 (可能會有預設值，以防止空白控制項) 。
-   有足夠的水準空間可容納標籤。
-   標籤的長度不會有太大的差異。
-   有許多資料行。 標籤和控制項會以視覺化方式顯示為單一資料行。

不過，在採用上述任何一種樣式之前，請考慮兩個因素：

-   偏好可在整個程式中一致使用的樣式。
-   靠左對齊的標籤會位於控制項左邊的控制項上方，是最常見的樣式，因此應該獲得喜好設定。

### <a name="balance"></a>餘額

當視窗或頁面的內容平均分佈在其表面上時，就會有平衡。 如果介面實際具有與視覺效果相同的加權，則平衡的版面配置不會提示至一側。

最常見的餘額問題是視窗或頁面左側有太多內容。 您可以透過下列方式建立餘額：

-   在左側使用較大的邊界，而不是右邊。
-   放置用來完成右邊工作的 UI 元素。
-   將在整個工作中使用的 UI 元素放在中央。
-   延長可調整大小或多行控制項。
-   策略性地使用中央對齊。

![左邊的印表機和右邊的文字螢幕擷取畫面 ](images/vis-layout-image37.png)

這種平衡式的 wizard 頁面版面配置會顯示較大的左邊界，以提升餘額。

如果這些技術沒有達到平衡，請考慮減少視窗或頁面的寬度，使其更符合其內容。

針對可調整大小的表面，請勿將內容置中以達到平衡。 相反地，請維持固定的左上原點、定義最大介面區，以及在使用的空間內平衡內容。

### <a name="grids"></a>方格

方格是不可見的基礎對齊系統。 網格可以進行對稱，但非對稱式方格也可以運作。 由單一視窗或頁面使用時，方格可協助組織介面內的內容。 重複使用時，方格會跨表面建立一致的版面配置。

格線數目會影響視覺複雜度的認知。 格線較少的版面配置會比具有更多格線的版面配置更簡單。

**視覺效果複雜：**

![雜亂對話方塊的螢幕擷取畫面 ](images/vis-layout-image38.png)

**以視覺化方式簡單：**

![[組織] 對話方塊的螢幕擷取畫面 ](images/vis-layout-image39.png)

不必要的格線會造成視覺效果複雜度。

您知道在下列情況中，版面配置會有效地使用方格：

-   具有類似內容或功能的 Windows 或頁面具有類似的版面配置。
-   重複的設計項目會出現在視窗和頁面的類似位置中。
-   沒有不必要的垂直和水準對齊格線。

### <a name="visual-simplicity"></a>視覺效果簡化

視覺效果簡單明瞭的是，版面配置比所需更複雜。

當版面配置：

-   消除不必要的視窗 chrome 層。
-   使用最多七個容易識別的群組來呈現內容。
-   使用輕量群組，例如版面配置和分隔符號，而不是群組方塊。
-   使用輕量控制項，例如次要命令的連結而非命令按鈕，以及下拉式清單，而不是選項的清單。
-   減少垂直和水準對齊格線的數目。
-   例如，在表面上只使用一或兩個命令按鈕寬度，以減少控制項大小的數目。
-   在需要時，會使用漸進式洩漏來隱藏 UI 元素。
-   使用足夠的空間，讓視窗或頁面不會覺得 cramped。
-   適當地調整視窗和控制項的大小，以消除不必要的滾動。
-   使用具有少量大小和文字色彩的單一字型。

一般而言，如果可以消除版面配置元素而不會損害 UI 的效果，則可能會是。

## <a name="guidelines"></a>指導方針

### <a name="screen-resolution-and-dpi"></a>螢幕解析度和 DPI

-   **支援最小的 Windows 有效解析度（800x600 圖元）。** 對於必須在安全模式下運作的重要 Ui，支援有效的640x480 圖元解析度。 請務必考慮工作列所使用的空間，其方式是針對顯示在工作列上的 windows 保留48垂直 [相對圖元](glossary.md) 。
-   **優化可調整大小的視窗版面配置，以有效解決1024圖元。** 以仍可正常運作的方式，自動調整這些視窗的大小以取得較低的螢幕解析度。
-   **請務必以每英寸96點測試您的 windows (DPI)  (為800x600 圖元) 、120 DPI (1024x768 圖元) ，以及 144 DPI (1200x900 圖元) 模式。** 檢查是否有版面配置問題，例如剪輯控制項、文字和視窗的裁剪，以及縮放圖示和點陣圖。
-   **針對具有觸控和行動使用案例的程式，請針對 120 DPI 進行優化。** 高 DPI 螢幕目前在觸控和行動電腦上很普遍。

### <a name="window-size"></a>視窗大小

-   **選擇適合其內容的預設視窗大小。** 如果您可以有效地使用空間，請不要害怕使用較大的初始視窗大小。
-   **使用對稱的高度與寬度的外觀比例。** 建議您最好使用3:5 和5:3 之間的外觀比例，雖然 (的外觀1:3 比例可以用於訊息對話方塊例如) 錯誤和警告。
-   **請盡可能使用可調整大小的視窗，以避免捲軸和截斷的資料。** 具有動態內容、檔、影像、清單和樹狀結構的 windows，可從可調整大小的視窗中獲益。
-   **針對文字檔，請考慮使用最大的行長度80個字元** ，讓文字更容易閱讀。  (字元包含字母、標點符號和空格。 ) 
-   固定大小的視窗：
    -   **固定大小的 windows 必須是完全可見且大小調整，才能納入工作區域中。**
-   可調整大小的視窗：
    -   **可調整大小的視窗可能已針對較高的解析度進行優化，但在顯示時視需要調整為實際螢幕解析度的大小。**
    -   **逐漸放大的視窗大小必須顯示漸進詳細資訊。** 請確定至少有一個視窗部分或控制項具有可調整大小的內容。
    -   **當調整視窗大小時，保留內容的左上角。** 請勿在視窗大小變更時移動來源以平衡內容。
    -   **如果內容可能太寬，請設定內容大小上限。** 如果內容變得很困難，請不要將內容區域的大小調整到寬度超過最大寬度，或變更內容的原點，因為視窗的大小變大。 相反地，請保留最大寬度和固定左上角。
    -   **設定最小的視窗大小（如果大小低於此值，內容就無法再使用）。** 針對可調整大小的控制項，請將可調整大小下限的元素大小設定為其最小的功能大小，例如清單視圖中的最小功能性 選擇性的 UI 元素應完全移除。
    -   **請考慮改變簡報，讓內容可以使用較小的大小。**

        ![media player 控制項的螢幕擷取畫面 ](images/vis-layout-image16.png)

        在此範例中，Windows Media Player 會在視窗對標準格式變得太小時變更其格式。

### <a name="control-size"></a>控制項大小

-   **讓所有互動式控制項至少具有相對16x16 圖元。** 如此一來，所有輸入裝置（包括 Windows 平板電腦和觸控技術）都能順利運作。
-   **大小控制項以避免截斷的資料。** 請勿截斷必須讀取才能執行工作的資料。 大小清單視圖資料行，以避免截斷的資料。
-   **用以消除不必要滾動的大小控制項。** 如果這樣做會消除捲軸，讓控制項稍微放大。 應該不需要太多垂直捲動條，也不需要任何不需要的水準捲軸。

    ![清單大小的螢幕擷取畫面，以避免捲軸 ](images/vis-layout-image40.png)

    在此範例中，下拉式清單會調整大小以消除捲軸。

-   **減少表面上的控制項大小數目。** 偏好使用 [標準建議的控制項大小](#recommended-sizing-and-spacing) ，如有必要，請使用幾個一致大小的較大或較小的控制項。 請嘗試使用單一寬度的清單方塊和樹狀檢視，而且不超過三個命令按鈕和下拉式清單的寬度。 不過，文字方塊和下拉式方塊寬度應建議其最長或預期的輸入長度。

    ![具有清單和按鈕之對話方塊的螢幕擷取畫面 ](images/vis-layout-image41.png)

    在此範例中，會一致地使用一個清單方塊和命令按鈕大小。

-   **針對以其文字為基礎調整大小的控制項，請針對將當地語系化的任何文字，在較短的文字) 中包含額外的 30% (高達200%。** 此指導方針假設版面配置是使用英文文字所設計。 另外也請注意，此指導方針參考當地語系化的文字，而非數位。
-   **將靜態文字控制項、核取方塊和選項按鈕擴充至版面配置所容納的最大寬度。** 這樣做可避免截斷變數長度文字和當地語系化。

    在此範例中，會不必要地截斷控制項文字。

### <a name="control-spacing"></a>控制項間距

-   **如果控制項沒有觸及，則至少有3個 Dlu (5 個相對圖元之間的空間) 。** 否則，使用者可以按一下控制項之間的非使用中空間。 由於按一下非使用中空間沒有結果或視覺效果的意見反應，因此使用者通常不確定發生什麼錯誤。

### <a name="placement"></a>放置

-   **將介面中的 UI 元素排列成自然地以左至右、由上到下的順序， (在西歐文化特性中) 。** UI 元素的位置會傳達其關聯性，而且應該鏡像步驟來執行工作。
-   **將在左上角或左上角起始工作的 UI 元素放在上方。** 提供使用者應該先查看最大視覺效果的 UI 元素。
-   **將完成工作的 UI 元素放在右下角。**
-   **將相關的 UI 元素放在一起，並分隔不相關的元素。**
-   **將必要的步驟放置在主要流程中。**
-   **在主要流程之外放置選擇性步驟，** 可能會使用適當的背景或漸進式洩漏來將其反白。
-   **將經常使用的元素放** 在掃描路徑中不常使用的元素之前。

### <a name="focus"></a>焦點

-   **選擇使用者必須先查看的單一 UI 元素，才能成為焦點。** 焦點應該是使用者需要快速尋找和瞭解的重要事項。
-   **將焦點放在左上角或左上角。**
-   **讓焦點成為最大的視覺效果，** 例如醒目的文字、預設選項或初始輸入焦點。

### <a name="alignment"></a>對齊

-   一般來說，使用靠左對齊。
-   針對數值資料使用靠右對齊，尤其是數值資料的資料行。
-   針對認可按鈕使用靠右對齊，以及對齊右視窗邊緣的控制項。
-   當靠左或靠右對齊不適當或顯示不平衡時，請使用中央對齊。
-   垂直對齊具有文字的控制項時，對齊文字基準以提供順暢的水準讀取流程。
-   針對標籤對齊，請參閱設計概念中的 [標籤對齊](#label-alignment) 區段。

### <a name="accessibility"></a>協助工具選項

-   **請勿使用版面配置作為傳達 UI 相關重要資訊的唯一方法。** 具有視覺障礙的使用者可能無法解讀此簡報。 例如，請確定控制項標籤會將其關聯性傳達給其他專案。
-   **請勿將次級控制項內嵌在控制項標籤中，以建立一個句子或片語。** 這類關聯純粹是以配置為基礎，而且不是由鍵盤流覽或協助工具輔助技術來妥善處理。 此外，這項技術並不會當地語系化，因為句子結構會因語言而異。

    **不正確：**

    ![句子中間文字方塊的螢幕擷取畫面 ](images/vis-layout-image43.png)

    在此範例中，文字方塊不正確地放在核取方塊標籤內。

    **正確：**

    ![句子結尾文字方塊的螢幕擷取畫面 ](images/vis-layout-image44.png)

    在此，文字方塊會置於核取方塊標籤之後。

-   **將群組設為可存取。** 視窗窗格、群組方塊、分隔符號、文字標籤和匯總工具所定義的群組，都是由協助工具輔助程式自動處理。 不過，只有位置和背景定義的群組不是，且必須以程式設計的方式定義以供存取。

如需詳細指導方針，請參閱 [協助工具](inter-accessibility.md)。
