# <a name="toolbars"></a>工具列

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新版本的 Windows 更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

工具列是一種將命令分組以進行有效率存取的方式。

![兩個具有標示元素之工具列的螢幕擷取畫面 ](images/cmd-toolbars-image1.png)

一些典型的工具列。

**除了功能表列之外，還可使用工具列。** 工具列的效率比功能表列更有效率，因為它們是直接的 (一律會顯示，而不是在滑鼠按一下) ，立即 (，而不需要額外的輸入) 並且包含最常使用的命令 (而不是完整的清單) 。 相較于功能表列，工具列不一定要有完整的功能，也不容易理解，只是快速、方便且有效率。

某些工具列是可自訂的，可讓使用者新增或移除工具列、變更其大小和位置，甚至變更其內容。 某些類型的工具列可以取消停駐，並產生 [調色板] 視窗。 如需有關工具列種類的詳細資訊，請參閱本文中的 [使用模式](#usage-patterns) 。

> [!Note]  
> 與 [功能表](cmd-menus.md)、 [命令按鈕](ctrl-command-buttons.md)和 [圖示](vis-icons.md) 相關的指導方針會顯示在不同的文章中。

 

## <a name="is-this-the-right-user-interface"></a>這是正確的使用者介面嗎？

若要決定使用時機，請考量下列問題：

-   **視窗是否為主視窗？** 工具列適用于主視窗，但通常對次要視窗而言很龐大。 針對次要視窗，請改用 [命令按鈕](ctrl-command-buttons.md)、 [功能表按鈕](ctrl-command-buttons.md)和 [連結](ctrl-command-links.md) 。
-   **是否有少量常用的命令？** 工具列無法處理任意數目的命令做為功能表列，因此效果最適合用來有效率地存取少量常用的命令。
-   **大部分的命令都會立即進行嗎？** 也就是說，它們大多是不需要額外輸入的命令嗎？ 為了有效率，工具列必須有直接且立即的感覺。 如果不是，功能表列更適合需要額外輸入的命令。
-   **可以直接呈現大部分的命令嗎？** 也就是說，使用者只要按一下，就能與他們互動呢？ 雖然有些命令可以使用功能表按鈕來呈現，但以這種方式呈現大部分的命令破壞工具列的效率，讓功能表列成為更好的選擇。
-   **命令是否適當地以圖示表示？** 工具列按鈕通常是以圖示而非文字標籤來表示 (雖然某些工具列按鈕會使用這兩個) ，而功能表命令則是以其文字表示。 如果命令圖示不是高品質且不容易理解，則功能表列可能是較佳的選擇。

如果您的程式具有沒有功能表列的工具列，且大部分的命令都可以透過功能表按鈕和 [分割按鈕](ctrl-command-buttons.md)間接存取，則此工具列基本上就是功能表列。 改為套用功能表指導方針中的 [工具列功能表](cmd-menus.md) 模式。

## <a name="design-concepts"></a>設計概念

良好的功能表列是所有可用最上層命令的完整目錄，而良好的工具列可讓您快速輕鬆地存取常用的命令。 工具列不會嘗試訓練使用者，只是讓使用者能夠有效率地進行定型。 一旦使用者學習如何存取工具列上的命令，就很少會從功能表列繼續存取命令。 基於這些理由，程式的功能表列和工具列不需要直接對應。

### <a name="toolbars-vs-menu-bars"></a>工具列與功能表列

傳統上，工具列是以下列方式與功能表列不同：

-   **頻率。** 工具列只會顯示最常使用的命令，而功能表列會將程式中所有可用的最上層命令類別目錄。
-   **即時。** 按一下工具列命令會立即生效，而功能表命令可能需要額外的輸入。 例如，功能表列中的 [列印] 命令會先顯示 [列印] 對話方塊，而 [列印] 工具列按鈕則會立即將檔的單一複本列印到預設印表機。

    ![已選取工具列 [印表機] 按鈕的螢幕擷取畫面 ](images/cmd-toolbars-image2.png)

    在此範例中，按一下 [列印] 工具列按鈕會立即將檔的單一複本列印到預設印表機。

-   **直接。** 只要按一下就可以叫用工具列命令，而功能表列命令則需要流覽功能表。
-   **數位和密度。** 工具列所需的螢幕空間與其命令的數目成正比，而且一律會使用空間，即使命令不是也一樣。 因此，工具列必須有效率地使用其空間。 相反地，功能表列命令通常會從 view 隱藏，而且其階層式結構允許任何數目的命令。
-   **大小和簡報。** 若要將許多命令封裝在一個小空間中，工具列通常會使用以工具提示為基礎的標籤 (的圖示型命令) ，而功能表列則使用以文字為基礎的命令 (並) 選擇性的圖示。 雖然工具列按鈕可以有標準文字標籤，但使用的空間會大幅增加。

    ![具有傳送/接收標籤的工具列螢幕擷取畫面 ](images/cmd-toolbars-image3.png)

    標記的工具列按鈕至少需要三倍的空間來作為未標記的空間。

-   **一目了然。** 設計完善的工具列需要最容易理解的圖示，因為使用者只需要使用工具提示就能有效率地找到命令。 但是，如果一些較不常用的命令不容易理解，工具列仍能順利運作。

    ![具有熟悉圖示的工具列螢幕擷取畫面 ](images/cmd-toolbars-image4.png)

    在此範例中，最常使用的圖示很容易理解。

-   **可辨識且可分辨。** 針對經常使用的命令，使用者會記住工具列按鈕屬性，例如位置、圖形和色彩。 使用設計完善的工具列，使用者即使不記得確切的圖示符號，也可以快速找到命令。 相反地，使用者會記住常用的功能表列命令位置，但會依賴命令標籤來進行選取。

    ![[剪取工具選項] 對話方塊的螢幕擷取畫面 ](images/cmd-toolbars-image5.png)

    針對工具列命令、獨特的位置、圖形和色彩，有助於讓圖示可辨識且可分辨。

    ![選取 [檔案] 命令的功能表列螢幕擷取畫面 ](images/cmd-toolbars-image6.png)

    若為功能表列命令，使用者最終會依賴其標籤。

### <a name="efficiency"></a>效率

根據其特性，工具列必須設計為高效率。 沒有效率的工具列就不會有任何意義。

**如果您只做一件事 .。。**

請確定您的工具列主要是為了提高效率而設計。 將工具列焦點放在經常使用、立即、直接且快速辨識的命令上。

### <a name="hiding-menu-bars"></a>隱藏功能表列

一般而言，工具列可搭配功能表列一起運作：良好的工具列提供效率和良好的功能表列提供全面性。 **有兩個功能表列和工具列都可讓他們專注于其優勢，而不會受到危害。**

令人驚訝的是，此模型會使用簡單的程式來細分。 對於只有幾個命令的程式，同時擁有功能表列和工具列並不合理，因為功能表列最後是一個多餘、沒有效率的工具列版本。

為了消除這種冗余，Windows Vista 中的許多簡單程式都只著重于透過工具列提供命令，並預設隱藏功能表列。 這類套裝程式括 Windows 檔案總管、Windows Internet Explorer、Windows Media Player 和 Windows 影像中心。

這是不太小的變更。 移除功能表列基本上會改變工具列的本質，因為這類工具列需要完整且以下列方式變更：

-   **頻率。** 移除功能表列表示所有無法直接從視窗或其內容功能表中使用的命令都必須可從工具列存取，不論其使用頻率為何。
-   **即時。** 移除功能表列可讓工具列成為命令的唯一可見存取點，要求工具列必須有功能完整的版本。 例如，如果沒有功能表列，工具列上的 [列印] 命令必須顯示 [列印] 對話方塊，而不是立即列印。  (雖然在這種情況下，使用分割按鈕是很棒的入侵。 請參閱標準 [列印分割] 按鈕的 [標準功能表和分割按鈕](#standard-menu-and-split-buttons) 。 ) 

    ![工具列和列印命令之選項的螢幕擷取畫面 ](images/cmd-toolbars-image7.png)

    在此範例中，Windows 影像中心中的 [列印] 工具列按鈕具有顯示 [列印] 對話方塊的 [列印] 命令。

-   **直接。** 為了節省空間並防止雜亂，較不常使用的命令可能會移到功能表按鈕，使其更不直接。

用來補充功能表列的工具列設計方式與設計用來搭配已移除或隱藏的功能表列的工具列不同。 而且因為您無法假設使用者將顯示隱藏的功能表列來執行單一命令，所以隱藏功能表列的方式應該和在進行設計決策時完全移除一樣。  (如果您預設隱藏功能表列，請不要假設使用者會想要顯示功能表列來尋找命令，甚至是找出如何顯示。 ) 

設計工具列以在沒有功能表列的情況下運作，通常會涉及一些折衷。 但為了提高效率，請不要入侵太多。 如果隱藏功能表列會導致沒有效率的工具列，請勿隱藏功能表列！

### <a name="keyboard-accessibility"></a>鍵盤協助工具

從鍵盤存取工具列與存取功能表列相當不同。 當使用者按下 ALT 鍵時，功能表列會收到輸入焦點，而且會失去使用 Esc 鍵的輸入焦點。 如果功能表列具有輸入焦點，則會獨立流覽視窗的其餘部分，並處理所有箭號、首頁、結尾和 Tab 鍵。 相反地，當使用者按下 Tab 鍵時，工具列會透過視窗的整個內容來接收輸入焦點。 因為工具列是最後一個定位順序，所以在忙碌頁面上啟動時可能需要投入大量的時間 (，除非使用者知道要使用 Shift + Tab 來向後移動) 。

協助工具在此提供一個難題：雖然滑鼠使用者更容易使用工具列，但鍵盤使用者的存取權較小。 如果功能表列和工具列都有，但如果功能表列已移除或隱藏，就不會有問題。

基於協助工具的考慮，請偏好保留功能表列，而不是完全以工具列的方式移除。 如果您必須選擇移除功能表列，並直接隱藏它，則偏好隱藏它。

## <a name="guidelines"></a>指導方針

### <a name="presentation"></a>簡報

-   **根據命令數目和其使用方式，選擇適當的工具列樣式。** 請參閱先前的工具列樣式表，以取得如何選擇的指引。 請避免使用在程式工作區中佔用太多空間的工具列設定。
-   **將工具列放在內容區域的正上方、** 功能表列和網址列下方（如果有的話）。
-   **如果空間是 premium，請透過下列方式節省空間：**

    -   省略已知圖示的標籤和較不常使用的命令。
    -   只使用部分工具列，而不是整個視窗寬度。
    -   使用功能表按鈕或分割按鈕來合併相關命令。
    -   使用 [溢](glossary.md) 位箭號來顯示較不常使用的命令。
    -   只有當命令套用至目前的內容時，才會顯示命令。

    ![工具列一般圖示未標示的螢幕擷取畫面 ](images/cmd-toolbars-image28.png)

    [Windows Internet Explorer] 工具列會藉由省略已知圖示的標籤、使用部分工具列，並針對較不常使用的命令使用溢位符號來節省空間。

-   **針對未加上標籤的圖示工具列模式，請使用預設設定，而不是兩個以上的工具列列。** 如果有兩個以上的資料列可能很有用，請將工具列設為可自訂。 從兩個以上的資料列開始可能會讓使用者負荷過多，並從程式工作區域佔用太多空間。

    **不正確：**

    ![功能表列和三列工具列的螢幕擷取畫面 ](images/cmd-toolbars-image29.png)

    具有兩列以上工具列的預設設定會導致太多視覺效果雜亂。

-   **停用未套用至目前內容的個別工具列按鈕，** 而不是將它們移除。 這樣做可讓工具列內容穩定且更容易尋找。
-   **停用個別的工具列按鈕，如果按一下它們會直接導致錯誤。** 您必須這麼做才能維持直接的感覺。
-   **針對未標記的圖示工具列模式，移除整個工具列（如果未套用至目前的內容）。** 只顯示在適用的模式中。

    ![[偵錯工具] 工具列的螢幕擷取畫面 ](images/cmd-toolbars-image30.png)

    在此範例中，只有在執行程式時，才會顯示 Debug 工具列。

-   **顯示靠左對齊的工具列按鈕。** [說明] 圖示（如果有的話）會靠右對齊。

    ![工具列的螢幕擷取畫面、靠右對齊的說明圖示 ](images/cmd-toolbars-image31.png)

    除了 [說明] 之外，所有工具列按鈕都會保持對齊。

    **例外狀況：** Windows 7 樣式工具列會靠左對齊程式專用的命令，但會靠右對齊標準、已知的命令，例如選項、View 和 Help。

-   **請勿動態變更工具列按鈕標籤。** 這麼做會造成混淆和非預期的情況。 不過，您可以變更圖示以反映目前的狀態。

    ![值區圖示杯畫的螢幕擷取畫面 ](images/cmd-toolbars-image26.png)

    在此範例中，圖示會變更以表示預設命令。

### <a name="controls-and-commands"></a>控制項和命令

-   **偏好最常使用的命令。**
    -   **針對主要工具列，請提供完整的命令。** 主要工具列並不需要像功能表列一樣完整，但必須提供所有不容易在其他地方探索的命令。 主要工具列不需要有下列命令：
        -   直接位於 UI 本身的命令。
        -   通常會透過內容功能表來存取命令。
        -   標準的知名命令，例如剪下、複製和貼上。
    -   **針對補充的工具列，提供最常使用的命令。** 功能表列命令是工具列命令的超集合，因此您不需要提供任何專案。 專注于快速且方便的命令存取，並略過其餘部分。
-   **偏好直接控制項。** 使用工具列按鈕的喜好設定順序如下：
    -   **圖示按鈕。** 直接使用，並以基本空間為限。
    -   **標示的圖示按鈕。** 直接，但需要更多空間。
    -   **分割按鈕。** 直接取得最常見的命令，但會處理命令的變化。
    -   **功能表按鈕。** 間接，但是提供許多命令。
-   **偏好立即命令。** 適用于可以立即進行或有額外輸入以提供彈性的命令：
    -   針對主要工具列，請使用彈性版本的命令， (例如 [列印] ) 。
    -   針對補充的工具列，請使用工具列中的 [立即] 版本 (例如 [列印) ]，然後在功能表列中使用彈性的版本 (例如 [列印 ...] ) 。
-   **提供常用命令的標籤，** 特別是如果它們的圖示不是已知的圖示。

    **可以接受：**

    ![沒有標示圖示的工具列螢幕擷取畫面 ](images/cmd-toolbars-image32.png)

    **較佳：**

    ![已標示一些圖示的工具列螢幕擷取畫面 ](images/cmd-toolbars-image33.png)

    Windows 傳真和掃描工具列有幾個命令，因此較佳的版本標籤最重要的標籤。

-   請勿將命令放在也直接在工具列上的工具列功能表中。

    **不正確：**

    ![工具列和功能表上的 [列印] 命令的螢幕擷取畫面 ](images/cmd-toolbars-image34.png)

    在此範例中，Print 直接位於工具列上，因此不需要在功能表中。

### <a name="organization-and-order"></a>組織與訂單

-   **將工具列內的命令組織成相關群組。**
-   **先放置最常使用的群組。在群組中，將命令放在其邏輯順序中。** 整體來說，這些命令應該要有邏輯流程，以方便尋找，同時仍會先顯示最常使用的命令。 這麼做是最有效率的方式，特別是在發生溢位時。
-   **只有在群組之間的命令弱結合時，才使用群組分隔。** 這樣做會讓群組清楚明瞭，並讓命令更容易找到。

    ![螢幕擷取畫面，顯示使用群組分隔圖示的工具列。](images/cmd-toolbars-image35.png)

    ![具有妥善組織圖示的工具列螢幕擷取畫面 ](images/cmd-toolbars-image36.png)

    從 Windows Mail 分組的工具列範例。

-   **避免在常用的命令旁放置破壞性命令。** 使用順序或群組來取得分隔。 此外，請考慮不要將破壞性命令放在工具列中，只是在功能表列或快顯功能表中。

    **可以接受：**

    ![相鄰 [列印] 和 [刪除] 按鈕的螢幕擷取畫面 ](images/cmd-toolbars-image37.png)

    **較佳：**

    ![分隔 [列印] 和 [刪除] 按鈕的螢幕擷取畫面 ](images/cmd-toolbars-image38.png)

    在更好的範例中，Delete 命令實際上是與 Print 分開的。

-   **使用溢位箭號表示並非所有命令都可以顯示。** 但只有在沒有足夠空間可顯示所有命令時，才使用溢位。

    **不正確：**

    ![[我的最愛] 列和隱藏命令的螢幕擷取畫面 ](images/cmd-toolbars-image39.png)

    溢位箭號表示並非所有的命令都顯示出來，但有更多命令可能會有較佳的版面配置。

-   **請確定最常使用的命令可以從工具列直接存取 (也就是，不會在小型視窗大小中) 溢位。** 如有必要，請重新排序命令、將較不常使用的命令移至功能表按鈕或分割按鈕，或甚至完全從工具列移除它們。 如果問題仍然存在，請重新考慮您選擇的工具列樣式。

### <a name="hiding-menu-bars"></a>隱藏功能表列

一般而言，工具列可與功能表列一起運作，因為兩者都可讓每個使用者專注于其優勢，而不會受到危害。

-   如果您的工具列設計有多餘的功能表列，預設會隱藏功能表列。
-   隱藏功能表列，而不是完全移除功能表列，因為鍵盤使用者更能存取功能表列。
-   若要還原功能表列，請在 [主要工具列] 的 [視圖] (中提供功能表列核取記號選項) 或次要工具列) 功能表類別的 [工具] (。 如需詳細資訊，請參閱 [標準功能表和分割按鈕](#standard-menu-and-split-buttons)。
-   當使用者按下 ALT 鍵，並在第一個功能表類別上設定輸入焦點時，會顯示功能表列。

-   **提供視窗管理命令的內容功能表。** 當使用者以滑鼠右鍵按一下標題列時，顯示此內容功能表。

    在此範例中，使用者可以在標題列上按一下滑鼠右鍵，以顯示快顯功能表。

-   **盡可能讓選擇區視窗變成可調整大小。** 指出視窗可以調整大小，並在視窗框架上方使用調整大小指標。
-   **當您重新顯示 [元件] 視窗時，會使用與上次存取相同的狀態來顯示它。** 當關閉時，請儲存視窗大小和位置。 重新調整時，請還原已儲存的視窗大小和位置。 此外，請考慮以每個使用者為基礎，將這些屬性保持在不同的程式實例之間。

### <a name="customization"></a>自訂

-   **針對由兩個或多個資料列所組成的工具列提供自訂。** 只有未標記的圖示樣式需要自訂。 只有少數命令的簡單工具列不需要自訂。
-   **提供良好的預設設定。** 在一般情況下，使用者不應自訂其工具列。 不要相依于使用者自訂不正確初始設定的方式。 假設大部分的使用者不會自訂其工具列。
-   **提供具有下列命令的內容功能表：**
    -   顯示可用工具列的核取方塊清單
    -   鎖定/解除鎖定工具列
    -   通知區域的 [自訂...]
-   **預設會鎖定可自訂的工具列**，以防止意外變更。
-   **針對 [自訂] 命令，顯示 [選項] 對話方塊** ，讓您可以選擇要顯示的工具列以及每個工具列上的命令。

    ![[自訂] 對話方塊和選項的螢幕擷取畫面 ](images/cmd-toolbars-image46.png)

    在此範例中，Visual Studio 提供選項對話方塊來自訂其工具列。

-   提供重設命令，以回到 [自訂選項] 對話方塊中的原始工具列設定。
-   **使用拖放功能，可讓您透過下列方式自訂工具列：**

    -   設定工具列順序和位置。
    -   設定工具列長度，顯示太小而無法顯示其內容與溢位的任何工具列。
    -   如果支援，請取消停駐工具列，使其成為調色盤視窗，反之亦然。

    當 [自訂選項] 對話方塊顯示時：

    -   設定工具列內容。
    -   設定工具列內容的順序。

    這麼做可讓使用者以更直接且有效率的方式進行變更。

-   以每一使用者為基礎 **，儲存所有工具列自訂** 專案。

### <a name="using-ellipses"></a>使用省略號

雖然工具列命令可用於立即動作，但有時需要更多資訊才能執行此動作。 使用省略號表示命令需要更多的資訊，才會生效。 將省略號放在工具提示和標籤的結尾（如果有的話）。

![列印工具提示文字與省略號的螢幕擷取畫面 ](images/cmd-toolbars-image47.png)

在此範例中，Print .。。命令會顯示 [列印] 對話方塊，以收集詳細資訊。

但是，如果命令無法立即生效，則不需要省略號。 因此，例如，共用設定沒有省略號，雖然它需要額外的資訊，因為命令可能不會立即生效。

![工具列、命令和工具提示的螢幕擷取畫面 ](images/cmd-toolbars-image48.png)

共用設定命令沒有省略號，因為它無法立即生效。

由於工具列會不斷地顯示，而空間是在 premium 上，因此 **應該不常使用省略號**。

> [!Note]  
> 針對工具列所顯示的功能表，套用 [功能表省略號指導方針](cmd-menus.md)。

### <a name="unlabeled-icon-buttons"></a>未標記的圖示按鈕

### <a name="menu-buttons-and-split-buttons"></a>功能表按鈕和分割按鈕

-   **偏好以動詞為基礎的功能表按鈕名稱。** 但是，如果動詞命令是建立、顯示、查看或管理，請省略該動詞。 例如，[ **工具** ] 和 [ **頁面** ] 功能表按鈕沒有動詞。
-   **使用可清楚且準確地描述功能表內容的單一特定單字。** 雖然這些名稱不一定要用來描述功能表中的所有專案，但它們應該是可預測的，如此一來，使用者在功能表中找到的內容就不會驚訝。
-   **雖然不是必要的，但請提供提示描述（如果有説明）。**

### <a name="menu-items"></a>功能表項目

-   **使用以動詞、名詞或名詞片語開頭的功能表項目名稱。**
-   **偏好以動詞為基礎的功能表名稱。** 但是，如果動詞命令是建立、顯示、查看或管理，請省略該動詞。 例如，下列命令不會使用動詞命令：
    -   關於
    -   進階
    -   全螢幕
    -   新增
    -   選項
    -   屬性
-   **使用特定的動詞命令。** 避免泛型、無用動詞，例如變更和管理。
-   **針對適用于單一物件的命令使用單數名詞，否則請** 使用複數名詞。
-   **針對互補命令配對，選擇清楚互補的名稱。** 範例：新增、移除、顯示、隱藏;Insert、Delete。
-   **根據使用者目標和工作（而非技術），選擇功能表項目名稱。**
-   針對指定的目的使用下列功能表項目名稱：
    -   **選項：** 以顯示程式選項。
    -   **自訂：** 顯示特別與機械 UI 設定相關的程式選項。
    -   **個人化：** 顯示常用 [個人](glossary.md) 化設定的摘要。
    -   **喜好設定：** 請勿使用。 請改用選項。
    -   **屬性：** 以顯示物件的屬性視窗。
    -   **設定：** 請勿使用做為功能表標籤。 請改用選項。
-   **顯示子功能表的功能表項目在其標籤上絕對沒有省略號。** 子功能表箭號表示需要另一個選取專案。

## <a name="documentation"></a>文件

參考工具列時：

-   如果只有一個工具列，請將它視為工具列。
-   如果有多個工具列，請依名稱參考這些工具列，後面接著單字工具列。 請參閱預設為開啟的主要工具列，並包含基本工作（例如開啟和列印檔案）的按鈕，作為標準工具列。
-   工具列是單一的 uncapitalized 單字。  (相反的，功能表列則是兩個單字。 ) 
-   依工具列按鈕的工具提示標籤來參考。 使用確切的標籤文字，包括其大小寫，但不包含任何省略號。
-   依工具列的標籤和 [word] 功能表來參考工具列功能表按鈕。 使用確切的標籤文字，包括其大小寫。
-   請將工具列控制項一般視為工具列按鈕。
-   若要描述使用者互動，請使用工具列按鈕和唯讀下拉式清單的 [按一下]，然後輸入可編輯的下拉式清單。 請勿使用 [選擇]、[選取] 或 [挑選]。
-   除非在程式設計檔中，否則請不要使用串聯、下拉式、下拉式或快顯來描述功能表按鈕。
-   請參閱無法使用的專案，而不是暗灰色、已停用或灰色。 在程式設計檔中使用 disabled。
-   可能的話，請使用粗體文字來格式化標籤。 否則，請只在必要時才將標籤放在引號中，以避免混淆。

範例：

-   在工具列的 [ **頁面** ] 功能表上，按一下 [ **以電子郵件傳送頁面**]。
-   在工具列上的 [字型] 方塊中，輸入「Segoe UI **」** 。
-   在 [ **格式化** ] 工具列上，指向 [ **顯示**]，然後按一下 [ **批註**]。
