# <a name="touch"></a>觸控

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

所有 Microsoft Windows 的應用程式都應該有絕佳的觸控體驗。 建立這類體驗比您想像的簡單。

觸控指的是使用一或多個手指透過裝置顯示來提供輸入，並與 Windows 和應用程式互動。 觸控優化應用程式有一個 UI 和互動模型，其設計目的是要容納較大型、較不精確的連絡人區域、觸控裝置的各種外型規格，以及許多做法和掌握使用者在使用觸控裝置時可能採用的程式。

![使用者使用觸控與平板電腦互動](images/inter_touch_image1.jpeg)

每個輸入裝置都有其優點。 鍵盤最適合用來輸入文字，並為命令提供最少量的手移動。 滑鼠最適合有效率、精確的指標。 觸控最適合用於物件操作並提供簡單的命令。 畫筆最適合用於自由格式的運算式，如同手寫和繪圖。

Windows 8.1 已針對回應性、準確度和易用性進行優化，同時完全支援傳統的輸入方法， (例如滑鼠、畫筆和鍵盤) 。 傳統輸入模式所提供的速度、精確度和 tactile 意見反應，對許多使用者來說都很熟悉且吸引人，而且可能更適合特定的互動案例。

您可以在個別主題中找到與滑鼠、畫筆和協助工具相關的指導方針。

當您考慮應用程式的互動體驗時：

請勿假設 UI 適用于滑鼠，也適用于觸控。 雖然良好的滑鼠支援是一開始，但良好的觸控體驗有一些額外的需求。

請假設，如果 UI 適用于手指，也適用于畫筆。 讓您的應用程式可觸式變得很長，也可以提供良好的畫筆支援。 主要的差異在於手指有 blunter 提示，因此需要較大的目標。

有了觸控功能，您就可以直接操作物件和 UI，以提供更快速、更自然且吸引人的體驗。

## <a name="provide-a-great-touch-experience"></a>提供絕佳的觸控體驗

您應該確保使用者可以使用觸控輸入有效率地執行重要和重要工作。 不過，特定應用程式功能（例如文字或圖元操作）可能不適合觸控，而且可以保留給最適合的輸入裝置。

如果您在開發觸控應用程式時沒有太多經驗，最好是透過進行學習。 取得具備觸控功能的電腦，將滑鼠和鍵盤放在旁邊，然後只使用您的手指與您的應用程式互動。 如果您有平板電腦，請嘗試將它保存在不同的位置，例如在您的膝上上、平面上的平面上，或在您的手中。 請嘗試在直向和橫向方向使用它。

適用于觸控互動的觸控優化應用程式通常是：

-   **自然又直覺。** 互動的設計目的是要與使用者在真實世界中與物件互動的方式相對應。
-   **較不具侵入性。** 使用觸控是無訊息的，因此會比輸入或按下更少干擾。
-   **可擕式。** 觸控裝置更精簡，因為許多工具可在沒有鍵盤、滑鼠、畫筆或觸控板的情況下完成。 它們也更有彈性，因為不需要工作介面。
-   **直接且吸引人。** 觸控讓您覺得直接操作螢幕上的物件。
-   **較不精確。** 相較于滑鼠或畫筆，使用者無法正確地以觸控的方式鎖定物件。

觸控可提供自然的真實世界風格來進行互動。 直接操作和動畫藉由為物件提供逼真、動態的移動和意見反應，來完成這項印象。 例如，請考慮使用卡片遊戲。 不但方便且輕鬆地使用手指拖曳卡片，當您可以像是實體的牌一樣，也能以吸引人的真實世界感覺來滑行和旋轉卡片。 當您嘗試移動無法移動的卡片時，有更好的體驗可讓卡片受到保護，但不能防止移動，並且在釋出時就地向後復原，以清楚地指出已辨識該動作，但無法完成。

幸運的是，如果您的應用程式已經過妥善設計，提供絕佳的觸控體驗很簡單。 基於這個目的，設計完善的程式：

-   **確保最重要的工作可以使用手指有效率地執行** (至少不涉及大量輸入或詳細圖元操作) 的工作。
-   **使用大型控制項進行觸控。** 通用控制項的最小大小為23x23 圖元 (13x13 Dlu) ，而最常使用的控制項至少是40x40 圖元 (23x22 Dlu) 。 為了避免沒有回應的行為，UI 元素至少應該要有5個圖元 (3 個 Dlu 之間的空格) 。 針對其他控制項，請確定它們至少有一個23x23 圖元 (13x13 DLU) 按一下 [目標]，即使其靜態外觀較小也一樣。 請參閱標準控制項大小調整。
-   **支援滑鼠輸入。** 互動式控制項具有明確的可見 affordances。 物件具有標準滑鼠互動的標準行為 (單一和按兩下、按一下滑鼠右鍵、拖放) 。
-   **支援鍵盤輸入。** 應用程式會提供標準的快速鍵指派，特別是針對也可以透過觸控手勢產生的導覽和編輯命令。
-   **確保存取範圍。** 使用消費者介面自動化或 Microsoft Active Accessibility (MSAA) ，以程式設計方式存取輔助技術的 UI。 應用程式會適當地回應方向、主題、地區設定和系統計量變更。
-   **消除不必要的互動。** 若要防止資料或系統存取遺失，請使用最安全且最安全的預設值。 如果安全性和安全性不是因素，則應用程式會選取最有可能或方便的選項。
-   **提供暫留的對等觸控。** 請勿以滑鼠停留作為執行動作的唯一方法。
-   **確保手勢立即生效。** 將連絡人點保持在使用者的手中，順暢地在筆勢中，這會將手勢對應的效果直接提供給使用者的動作。
-   **盡可能使用標準手勢。** 僅適用于您的應用程式特有互動的自訂筆勢。
-   **確保可以反轉或更正不想要或破壞性的命令。** 使用觸控時，很可能會發生意外的動作。

## <a name="guidelines-for-touch-input"></a>觸控輸入的指導方針

有了觸控，您的 Windows 應用程式就可以使用實體手勢來模擬 UI 元素的直接操作。

設計具備觸控功能的應用程式時，請考慮下列最佳做法：

**回應能力對於建立直接且吸引人的觸控體驗而言是不可或缺的。** 若要直接操作，手勢必須立即生效，而物件的接觸點必須在使用者的手中順暢地停留在整個手勢中。 觸控輸入的效果應該會直接對應到使用者的動作，因此，例如，如果使用者將手指旋轉90度，則物件也應旋轉90度。 任何延遲、不穩定的回應、連絡人遺失或不正確的結果，都可終結直接操作和品質的認知。

**一致性對於建立自然且直覺的觸控式體驗而言是不可或缺的。** 當使用者學習標準手勢之後，他們會預期筆勢在所有應用程式中都具有相同的效果。 為了避免混淆和挫折，絕對不要將非標準意義指派給標準手勢。 相反地，請使用自訂手勢進行程式的唯一互動。

接下來，我們將說明 Windows 的觸控式語言，但在繼續之前，以下是基本觸控輸入詞彙的簡短清單。

-   **手勢**

    手勢是在輸入裝置 (手指、手指、畫筆/手寫筆、滑鼠等) 上執行的實體 act 或動作。 例如，若要啟動、啟動或叫用命令，您可以使用單一手指點來執行觸控或觸控板裝置， (相當於滑鼠左鍵、使用畫筆來點一下，或輸入鍵盤) 。

-   **操作**

    操作是物件或 UI 對手勢的即時反應或回應。 例如，投影片和滑動手勢通常會讓元素或 UI 以某種方式移動。

    操作的最終結果、畫面上的物件和 UI 中的物件所顯示的結果是互動。

-   **互動**

    互動取決於操作的解讀方式，以及操作所產生的命令或動作。 例如，您可以使用投影片和滑動手勢來移動物件，但結果會因距離臨界值是否超過而有所不同。 您可以使用投影片來拖曳物件或平移視圖，並使用滑動來選取專案或顯示應用程式行。

### <a name="the-windows-touch-language"></a>Windows 觸控語言

Windows 提供一組在整個系統中使用的精確觸控互動。 以一致的方式套用這種觸控語言，可讓您的應用程式感到熟悉的使用者。 這樣可讓您的應用程式更容易學習和使用，進而提高使用者的信心。 若要深入瞭解觸控語言的執行，請參閱手勢、操作和互動。

**按住以學習**

> [!IMPORTANT]
> 您可以在水準和垂直移動都啟用的情況下，使用按住以進行選取。

移動：沒有動作。

結束狀態：最後一個手指會結束手勢。

效果：顯示詳細資訊。

可惜的是，觸控技術並不支援暫止，所以使用者在使用手指時無法使用滑鼠停留。 這個問題的簡單解決方法是充分利用 [暫止]，但只能以執行動作所需的方式來使用。 在實務上，這通常表示動作也可以藉由按一下來執行，但不一定會以完全相同的方式執行。

**捏合和伸展以進行縮放**

### <a name="control-usage"></a>控制使用方式

在這裡，我們會提供一些指導方針，讓您優化觸控使用的控制項。

-   **使用通用控制項。** 最常見的控制項是設計來支援良好的觸控體驗。
-   **選擇設計來支援觸控的自訂控制項。** 您可能需要自訂控制項來支援程式的特殊體驗。 選擇自訂控制項，其：
    -   大小可以大到足以方便進行目標和操作。
    -   在操作時，移動和反應真實世界物件的移動和反應方式，例如有動力和摩擦。
    -   的容許方式是讓使用者可以輕鬆地更正錯誤。
    -   是誤差的容許，並按一下和拖曳。 放在其目的地附近的物件應該會落在正確的位置。
    -   當手指在控制項上方時，有清楚的視覺效果意見反應。
-   **使用受限制的控制項。** 清單和滑杆等受限制的控制項，在設計方便觸控目標時，可以比不受限制的控制項（例如文字方塊）更好，因為它們可減少文字輸入的需要。
-   **提供適當的預設值。** 依預設，請選取最安全的 (，以防止遺失資料或系統存取) 和最安全的選項。 如果安全性和安全性不是因素，請選取最有可能或方便的選項，以避免不必要的互動。
-   **提供文字自動完成。** 提供最可能值或最近輸入值的清單，讓文字輸入更容易。
-   **針對使用多重選取的重要** 工作，如果通常使用標準的多重選取清單，請提供選項來改用核取方塊清單。

### <a name="control-sizes-and-touch-targeting"></a>控制項大小和觸控目標

由於 fingertip 的大型介面區，太接近的小型控制項可能很難精確地鎖定目標。

一般而言，23x23 圖元的控制項大小 (13x13 Dlu) 是適用于任何輸入裝置的最小互動式控制項大小。 相反地，15x11 圖元的微調控制項太小，無法有效使用觸控。

![顯示向上和向下選取按鈕的寬度和高度、測量 9 Dlu (15 圖元) 寬 x 5 Dlu (9 圖元) 高的螢幕擷取畫面。](images/inter-touch-image14.png)

請記住，大小下限實際上是以實體區域為基礎，而不是圖元或 Dlu 等版面配置度量。 研究表示使用手指進行有效率、準確互動的最小目的地區域是6x6 毫米 (mm) 。 此區域會轉譯成如下的版面配置度量：



| 字型             | 公釐 | 相對圖元 | 單位  |
|------------------|-------------|-----------------|-------|
| 9點 Segoe UI | 6x6         | 23x23           | 13x13 |
| 8點 Tahoma   | 6x6         | 23x23           | 15x14 |



 

此外，研究顯示 10x10 mm 的最小大小 (40x40 圖元) 可提供更佳的速度和正確性，也覺得更適合使用者使用。 在可行的情況下，針對最重要或常用命令所使用的命令按鈕使用此較大的大小。

目標不是有巨大的控制項，只是很容易接觸觸控的控制項。

![顯示 Microsoft Word 工具列的螢幕擷取畫面，其中已醒目提示 [A B C 拼寫 & 文法] 按鈕，並以 41 dlu 高度和 40 DLU 寬度顯示。](images/inter-touch-image15.png)

在此範例中，Microsoft Word 使用大於 10x10 mm 的按鈕來取得最重要的命令。

![顯示 Windows 計算機的螢幕擷取畫面。](images/inter-touch-image16.png)

此版本的計算機針對其最常使用的命令使用大於 10x10 mm 的按鈕。

觸控目標沒有完美的大小。 不同的大小適用于不同的情況。 具有嚴重後果的動作 (例如刪除和關閉) 或經常使用的動作，都應該使用大型的觸控目標。 不常使用的動作若有輕微的結果，則可以使用較小的目標。

**自訂控制項的目標大小指導方針**



| 大小指導方針                                                                                 | Description                                                                                                                                                                                                                                                                                 |
|----------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| ![7x7 建議的最小大小](images/inter_touch_image10.jpeg)<br/>      | **7x7 mm：建議的最小大小**<br/> 如果觸及錯誤的目標可在一或兩個手勢中或在五秒內更正，7x7 mm 是很好的最小大小。 目標之間的填補與目標大小一樣重要。<br/>                               |
| ![9x9 精確度的建議大小](images/inter_touch_image11.jpeg)<br/> | **當精確度很重要時**<br/> 具有嚴重後果的關閉、刪除和其他動作，無法承受意外的點擊。 如果觸及錯誤的目標需要兩個以上的手勢、五秒或主要內容變更才能正確，請使用 9x9 mm 目標。<br/>     |
| ![5x5 大小下限](images/inter_touch_image12.jpeg)<br/>                  | **當它不符合時**<br/> 如果您發現自己 cramming 要符合的專案，只要觸及錯誤的目標，就可以使用一個手勢來修正。 在此情況下，在目標之間使用 2 mm 填補非常重要。<br/> |



 

**通用控制項的目標大小指導方針**

-   **針對通用控制項，請使用建議的控制項大小。** 建議的控制項調整大小滿足23x23 圖元 (13x13 DLU) 最小大小，除了核取方塊和選項按鈕 (其文字寬度會稍微補償) 、微調控制項 (無法使用觸控，但卻是重複的) 和分隔器。

    ![顯示通用控制項範例的螢幕擷取畫面，包括音訊控制項、[立即流覽網際網路] 按鈕，以及檔案總管視窗。](images/inter-touch-image17.png)

    建議的控制項大小很容易可觸式。

-   **針對最重要或經常使用的命令所使用的命令按鈕，請盡可能使用40x40 圖元的最小大小 (23x22 Dlu) 。** 這樣做會產生更佳的速度和精確度，而且對使用者來說也更熟悉。

    ![顯示多個電子郵件 [傳送] 按鈕的螢幕擷取畫面，從左至右最小到最大的大小。](images/inter-touch-image18.png)

    在實際情況下，請使用較大的命令按鈕來進行重要或常用的命令。

-   **針對其他控制項：**

    -   **使用較大的點擊目標。** 若為小型控制項，請將目標大小設為大於靜態可見的 UI 元素。 例如，16x16 圖元圖示按鈕可以有23x23 圖元的 [目標] 按鈕，而文字元素的選取範圍會比文字和23圖元高的選取矩形8圖元。

        不正確：

        ![螢幕擷取畫面，顯示具有不正確大小之目的地區域的 U I 樹狀結構。](images/inter-touch-image20.png)

        正確：

        ![顯示具有正確大小之目的地區域之 U I 樹狀結構的螢幕擷取畫面。](images/inter-touch-image21.png)

        在正確的範例中，按一下目標大於靜態可見的 UI 元素。

    -   **使用重複的按一下目標。** 如果控制項有重複的功能，則可接受點擊目標小於最小大小。

        例如，樹狀檢視控制項所使用的漸進式洩漏三角形僅6x9 圖元，但其功能在其相關聯的專案標籤中是多餘的。

        遵循系統計量。 請勿將大小硬式編碼。 如有必要，使用者可以變更系統計量或 DPI 來滿足其需求。 不過，將此視為最後的手段，因為使用者通常不應該調整系統設定來讓 UI 可供使用。

編輯文字

使用手指時，編輯文字是最具挑戰性的互動之一。 使用受限制的控制項、適當的預設值和自動完成功能，可以消除或減少輸入文字的需要。 但是，如果您的應用程式牽涉到編輯文字，您可以在使用觸控時，自動將輸入 UI 自動縮放至150%，讓使用者更具生產力。

例如，電子郵件程式可能會以一般可觸式大小顯示 UI，但會將輸入 UI 縮放至150% 來撰寫訊息。

![顯示電子郵件 U I 的螢幕擷取畫面。](images/inter-touch-image24.png)

### <a name="control-layout-and-spacing"></a>控制項版面配置和間距

控制項之間的間距是讓控制項易於可觸式的重要因素。 使用手指作為指標裝置時，鎖定目標會更快速但較不精確，因此使用者通常會在其預定目標以外的地方點擊。 當互動式控制項放在一起時，但實際上並不觸及時，使用者可以按一下控制項之間的非使用中空間。 因為按一下非使用中空間沒有結果或視覺效果的意見反應，所以使用者通常不確定發生什麼錯誤。

**根據所使用的輸入裝置動態調整間距。** 這在暫時性的 UI （例如功能表和 flyouts）中特別有用。

**在互動式控制項的目的地區域之間，提供最少5圖元的 (3 Dlu) 空間。** 如果小控制項的間距太大，則使用者需要利用精確的精確度來避免點擊錯誤的物件。

**使用超過控制項之間建議的垂直間距，讓群組內的控制項更容易區分。** 例如，在19圖元高的選項按鈕，其小於建議的最小值23圖元。 當您有可用的垂直空間時，您可以將額外4圖元的間距新增至標準7圖元，以達到與建議大小大致相同的效果。

正確：

![顯示控制項之間垂直間距標準範例的螢幕擷取畫面。](images/inter-touch-image25.png)

較佳：

![顯示有更多垂直間距的控制項範例的螢幕擷取畫面。](images/inter-touch-image26.png)

在更好的範例中，選項按鈕之間的額外間距可讓您更輕鬆地區分。

在某些情況下，使用觸控（但不是使用滑鼠或鍵盤時）時，可能會需要額外的間距。 在這種情況下，只有在使用觸控來起始動作時，才使用更 spacious 的設計。

**選擇將控制項放在最可能使用之位置的版面配置。** 盡可能在社區域內保持工作互動，並找出靠近最可能使用的控制項。 避免長距離的移動，特別是針對一般工作和拖曳。

請考慮目前的指標位置是最接近的目標，因此很容易取得。 因此，操作功能表可充分利用 Fitts 的定律，就像 Microsoft Office 使用的迷你工具列一樣。

![顯示 Microsoft Office 並列的內容功能表和迷你工具列範例的螢幕擷取畫面。](images/inter-touch-image27.png)

**避免將小型控制項放在應用程式邊緣附近或顯示。** 接近邊緣的小型目標可能很難接觸 (顯示 bezels 可能會干擾邊緣手勢) 。 為了確保控制項在視窗最大化時可以輕鬆地設為目標，請將其設為至少23x23 圖元 (13x13 Dlu) 或將它們放在視窗邊緣以外。

**使用建議的間距。** 建議的間距適合觸控。 不過，如果您的應用程式可受益于較大的大小和間距，請考慮建議的大小調整，並在適當時調整間距。

**在互動式控制項之間提供至少5個圖元 (三個 Dlu) 空間。** 這樣做可避免使用者在想要的目標之外時，產生混淆。

請考慮在控制項群組（例如命令連結、核取方塊和選項按鈕，以及群組之間）新增超過建議的垂直間距。 這麼做可讓您更輕鬆地區分。

**當使用觸控來起始動作時，請考慮以動態方式新增超過建議的垂直間距。** 這麼做可讓物件更容易區分，但在使用鍵盤或滑鼠時，不會佔用更多空間。 將間距增加為其正常大小的第三個，或至少8個圖元。

### <a name="interaction"></a>互動

使用正確的控制項可讓您只是觸控優化應用程式的一部分，您也必須考慮這些控制項支援的整體互動模型。 以下是一些指導方針，可協助您進行此工作。

-   **將滑鼠暫留為多餘。** 大部分的觸控技術都不支援暫止，所以具有這類 touchscreens 的使用者無法執行任何需要停留的工作。
-   **對於需要文字輸入的應用程式，請透過下列方式將觸控鍵盤功能完全整合** ：
    -   為使用者輸入提供適當的預設值。
    -   適當時提供自動完成的建議。

    > [!Note]  
    > 開發人員：如需有關整合觸控鍵盤的詳細資訊，請參閱 [**ITextInputPanel**](/windows/desktop/api/peninputpanel/nn-peninputpanel-itextinputpanel)。

     

-   **如果您的程式有需要編輯文字的工作，可讓使用者縮放內容 UI。** 使用觸控時，請考慮自動縮放至150%。
-   **在適當的情況下提供順暢、回應式的移動和縮放。** 在平移或縮放之後快速重繪，以保持回應。 這麼做是為了讓直接操作感覺成為真正的直接操作所需。
-   **在移動流覽或縮放期間，請確定連絡人點在筆勢中保持在手指下。** 否則，就難以控制平移或縮放。
-   **因為手勢是記下的，所以請在應用程式之間指派一致的意義。** 請勿對具有固定語義的手勢提供不同意義。 請改用適當的應用程式特定手勢。

### <a name="forgiveness"></a>寬恕

直接操作可讓觸控自然、具表達性、有效率且吸引人。 不過，在有直接操作的情況下，可能會有意外的操作，因此需要容許。

容許可讓您輕鬆地反向或更正不想要的動作。 您可以藉由提供復原、提供良好的視覺反應、在常用的命令與破壞性命令之間進行清楚的實體分隔，讓使用者輕鬆地更正錯誤，來容許觸控體驗。 與容許相關聯，會防止在一開始發生不想要的動作，您可以使用限制的控制項，並確認具有非預期結果的具風險動作或命令。

-   **提供復原命令。** 最好提供簡單的方法來復原所有命令，但您的應用程式可能會有一些無法復原其效果的命令。
-   **只要可行，請在手指下提供良好的意見反應，但不要採取動作，直到跟上為止。** 這麼做可讓使用者在他們進行之前更正錯誤。
-   **只要可行，就能讓使用者輕鬆地更正錯誤。** 如果動作會在手指上生效，可讓使用者在手指仍關閉時滑動，以修正錯誤。
-   **在實際情況下，表示 resisting 移動時無法執行直接操作。** 允許進行移動，但在釋出時，物件會就地向後復原，以明確指出已辨識該動作，但無法完成。
-   **在經常使用的命令和破壞性命令之間有清楚的實體分隔。** 否則，使用者可能不小心接觸到破壞性命令。 如果命令的效果很大，而且無法輕易復原或效果無法立即察覺，則會將其視為具破壞性的命令。
-   **確認具有非預期結果的具風險動作或命令的命令。** 基於此目的，請使用確認對話方塊。
-   **請考慮確認使用者在使用觸控時可能會不小心執行的任何其他動作，而這可能會忽略或難以復原。** 一般來說，這些稱為「例行確認」，建議您根據使用者不常使用滑鼠或鍵盤來發出這類命令的假設。 若要避免不必要的確認，請只在使用觸控來起始命令時才顯示這些確認。

    使用者通常會不小心使用觸控的互動，可以接受例行確認。

    開發人員：您可以使用 [**輸入 \_ 訊息 \_ 來源**](/windows/win32/api/winuser/ns-winuser-input_message_source) API 來區分滑鼠事件和觸控事件。
