# <a name="how-to-design-a-great-user-experience-for-desktop-applications"></a>如何設計桌面應用程式的絕佳使用者體驗

> [!NOTE]
> 此設計指南是針對 Windows 7 所建立，而且尚未針對較新的 Windows 版本進行更新。 大部分的指引仍然適用于準則，但展示和範例不會反映我們目前的 [設計指引](/windows/uwp/design/)。

絕佳的桌面應用程式功能強大，而且也很簡單。 透過仔細平衡的特徵選取和呈現，您可以同時達到強大和簡單性。

**強大：**

**功能強大且簡單：**

![可能拼寫修訂清單的螢幕擷取畫面 ](images/powerful-simple-image2.png)

理想的 Windows 應用程式既強大又簡單。 當然，您想要讓應用程式具有強大的功能，當然您希望它變得很簡單，但您可以同時達到兩者？ 這些目標之間會有自然的張力，但這種程度的 irreconcilable。 您可以透過仔細平衡的特徵選取和簡報，達成您的威力和簡單性。

## <a name="what-makes-an-application-powerful"></a>什麼讓應用程式變得強大？

「電源」在軟體方面真正的意義為何？ 如果應用程式卡在紙上的功能不足，可能會被視為強大的功能，而且在嘗試所有使用者都有相當廣泛的功能。 這種設計不太可能會成功，因為 untargeted 的功能集不太可能滿足任何人的需求。 這並不是我們所用的電源類型。

當應用程式具有這些特性的正確組合時，就會有強大的功能：

-   **使。** 應用程式滿足其目標使用者的需求，讓他們能夠執行無法以其他方式執行的工作，並有效達成其目標。
-   **Efficient。** 應用程式可讓使用者在不可能的情況下，以更高的產能和規模來執行工作。
-   **多才多藝。** 應用程式可讓使用者在各種情況下，有效地執行各種工作。
-   **直接。** 應用程式看起來就像是直接協助使用者達成其目標，而不是取得或要求不必要的步驟。 快速鍵、鍵盤存取和宏等功能改善了 directness 的意義。
-   **靈活。** 應用程式可讓使用者完成其工作的精確控制。
-   **綜合。** 應用程式與 Microsoft Windows 緊密整合，讓它能夠與其他應用程式共用資料。
-   **進階：** 應用程式具有在競爭解決方案中找不到的卓越、創新、最先進的功能。

其中某些特性取決於使用者的認知，以及相對於使用者目前的功能。 視為強大的功能可能會隨著時間而改變，因此現今的先進搜尋功能可能會在未來很普遍。

所有這些特性都可以合併為我們的強大定義：

**當應用程式可讓其目標使用者有效率地發揮其完整潛力時，就會有強大的功能。**

因此，強大的電源量值是生產力，而不是功能的數目。

不同的使用者需要協助，才能以不同的方式達到其完整潛力。 針對某些使用者所啟用的功能可能會傷害他人的多樣化、directness 和控制。 設計完善的軟體必須適當地平衡這些特性。 例如，針對 nonprofessionals 而設計的桌面發佈系統可能會使用嚮導來引導使用者完成複雜的工作。 這類的嚮導可讓目標使用者執行他們無法執行的工作。 相較之下，適用于專業人員的桌面發佈系統可能著重于 directness、效率和完整的控制。 針對這類應用程式的使用者，可能會限制和挫折。

**如果您只做一件事 .。。**

瞭解您的目標使用者目標，並製作功能組，讓他們能夠有效率地達成這些目標。

## <a name="what-makes-a-user-experience-simple"></a>如何讓使用者體驗變得簡單？

我們定義了簡單的方法，如下所示：

**簡單來說，您可以減少或排除目標為使用者感知的設計屬性，並考慮 unessential。**

在實務上，您可以藉由選取正確的功能集，並以正確的方式呈現功能，來達成簡單性。 這可減少實際和認知的 unessential。

簡單性取決於使用者的認知。 考慮自動傳輸的效果如何取決於使用者的觀點：

-   針對 (目標使用者) 的一般驅動程式，自動傳輸不需要手動齒輪 shift 和離合器，讓汽車更容易推動。 手動齒輪 shift 和離合器對駕駛工作而言並不重要，因此會移除它們以達成簡潔性。
-   若為專業的競爭汽車驅動程式，可直接控制傳輸的關鍵是具競爭力。 自動傳輸會對車輛的效能造成負面影響，因此不會被視為簡化的結果。
-   針對技師修理，自動傳輸是更複雜的機制，因此不會比手動傳輸更容易修復或維護。 與技師修理不同的是，目標使用者知道不知道此內部複雜度。

雖然不同的使用者會以不同的方式來考慮自動傳輸，但它會成功，因為它不需要 unessential 其目標使用者的知識、技能和精力。 針對一般驅動程式，自動傳輸是很棒的功能，因為它是正常運作的功能。

### <a name="simplicity-vs-ease-of-use"></a>簡化與方便使用

在正確套用的情況下，簡單明瞭會導致使用起來更容易。 但簡單且便於使用的概念並不相同。 當使用者可以自行順利執行工作，而不會在適當的時間內發生困難或混淆時，就能輕鬆使用。 有許多方法可讓您輕鬆使用，簡單來說，減少 unessential，只是其中之一。

所有使用者只要複雜，就會想要以最少的不必要工作來完成工作。 所有使用者（甚至是 advanced users）主要是負責完成工作，而不是瞭解電腦或您的應用程式。

簡化是達成易用性的最有效方式，而且便於使用的 equals 用途。 但不會使用複雜、難以使用的功能。 相反地，執行其功能的簡單簡潔設計，是一項很好的使用方式。 他們叫用正面的情緒回應。

例如，請考慮 Microsoft Windows XP 中的無線網路支援。 Microsoft 可能已新增嚮導來引導使用者完成設定流程。 這種方法可讓您輕鬆使用，但無法簡化，因為 unessential 功能 (wizard) 已加入。 Microsoft 會改為自動設定無線網路功能。 使用者最後並不在意設定的詳細資料，只要它是可靠且安全地運作。 這種無線網路技術的威力和簡單性組合，導致其受歡迎且快速採用。

**如果您只做一件事 .。。**

使用工作正常的最簡單設計來開始設計流程。

如果您對目前的設計感到不滿意，請從移除所有 unessential 元素開始。 您會發現剩下的內容通常很好用。

## <a name="obtaining-simplicity-while-maintaining-power"></a>在維護電源時取得簡單性

### <a name="design-principles"></a>設計原則

為了簡單起見，請一律針對可能的設計進行設計，而不是可行的。

可能的

設計決策是以可能導致複雜的使用者介面（例如登錄編輯程式）為基礎，而設計會假設所有動作都同樣可能，因此需要相當的努力。 由於有任何可能的情況，因此不會在設計決策中考慮使用者目標。

可能的

設計決策是根據可能的簡化、目標和工作解決方案的解決方案，其中可能的案例會獲得焦點並需要最少量的精力來執行。

簡易性設計原則

**若要取得簡單的資訊，請專注于可能的情況;減少、隱藏或移除不太可能的情況;並消除不可能的情況。**

使用者所做的工作，與設計的相關程度更高。

### <a name="design-techniques"></a>設計技巧

若要取得更簡單的功能，同時保有電源，請選擇 **正確的功能組**， **在適當的位置** 找出功能，並減少使用這些功能 **的工作** 。 本節提供一些可達成這些目標的常見技術。

選擇正確的功能集

「達到完美的目的，而不需要新增任何專案，

但是，如果沒有任何離開的東西， — Antoine de Saint-Exupery

下列設計技術可為您的使用者提供所需的功能，同時透過實際的縮減或移除來達到簡易性：

-   **判斷您的使用者所需的功能。** 透過目標、案例和工作分析，瞭解您的使用者需求。 判斷瞭解這些目標的一組功能。
-   **移除不必要的元素。** 移除不太可能使用的元素，或有偏好的替代方案。
-   **移除不必要的冗余。** 可能有數種有效的方式可以執行工作。 為了達成簡單性，請進行硬性決策，並為目標使用者選擇最適合的專案，而不是提供所有選項並選擇選項。
-   **讓它自動「工作」。** 專案是必要的，但任何使用者互動以使其運作，都不是因為有可接受的預設行為或設定。 為了達成簡單性，請讓它自動運作，並將其完全隱藏在使用者之外，或大幅降低其暴露程度。

簡化簡報

「簡化方法以避免不必要的情況

這可能是必要的。」 — Hans Hofmann

您可以使用下列設計技術來保有電源，並透過減少或移除的認知來達成簡單性：

-   **結合應結合的內容。** 將支援工作的基本功能放在一起，讓工作可以在一個位置執行。 工作的步驟應該具有統一的簡化流程。 將複雜的工作細分為一組簡單、明確的步驟，讓「一」位置可能包含數個 UI 介面，例如 wizard。
-   **分隔應分隔的內容。** 並非所有專案都可以在單一位置呈現，因此一定要有清楚、明確選擇的界限。 製作支援核心案例的功能，並隱藏選用功能或使其成為週邊環境。 分隔個別的工作，並提供相關工作的連結。 例如，與管理相片相關的工作應該清楚地與管理相片集合相關的工作分開，但是應該可以輕鬆地彼此存取。
-   **消除可刪除的內容。** 製作您設計的列印，並醒目提示用來執行最重要工作的元素。 甚至會反白顯示可傳達有用資訊的 UI 文字中的個別文字。 現在複習未醒目提示的內容，並考慮將它從設計中移除。 如果您移除該專案，是否會發生任何錯誤？ 如果沒有，請移除！
-   一致性、可配置和一般化通常是理想的品質，但可能會導致不必要的複雜性。 檢查您的誤導在一致性 (的設計，例如有重複的文字) 、一般化 (例如，當兩個區域的足夠) 時，有任意數量的時區，以及可進行的配置 (，例如使用者不太可能變更) 的選項，以及消除可消除的內容。
-   **將專案放在正確的位置。** 在視窗中，元素的位置應該遵循其公用程式。 基本的控制項、指示和說明都應該以邏輯順序在內容中。 如果需要更多選項，請按一下箭號或類似的機制，在內容中公開這些選項。如果需要更多資訊，請在滑鼠停留時顯示資訊提示。 將不重要的工作、選項和說明資訊放在另一個視窗或頁面中的主要流程之外。 視需要顯示額外詳細資料的技巧稱為「漸進式洩漏」。
-   **使用有意義的高層級組合。** 選取和操作相關專案群組的方式通常比個別元素更簡單且更具擴充性。 高層級組合的範例包括資料夾、主題、樣式和使用者群組。 這類組合通常會對應至使用者目標，或不是個別元素明顯的意圖。 例如，高對比黑色色彩配置背後的目的遠比黑色視窗背景更明顯。
-   **選取正確的控制項。** 設計項目是由您用來表示它們的控制項所代表，因此選取正確的控制項對有效率的簡報而言很重要。 例如，Microsoft Word 使用的字型選取方塊會顯示字型的預覽，以及最近使用的字型。 同樣地，Word 顯示潛在拼寫和文法錯誤的方式比對話方塊的替代方案更簡單，如本文開頭所示。

減少投入時間

「簡單的事情應該很簡單。

應該要有複雜的東西。」— Alan Kay

下列設計技術可減少使用者的投入時間：

-   **讓工作可供探索並顯示。** 在使用者介面中，您應該可以輕鬆探索所有工作，但特別是經常發生的工作。 執行工作所需的步驟應該是可見的，且不應該依賴記住需求。
-   **在使用者的網域中顯示工作。** 複雜的軟體需要使用者將其問題對應到技術。 簡單的軟體是藉由呈現自然的方式來進行對應。 例如，紅色的縮減功能會直接對應到問題空間，而不需要使用者考慮像是色調和漸層等詳細資料。
-   **將領域知識放入計畫中。** 使用者不需要存取外部資訊，就能成功使用您的應用程式。 領域知識可以從複雜的資料和演算法，到單純地清楚輸入有效的型別。
-   **使用使用者瞭解的文字。** 設計完善的文字對於有效與使用者的通訊非常重要。 使用您的使用者熟悉的概念和詞彙。 完整說明以純語言詢問的內容，讓使用者能夠做出明智的明智決策。
-   **使用安全、安全、可能的預設值。** 如果設定在大部分情況下都有適用于大部分使用者的值，而且該設定是安全且安全的，請使用它作為預設值。 只有在必要時，才讓使用者指定值。
-   **使用條件約束。** 如果有許多方式可以執行工作，但只有部分是正確的，則會將工作限制為這些正確的方式。 不應允許使用者立即 preventable 錯誤。

### <a name="simplicity-does-not-mean-simplistic"></a>簡單性不表示簡單

「一切都應該儘量簡單，

但不簡單。」— Albert Einstein

我們相信簡單性對有效、期望的使用者體驗而言是很重要的，但永遠都可能會有太大的進展。 簡單性的本質是減少或排除 unessential。 移除必要項只會產生不良的設計。 如果您的「簡化」會導致使用者變得感到挫折、混淆、unconfident 或無法順利完成工作，您已移除太多工作。

### <a name="simplicity-does-mean-more-effort-for-you"></a>簡化對您而言的工作

「我已經花了太長的時間，

不是讓它變得較短的時間。」— Blaise Pascal

取得簡單性，同時保有電源通常需要大量的內部複雜性。 您通常可以更輕鬆地設計公開所有技術配管的軟體，而不是設計隱藏它的軟體，後者需要對目標使用者及其目標有絕佳的瞭解。 移除功能需要專業領域，因為它會決定是否要加入真正不實用的酷炫功能。 簡化需要進行硬性的設計選擇，而不是讓所有專案都可設定。 複雜的軟體通常是由誤解的使用者所造成：它們的價值是未使用的功能，或它們無法理解的過於複雜功能。

## <a name="powerful-and-simple"></a>功能強大且簡單

強大的功能就是讓您的使用者能夠更有效率地進行。 簡單的說，就是移除 unessential，並以正確的方式呈現功能。 藉由瞭解您的目標使用者並取得功能和簡報的適當平衡，您可以設計同時進行這兩項作業的 Windows 架構應用程式。
