# <a name="protocol-basics-listen-connect-accept"></a><span data-ttu-id="f9855-103">通訊協定基本概念：接聽、連接、接受</span><span class="sxs-lookup"><span data-stu-id="f9855-103">Protocol Basics: Listen, Connect, Accept</span></span>

<span data-ttu-id="f9855-104">在用戶端-伺服器範例中，建立通訊端連線所牽涉到的基本作業最方便說明。</span><span class="sxs-lookup"><span data-stu-id="f9855-104">The basic operations involved with establishing a socket connection can be most conveniently explained in terms of the client-server paradigm.</span></span>

<span data-ttu-id="f9855-105">伺服器端會先建立通訊端、將其系結至已知的本機位址 (以便讓用戶端找到) ，然後透過 [**WSPListen**](/previous-versions/windows/hardware/network/ff566297(v=vs.85))將通訊端置於接聽模式，以準備任何連入連線要求，並指定連線待處理專案佇列的長度。</span><span class="sxs-lookup"><span data-stu-id="f9855-105">The server side will first create a socket, bind it to a well known local address (so that the client can find it), and put the socket in listening mode, through [**WSPListen**](/previous-versions/windows/hardware/network/ff566297(v=vs.85)), in order to prepare for any incoming connection requests and to specify the length of the connection backlog queue.</span></span> <span data-ttu-id="f9855-106">服務提供者會將暫止的連接要求保存在待處理專案佇列中，直到伺服器處理這些要求或因為用戶端的超時) 而撤銷 (為止。</span><span class="sxs-lookup"><span data-stu-id="f9855-106">Service providers hold pending connection requests in a backlog queue until they are acted upon by the server or are withdrawn (due to a time-out) by the client.</span></span> <span data-ttu-id="f9855-107">服務提供者可能會以無訊息方式忽略要求，將待處理專案佇列的大小延伸到超出提供者定義的上限。</span><span class="sxs-lookup"><span data-stu-id="f9855-107">A service provider may silently ignore requests to extend the size of the backlog queue beyond a provider-defined upper limit.</span></span>

<span data-ttu-id="f9855-108">此時，如果正在使用封鎖通訊端，伺服器端可能會立即呼叫 [**WSPAccept**](/windows/desktop/api/Ws2spi/nc-ws2spi-lpwspaccept) ，這會封鎖直到連線要求擱置為止。</span><span class="sxs-lookup"><span data-stu-id="f9855-108">At this point, if a blocking socket is being used, the server side may immediately call [**WSPAccept**](/windows/desktop/api/Ws2spi/nc-ws2spi-lpwspaccept) which will block until a connection request is pending.</span></span> <span data-ttu-id="f9855-109">相反地，伺服器也可以使用先前討論的其中一個網路事件指示機制，來排列連入連線要求的通知。</span><span class="sxs-lookup"><span data-stu-id="f9855-109">Conversely, the server may also use one of the network event indication mechanisms discussed previously to arrange for notification of incoming connection requests.</span></span> <span data-ttu-id="f9855-110">視所選的通知機制而定，提供者會在連接要求抵達時發出 Windows 訊息或發出事件物件的信號。</span><span class="sxs-lookup"><span data-stu-id="f9855-110">Depending on the notification mechanism selected, the provider will either issue a Windows message or signal an event object when connection requests arrive.</span></span> <span data-ttu-id="f9855-111">請參閱 [網路事件通知](notification-of-network-events-2.md) ，以瞭解如何註冊 FD \_ 接受網路事件。</span><span class="sxs-lookup"><span data-stu-id="f9855-111">See [Notification of Network Events](notification-of-network-events-2.md) for how to register for the FD\_ACCEPT network event.</span></span>

<span data-ttu-id="f9855-112">用戶端會建立適當的通訊端，並藉由呼叫 [**WSPConnect**](/previous-versions/windows/hardware/network/ff566275(v=vs.85))來起始連線，並指定已知的伺服器位址。</span><span class="sxs-lookup"><span data-stu-id="f9855-112">The client side will create an appropriate socket, and initiate the connection by calling [**WSPConnect**](/previous-versions/windows/hardware/network/ff566275(v=vs.85)), specifying the known server address.</span></span> <span data-ttu-id="f9855-113">用戶端通常不會在起始連接之前 [**執行明確的**](/windows/desktop/api/winsock/nf-winsock-bind) 系結作業，以允許服務提供者代表他們執行隱含系結。</span><span class="sxs-lookup"><span data-stu-id="f9855-113">Clients usually do not perform an explicit [**bind**](/windows/desktop/api/winsock/nf-winsock-bind) operation prior to initiating a connection, allowing the service provider to perform an implicit bind on their behalf.</span></span> <span data-ttu-id="f9855-114">如果通訊端處於封鎖模式，則 **WSPConnect** 會封鎖，直到伺服器收到 (的連線要求，或直到超時) 。</span><span class="sxs-lookup"><span data-stu-id="f9855-114">If the socket is in blocking mode, **WSPConnect** will block until the server has received and acted upon the connection request (or until a time-out occurs).</span></span> <span data-ttu-id="f9855-115">否則，用戶端應該使用先前討論的其中一個網路事件指示機制，來排列建立新連線的通知。</span><span class="sxs-lookup"><span data-stu-id="f9855-115">Otherwise, the client should use one of the network event indication mechanisms discussed previously to arrange for notification of a new connection being established.</span></span> <span data-ttu-id="f9855-116">視所選的通知機制而定，提供者會透過 Windows 訊息或透過通知事件物件來指出這一點。</span><span class="sxs-lookup"><span data-stu-id="f9855-116">Depending on the notification mechanism selected, the provider will indicate this either through a Windows message or by signaling an event object.</span></span>

<span data-ttu-id="f9855-117">當伺服器端叫用 [**WSPAccept**](/windows/desktop/api/Ws2spi/nc-ws2spi-lpwspaccept)時，服務提供者會呼叫應用程式提供的條件函式，並使用函式參數從連接要求待處理專案佇列中的最上層專案傳遞至伺服器資訊。</span><span class="sxs-lookup"><span data-stu-id="f9855-117">When the server side invokes [**WSPAccept**](/windows/desktop/api/Ws2spi/nc-ws2spi-lpwspaccept), the service provider calls the application-supplied condition function, using function parameters to pass into the server information from the top entry in the connection request backlog queue.</span></span> <span data-ttu-id="f9855-118">這項資訊包括連線主機的位址、任何可用的使用者資料，以及任何可用的 QoS 資訊。</span><span class="sxs-lookup"><span data-stu-id="f9855-118">This information includes such things as address of connecting host, any available user data, and any available QoS information.</span></span> <span data-ttu-id="f9855-119">使用這種資訊，伺服器的條件函式會決定是否要接受要求、拒絕要求，還是延遲在稍後進行決策。</span><span class="sxs-lookup"><span data-stu-id="f9855-119">Using this information the server's condition function determines whether to accept the request, reject the request, or defer making a decision until later.</span></span> <span data-ttu-id="f9855-120">這項決定是透過 condition 函式的傳回值來表示。</span><span class="sxs-lookup"><span data-stu-id="f9855-120">This decision is indicated through the return value of the condition function.</span></span> <span data-ttu-id="f9855-121">請參閱 [網路事件通知](notification-of-network-events-2.md) ，以瞭解如何註冊 FD \_ CONNECT 網路事件。</span><span class="sxs-lookup"><span data-stu-id="f9855-121">See [Notification of Network Events](notification-of-network-events-2.md) for how to register for the FD\_CONNECT network event.</span></span>

<span data-ttu-id="f9855-122">如果伺服器決定接受要求，則提供者必須使用所有與接聽通訊端相同的屬性和事件註冊來建立新的通訊端。</span><span class="sxs-lookup"><span data-stu-id="f9855-122">If the server decides to accept the request, the provider must create a new socket with all of the same attributes and event registrations as the listening socket.</span></span> <span data-ttu-id="f9855-123">原始通訊端會保持在接聽狀態，以便接收後續的連接要求。</span><span class="sxs-lookup"><span data-stu-id="f9855-123">The original socket remains in the listening state so that subsequent connection requests can be received.</span></span> <span data-ttu-id="f9855-124">透過 condition 函數的 output 參數，伺服器也可以提供任何回應使用者資料並指派 QoS 參數， (假設服務提供者) 支援這些作業。</span><span class="sxs-lookup"><span data-stu-id="f9855-124">Through output parameters of the condition function, the server may also supply any response user data and assign QoS parameters (assuming that these operations are supported by the service provider).</span></span>
