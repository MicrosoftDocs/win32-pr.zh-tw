# <a name="graphics-low-level-client-support"></a>用戶端支援的圖形 Low-Level

\[這些函式可能會隨著每個作業系統的修訂而變更。 請改用 Microsoft DirectDraw 和 Microsoft Direct3DAPIs;這些 Api 會防止應用程式進行這類作業系統變更，並隱藏與顯示驅動程式直接互動的許多其他難題。 如需詳細資訊，請參閱 [顯示簡介](/windows-hardware/drivers/display/introduction-to-display--windows-2000-model-)。\]

DirectDraw 和 Direct3D 使用一些核心模式常式來與作業系統和顯示器驅動程式進行通訊。

從使用者模式元件的觀點來看，作業系統核心似乎是 DirectDraw/Direct3D 顯示器驅動程式。 不過，核心和顯示器驅動程式之間有一些差異。 核心模式元件會保留代表 DirectDraw/Direct3D 物件的結構複本，例如裝置和表面。 DirectDraw/Direct3D 的使用者模式元件會要求建立這些結構，這些結構是由核心模式所傳回的控制碼所參考。 因此，這些差異大多在傳遞至常式的引數中：核心接受控制碼，而驅動程式通常會接受指標。

## <a name="evolution-of-kernel-mode-entry-points"></a>核心模式進入點的演進

DirectDraw 會使用數個進入點，使其透過 Gdi32.dll 的所有連接到核心模式。 這些進入點通常只會封送處理參數。 在建立裝置的過程中，Gdi32.dll 會將指標傳回給其本身的常式，然後供 DirectDraw 和 Direct3D 7.0 使用。

Direct3D 8。*x* 會讓 Gdi32.dll 的清除中斷，並直接呼叫核心模式進入點。

## <a name="calling-the-kernel"></a>呼叫核心

這些進入點很類似，而且在許多情況下，與顯示器驅動程式本身的進入點相同，因此瞭解驅動程式開發工具組 (DDK 和 Direct3D 的 DDK) 教材，是使用這些功能的必要必要條件。

呼叫這些函式與作業系統不同。 在 Windows 2000 上，W2KUMODE。LIB 程式庫包含可讓呼叫執行緒轉換成核心模式的常式。 這些常式不會進行任何參數封送處理，而是 Windows 2000 上的理想呼叫機制。 讓轉換成核心模式的機制依賴資料表的順序。 此順序會在作業系統修訂間變更。 因此，建議您不要建立依賴 W2KUMODE 的應用程式。適用于 Windows 2000 的 LIB，並預期它會在 Windows XP 上執行取消修改。 下一節將說明用來建立作業系統可攜性的機制。

在 Windows XP 上，D3D8THK.DLL 會匯出所有核心模式函式，但是有稍微裝飾的名稱。 下列範例顯示應用程式如何使用預處理器來為低層級用戶端函式的別名。 在這麼做之前，請以動態方式連結至 D3D8THK.DLL。


```

#define NtGdiDdUnlock OsThunkDdUnlock
#include "ntgdi.h"      // Brings in the definition of "NtGdiDdUnlock", 
.                       //   which is now seen as OsThunkDdUnlock.
.
.
OsThunkDdUnlock(hSurface, puUnlockData);
```



## <a name="writing-portable-applications"></a>撰寫可移植的應用程式

有數種技術可讓應用程式跨作業系統進行移植。 最穩固的是在 Windows 2000 上安裝 Microsoft DirectX 8.x 可轉散發套件，作為您的應用程式安裝程式的一部分。 這可保證 D3D8THK.DLL 將可在 Windows 2000 系統上使用，並允許應用程式中的單一程式碼路徑，因為您可以使用上述 Windows XP 所述的相同技術。

應用程式也可以建立自己的絕緣 DLL，並執行該 DLL 的兩個版本，一個適用于使用上述技術的 Windows 2000，另一個則是使用 D3D8THK.DLL 的 Windows XP。

## <a name="device-creation"></a>裝置建立

DirectDraw 和 Direct3D 必須先建立驅動程式物件的核心抽象概念實例。 在 Gdi32.dll 和 Ddraw.dll 中，藉由呼叫 [**DdCreateDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatedirectdrawobject)、 [**DdQueryDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddquerydirectdrawobject) 和 [**DdReenableDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddreenabledirectdrawobject)來達成。 在 Direct3D 8 中。*x*，藉由呼叫 [**NtGdiDdCreateDirectDrawObject**](-dxgkernel-ntgdiddcreatedirectdrawobject.md)、 [**NtGdiDdQueryDirectDrawObject**](-dxgkernel-ntgdiddquerydirectdrawobject.md) 和 [**NtGdiDdReenableDirectDrawObject**](-dxgkernel-ntgdiddreenabledirectdrawobject.md)來達成。 請注意，GdiEntry \* 進入點僅適用于核心模式進入點的封送處理器。

呼叫這些函式之後，會傳回一個控制碼，代表 DirectDraw 裝置的核心物件抽象概念。 此核心物件代表顯示驅動程式所擁有的驅動程式實例。 這個控制碼可用於後續的呼叫，例如 [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (或 Direct3D 8 中的 [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) 。*x*) 建立進一步的物件。

## <a name="kernel-objects"></a>核心物件

有幾個進入點用來管理某些物件的核心模式標記法。 Direct3D 8 中的 [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (或 [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) 。*x*) 是很好的範例。 這個進入點會建立代表系統記憶體片段的核心模式物件。 進入點會傳回控制碼，而此控制碼可用於後續的繪圖呼叫。 核心衍生自這些控制碼中自己結構的指標，並將該指標傳遞至顯示驅動程式，然後執行作業。
