# <a name="using-a-three-tier-architecture-model"></a>使用 Three-Tier 架構模型

三層式架構模型（ [邏輯設計模型](the-logical-model--application-definition-and-planning.md)的基本架構）會將應用程式的元件分割成三個服務層級。 這些層級不一定會對應到網路上不同電腦上的實體位置，而是對應到應用程式的邏輯層。 根據系統需求而定，應用程式的片段在實體拓撲中的散發方式可能會變更。

以下是配置給每一層的服務的簡短說明：

-   **展示層（或使用者服務層）可讓使用者存取應用程式。** 這一層會向使用者呈現資料，並選擇性地允許資料操作和資料輸入。 這一層的兩個主要使用者介面類別型是傳統的應用程式和 Web 應用程式。 Web 應用程式現在通常包含傳統應用程式所使用的大部分資料操作功能。 這可透過使用動態 HTML 和用戶端資料來源和資料指標來完成。

    > [!Note]  
    > 在三層式應用程式中，用戶端應用程式將會 skinnier 至用戶端伺服器應用程式，因為它不會包含目前位於仲介層的服務元件。 這會降低使用者的負擔，但系統會有更多的網路流量，因為元件是分散在不同的電腦上。

     

-   **仲介層（或商務服務層）包含商務和資料規則。** 中介層也稱為 *商務邏輯層*，可讓 com + 開發人員解決任務關鍵性商務問題，並達到主要的生產力優勢。 組成此層的元件可以存在於伺服器機器上，以協助資源分享。 這些元件可以用來強制執行商務規則，例如商務演算法和法律或政府法規，以及資料規則，這些規則是設計用來在特定或多個資料庫中保持一致的資料結構。 因為這些中介層元件未系結至特定的用戶端，所以可供所有應用程式使用，並可移至不同的位置，因為回應時間和其他規則需要。 例如，您可以在用戶端上放置簡單的編輯，以將網路往返降到最低，或將資料規則放在預存程式中。

-   **資料層或資料服務層會與持續性資料互動，通常儲存在資料庫或永久儲存體中。** 這是實際的 DBMS 存取層。 它可透過商務服務層存取，並在使用者服務層級進行。 這一層包含 (的資料存取元件，而不是原始的 DBMS 連接) 協助資源分享，並允許設定用戶端，而不需要在每個用戶端上安裝 DBMS 程式庫和 ODBC 驅動程式。

在應用程式的生命週期期間，三層式方法可提供重複使用性、彈性、管理性、可維護性和擴充性等優點。 您可以共用和重複使用您所建立的元件和服務，也可以視需要將它們分散到電腦的網路上。 您可以將大型和複雜的專案分成更簡單的專案，並將它們指派給不同的程式設計人員或程式設計團隊。 您也可以在伺服器上部署元件和服務來協助保持最新的變更，而且您可以在應用程式的使用者群、資料和交易量增加的成長中重新部署它們。
