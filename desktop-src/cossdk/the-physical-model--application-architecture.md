# <a name="the-physical-model-application-architecture"></a>實體模型：應用程式架構

概念和邏輯模型完成之後，您就可以在應用程式的實體實作為決策。 若要建立實體模型，您必須瞭解應用程式的各種服務的所在位置，以及應該如何執行這些服務。 判斷各服務的所在位置應該在服務的執行之前。

判斷各種服務所在位置的其中一個基本規則：將元件放在使用的位置。 例如，如果元件顯示基礎用戶端的資訊，則應該在使用者的電腦上。 如果元件會驗證來自基底用戶端的資訊，它也應該位於基礎用戶端的電腦上。 如果元件更新資料庫中的資訊，它應該位於資料庫伺服器上。

當然，還有其他考慮會造成此規則的例外狀況。 效能和安全性問題也可以規定元件的所在位置。 請考慮下列事項：

-   元件是否會經常變更，使散發更新變得很困難？
-   元件是否會由其他應用程式使用，例如一般安全性驗證元件？
-   元件是否會進行冗長的計算，或執行可卸載至伺服器的功能（例如列印）？
-   將元件放在伺服器上可以增強元件的安全性嗎？

如果資料的系統或位置有所變更，適當地找出應用程式的元件也可以將開發小組與昂貴的編碼進行隔離。 例如，將資料存取規則放在資料層，而不是在預存程式中，則應用程式更容易與特定 DBMS 的相依性隔離。 不只是變更的限制和測試劃分，還可以改變數據源，而且可以散發資料，而不需要以根本上變更應用程式。

最後，找出元件應該利用系統效率。 將商務物件放在網路上的集中位置是時間且符合成本效益的。 物件可以在應用程式之間共用，而且可以在部署任何元件之前完成單元測試。 您也可以減少維護成本，因為規則變更只會在單一點發生。
