# <a name="benefits-of-queued-processing"></a>佇列處理的優點

設計新的應用程式時，開發人員必須考慮程式碼撰寫元件的影響，以進行即時 (同步) 處理，與佇列 (非同步) 處理。 選擇取決於基礎商務邏輯所決定的特定應用程式需求。 作為指導方針，佇列處理可提供下列優於即時處理的優點：

-   降低元件可用性的相依性
-   較短的元件存留期
-   使用已中斷連線的應用程式時的生產力不中斷
-   訊息可靠性
-   有效率的伺服器排程

## <a name="component-availability"></a>元件可用性

在即時處理應用程式中，如果交易的某個元件無法使用（也許是因為伺服器多載或網路問題），整個程式會遭到封鎖而無法完成。 相反地，使用 COM + 佇列元件服務的應用程式會將交易區分為必須立即完成的活動，以及稍後可以完成的活動。 例如，您可以將訊息排入佇列以供稍後處理，讓提出要求的元件在其他工作中都是免費的。

## <a name="component-lifetimes"></a>元件存留期

使用佇列元件服務的應用程式可讓伺服器元件獨立于用戶端運作。 因此，伺服器元件可以更快完成。 在即時系統中，伺服器元件是從建立時開始，到最後釋放物件為止。 伺服器會等候用戶端進行方法呼叫，並傳回結果，這會使伺服器物件快速迴圈，並限制伺服器的擴充性。

## <a name="disconnected-applications"></a>中斷連線的應用程式

膝上型電腦、筆記本電腦和掌上電腦的增加使用，為服務偶爾中斷用戶端或行動使用者的應用程式建立了需求。 在已排入佇列的系統中，這些使用者可以在中斷連線的情況下繼續工作，或在未連接到伺服器時繼續運作，而且稍後可以連接到資料庫或伺服器來處理其要求。 例如，銷售人員可以從客戶取得訂單，並于稍後連接到出貨部門來處理這些訂單。

如果您有可執行連線或中斷連線的元件，訊息會以單向方式進行，而且很少需要來回切換。 例如，在訂單採取案例中，出貨元件會接收訊息並進行處理。 它可能會產生另一個用於計費或審核的元件。 用戶端會在伺服器開始之前認可。 在應用程式認可之前，不會傳送訊息。

## <a name="message-reliability"></a>訊息可靠性

訊息佇列是一種功能強大的工具，可使用資料庫技術，以健全的方式協助保護資料。 萬一伺服器失敗，訊息佇列可確保回復交易，讓訊息不會遺失，而且資料也不會損毀。

## <a name="server-scheduling"></a>伺服器排程

使用已排入佇列之元件的應用程式非常適用于經過時間改變的元件執行，這會將 uncritical 工作延後到離峰時段。 這與傳統的批次模式處理相同，也是相當有用的概念。 您可以將類似的要求延後以供伺服器連續執行，而不需要讓伺服器立即回應各式各樣的要求。
