## <a name="description"></a><span data-ttu-id="0f652-104">Description</span><span class="sxs-lookup"><span data-stu-id="0f652-104">Description</span></span>

<span data-ttu-id="0f652-105">搭配 [SetWindowsHookEx](/windows/desktop/api/winuser/nf-winuser-setwindowshookexw) 函式使用的應用程式定義或程式庫定義的回呼函數。</span><span class="sxs-lookup"><span data-stu-id="0f652-105">An application-defined or library-defined callback function used with the [SetWindowsHookEx](/windows/desktop/api/winuser/nf-winuser-setwindowshookexw) function.</span></span>
<span data-ttu-id="0f652-106">每次有新的鍵盤輸入事件張貼到執行緒輸入佇列時，系統就會呼叫這個函式。</span><span class="sxs-lookup"><span data-stu-id="0f652-106">The system calls this function every time a new keyboard input event is about to be posted into a thread input queue.</span></span>

<span data-ttu-id="0f652-107">**注意：**  當呼叫此回呼函式以回應索引鍵的狀態變更時，會先呼叫回呼函式，然後再更新索引鍵的非同步狀態。</span><span class="sxs-lookup"><span data-stu-id="0f652-107">**Note:**  When this callback function is called in response to a change in the state of a key, the callback function is called before the asynchronous state of the key is updated.</span></span>

```cpp
  _In_ int    nCode,
  _In_ WPARAM wParam,
  _In_ LPARAM lParam
);
```

## <a name="remarks"></a><span data-ttu-id="0f652-133">備註</span><span class="sxs-lookup"><span data-stu-id="0f652-133">Remarks</span></span>

<span data-ttu-id="0f652-134">應用程式會在 **SetWindowsHookEx** 函式的呼叫中指定 **WH_KEYBOARD_LL** 攔截類型和攔截程式指標，以安裝攔截程式。</span><span class="sxs-lookup"><span data-stu-id="0f652-134">An application installs the hook procedure by specifying the **WH_KEYBOARD_LL** hook type and a pointer to the hook procedure in a call to the **SetWindowsHookEx** function.</span></span>

<span data-ttu-id="0f652-135">此攔截器會在其安裝所在的執行緒內容中呼叫。</span><span class="sxs-lookup"><span data-stu-id="0f652-135">This hook is called in the context of the thread that installed it.</span></span>
<span data-ttu-id="0f652-136">呼叫是藉由將訊息傳送至已安裝攔截器的執行緒進行。</span><span class="sxs-lookup"><span data-stu-id="0f652-136">The call is made by sending a message to the thread that installed the hook.</span></span>
<span data-ttu-id="0f652-137">因此，安裝攔截的執行緒必須有訊息迴圈。</span><span class="sxs-lookup"><span data-stu-id="0f652-137">Therefore, the thread that installed the hook must have a message loop.</span></span>

<span data-ttu-id="0f652-138">鍵盤輸入可以來自本機鍵盤驅動程式，或來自 [keybd_event](/windows/desktop/api/winuser/nf-winuser-keybd_event) 函式的呼叫。</span><span class="sxs-lookup"><span data-stu-id="0f652-138">The keyboard input can come from the local keyboard driver or from calls to the [keybd_event](/windows/desktop/api/winuser/nf-winuser-keybd_event) function.</span></span>
<span data-ttu-id="0f652-139">如果輸入來自 **keybd_event** 的呼叫，則輸入為「已插入」。</span><span class="sxs-lookup"><span data-stu-id="0f652-139">If the input comes from a call to **keybd_event**, the input was "injected".</span></span>
<span data-ttu-id="0f652-140">不過，WH_KEYBOARD_LL 的勾點不會插入另一個進程。</span><span class="sxs-lookup"><span data-stu-id="0f652-140">However, the WH_KEYBOARD_LL hook is not injected into another process.</span></span>
<span data-ttu-id="0f652-141">相反地，內容會切換回已安裝攔截的進程，並在其原始內容中呼叫。</span><span class="sxs-lookup"><span data-stu-id="0f652-141">Instead, the context switches back to the process that installed the hook and it is called in its original context.</span></span>
<span data-ttu-id="0f652-142">然後，內容會切換回產生事件的應用程式。</span><span class="sxs-lookup"><span data-stu-id="0f652-142">Then the context switches back to the application that generated the event.</span></span>

<span data-ttu-id="0f652-143">攔截程式應處理訊息的時間，比下列登錄機碼中 **LowLevelHooksTimeout** 值指定的資料輸入少： **HKEY_CURRENT_USER\Control Panel\Desktop**。</span><span class="sxs-lookup"><span data-stu-id="0f652-143">The hook procedure should process a message in less time than the data entry specified in the **LowLevelHooksTimeout** value in the following registry key: **HKEY_CURRENT_USER\Control Panel\Desktop**.</span></span>

<span data-ttu-id="0f652-144">值會以毫秒來表示。</span><span class="sxs-lookup"><span data-stu-id="0f652-144">The value is in milliseconds.</span></span>
<span data-ttu-id="0f652-145">如果攔截程式超時，系統會將訊息傳遞至下一個攔截。</span><span class="sxs-lookup"><span data-stu-id="0f652-145">If the hook procedure times out, the system passes the message to the next hook.</span></span>
<span data-ttu-id="0f652-146">不過，在 Windows 7 和更新版本上，會以無訊息方式移除攔截，而不會被呼叫。</span><span class="sxs-lookup"><span data-stu-id="0f652-146">However, on Windows 7 and later, the hook is silently removed without being called.</span></span>
<span data-ttu-id="0f652-147">應用程式沒有辦法知道是否已移除攔截。</span><span class="sxs-lookup"><span data-stu-id="0f652-147">There is no way for the application to know whether the hook is removed.</span></span>

<span data-ttu-id="0f652-148">注意：偵錯工具攔截無法追蹤這種類型的低層級鍵盤攔截。</span><span class="sxs-lookup"><span data-stu-id="0f652-148">Note:  Debug hooks cannot track this type of low level keyboard hooks.</span></span>
<span data-ttu-id="0f652-149">如果應用程式必須使用低層級的勾點，則應該在將工作關閉到工作者執行緒的專用線程上執行勾點，然後立即傳回。</span><span class="sxs-lookup"><span data-stu-id="0f652-149">If the application must use low level hooks, it should run the hooks on a dedicated thread that passes the work off to a worker thread and then immediately returns.</span></span>
<span data-ttu-id="0f652-150">在大部分的情況下，應用程式必須使用低層級的勾點，而改為監視原始輸入。</span><span class="sxs-lookup"><span data-stu-id="0f652-150">In most cases where the application needs to use low level hooks, it should monitor raw input instead.</span></span>
<span data-ttu-id="0f652-151">這是因為原始輸入可以更有效率地監視以其他執行緒為目標的滑鼠和鍵盤訊息，而非低層級的勾點。</span><span class="sxs-lookup"><span data-stu-id="0f652-151">This is because raw input can asynchronously monitor mouse and keyboard messages that are targeted for other threads more effectively than low level hooks can.</span></span>
<span data-ttu-id="0f652-152">如需原始輸入的詳細資訊，請參閱 [原始輸入](/windows/desktop/inputdev/raw-input)。</span><span class="sxs-lookup"><span data-stu-id="0f652-152">For more information on raw input, see [Raw Input](/windows/desktop/inputdev/raw-input).</span></span>
