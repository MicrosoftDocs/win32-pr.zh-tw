# <a name="backups-without-writer-participation"></a>不含寫入器參與的備份

如果在沒有寫入器介入的情況下進行 VSS 備份作業，仍可能會建立陰影複製。

如先前所述 (查看 [預設陰影複製狀態](shadow-copies-and-shadow-copy-sets.md)) ，這種陰影複製類型的結果是磁片區，可反映陰影複製時磁片的狀態：陰影複製的磁片區上的資料可能會反映不完整或部分 i/o 作業，而且會描述為處於損 [*毀一致狀態*](vssgloss-c.md)。

有幾種情況需要備份應用程式處理損毀一致的陰影複製資料：

-   **資料是由 VSS 感知的應用程式所管理**

    幾乎每個系統都有一些應用程式（文字編輯器、郵件讀取器、文字處理器等等）不是 VSS 感知，所以陰影複製上的部分資料一律會被視為處於損毀一致狀態。

    這類資料通常不是系統或服務關鍵，因此備份它應該不會有問題，或至少在傳統備份期間沒有任何問題。

    如同傳統備份的準備工作，在可能的情況下，備份操作員應該在啟動 VSS 備份作業之前，先嘗試暫停或終止這類應用程式。

-   **系統，沒有與 VSS 相容的寫入器**

    這種情況可能相當罕見，因為 VSS 備份應用程式可以備份的大部分系統都會有啟用 VSS 的 Windows 版本，其應包含寫入器。

    不過，在某些情況下可能會發生此問題，例如，如果您是在不使用 VSS 支援的情況下備份系統，但系統使用具備 VSS 功能的網路設備來儲存它。

    雖然從損毀一致映射備份系統的狀態並不是最佳的，但讓電腦重新開機操作狀態可能就已足夠。 在許多情況下，電腦會從任何未完成的 i/o 作業復原到檔案系統，而且會正常運作。

-   **要求者選擇不使用系統寫入器**

    如果要求者選擇不新增寫入器元件，或停用所有寫入器，就會發生這種情況。

    通常不建議以這種方式執行備份。 雖然陰影複製的磁片區可能足以將系統還原為執行中，但並不是理想的做法。 事實上，假設在系統上執行的寫入器是以可與 VSS 和陰影複製合作的功能來設計的，因此以這種方式備份資料可能會造成不穩定的情形。
