# <a name="handling-passwords"></a>處理密碼

目前，使用者名稱和密碼認證是用來進行驗證的最常見認證。 即使其他類型的認證（例如憑證和生物識別）開始在系統和網路的世界中找出其使用方式，但它們通常會以密碼進行備份。 而且，即使使用憑證，也必須保護其加密金鑰。 因此，使用者名稱和密碼將繼續用於可預見的未來的認證。

由於密碼和加密金鑰即將存在一段時間，因此軟體系統必須以安全的方式使用它們。 如果網路或電腦系統保持安全，則必須保護密碼免于受到入侵者的攻擊。 乍看之下，這可能很簡單。 不過，在網路遭入侵之後系統和網路之後的系統，是因為攻擊者可以登出使用者的密碼。 這些問題的範圍包括將其密碼與某人共用的使用者，到可由攻擊者滲透的軟體。

以完全安全的方式將秘密資訊儲存在軟體中是不可能的。 而且因為軟體系統中儲存密碼和加密金鑰絕對不會完全安全，建議您不要將密碼和加密金鑰儲存在軟體系統中。

不過，當密碼必須儲存在軟體系統（通常是這種情況下）時，有一些預防措施可以採取。 主要的預防措施是讓入侵者更難探索密碼。 就像鎖定您的房屋大門一樣，如果有人判斷中斷了，就幾乎不可能阻止他們這麼做。 但希望您能夠更輕鬆地提高攻擊者的程度，而不會發現更容易受害者。

有許多方法可讓攻擊者更難探索密碼的作業。 不過，實際上可以完成的工作範圍通常是與網路或系統使用者願意使用的功能取捨。 例如，採用「單一登入」未使用的情況，並在每次啟動應用程式時提示使用者輸入密碼。 在大部分的情況下，這會對使用者造成大量的負擔，而且可能會抱怨。 但不只是這樣，但缺少單一登入沒有效率，也會降低使用者的生產力。 因此，幾乎說的，使用者通常不會從使用者收集密碼，除非登入時間。

由於密碼通常必須儲存在軟體系統上，因此請務必確保密碼盡可能保持安全，並維護使用者的便利性。 如需詳細資訊，請參閱下列主題：

-   [密碼威脅評估](password-threat-assessment.md)
-   [威脅緩和技巧](threat-mitigation-techniques.md)

> [!Note]  
> 當您完成使用應用程式中的密碼時，請呼叫 [**SecureZeroMemory**](/previous-versions/windows/desktop/legacy/aa366877(v=vs.85)) 函式，以清除記憶體中的敏感資訊。
