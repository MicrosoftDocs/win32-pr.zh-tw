# <a name="determining-push-or-pull-status"></a>判斷推送或提取狀態

您可以將類別提供者模型為 push 或 pull 提供者，以指定提供者如何預期與 WMI 互動。 提取提供者會收到來自 WMI 的要求，並透過動態產生資料或從本機快取抓取來滿足要求。 提取提供者也必須執行大量的介面。

提取提供者會動態產生類別定義。 通常，提取提供者所管理的資料會經常變更，要求提供者必須動態產生類別，或在應用程式發出要求時，從本機快取取出類別。 提取提供者必須執行自己的資料抓取、快取和事件通知機制。 因為大部分的提供者都是提取提供者，所以此檔案中的檔會假設您要建立提取提供者，除非另行明確陳述。

相反地，WMI 會使用 WMI 儲存機制中的資料來處理推播提供者的所有應用程式要求。 推播提供者也會使用較少的介面方法，因此更容易執行。 推入提供者會使用 WMI 存放庫做為受管理物件資訊的儲存區域，而且只會在初始化期間更新該資訊。 例如，Microsoft Windows 軟體開發套件 (SDK) 的 WMI 區段中所包含的 WDM 類別提供者，會模型化為推播提供者。

藉由使用 WMI 儲存機制作為存放區域，推播提供者可以透過提取提供者獲得下列優點：

-   提供者不需要執行本機快取來儲存資料。
-   提供者不需要支援資料抓取;相反地，提供者可以依賴 WMI 提供抓取支援。
-   當應用程式要求提供者提供的資料時，WMI 會滿足該要求。
-   提供者也可以依賴 WMI 來支援事件通知。

不過，因為推播提供者只會在初始化期間更新，所以對類別所做的任何變更可能會有一段時間不會反映在 WMI 存放庫中。 因此，推播提供者模型最適合用於幾乎不會變更或完全靜態的類別。
